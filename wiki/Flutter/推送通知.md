# 推送通知

## 定义

推送通知（Push Notification）是一种从服务器向移动设备发送即时消息的机制，允许应用在用户未主动打开应用时触达用户。在 Flutter 生态中，推送通知主要通过 **Firebase Cloud Messaging (FCM)** 实现跨平台的统一消息推送，同时在中国市场需要适配各手机厂商的自有推送通道（小米 MiPush、华为 HMS Push、OPPO Push 等）。

推送通知是 Flutter 应用与用户保持连接的关键通道，涵盖消息接收、展示、点击跳转、后台处理等完整链路。它不仅是运营工具，也是实时通信、即时协作等场景的核心基础设施。

推送通知在 Flutter 中的实现涉及以下核心概念：

- **FCM (Firebase Cloud Messaging)**：Google 提供的跨平台消息推送服务，支持 Android/iOS/Web
- **APNs (Apple Push Notification service)**：Apple 的原生推送服务，iOS 端必须通过 APNs 发送
- **厂商通道**：国内 Android 手机厂商的自有推送通道，解决 FCM 在国内不可用的问题
- **后台消息 (Background Message)**：应用处于后台或被系统杀死时的消息处理
- **通知渠道 (Notification Channel)**：Android 8.0+ 的通知分类机制
- **Token 管理**：设备推送令牌的获取、更新与失效处理

## 核心原理

### 1. FCM/APNs 双通道架构

FCM 是 Flutter 推送通知的核心，其工作流程如下：

```
应用服务器 → FCM HTTP v1 API → FCM 服务器
                                    ↓
                    Android: GCM/FCM 长连接
                    iOS: APNs → APNs 长连接
                                    ↓
                           客户端 SDK 接收
```

**FCM 核心组件：**

- **FCM Token**：每个设备实例的唯一标识，用于定向推送。Token 会随应用重装、清除数据或长时间未使用而更新
- **Topic**：基于发布/订阅模式的主题推送，适合广播类消息
- **Condition**：基于 Topic 逻辑组合的条件推送（如 `topicA && !topicB`）
- **Priority**：消息优先级，`high` 优先级会立即显示通知，`normal` 优先级可能被系统延迟
- **Data Payload**：纯数据载荷，不会自动显示通知，需要应用自行处理
- **Notification Payload**：系统自动处理的通知载荷，包含标题、正文、图标等

**APNs 集成流程：**

1. 在 Apple Developer Portal 创建 APNs 证书或 Key
2. 在 Firebase Console 上传 APNs 凭证
3. FCM 自动处理 APNs 转发，开发者无需直接对接 APNs
4. iOS 端需获取用户推送权限授权

### 2. 国内厂商通道适配

由于 FCM 在中国大陆不可用，需要适配各厂商通道：

**小米 MiPush：**
- 注册小米开发者账号，创建应用获取 AppID/AppKey/APP Secret
- 在 App-level `build.gradle` 中集成 MiPush SDK
- 配置推送服务和广播接收器
- 通过小米推送 API 发送消息

**华为 HMS Push：**
- 在华为 AppGallery Connect 创建项目
- 下载 `agconnect-services.json` 配置文件
- 集成 HMS Push SDK，处理 Token 获取和消息接收
- HMS Core 覆盖率约 3 亿华为设备

**OPPO Push：**
- 在 OPPO 开放平台注册应用
- 获取 AppKey/AppSecret
- 集成 OPPO Push SDK，适配其独特的通知样式
- 需要注意 OPPO 的通知展示策略限制

**其他厂商：**
- vivo Push（vivo/iQOO 设备）
- 魅族 Flyme Push
- 每个厂商都需要独立的注册、集成和发送流程

### 3. 消息接收与处理链路

Flutter 推送消息的处理分为三个阶段：

**前台消息处理：**
```dart
// 应用处于前台时的消息处理
FirebaseMessaging.onMessage.listen((RemoteMessage message) {
  // 展示自定义 UI 或 Flutter Local Notifications
  showCustomNotification(message);
});
```

**后台消息处理：**
```dart
// 应用在后台但未被杀死
FirebaseMessaging.onMessageOpenedApp.listen((RemoteMessage message) {
  // 处理用户点击通知后的跳转
  handleNotificationTap(message);
});
```

**终止状态处理：**
```dart
// 应用被系统杀死后恢复
final initialMessage = await FirebaseMessaging.instance.getInitialMessage();
if (initialMessage != null) {
  handleNotificationTap(initialMessage);
}
```

### 4. 后台消息处理 (Background Handler)

当应用处于后台或被杀死时，Flutter 的 Dart 代码无法直接执行。FCM 提供了后台消息处理器：

```dart
@pragma('vm:entry-point')
Future<void> _firebaseMessagingBackgroundHandler(RemoteMessage message) async {
  // 必须初始化 Firebase
  await Firebase.initializeApp();
  // 处理后台消息（如静默推送、数据同步）
  await _handleBackgroundMessage(message);
}

void main() {
  FirebaseMessaging.onBackgroundMessage(_firebaseMessagingBackgroundHandler);
  runApp(MyApp());
}
```

**后台处理器的限制：**
- 必须是顶层函数或静态方法
- 必须标记 `@pragma('vm:entry-point')`（Android）
- 只能执行简短任务，不能执行 UI 操作
- 执行时间有限（Android 约 30 秒）

### 5. Token 管理策略

FCM Token 的生命周期管理至关重要：

```dart
class PushNotificationService {
  String? _fcmToken;
  
  Future<void> initialize() async {
    // 获取当前 Token
    _fcmToken = await FirebaseMessaging.instance.getToken();
    print('FCM Token: $_fcmToken');
    
    // 监听 Token 刷新
    FirebaseMessaging.instance.onTokenRefresh.listen((newToken) {
      _fcmToken = newToken;
      _uploadTokenToServer(newToken);
    });
    
    // 上传 Token 到服务器
    if (_fcmToken != null) {
      await _uploadTokenToServer(_fcmToken!);
    }
  }
  
  Future<void> _uploadTokenToServer(String token) async {
    // POST 到后端 API，关联用户与 Token
  }
}
```

```yaml
      - id: 'system'
        name: '系统通知'
        description: '应用更新等系统消息'
        importance: high
```

## 实战案例

### 案例 1：FCM 双通道集成

**场景**：一个面向全球用户的电商应用，需要同时支持 Android 和 iOS 的推送通知。

**实现步骤：**

1. **项目配置**
```yaml
# pubspec.yaml
dependencies:
  firebase_core: ^2.15.0
  firebase_messaging: ^14.6.0
  flutter_local_notifications: ^15.1.0+1
```

2. **初始化服务**
```dart
class PushService {
  final FirebaseMessaging _messaging = FirebaseMessaging.instance;
  
  Future<void> initialize() async {
    // 请求权限
    NotificationSettings settings = await _messaging.requestPermission(
      alert: true,
      badge: true,
      sound: true,
      provisional: false,
    );
    
    if (settings.authorizationStatus == AuthorizationStatus.authorized) {
      // 获取 Token
      String? token = await _messaging.getToken();
      await _registerToken(token);
      
      // 监听前台消息
      FirebaseMessaging.onMessage.listen(_handleForegroundMessage);
      
      // 监听应用从后台打开
      FirebaseMessaging.onMessageOpenedApp.listen(_handleMessageOpenedApp);
      
      // 处理冷启动
      RemoteMessage? initialMessage = await _messaging.getInitialMessage();
      if (initialMessage != null) {
        _handleMessageOpenedApp(initialMessage);
      }
    }
  }
  
  void _handleForegroundMessage(RemoteMessage message) {
    // 使用 flutter_local_notifications 展示通知
    _showLocalNotification(message);
  }
}
```

3. **后台处理器**
```dart
@pragma('vm:entry-point')
Future<void> _onBackgroundMessage(RemoteMessage message) async {
  await Firebase.initializeApp();
  // 静默处理：更新本地缓存、同步数据等
}

void main() {
  FirebaseMessaging.onBackgroundMessage(_onBackgroundMessage);
  runApp(MyApp());
}
```

### 案例 2：国内厂商通道统一适配

**场景**：面向中国市场的社交应用，需要适配主流 Android 手机厂商的推送通道。

**适配策略：**

1. **设备检测与通道选择**
```dart
class VendorPushService {
  static VendorPushService? _instance;
  late final VendorPushChannel _channel;
  
  VendorPushService._() {
    if (Platform.isAndroid) {
      final brand = _getDeviceBrand();
      switch (brand) {
        case 'xiaomi':
          _channel = MiPushChannel();
          break;
        case 'huawei':
          _channel = HuaweiPushChannel();
          break;
        case 'oppo':
          _channel = OppoPushChannel();
          break;
        case 'vivo':
          _channel = VivoPushChannel();
          break;
        default:
          _channel = FCMChannel();
      }
    } else {
      _channel = FCMChannel(); // iOS 统一走 FCM/APNs
    }
  }
  
  static VendorPushService get instance {
    _instance ??= VendorPushService._();
    return _instance!;
  }
}
```

2. **统一接口封装**
```dart
abstract class VendorPushChannel {
  Future<String?> getToken();
  void onTokenRefresh(void Function(String) callback);
  void onMessageReceived(void Function(Map<String, dynamic>) handler);
  Future<void> subscribeToTopic(String topic);
  Future<void> unsubscribeFromTopic(String topic);
}
```

### 案例 3：通知点击跳转路由

**场景**：用户点击通知后，根据通知内容跳转到应用内对应页面。

**路由处理：**
```dart
class NotificationRouter {
  static void handleNotificationTap(RemoteMessage message) {
    final data = message.data;
    final routeType = data['route_type'];
    final routeId = data['route_id'];
    
    switch (routeType) {
      case 'product':
        Get.toNamed('/product/$routeId');
        break;
      case 'order':
        Get.toNamed('/order/detail/$routeId');
        break;
      case 'chat':
        Get.toNamed('/chat/$routeId');
        break;
      case 'promotion':
        Get.toNamed('/promotion/$routeId');
        break;
      default:
        // 默认跳转首页
        Get.toNamed('/');
    }
  }
}
```

### 案例 4：静默推送与数据同步

**场景**：应用需要在后台静默同步最新数据，不展示通知。

**实现：**
```dart
@pragma('vm:entry-point')
Future<void> _onSilentPush(RemoteMessage message) async {
  await Firebase.initializeApp();
  
  final syncType = message.data['sync_type'];
  
  switch (syncType) {
    case 'order_update':
      await OrderRepository.syncOrders();
      break;
    case 'new_message':
      await MessageRepository.syncUnread();
      break;
    case 'config_update':
      await ConfigRepository.refresh();
      break;
  }
  
  // 更新本地通知角标
  final unreadCount = await MessageRepository.getUnreadCount();
  await FlutterAppBadger.updateBadgeCount(unreadCount);
}
```

## 常见问题

### Q1：FCM Token 什么时候会失效？

FCM Token 可能在以下情况失效：
- 用户卸载并重新安装应用
- 用户清除应用数据
- 应用长时间未打开（约 2-3 个月无活动）
- 用户在多设备登录后退出
- Token 使用超过 60 天未刷新

**最佳实践**：始终监听 `onTokenRefresh` 并及时同步到服务器，服务器端维护 Token 有效性映射。

### Q2：iOS 推送权限被拒绝后如何处理？

iOS 用户拒绝推送权限后，应用无法再次弹出授权对话框。解决方案：
1. 在应用内设置页面添加"开启通知"引导
2. 使用 `canShowNotification` 检查当前权限状态
3. 引导用户到系统设置手动开启
4. 使用 Provisional Authorization（iOS 12+）实现静默授权

### Q3：国内厂商通道的 Token 如何统一管理？

建议方案：
- 在服务器端建立统一的 Token 管理表，记录设备型号、厂商类型、Token
- 发送时根据设备厂商选择对应通道
- 维护 Token 失效重试机制
- 使用厂商提供的批量 Token 验证接口

### Q4：后台消息处理器执行时间有限怎么办？

Android 的后台消息处理器有时间限制（约 30 秒）。优化策略：
- 将耗时任务放入 WorkManager 或后台服务
- 后台处理器只负责触发任务，不做复杂处理
- 使用 Isolate 处理 CPU 密集型任务
- 考虑使用数据载荷（Data Payload）触发应用内同步

### Q5：如何处理应用在前台时的通知展示？

应用在前台时，FCM 默认不会自动展示通知。解决方案：
1. 使用 `flutter_local_notifications` 插件手动展示
2. 创建自定义通知 UI（如横幅、弹窗）
3. 根据通知类型选择不同的展示策略
4. 注意 iOS 的通知权限限制

### Q6：推送通知的安全性如何保障？

- **Token 安全**：不要在客户端暴露 API 密钥，Token 上传使用 HTTPS
- **消息验证**：验证消息来源的合法性
- **Payload 加密**：敏感数据使用端到端加密
- **权限控制**：服务器端校验推送目标的权限
- **审计日志**：记录所有推送操作
