# 错误监控与崩溃收集

## 定义

错误监控与崩溃收集是保障 Flutter 应用质量的核心基础设施，通过在应用运行时捕获、记录和分析异常错误，帮助开发者快速定位和修复问题。在 Flutter 生态中，主流方案包括 **Sentry**（开源错误追踪平台）和 **Firebase Crashlytics**（Google 的崩溃报告工具）。

错误监控系统的核心目标：
- **实时捕获**：在错误发生时立即捕获，包括 Dart 异常、原生崩溃、UI 渲染错误
- **上下文收集**：记录错误发生时的设备信息、用户行为、运行环境
- **智能聚合**：将相同错误聚合，减少噪音，突出关键问题
- **趋势分析**：跟踪错误发生频率、影响用户数、版本趋势
- **告警通知**：当错误率超过阈值时及时通知团队

完整错误监控体系包含以下概念：

- **Error Boundary**：Flutter 的错误边界机制，用于捕获子 Widget 树的渲染错误
- **Breadcrumbs**：面包屑追踪，记录用户操作轨迹，便于还原错误现场
- **Release Health**：版本健康度监控，追踪每个版本的崩溃率和 ANR 率
- **Performance Monitoring**：性能监控，与错误监控结合提供完整的应用健康视图
- **Source Map**：源码映射，将 minified 代码还原为可读的 Dart 源码

## 核心原理

### 1. Sentry 集成架构

Sentry 是一个功能强大的开源错误追踪平台，支持 Flutter 的全面监控：

**Sentry 工作流程：**
```
Flutter 应用 → Sentry SDK 捕获异常 → 本地缓存/队列
    ↓
Sentry Relay（数据中转）
    ↓
Sentry Server（存储/分析/聚合）
    ↓
Dashboard / Webhooks / 邮件告警
```

**核心组件：**

- **Dart SDK (`sentry_flutter`)**：Flutter 专用的 Sentry 集成
- **Performance SDK**：性能监控，追踪 Traces 和 Transactions
- **Release SDK**：版本健康度追踪
- **User Feedback SDK**：用户反馈收集

**Sentry 集成示例：**
```dart
import 'package:sentry_flutter/sentry_flutter.dart';

Future<void> main() async {
  await SentryFlutter.init(
    (options) {
      options.dsn = 'YOUR_SENTRY_DSN';
      options.tracesSampleRate = 1.0; // 性能追踪采样率
      options.attachScreenshot = true; // 附加截图
      options.maxBreadcrumbs = 100; // 面包屑数量
      options.environment = kReleaseMode ? 'production' : 'development';
    },
    appRunner: () => runApp(MyApp()),
  );
}
```

**自动捕获：**
- Dart 异常（未捕获的 Exception 和 Error）
- Flutter 框架错误（FlutterError.onError）
- 平台异常（iOS/Android 原生崩溃）
- HTTP 请求错误
- 自定义日志事件

### 2. Firebase Crashlytics 集成

Firebase Crashlytics 是 Google 提供的轻量级崩溃报告工具：

**Crashlytics 核心特性：**
- 自动捕获 Dart 异常和原生崩溃
- 按版本、设备、用户维度聚合崩溃
- 与 Firebase Console 深度集成
- 实时告警和趋势分析
- 支持非致命异常记录

**集成示例：**
```dart
import 'package:firebase_crashlytics/firebase_crashlytics.dart';

void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await Firebase.initializeApp();
  
  // 设置 Crashlytics
  FlutterError.onError = FirebaseCrashlytics.instance.recordFlutterError;
  
  // 捕获异步错误
  runZonedGuarded<Future<void>>(() async {
    runApp(MyApp());
  }, (error, stack) {
    FirebaseCrashlytics.instance.recordError(error, stack);
  });
}
```

**非致命异常记录：**
```dart
try {
  await riskyOperation();
} catch (error, stack) {
  await FirebaseCrashlytics.instance.recordError(
    error,
    stack,
    reason: '用户操作导致的异常',
    context: {'operation': 'riskyOperation', 'userId': currentUserId},
  );
}
```

### 3. Error Boundary 错误边界

Flutter 的错误边界机制用于捕获子 Widget 树的渲染错误：

```dart
class ErrorBoundary extends StatefulWidget {
  final Widget child;
  final Widget Function(Object error)? errorBuilder;
  
  const ErrorBoundary({
    Key? key,
    required this.child,
    this.errorBuilder,
  }) : super(key: key);

  @override
  State<ErrorBoundary> createState() => _ErrorBoundaryState();
}

class _ErrorBoundaryState extends State<ErrorBoundary> {
  Object? _error;
  StackTrace? _stackTrace;

  @override
  void initState() {
    super.initState();
    FlutterError.onError = (FlutterErrorDetails details) {
      setState(() {
        _error = details.exception;
        _stackTrace = details.stack;
      });
      // 上报错误
      Sentry.captureException(
        details.exception,
        stackTrace: details.stack,
      );
    };
  }

  @override
  Widget build(BuildContext context) {
    if (_error != null) {
      if (widget.errorBuilder != null) {
        return widget.errorBuilder!(_error!);
      }
      return ErrorFallbackWidget(error: _error);
    }
    return widget.child;
  }
}

// 使用方式
ErrorBoundary(
  child: ProductDetailPage(productId: '123'),
  errorBuilder: (error) => ProductLoadErrorWidget(),
)
```

### 4. Breadcrumbs 面包屑追踪

面包屑记录用户操作轨迹，帮助还原错误现场：

```dart
class BreadcrumbService {
  static void recordNavigation(String from, String to) {
    Sentry.configureScope((scope) {
      scope.addBreadcrumb(Breadcrumb(
        message: '页面导航',
        category: 'navigation',
        data: {'from': from, 'to': to},
        level: SentryLevel.info,
      ));
    });
  }

  static void recordUserAction(String action, {Map<String, String>? params}) {
    Sentry.addBreadcrumb(Breadcrumb(
      message: '用户操作: $action',
      category: 'user',
      data: params,
      level: SentryLevel.info,
    ));
  }

  static void recordNetworkRequest(String url, int statusCode) {
    Sentry.addBreadcrumb(Breadcrumb(
      message: 'HTTP 请求',
      category: 'http',
      data: {
        'url': url,
        'status_code': statusCode.toString(),
      },
      level: statusCode >= 400 ? SentryLevel.warning : SentryLevel.info,
    ));
  }
}
```

### 5. Release Health 版本健康度

Release Health 追踪每个应用版本的崩溃率和稳定性指标：

**关键指标：**
- **Crash Free Users**：无崩溃用户百分比
- **Crash Free Sessions**：无崩溃会话百分比
- **ANR Rate**：应用无响应率（Android）
- **用户影响**：受影响的用户数量

**版本管理：**
```dart
// 在 Sentry 中设置版本信息
Sentry.configureScope((scope) {
  scope.setTag('version', appVersion);
  scope.setTag('build_number', buildNumber);
  scope.setUser(SentryUser(
    id: userId,
    email: userEmail,
    extras: {
      'device_model': deviceModel,
      'os_version': osVersion,
    },
  ));
});
```

**发布工作流：**
1. 新版本发布后监控 Crash Free Users
2. 设置告警阈值（如 Crash Free Users < 99.5%）
3. 高崩溃率版本触发回滚机制
4. 与 CICD 集成，实现自动化发布控制

### 6. 错误分类与优先级

根据错误类型和影响范围进行分类：

**按严重程度：**
- **Fatal**：导致应用崩溃的错误
- **Error**：功能异常但应用未崩溃
- **Warning**：潜在问题，可能影响用户体验
- **Info**：一般信息记录

**按影响范围：**
- **P0（紧急）**：影响核心功能，用户无法正常使用
- **P1（高）**：影响重要功能，部分用户受影响
- **P2（中）**：影响次要功能，小范围用户受影响
- **P3（低）**：体验问题，不影响功能使用

```dart
class PerformanceTracker {
  static Future<T> trackTransaction<T>(
    String name,
    String operation,
    Future<T> Function() task,
  ) async {
    final transaction = Sentry.startTransaction(
      name,
      operation: operation,
    );
    
    try {
      final result = await task();
      transaction.status = const SpanStatus.ok();
      return result;
    } catch (e) {
      transaction.status = const SpanStatus.internalError();
      rethrow;
    } finally {
      await transaction.finish();
    }
  }
}
```

### 案例 3：自定义 Error Boundary 实战

**场景**：需要在关键页面（如支付、登录）实现错误边界保护。

**实现：**
```dart
// payment_error_boundary.dart
class PaymentErrorBoundary extends StatefulWidget {
  final Widget child;
  
  const PaymentErrorBoundary({Key? key, required this.child}) : super(key: key);

  @override
  State<PaymentErrorBoundary> createState() => _PaymentErrorBoundaryState();
}

class _PaymentErrorBoundaryState extends State<PaymentErrorBoundary> {
  Object? _error;

  @override
  void initState() {
    super.initState();
    _setupErrorHandling();
  }

  void _setupErrorHandling() {
    FlutterError.onError = (FlutterErrorDetails details) {
      setState(() => _error = details.exception);
      
      // 记录支付相关错误
      Sentry.captureException(
        details.exception,
        stackTrace: details.stack,
        hint: Hint.withMap({
          'page': 'payment',
          'severity': 'critical',
        }),
      );
      
      // 发送告警
      _sendAlert('支付页面错误: ${details.exception}');
    };
  }

  @override
  Widget build(BuildContext context) {
    if (_error != null) {
      return PaymentErrorWidget(
        error: _error,
        onRetry: () {
          setState(() => _error = null);
          // 重试支付流程
        },
        onFallback: () {
          // 降级到备用支付方式
        },
      );
    }
    return widget.child;
  }
}
```

### 案例 5：Release Health 自动化监控

**场景**：新版本发布后自动监控崩溃率，超阈值自动告警。

**实现：**
```dart
// release_health_monitor.dart
class ReleaseHealthMonitor {
  static const double CRASH_RATE_THRESHOLD = 0.5; // 0.5% 崩溃率阈值
  
  static Future<void> checkReleaseHealth() async {
    final healthData = await SentryApi.getReleaseHealth(
      app: 'my-app',
      environment: 'production',
      latestReleases: 5,
    );
    
    for (final release in healthData.releases) {
      if (release.crashFreeUsers < (1 - CRASH_RATE_THRESHOLD)) {
        await _sendCrashRateAlert(release);
        
        // 如果崩溃率过高，通知 CICD 系统
        if (release.crashFreeUsers < 0.95) {
          await _triggerRollback(release.version);
        }
      }
    }
  }
  
  static Future<void> _sendCrashRateAlert(ReleaseInfo release) async {
    // 发送邮件/Slack/钉钉告警
    await AlertService.send({
      'title': '崩溃率告警',
      'message': '版本 ${release.version} 崩溃率: ${release.crashRate}%',
      'users_affected': release.affectedUsers,
      'timestamp': DateTime.now().toIso8601String(),
    });
  }
}
```

## 常见问题

### Q1：Sentry 和 Firebase Crashlytics 如何选择？

| 维度 | Sentry | Firebase Crashlytics |
|------|--------|---------------------|
| **开源性** | 开源，可自托管 | 闭源，仅 Firebase 提供 |
| **功能丰富度** | 功能全面（错误、性能、会话重放） | 功能精简（崩溃、ANR） |
| **集成复杂度** | 配置项多，学习曲线陡 | 集成简单，开箱即用 |
| **定价** | 自托管免费，SaaS 按量计费 | 免费 |
| **生态整合** | 支持多平台、多框架 | 深度整合 Firebase 生态 |
| **适用场景** | 大型项目、复杂需求 | 中小型项目、Firebase 用户 |

**建议**：如果已在使用 Firebase 生态，优先选择 Crashlytics；如果需要更丰富的功能或自托管，选择 Sentry。

### Q2：如何减少错误监控的性能开销？

- **采样率控制**：生产环境设置合理的采样率（如 10-20%）
- **本地缓存**：错误上报前先缓存本地，批量发送
- **异步上报**：错误上报使用异步操作，不阻塞主线程
- **去重机制**：相同错误短时间内只上报一次
- **轻量级 SDK**：移除不必要的功能模块

### Q3：如何处理 Source Map 上传问题？

**Sentry Source Map 上传：**
```bash
# 使用 sentry-cli 上传 Source Map
sentry-cli upload-sourcemaps \
  --org=your-org \
  --project=your-project \
  --release=1.0.0 \
  build/app/outputs/flutter-apk \
  --url-prefix '~/'
```

**常见问题：**
- **构建优化**：确保在 release 模式下构建，debug 模式不上传 Source Map
- **版本匹配**：Source Map 的 release 版本必须与应用运行时一致
- **路径正确**：`--url-prefix` 必须与运行时路径匹配
- **CI/CD 集成**：在发布流程中自动上传 Source Map

### Q4：如何实现错误的智能聚合？

### Q5：错误监控数据如何与 CICD 集成？

**集成方案：**
1. **发布前检查**：在 CICD 流程中检查最新版本的崩溃率
2. **自动回滚**：崩溃率超过阈值自动触发回滚
3. **质量门禁**：将崩溃率作为发布的质量门禁
4. **报告生成**：自动生成发布质量报告

```yaml
# GitHub Actions 示例
- name: Check Crash Rate
  run: |
    CRASH_RATE=$(sentry-cli get-release-info --org=your-org --project=your-project --release=$VERSION | jq '.crashFreeUsers')
    if [ $(echo "$CRASH_RATE < 99.5" | bc) -eq 1 ]; then
      echo "崩溃率过高: $CRASH_RATE%，阻止发布"
      exit 1
    fi
```

### Q6：如何保护用户隐私的同时收集错误信息？

- **数据脱敏**：自动移除敏感信息（密码、Token、银行卡号）
- **用户授权**：获取用户同意后再收集错误信息
