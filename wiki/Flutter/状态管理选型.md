# 状态管理选型

## 定义

状态管理（State Management）是指在 Flutter 应用中管理和共享应用数据（状态）的方式和模式。随着应用规模增长，单纯的 `setState()` 难以维护，需要引入系统化的状态管理方案来处理状态的创建、传递、更新和销毁。

**核心问题：**

- **状态提升（Lifting State Up）**：多个 Widget 共享同一数据时，状态需要提升到公共祖先
- **状态传播**：如何高效地将状态变化通知到依赖它的 Widget
- **副作用管理**：异步操作、API 调用、缓存等与 UI 无关的逻辑
- **可测试性**：业务逻辑是否可以脱离 UI 独立测试

**主流方案：**

| 方案 | 维护方 | 核心思想 | 学习曲线 |
|------|--------|----------|----------|
| Riverpod | Remi Rousselet | Provider 的改进版，编译时安全 | 中等 |
| Bloc | Felix Angelov | 事件驱动，BLoC 模式 | 较高 |
| GetX | Jonatas Borges | 轻量全家桶，约定优于配置 | 低 |

## 核心原理

### 1. 响应式编程基础

Flutter 状态管理建立在响应式编程范式之上：

- **Stream / StreamController**：Dart 原生的异步事件流
- **ChangeNotifier / ValueNotifier**：Flutter 框架提供的监听者模式
- **InheritedWidget**：Flutter 的状态向下传播机制（所有 Provider 类方案的底层依赖）
- **StreamBuilder / ValueListenableBuilder**：根据数据流自动重建 Widget

### 2. Riverpod

Riverpod 是 Provider 的作者推出的下一代方案，解决了 Provider 的核心痛点：

**核心特性：**
- 不依赖 BuildContext（可在任何地方读取）
- 编译时类型安全
- 自动销毁未使用的状态
- 内置异步状态处理（`AsyncValue`）

```dart
// 定义 Provider
final counterProvider = StateNotifierProvider<CounterNotifier, int>((ref) {
  return CounterNotifier();
});

class CounterNotifier extends StateNotifier<int> {
  CounterNotifier() : super(0);
  void increment() => state++;
}

// Widget 中使用
class CounterWidget extends ConsumerWidget {
  @override
  Widget build(BuildContext context, WidgetRef ref) {
    final count = ref.watch(counterProvider);
    return Text('$count');
  }
}
```

**Provider 类型：**
- `Provider`：只读，不可修改
- `StateProvider`：简单状态
- `StateNotifierProvider`：复杂状态逻辑
- `FutureProvider`：异步数据
- `StreamProvider`：流式数据
- `NotifierProvider`（Riverpod 2.x 新推荐）

### 3. BLoC（Business Logic Component）

BLoC 模式通过 Event → State 的单向数据流管理状态：

**核心概念：**
- **Event**：用户操作或系统事件
- **State**：UI 的状态描述
- **Bloc**：接收 Event，产出 State 的业务逻辑组件

```dart
// Event
abstract class CounterEvent {}
class Increment extends CounterEvent {}
class Decrement extends CounterEvent {}

// Bloc
class CounterBloc extends Bloc<CounterEvent, int> {
  CounterBloc() : super(0) {
    on<Increment>((event, emit) => emit(state + 1));
    on<Decrement>((event, emit) => emit(state - 1));
  }
}

// Widget 中使用
BlocProvider(
  create: (_) => CounterBloc(),
  child: BlocBuilder<CounterBloc, int>(
    builder: (context, count) => Text('$count'),
  ),
)
```

**BLoC 的优势：**
- 事件与状态完全分离，可追溯
- 强制单向数据流，逻辑清晰
- 优秀的可测试性（输入事件，断言状态）
- 配合 `flutter_bloc` 包提供 `BlocBuilder`、`BlocListener`、`BlocConsumer`

### 4. GetX

GetX 追求极简和约定优于配置：

```dart
// Controller
class CounterController extends GetxController {
  var count = 0.obs;  // .obs 使变量可观察
  void increment() => count++;
}

// Widget 中使用
final controller = Get.put(CounterController());

Obx(() => Text('${controller.count}'))
```

**GetX 三大功能：**
- **状态管理**：`.obs` + `Obx()` / `GetBuilder`
- **路由管理**：`Get.to()`, `Get.off()`, 命名路由
- **依赖注入**：`Get.put()`, `Get.lazyPut()`, `Get.find()`

### 5. Provider 模式（原始 Flutter Provider）

基于 InheritedWidget 的封装，是 Flutter 官方推荐的入门方案：

```dart
ChangeNotifierProvider(
  create: (_) => CounterModel(),
  child: Consumer<CounterModel>(
    builder: (context, model, child) => Text('${model.count}'),
  ),
)
```

### 6. 依赖注入

依赖注入（DI）是状态管理的重要组成部分：

- **Riverpod**：Provider 定义本身就是 DI 容器
- **BLoC**：通常配合 `get_it` 或 `injectable` 使用
- **GetX**：内置 DI（`Get.put`, `Get.lazyPut`）
- **Provider**：`Provider.value`, `ChangeNotifierProvider`

## 常见问题

### Q1: 小项目该用哪种方案？

- **1-5 个页面**：`setState` + `ValueNotifier` 足够
- **5-20 个页面**：Riverpod 或 Provider
- **大型项目/团队**：BLoC（强约束、可测试性好）
- **快速原型**：GetX（但大型项目可能维护困难）

### Q2: Riverpod vs Provider 的区别？

Riverpod 不依赖 InheritedWidget，不需要 `context`，可以在 `main()` 中使用。它提供编译时检查、自动销毁、更灵活的 Provider 组合（`ref.watch` / `ref.read`）。Provider 是 Riverpod 的前身，新项目推荐直接用 Riverpod。

### Q3: BLoC 的样板代码太多怎么办？

使用 `flutter_bloc` 包配合代码生成（`bloc_generator`），或采用 Cubit 简化版：

```dart
class CounterCubit extends Cubit<int> {
  CounterCubit() : super(0);
  void increment() => emit(state + 1);
}
```

Cubit 去掉了 Event 层，直接调用方法 emit 新状态，适合简单场景。

### Q4: GetX 有什么争议？

GetX 的争议点：过度全局化（`Get.put` 污染全局）、路由不遵循 Navigator 2.0 规范、维护频率下降、隐式行为多不利于大型团队协作。适合个人项目和快速原型，企业级项目建议选择 Riverpod 或 Bloc。

### Q5: 如何避免不必要的 Widget 重建？

- **Riverpod**：使用 `ref.select()` 只监听需要的部分
- **BLoC**：`BlocBuilder` 的 `buildWhen` 条件
- **GetX**：`Obx` 自动只重建依赖了 `.obs` 变量的 Widget
- **通用**：将变化部分拆分为独立 Widget，使用 `const` 构造

### Q6: 状态应该放在哪一层？

推荐分层架构：
1. **Presentation 层**：Widget + 状态管理（BLoC/Notifier）
2. **Domain 层**：业务实体和用例
3. **Data 层**：Repository + 数据源（API/DB）

状态管理负责 Presentation 层与 Domain 层之间的桥接。
