# 消息可靠性保障

## 定义
消息可靠性保障是确保消息从生产到消费全链路不丢失、不重复的工程化方案。核心环节包括：生产端确认、Broker 持久化、消费端 ACK、幂等消费、死信处理。

## 核心原理

### 消息丢失的三个环节

```
Producer ──→ Broker ──→ Consumer
   │           │           │
   ▼           ▼           ▼
发送丢失    存储丢失    消费丢失
```

### 生产端保障
| 策略 | 说明 | 适用场景 |
|------|------|----------|
| **同步发送** | 发送后等待 Broker 确认 | 关键业务事件 |
| **异步回调** | 发送后通过回调确认 | 高吞吐场景 |
| **本地事务** | 先写 DB 再发消息 | Outbox 模式 |
| **重试机制** | 发送失败自动重试 | 网络不稳定 |

### Broker 端保障
| 策略 | 说明 | Kafka | RabbitMQ | NATS |
|------|------|-------|----------|------|
| **持久化** | 消息写入磁盘 | ✅ 默认 | ✅ 需配置 | JetStream |
| **多副本** | 数据冗余 | ✅ ISR | ✅ 镜像队列 | ✅ 集群 |
| **确认机制** | 写入确认 | acks=all | confirm | pub ack |

### 消费端保障
```
消费流程：
1. 从 Queue 拉取消息
2. 处理业务逻辑
3. 确认消费（ACK）
4. 如果处理失败 → 重试或进入死信队列
```

## 幂等消费

### 为什么需要幂等？
- 网络抖动导致 ACK 丢失，Broker 重新投递
- 消费者重启后重新消费未确认消息
- 重试机制导致消息重复

### 幂等实现方案

| 方案 | 实现 | 适用场景 |
|------|------|----------|
| **数据库唯一约束** | INSERT IGNORE 或 ON DUPLICATE KEY | 创建类操作 |
| **幂等键表** | 记录已处理的消息 ID | 通用方案 |
| **Redis SETNX** | 检查消息 ID 是否已处理 | 高并发场景 |
| **业务状态机** | 检查当前状态是否允许转换 | 状态变更类 |

### Laravel 实现示例
```php
// 幂等消费中间件
class IdempotentMiddleware
{
    public function handle($message, Closure $next)
    {
        $messageId = $message->getMessageId();
        
        // 检查是否已处理
        if (ProcessedMessage::where('message_id', $messageId)->exists()) {
            return; // 跳过重复消息
        }
        
        // 处理消息
        $next($message);
        
        // 记录已处理
        ProcessedMessage::create(['message_id' => $messageId]);
    }
}
```

## 死信队列（DLQ）

### 什么是死信？
- 消费失败超过最大重试次数的消息
- 消息格式错误无法解析
- 业务规则拒绝的消息

### 死信处理流程
```
正常 Queue → 消费失败 → 重试（N 次）→ 死信 Queue → 人工处理/告警
```

### 死信队列设计
1. **独立存储**：死信与正常消息分离
2. **告警机制**：死信数量超阈值触发告警
3. **人工干预**：提供管理界面查看和重发死信
4. **分析归档**：定期分析死信原因，优化消费逻辑

## 重试策略

### 重试模式
| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **立即重试** | 失败后立即重试 | 瞬时故障 |
| **固定间隔** | 每次重试间隔固定 | 一般场景 |
| **指数退避** | 间隔指数增长 | 网络不稳定 |
| **抖动退避** | 指数退避 + 随机抖动 | 高并发场景 |

### 重试配置建议
```
重试次数：3-5 次
初始间隔：1 秒
最大间隔：60 秒
退避因子：2
抖动范围：±20%
```

## 常见问题

### 消息丢失 vs 消息重复，优先解决哪个？
- **消息丢失**：业务数据不一致，影响更大
- **消息重复**：通过幂等消费解决，影响较小
- 决策：优先保证不丢失，再通过幂等解决重复

### 如何监控消息可靠性？
1. **生产端**：发送成功率、发送延迟
2. **Broker**：消息堆积量、消费延迟
3. **消费端**：消费成功率、消费延迟、死信数量
4. **端到端**：消息从生产到消费的总延迟
