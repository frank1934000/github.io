# 构建优化策略

## 定义
前端构建优化是指通过代码分割（Code Splitting）、Tree Shaking、预构建、缓存策略、图片优化等手段，减小打包体积、加快构建速度和页面加载速度。

## 核心原理

### 代码分割（Code Splitting）
```typescript
// 路由级懒加载
const routes = [
  { path: '/dashboard', component: () => import('./views/Dashboard.vue') },
  { path: '/orders', component: () => import('./views/Orders.vue') }
]

// Vite 手动分包
export default defineConfig({
  build: {
    rollupOptions: {
      output: {
        manualChunks: {
          vendor: ['vue', 'vue-router', 'pinia'],
          charts: ['echarts', 'chart.js']
        }
      }
    }
  }
})
```

### 缓存策略
```
index.html      → no-cache（确保获取最新引用）
JS/CSS 带 hash  → immutable, max-age=31536000（一年缓存）
图片/字体       → max-age=2592000（30天）
API 响应        → no-store
```

### 预构建优化（Vite）
```typescript
// vite.config.ts
export default defineConfig({
  optimizeDeps: {
    include: ['lodash-es', 'axios'],  // 强制预构建
    exclude: ['your-local-package']    // 排除本地包
  }
})
```

### 图片优化
- WebP/AVIF 格式替代 PNG/JPEG
- 响应式图片 `<img srcset>`
- 懒加载 `loading="lazy"`
- CDN 分发

### Sourcemap 策略
| 环境 | 策略 | 原因 |
|------|------|------|
| 开发 | `eval-source-map` | 快速重建 + 完整映射 |
| 测试 | `source-map` | 完整映射便于调试 |
| 生产 | `hidden-source-map` | 不暴露但可上传 Sentry |

## 常见问题

**Q: 分包越多越好吗？**
A: 不是。过多的小文件增加 HTTP 请求数。一般按 vendor/third-party/app 三层分包即可。

**Q: Gzip 还是 Brotli？**
A: Brotli 压缩率更高（约 15-20%），但压缩速度慢。生产环境用 Brotli，开发用 Gzip。
