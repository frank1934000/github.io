# 微前端架构

## 定义
微前端是一种将大型前端应用拆分为多个独立子应用的架构模式，每个子应用可独立开发、独立部署、独立运行。qiankun 是基于 single-spa 的微前端框架，提供 JS 沙箱、样式隔离等能力。

## 核心原理

### qiankun 核心机制

```
主应用（Laravel + Vue）
├── 注册子应用列表
├── 路由匹配 → 加载子应用
├── JS 沙箱隔离（Proxy 沙箱）
└── CSS 隔离（Shadow DOM / Scoped）
    ├── 子应用 A（Vue 管理后台）
    ├── 子应用 B（React 数据看板）
    └── 子应用 C（原生 JS 工具）
```

### 主应用配置
```typescript
import { registerMicroApps, start } from 'qiankun'

registerMicroApps([
  {
    name: 'admin-app',
    entry: '//localhost:8081',
    container: '#micro-container',
    activeRule: '/admin'
  },
  {
    name: 'dashboard-app',
    entry: '//localhost:8082',
    container: '#micro-container',
    activeRule: '/dashboard'
  }
])

start({ prefetch: 'all' })
```

### 子应用生命周期
```typescript
export async function bootstrap() { /* 初始化 */ }
export async function mount(props) { /* 挂载 */ }
export async function unmount(props) { /* 卸载 */ }
```

### 样式隔离方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| Shadow DOM | 完全隔离 | 第三方库兼容问题 |
| Scoped CSS | 简单 | 隔离不彻底 |
| CSS Modules | 编译时隔离 | 需要约定命名 |
| CSS-in-JS | 运行时隔离 | 性能开销 |

### 常见踩坑点
1. **路由冲突** - 子应用路由前缀必须与 `activeRule` 一致
2. **静态资源路径** - 子应用需要配置 `publicPath`
3. **Cookie/Token 共享** - 主子应用间鉴权信息传递
4. **全局样式污染** - 子应用样式影响主应用

## 常见问题

**Q: qiankun vs Module Federation vs iframe？**
A: qiankun 运行时集成，适合异构技术栈；MF 构建时共享，适合同技术栈；iframe 最隔离但体验差。

**Q: 微前端适合什么场景？**
A: 大型团队协作、遗留系统渐进迁移、多技术栈共存。小团队不建议使用。
