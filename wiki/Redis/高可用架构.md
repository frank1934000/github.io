# 高可用架构

## 概览
当 Redis 从“缓存组件”升级为“核心基础设施”后，单实例模式会逐渐暴露容量、吞吐、扩容、故障转移等瓶颈。文章体系中的高可用主题主要围绕：**主从、Sentinel、Cluster、故障转移与分片扩容**。

- 关联页面：[缓存策略](缓存策略.md)、[分布式锁](分布式锁.md)、[性能优化](性能优化.md)、[Laravel集成](Laravel集成.md)

## 一、为什么需要高可用架构
随着 Redis 承担更多职责：
- 缓存
- Session
- 分布式锁
- 排行榜
- 限流计数器
- 队列与事件流

系统就会遇到：
- 单实例内存天花板
- 写入瓶颈
- 主从切换延迟
- 无法水平扩展

## 二、Sentinel 与 Cluster

### Sentinel
特点：
- 解决主从故障转移
- 不解决数据分片
- 所有数据仍在主节点

适合：
- 数据量还未大到需要水平切分
- 更关注高可用而非扩容

### Cluster
特点：
- 16384 个 hash slot
- 自动分片
- 主从复制 + 自动故障转移
- 可水平扩容

适合：
- 数据量大
- 单主写入压力高
- 需要在线扩容

### 从文章中提炼的核心判断
- 如果主要问题是“Redis 宕机后恢复慢”，Sentinel 可能够用
- 如果主要问题是“内存与写吞吐到顶”，就要上 Cluster

## 三、Redis Cluster 的关键概念

### 1. Hash Slot
Cluster 把 key 分布到 16384 个槽位。
- 节点不是直接存“部分 key”，而是负责“部分 slot”
- 扩容/迁移本质是 slot 迁移

### 2. 主从结构
每个 master 可以带 replica：
- master 负责写
- replica 参与故障转移与读扩展（视客户端/架构而定）

### 3. 故障转移
当 master 异常：
- 集群检测失联
- 选举 replica 晋升为新 master
- 客户端重新路由

### 4. 客户端要求
Cluster 需要**智能客户端**理解 MOVED/ASK 重定向。
这也是 Laravel/Predis 集成中的关键坑点之一。

## 四、高可用与缓存问题的关系

### 1. 缓存雪崩会推动架构升级
当 Redis 自身故障会造成大面积缓存失效时，仅靠业务代码难以兜底，必须靠：
- 主从
- Sentinel
- Cluster
- 多级缓存
- 降级系统

所以高可用不是独立话题，而是 [缓存策略](缓存策略.md) 的终局部分。

### 2. 分布式锁在 Cluster 下更复杂
锁在 Cluster 中涉及：
- 单 key 路由到特定 slot
- 多节点一致性预期
- RedLock 是否必要
- 客户端故障时的重试策略

这与 [分布式锁](分布式锁.md) 强相关。

## 五、部署与运维关注点
从两篇 Cluster 文章可提炼出：
- 端口、数据目录、集群配置文件要独立
- 节点间密码与认证配置要一致
- 故障转移必须做演练，不能只停留在“理论可用”
- 监控项要覆盖：内存、复制延迟、slot 分布、故障切换、慢查询
- 扩容时评估 slot 迁移对业务延迟的影响

## 六、架构选型速查

| 问题 | 更适合的方案 |
|---|---|
| 只需主从切换 | Sentinel |
| 需要水平扩容 | Cluster |
| 容量逼近单机上限 | Cluster |
| 写压力过高 | Cluster |
| 只想快速提升可用性 | 主从 + Sentinel |

## 七、与 Laravel 的连接点
Laravel/Predis 集成高可用 Redis 时需要关注：
- Cluster 模式客户端配置
- Horizon/Queue/Cache 是否共用连接
- 超时与重试策略
- 故障转移期间的异常处理与降级

详见 [Laravel集成](Laravel集成.md)。
