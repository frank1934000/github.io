# Redis 持久化

## 定义

Redis 持久化是将内存中的数据保存到硬盘的过程，确保 Redis 重启后数据不丢失。Redis 提供两种持久化方案：**RDB（快照）** 和 **AOF（追加日志）**，以及 Redis 4.0 引入的 **混合持久化** 模式。

## 核心原理

### 1. RDB 持久化

RDB 持久化是把当前进程数据生成**快照**保存到硬盘的过程。RDB 文件是一个压缩的二进制文件，通过它可以还原某个时刻数据库的状态。

**触发方式**：

| 触发方式 | 说明 | 是否阻塞 |
|---------|------|---------|
| `save` 命令 | 手动触发，阻塞直到完成 | **是（线上禁用）** |
| `bgsave` 命令 | 手动触发，fork 子进程执行 | 仅 fork 阶段短暂阻塞 |
| `save m n` 配置 | m 秒内 n 次修改自动触发 | 仅 fork 阶段短暂阻塞 |
| 主从全量复制 | 主节点自动 bgsave | 仅 fork 阶段短暂阻塞 |
| `shutdown` 命令 | 无 AOF 时自动触发 | 阻塞 |

**RDB 文件格式**：
```
REDIS | db_version | databases | EOF | check_sum
```

**优点**：
- 紧凑的二进制文件，适合备份与全量复制
- 恢复速度远快于 AOF
- 容灾性好，可拷贝到远程机器

**缺点**：
- 实时性低，间隔期间故障会丢数据
- 存在版本兼容问题（老版本 Redis 可能无法加载新版本 RDB）
- fork 子进程时内存翻倍风险

### 2. AOF 持久化

AOF（Append Only File）以独立日志的方式记录每次写命令，重启时重新执行 AOF 文件中的命令恢复数据。AOF 解决了数据持久化的实时性问题，是目前 Redis 持久化的**主流方式**。

**工作流程**：

```
写入命令 → aof_buf 缓冲区 → 文件同步（fsync） → AOF 文件
                                                    ↓
                                              定期重写（rewrite）
                                                    ↓
                                              重启加载（load）
```

**fsync 策略**（`appendfsync` 配置）：

| 策略 | 说明 | 数据安全性 | 性能 |
|------|------|-----------|------|
| `always` | 每条命令都 fsync | 最高（不丢数据） | 最差 |
| `everysec` | 每秒 fsync（默认） | 最多丢 1 秒数据 | **推荐** |
| `no` | 由操作系统决定 | 不保证 | 最好 |

**AOF 重写（Rewrite）**：
- 随着 AOF 文件越来越大，需要定期重写压缩
- 重写时读取数据库最新状态，生成最小命令集
- Redis 7.0+ 支持 Multi-Part AOF，将 AOF 拆分为 base + incremental

**优点**：
- 实时性好，可配置每条命令都持久化
- 通过 append 模式写文件，中途宕机可通过 `redis-check-aof` 工具修复

**缺点**：
- 文件比 RDB 大，恢复速度慢
- 数据集大时启动效率低

### 3. 混合持久化（Redis 4.0+）

Redis 4.0 引入混合持久化，将 RDB 的快速加载与 AOF 的低数据丢失结合：

```
[RDB 快照内容] + [增量 AOF 日志]
```

**工作原理**：
- AOF 重写时，前半段写入 RDB 格式的全量数据
- 后半段写入重写期间的增量 AOF 命令
- 重启时先加载 RDB 部分（快），再重放增量 AOF（小）

**配置**：
```conf
aof-use-rdb-preamble yes  # 默认开启（Redis 5.0+）
```

### 4. 数据恢复优先级

Redis 启动时加载数据的流程：

```
1. AOF 开启且文件存在 → 加载 AOF
2. AOF 关闭或文件不存在 → 加载 RDB
3. 加载成功 → 启动成功
4. 文件有错误 → 启动失败并打印错误
```

## 选型建议

| 场景 | 推荐方案 |
|------|---------|
| 追求数据安全 | AOF（everysec） + RDB 定时备份 |
| 可接受分钟级丢失 | 仅 RDB |
| 大数据集 + 快速恢复 | 混合持久化 |
| 仅缓存不重要数据 | 关闭持久化 |

## 生产实践要点

1. **Master 节点不要做持久化**：由 Slave 开启 AOF，策略为每秒同步
2. **主从在同一局局域网**：保证复制速度和连接稳定性
3. **监控 fork 耗时**：`latest_fork_usec` 指标，超过 1s 需告警
4. **关闭 Transparent HugePages**：避免写操作阻塞放大 512 倍
5. **AOF 刷盘阻塞监控**：主线程发现距上次 fsync 超过 2 秒会阻塞等待

## 常见问题

### Q: RDB 和 AOF 应该选哪个？
A: 生产环境推荐**同时使用两种**，AOF 保证数据安全性，RDB 用于备份和快速恢复。Redis 重启时优先加载 AOF。

### Q: AOF 重写会阻塞主线程吗？
A: 重写过程由子进程完成，不会阻塞主线程。但重写期间的新写命令需要同时追加到旧 AOF 和重写缓冲区，重写完成后由主线程进行原子替换。

### Q: 混合持久化是默认开启的吗？
A: Redis 5.0+ 默认开启（`aof-use-rdb-preamble yes`）。如果需要关闭，设置为 `no`。

### Q: 如何监控持久化状态？
A: 使用 `INFO persistence` 命令，关注 `rdb_last_bgsave_status`、`aof_last_bgrewrite_status`、`aof_last_write_status` 等指标。
