# 分布式锁

## 概览
Redis 分布式锁在这批文章中主要用于：**库存预占、优惠券发放、缓存重建互斥、热点任务串行化**。核心主题不是“怎么加锁”，而是“锁为什么会失效，以及如何把失效风险降到可控”。

- 关联页面：[缓存策略](缓存策略.md)、[性能优化](性能优化.md)、[高可用架构](高可用架构.md)、[Laravel集成](Laravel集成.md)

## 一、为什么需要分布式锁
在单实例应用里，进程内锁可能足够；但在 Laravel 多实例、PHP-FPM、多 worker、微服务拆分后，单机锁已失效，需要一个跨实例共享的互斥机制。

典型场景：
- 秒杀库存扣减
- 用户优惠券原子发放
- 热点缓存重建
- 定时任务去重执行
- 多实例预热任务互斥

## 二、正确的基础模型

### 1. 加锁
最基础实现是：
- `SET key value NX EX seconds`
- value 必须是**唯一标识**，如 UUID/request_id
- 不能只用 `SETNX` 而不带过期时间，否则异常退出会造成死锁

### 2. 解锁
不能直接 `DEL key`，否则可能删掉别人的锁。正确做法：
- 先比较 value 是否与自己持有的一致
- 一致才删除
- 最稳妥的是通过 **Lua 脚本原子比较并删除**

### 3. 锁超时
TTL 必须覆盖业务执行时间，但不能过长。
- 太短：业务未完成锁就过期，导致并发冲入
- 太长：出现故障时恢复变慢

因此很多生产场景需要：
- 动态 TTL
- 看门狗续期
- 长任务拆小

## 三、常见失效场景

### 1. 死锁
典型原因：
- 加锁成功后业务进程崩溃
- 没设置过期时间
- 解锁逻辑没执行

应对：
- 强制 TTL
- 业务超时控制
- 监控长时间占锁 key

### 2. 锁误删
典型原因：
- 客户端 A 锁过期
- 客户端 B 获得新锁
- 客户端 A 恢复后执行 `DEL key`
- 导致 B 的锁被删掉

应对：
- 锁值必须唯一
- 用 Lua 比较 value 后删除

### 3. 锁提前过期
典型原因：
- 业务处理时间超过 TTL
- 网络抖动、GC、下游慢调用

应对：
- TTL 估算要留缓冲
- 长任务用续期机制
- 重逻辑迁到 Lua/异步任务/数据库事务协调

### 4. 热点 Key 锁竞争
大量请求同时竞争同一把锁，会带来：
- 自旋重试放大 Redis 压力
- 线程堆积
- 延迟陡升

应对：
- 指数退避
- 快速失败与业务降级
- 拆分热点资源粒度
- 回到 [性能优化](性能优化.md) 做批量化与流程压缩

### 5. 集群一致性争议
文章中提到 RedLock 与多节点一致性问题，核心不是“理论上绝对完美”，而是：
- 业务是否真的需要跨主节点多数派加锁
- 你能否接受极端网络分区下的小概率异常
- 是否有数据库唯一约束、幂等设计等第二道防线

这类问题与 [高可用架构](高可用架构.md) 强相关。

## 四、Lua 与分布式锁的关系
Lua 并不是锁的替代品，而是锁实现中的关键增强组件。

主要作用：
1. **原子解锁**：compare-and-del
2. **原子执行业务逻辑**：检查库存并扣减，不必显式先加锁再改值
3. **降低客户端往返**：减少 TOCTOU 竞态窗口

因此很多文章会从“分布式锁”自然延伸到 [性能优化](性能优化.md) 里的 Lua 脚本实践。

## 五、分布式锁与缓存击穿
缓存击穿时常见做法是"只允许一个请求回源构建缓存"，本质上就是锁。

### 进阶：三重纵深防御
简单的互斥锁是缓存击穿防护的基础，但高并发场景需要更精细的策略。详见 [Cache Stampede 防护](Cache-Stampede防护.md) 了解分布式锁 + XFetch + SWR 三重防御体系。

处理链路：
1. 请求发现热点 key 失效
2. 尝试获取重建锁
3. 拿到锁的请求查 DB 并回填缓存
4. 其他请求等待、快速失败或返回旧值

所以缓存重建锁是 [缓存策略](缓存策略.md) 的一个具体落地点。

## 七、最佳实践清单

| 风险点 | 推荐做法 |
|---|---|
| 死锁 | `SET NX EX`，禁止无 TTL 锁 |
| 误删锁 | 唯一 value + Lua compare-and-del |
| TTL 不准 | 按业务耗时动态估算，必要时续期 |
| 热点争用 | 退避、降级、拆 key、限制重试 |
| 集群不一致 | 幂等、唯一约束、补偿机制 |
| 缓存重建风暴 | 锁 + 逻辑过期 + 旧值兜底 |
