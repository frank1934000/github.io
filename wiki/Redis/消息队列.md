# 消息队列

## 概览
Redis 在文章中承担了两类消息能力：
1. **基础队列**：List/阻塞队列、Pub/Sub
2. **可确认消费队列**：Redis Stream + Consumer Group

如果只是轻量异步，Redis 足够；如果需要极高吞吐、复杂投递语义，再考虑 Kafka/RabbitMQ。

- 关联页面：[数据结构](数据结构.md)、[性能优化](性能优化.md)、[Laravel集成](Laravel集成.md)

## 一、Redis 做消息队列的定位
总览文章认为 Redis 可用于消息队列，但“能力够不够”取决于场景：
- 轻量异步任务：适合
- 已有 Redis 基础设施：适合
- 需要消费者组、ACK、Pending：优先 Stream
- 需要强投递保障、复杂路由、超大吞吐：Redis 可能不是最终形态

## 二、List 队列

### 典型模式
- 生产：`LPUSH` / `RPUSH`
- 消费：`LPOP` / `RPOP`
- 阻塞消费：`BLPOP` / `BRPOP`

### 优点
- 简单直接
- 命令少，接入快
- 适合短链路、低复杂度异步处理

### 局限
- 缺少内建 ACK
- 消费者宕机后的消息追踪能力弱
- 多消费者协同能力有限
- 不如 Stream 易于管理未确认消息

### 适用场景
- 简易任务队列
- 非关键通知
- 单消费者/少量消费者模型

---

## 三、Pub/Sub

### 特点
- 发布即广播
- 消息不持久
- 没有 ACK
- 消费者离线就丢消息

### 适合
- 实时通知
- 在线广播
- 对“消息不必重放”容忍的场景

### 不适合
- 订单事件
- 审计日志
- 必须确认处理的任务

## 四、Redis Stream

### 为什么重要
`redis-stream-guide-laravel.md` 说明：Stream 填补了 Redis 在“轻量 MQ + 消费确认”之间的空白。

它尤其适合：
- 订单状态变更通知
- 库存扣减事件广播
- 用户行为埋点
- 不想额外引入 RabbitMQ/Kafka 的中等吞吐异步任务

### 核心能力
- 追加消息：`XADD`
- 读取消息：`XREAD`
- 消费者组：`XGROUP`
- 组内消费：`XREADGROUP`
- 确认消息：`XACK`
- 待处理列表：PEL / Pending Entries List
- 卡住消息认领：`XCLAIM` / `XAUTOCLAIM`

### 与 List 最大差异
| 能力 | List | Stream |
|---|---|---|
| 消息 ID | 弱 | 强，自动生成 |
| ACK | 无内建 | 有 |
| 消费者组 | 无 | 有 |
| Pending 可追踪 | 弱 | 强 |
| 重试/认领 | 需自建 | 原生支持 |

### 文章里的重点坑位
- 消费成功后必须 `XACK`
- Pending 长期堆积意味着消费者异常或逻辑卡死
- Stream 长度需要治理，避免无限增长
- Consumer Group 初始化和幂等创建要处理好
- Laravel Worker 退出/重启后的消息认领要有恢复方案

## 五、消息可靠性要点
Redis 不是默认“绝对可靠”的 MQ，仍需工程补强：
- 消费端幂等
- 失败重试机制
- 死信/异常队列设计
- 监控 Pending 数量、消费延迟、失败率
- 消费者优雅停机

这些能力常常需要与 [Laravel集成](Laravel集成.md) 中的 Queue Worker 封装配合。

## 六、选型建议

| 场景 | 推荐 |
|---|---|
| 简单阻塞队列 | List |
| 广播通知 | Pub/Sub |
| 轻量可靠异步任务 | Stream |
| 高吞吐复杂消息系统 | 评估 Kafka / RabbitMQ |
