## 一、通用缓存模式

### 1. Cache Aside
最常见模式：读请求先查缓存，未命中再查数据库并回填缓存；写请求更新数据库后删除或刷新缓存。

适用场景：
- 商品详情
- 订单聚合数据
- 用户信息
- API Response 预热

来源文章指出的核心问题：如果 TTL 设计粗糙、热点分布不均、失效时没有回源控制，就会放大数据库压力。

### 2. TTL 设计不是“随便给个时间”
`redis-guide-cache.md` 强调：固定 TTL 在订单、库存、状态机类业务里很危险。更稳妥的做法是：
- 根据业务时间动态计算 TTL
- 订单状态变更时主动刷新
- 关键事件触发时手动失效
- 定时扫描兜底修正

### 3. 热点 Key 管理
热点数据要重点治理：
- 对超级热点 Key 考虑永不过期或逻辑过期
- 增加本地缓存/多级缓存
- 缓存重建时加互斥控制
- 将大对象拆分，避免单 Key 负载过高

### 4. 空值缓存与预热
- 空值缓存：降低对不存在数据的重复回源
- 缓存预热：在活动开始前主动构建缓存
- 随机过期：避免同一时刻批量失效

## 二、缓存穿透

### 定义
请求查询的 key 对应数据根本不存在，导致每次都穿过缓存直达数据库。

### 典型场景
- 恶意构造不存在的商品 ID
- 参数无效但未校验
- 数据删除后缓存与布隆层未同步

### 风险
- DB 被无意义查询打满
- 连接池耗尽
- 对攻击流量缺少第一层拦截

### 常见方案
1. **参数校验**：先在应用层过滤非法 ID、非法格式
2. **空值缓存**：数据库查不到时缓存空对象/Null Sentinel
3. **布隆过滤器**：先判断 key 是否“可能存在”
4. **限流与熔断**：异常流量直接拦截

### 文章交叉观点
- 老文章更强调“空值缓存 + 布隆过滤器”
- 新实战文章进一步加入 API 层校验、可观测性和降级策略

## 三、缓存击穿

### 定义
某个**热点 Key**在高并发场景下恰好过期，导致大量请求同时回源数据库。

### 典型场景
- 爆款商品详情页 TTL 到点失效
- 榜单、活动页在高峰期间过期
- 缓存重建逻辑过慢

### 常见方案
1. **互斥锁/单飞机制**：只允许一个线程回源重建缓存
2. **逻辑过期**：缓存值带过期时间，后台异步刷新
3. **热点 Key 永不过期**：业务允许时直接长期驻留
4. **请求降级**：返回旧值、兜底页或静态数据

### 与分布式锁关系
缓存击穿本质是"缓存重建阶段的并发控制问题"，所以常和 [分布式锁](分布式锁.md) 联动。

### 进阶：三重纵深防御
对于高并发热点 Key，简单的互斥锁可能不够。进阶方案采用**分布式锁 + XFetch 概率性提前过期 + SWR 后台异步刷新**三重防御体系，详见 [Cache Stampede 防护](Cache-Stampede防护.md)。

## 四、缓存雪崩

### 定义
大量 Key 在同一时间失效，或 Redis 整体不可用，导致请求集中涌向后端数据库。

### 触发原因
- 所有缓存使用相同 TTL
- 批量预热后统一过期
- Redis 实例/集群异常
- 批量删除引发回源洪峰

### 常见方案
1. **TTL 加随机抖动**
2. **分批预热/分批过期**
3. **熔断、限流、降级**
4. **多级缓存**
5. **高可用 Redis 架构**：主从、Sentinel、Cluster
6. **数据库容灾**：读写分离、分库分表、只读兜底

### 架构层关联
雪崩问题最终会推动系统走向 [高可用架构](高可用架构.md)。
