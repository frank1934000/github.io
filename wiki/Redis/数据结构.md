# 数据结构

## 概览
这批文章既覆盖 Redis 五大常规结构，也重点展开了三类特殊结构：**HyperLogLog、Bitmap、GEO**。整体主线是“按场景选结构”，而不是把 Redis 只看作一个 String 缓存。

- 关联页面：[消息队列](消息队列.md)、[性能优化](性能优化.md)、[Laravel集成](Laravel集成.md)

## 一、基础数据结构速览

来自总览/面试类文章的共识：
- **String**：缓存、计数器、分布式锁、Bitmap/HyperLogLog 的底层承载
- **Hash**：用户信息、会话、购物车属性聚合
- **List**：简单队列、时间线、阻塞队列
- **Set**：去重、标签、共同关系
- **ZSet**：排行榜、范围排序、GEO 底层

## 二、HyperLogLog

### 适用场景
- UV 统计
- 去重后的大规模基数估算
- 多天/多渠道访客合并估算

### 核心特点
- 固定约 12KB 内存
- 适合“估算有多少个不同元素”
- 精度不是 100%，但对报表很实用
- 不支持查询具体成员，也不适合删除成员

### 文章提炼
`redis-hyperloglog-guide-uv.md` 强调的不是命令本身，而是**与 Set 的取舍**：
- 要精确去重集合 → Set
- 只要 UV 统计数量 → HyperLogLog
- 数据规模极大时，HyperLogLog 以内存优势取胜

相关命令：
- `PFADD`
- `PFCOUNT`
- `PFMERGE`

## 三、Bitmap

### 适用场景
- 用户签到
- 在线状态
- 布尔特征标记
- 批量状态计算

### 核心特点
- 一个 bit 表示一个布尔状态
- 空间压缩极强
- 单点读写 O(1)
- 适合 ID 可映射成 offset 的场景

### 文章提炼
`redis-bitmap-guide.md` 将 Bitmap 用在三类高频业务：
1. 用户签到
2. 在线状态
3. 活动/优惠券/黑名单等特征标记

常用能力：
- `SETBIT`
- `GETBIT`
- `BITCOUNT`
- `BITOP`
- `BITFIELD`

设计重点：
- 明确 offset 映射规则
- 区分“按用户建 key”还是“按日期建 key”
- 控制超大 offset 带来的稀疏浪费

## 四、GEO

### 适用场景
- 附近门店
- 周边景点
- 配送范围校验
- 按距离排序的 POI 查询

### 核心特点
- 底层基于 GeoHash + ZSet
- 支持范围查询与距离排序
- 更适合中等精度、实时检索场景

### 文章提炼
`redis-geo-guide.md` 的重点包括：
- `longitude` 在前、`latitude` 在后，顺序易错
- Redis 6.2+ 更推荐 `GEOSEARCH`
- GEO 非 GIS 全功能数据库，不替代复杂空间分析系统
- 在 Cluster 下仍要注意 key 组织与查询模式

常用命令：
- `GEOADD`
- `GEOSEARCH`
- `GEODIST`
- `GEOHASH`

## 六、结构选型速查

| 需求 | 推荐结构 | 原因 |
|---|---|---|
| 普通缓存 | String / Hash | 简单直接，通用性强 |
| 用户会话 | Hash | 字段化读取、原子更新方便 |
| 购物车 | Hash + List | 商品维度字段聚合，顺序操作可选 |
| UV 统计 | HyperLogLog | 固定内存、适合基数估算 |
| 签到/在线状态 | Bitmap | 1 bit 压缩，适合布尔状态 |
| 附近门店 | GEO | 原生距离查询与排序 |
| 排行榜 | ZSet | 天然按 score 排序 |
| 简单队列 | List / Stream | 取决于是否需要 ACK 与消费者组 |
