# 缓存写入模式

## 定义

缓存写入模式定义了应用在写入数据时，缓存层与持久化存储（数据库）之间的协作方式。不同的写入模式在**一致性、延迟、吞吐量**之间做出不同的权衡。Redis 作为最常用的缓存层，与 MySQL/PostgreSQL 配合形成多种经典模式。

## 核心原理

### 1. Cache-Aside（旁路缓存）

Cache-Aside 是最常用的缓存模式，应用代码直接管理缓存的读写。

**读流程**：
```
应用 → 查询 Redis
  ├─ 命中 → 返回数据
  └─ 未命中 → 查询 MySQL → 写入 Redis → 返回数据
```

**写流程**：
```
应用 → 更新 MySQL → 删除（或更新）Redis
```

**关键设计决策**：先更新 DB 再删缓存（而非先删缓存再更新 DB）

| 顺序 | 问题 |
|------|------|
| 先删缓存再更新 DB | 并发读请求可能将旧数据写回缓存 |
| **先更新 DB 再删缓存** | 理论上有极小概率不一致，但实际几乎不会发生 |

**优点**：
- 实现简单，应用完全控制缓存逻辑
- 缓存只存储热数据，内存利用率高
- 适合读多写少的场景

**缺点**：
- 首次请求必然缓存未命中（Cold Start）
- 缓存与数据库之间存在短暂不一致窗口

**适用场景**：商品详情、用户信息、API 响应缓存

### 2. Write-Through（写穿透）

Write-Through 在写入缓存时同步写入数据库，保证缓存与数据库始终一致。

**写流程**：
```
应用 → 写入 Redis → 同步写入 MySQL → 返回成功
```

**读流程**：
```
应用 → 查询 Redis → 命中返回（理论上永远命中）
```

**优点**：
- 数据强一致性，缓存始终是最新数据
- 读取延迟低（缓存永远命中）

**缺点**：
- 每次写操作都产生数据库 I/O
- 写入延迟 = Redis 延迟 + MySQL 延迟
- 高写入场景下数据库成为瓶颈

**适用场景**：读多写少、对一致性要求极高的场景（如配置信息、字典数据）

### 3. Write-Back（回写/写回）

Write-Back 是本文重点介绍的高级模式：写操作仅写入缓存层，异步批量回写到数据库。

**写流程**：
```
应用 → 写入 Redis（标记 dirty）→ 立即返回成功
后台任务 → 检测脏数据达到阈值/定时触发 → 批量写入 MySQL
```

**核心组件**：
1. **WriteBackCacheService**：负责缓存写入、脏数据标记、回写调度
2. **Redis Buffer Layer**：使用 Hash/Set 缓存变更数据
3. **FlushJob**：Laravel 队列任务，执行批量 DB 写入
4. **WAL（Write-Ahead Log）**：预写日志，保障崩溃恢复

**优点**：
- 极低的写入延迟（仅 Redis 操作）
- 大幅降低数据库写入频率（N 次单条 → 1 次批量）
- 将随机写转化为顺序写，提升 DB 吞吐

**缺点**：
- 数据存在短暂不一致窗口
- 缓存故障可能导致数据丢失（需 WAL 保障）
- 实现复杂度高

**适用场景**：
- 实时分析计数器（PV/UV 统计）
- 用户行为日志（点击流、浏览记录）
- 活动 Feed 流
- 分布式会话存储
- 高写入、可容忍短暂不一致的场景

**WAL（预写日志）保障机制**：
```
写入 Redis → 同时写入 WAL 文件（顺序写，极快）
后台回写成功 → 标记 WAL 条目为已完成
Redis 崩溃恢复 → 重放未完成的 WAL 条目
```

### 4. Write-Around（写绕过）

Write-Around 将数据直接写入数据库，完全绕过缓存。缓存仅在读取时按需加载。

**写流程**：
```
应用 → 直接写入 MySQL（缓存不参与）
```

**读流程**：
```
应用 → 查询 Redis → 未命中 → 查询 MySQL → 写入 Redis → 返回
```

**优点**：
- 避免缓存被一次性写入数据污染（write-only 数据）
- 实现最简单

**缺点**：
- 写入后首次读取必然缓存未命中
- 适合写后不常读的数据

**适用场景**：日志数据、审计记录、写入后很少读取的数据

## 四种模式全景对比

| 维度 | Cache-Aside | Write-Through | Write-Back | Write-Around |
|------|------------|---------------|------------|--------------|
| 写入延迟 | 中（先 DB） | **高（同步 DB）** | **低（仅缓存）** | 高（直接 DB） |
| 读取延迟 | 低（缓存命中） | **低（永远命中）** | **低（缓存命中）** | 高（首次未命中） |
| 数据一致性 | 最终一致 | **强一致** | 最终一致 | **强一致** |
| DB 写入频率 | 与写入 1:1 | 与写入 1:1 | **1:N 批量** | 与写入 1:1 |
| 数据丢失风险 | 无 | 无 | **存在（需 WAL）** | 无 |
| 实现复杂度 | 低 | 中 | **高** | 低 |
| 适用场景 | 读多写少 | 强一致 + 读多 | **高写入** | 写后不常读 |

## 缓存一致性保障

### 延迟双删策略
```
1. 先删除缓存
2. 更新数据库
3. 延迟 N 毫秒后再次删除缓存
```
目的：防止并发读请求将旧数据写回缓存。

### 基于消息队列的最终一致
```
1. 更新数据库
2. 发送变更消息到 MQ
3. 消费者更新/删除缓存
```
适用于分布式系统中多个服务共享缓存的场景。

### 基于 binlog 的 CDC
```
1. 更新数据库
2. Debezium/Canal 捕获 binlog
3. 消费者更新/删除缓存
```
最可靠的方案，不侵入业务代码。

## 常见问题

### Q: Cache-Aside 为什么推荐"先更新 DB 再删缓存"？
A: 先删缓存再更新 DB 存在并发问题：线程 A 删缓存 → 线程 B 读 DB 旧值写入缓存 → 线程 A 更新 DB。此时缓存是脏数据。先更新 DB 再删缓存的不一致窗口极小（仅在 DB 更新成功但缓存删除失败时）。

### Q: Write-Back 如何保证数据不丢失？
A: 通过 WAL（预写日志）。每次写入 Redis 时同步写入 WAL 文件（顺序写，性能极好）。后台回写成功后标记 WAL 条目为已完成。Redis 崩溃恢复时重放未完成的 WAL 条目。

### Q: 什么时候该用 Write-Back 而不是 Cache-Aside？
A: 当写入频率很高（如每秒数千次），且可以容忍短暂不一致（秒级）时，Write-Back 能显著降低数据库压力。典型场景：PV 计数、用户行为日志、实时分析。

### Q: 缓存和数据库的双写一致性有完美方案吗？
A: 没有完美的方案。强一致性需要分布式事务（性能差），最终一致性是务实选择。推荐：Cache-Aside + 延迟双删 + 兜底校验（定时任务对账）。
