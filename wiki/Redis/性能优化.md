# 性能优化

## 概览
性能优化主题贯穿多篇文章，核心不在“Redis 很快”，而在于：**Redis 再快，也会被 RTT、热点、脚本阻塞、连接池争用、回源风暴拖垮**。

- 关联页面：[缓存策略](缓存策略.md)、[分布式锁](分布式锁.md)、[高可用架构](高可用架构.md)、[Laravel集成](Laravel集成.md)

## 一、Pipeline：优先解决 RTT 问题

### 核心思想
当一次请求需要执行几十条 Redis 命令时，性能瓶颈往往不是 Redis 计算，而是客户端与 Redis 之间的**网络往返次数**。

`redis-pipeline-guide-commandsoptimization.md` 的重点是：
- 多个 `GET` 串行调用会让 RTT 累加
- Pipeline 把 N 次往返压缩成 1 次
- 在 Laravel/Predis 中非常适合批量读取 SKU、标签、个性化标记

### 适用场景
- 批量读取多个 key
- 批量写入预热缓存
- 统计类批处理

### 注意事项
- Pipeline 不是事务，不保证回滚
- 批次过大可能吃掉客户端内存
- 与阻塞命令、长脚本混用要谨慎
- 在 Cluster 中要关注 key 分布与客户端能力

## 二、Lua：把多步逻辑压缩成原子执行

### 适用问题
- 限流
- 库存扣减
- 原子解锁
- 排行榜复杂更新
- “检查 + 修改”组合逻辑

### 优势
- 在 Redis 服务端原子执行
- 消除客户端多次往返带来的竞态
- 把复杂逻辑集中到脚本层管理

### 风险
- 脚本过长会阻塞 Redis 单线程
- 需要控制执行时间
- 脚本版本管理、缓存与 `EVALSHA` 失效要处理

## 三、高并发下的问题并不在 Redis“线程安全”
`high-concurrency.md` 指出：Redis 单线程顺序执行命令，本身不太会有传统共享内存式并发问题；真正的问题多出在：
- 客户端连接混乱
- 网络延迟
- 热点 key 排队
- 读写成对操作在客户端拆开执行

### 典型表现
- 连接超时
- 请求堆积
- 数据转换异常
- 高峰期延迟上升

### 应对方向
1. 连接池化
2. 请求批量化
3. 热点 key 隔离
4. 原子逻辑迁移到 Lua
5. 加锁或幂等控制

## 四、热点 Key 治理
热点问题出现在多个主题中：缓存、锁、排行榜、活动页、库存 key。

### 常见手段
- 永不过期或逻辑过期
- 读写分离或本地缓存
- 拆分大 key / 单 key
- 在回源重建时加锁
- 对热点流量限速

热点治理通常要与 [缓存策略](缓存策略.md) 和 [分布式锁](分布式锁.md) 联动看。

- Worker 与 Web 请求共用 Redis 时要留意争用

## 七、何时该从“优化命令”升级到“优化架构”
如果已经做了 Pipeline、Lua、连接池、热点治理，但仍然出现：
- CPU 长期高位
- 内存逼近阈值
- 单实例写入瓶颈
- 故障切换时间不可接受

说明问题已经不只是"写法"，而是要走向 [高可用架构](高可用架构.md)。
