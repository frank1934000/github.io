# 分布式限流算法

## 定义

分布式限流是在微服务架构中，通过共享的 Redis 计数器对全局流量进行控制的机制。核心诉求：**准确性**（所有节点共享计数器）、**高性能**（热路径不能成为瓶颈）、**原子性**（并发无竞态）、**容错性**（Redis 不可用时有降级策略）。

## 核心原理

### 五大限流算法对比

| 算法 | 原理 | 优点 | 缺点 | 适用场景 |
|------|------|------|------|---------|
| **固定窗口** | 等长时间窗口 + 计数器 | 实现简单 | 窗口交界处可突发 2x 流量 | 内部系统、精度要求低 |
| **滑动窗口计数器** | Redis Sorted Set + 时间戳 | 平滑、内存友好 | 略有精度损失 | **生产推荐**、API 限流 |
| **滑动窗口日志** | 存储每条请求的时间戳 | 精确 | 内存开销大（O(n)） | QPS 不高的高精度场景 |
| **令牌桶** | 固定速率放令牌 + 桶缓冲 | 允许突发流量 | 实现较复杂 | API 网关、允许短时突发 |
| **漏桶** | 固定速率排水 | 强制匀速输出 | 无法应对突发 | 后端保护、匀速处理 |

### Redis Cell 模块

Redis Cell 是 Redis 原生令牌桶模块，一行命令实现：

```
CL.THROTTLE user:123 10 5 60
# key, max_burst, tokens_per_period, period_seconds
# 返回 [0=allowed/1=denied, limit, remaining, retry_after, reset_after]
```

## 滑动窗口计数器（生产推荐）

### Redis Lua 实现

```lua
-- sliding_window.lua
local key = KEYS[1]
local now = tonumber(ARGV[1])      -- 当前时间戳(ms)
local window = tonumber(ARGV[2])   -- 窗口大小(ms)
local limit = tonumber(ARGV[3])    -- 限流阈值

-- 清除窗口外的过期成员
redis.call('ZREMRANGEBYSCORE', key, 0, now - window)

-- 当前窗口内的请求数
local count = redis.call('ZCARD', key)

if count < limit then
    -- 允许：添加当前请求
    redis.call('ZADD', key, now, now .. ':' .. math.random(1000000))
    redis.call('PEXPIRE', key, window)
    return 1  -- 允许
else
    return 0  -- 拒绝
end
```

### Laravel 中间件集成

```php
// app/Http/Middleware/RateLimitSlidingWindow.php
class RateLimitSlidingWindow
{
    public function handle(Request $request, Closure $next, int $limit = 60, int $window = 60)
    {
        $key = 'rate_limit:' . $request->user()?->id ?? $request->ip();
        $now = now()->getTimestampMs();
        $windowMs = $window * 1000;

        $allowed = Redis::eval(self::SCRIPT, 1, $key, $now, $windowMs, $limit);

        if (!$allowed) {
            return response()->json(['message' => 'Too Many Requests'], 429);
        }

        return $next($request);
    }
}
```

## 令牌桶（允许突发）

### 核心逻辑

```
桶容量 = max_burst
令牌生成速率 = tokens_per_second
每次请求消耗 1 个令牌
桶满时丢弃新令牌
令牌不足时拒绝请求（或等待）
```

### 适用场景

- API 网关：允许短时突发，但限制平均速率
- 第三方 API 调用：遵守上游限流策略
- 用户行为限制：如发帖、评论等

## 多维限流

生产环境通常需要多维限流组合：

```
全局限流：60,000 次/分钟（保护后端总容量）
  └── 用户级限流：60 次/分钟（防止单用户滥用）
        └── 接口级限流：10 次/分钟（敏感接口加强限制）
```

```php
// Laravel RateLimiter 多维定义
RateLimiter::for('api', function (Request $request) {
    return [
        Limit::perMinute(60000),                           // 全局
        Limit::perMinute(60)->by($request->user()?->id),   // 用户级
    ];
});

RateLimiter::for('sensitive', function (Request $request) {
    return Limit::perMinute(10)->by($request->user()?->id);
});
```

## 生产踩坑

| 问题 | 根因 | 解决方案 |
|------|------|---------|
| 热点 Key 单节点过载 | 所有请求打到同一个 Redis Key | 按用户/IP 分片 Key |
| Lua 脚本阻塞 Redis | 复杂 Lua 执行时间过长 | 简化脚本、用 EVALSHA |
| Redis 故障导致全站不可用 | 限流成为单点 | 降级为本地限流或放行 |
| 限流被绕过 | 不同入口未统一限流 | API Gateway 统一入口 |
| 时钟漂移 | 多节点时间不一致 | 使用 Redis 服务器时间 |

## 常见问题

**Q: 滑动窗口 vs 令牌桶怎么选？**
A: 大多数场景用滑动窗口（简单、Redis 内存友好）。需要允许短时突发流量时用令牌桶。

**Q: Redis 不可用时怎么办？**
A: 三种降级策略：(1) 放行所有请求 (2) 降级为本地限流（单节点限流） (3) 返回 503 告诉客户端稍后重试。

**Q: 如何限流第三方 API 调用？**
A: 用令牌桶 + Redis 全局计数。桶容量等于第三方的限额，令牌生成速率等于其恢复速率。
