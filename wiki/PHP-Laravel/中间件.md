# Laravel 中间件（Middleware）

## 定义

中间件是 HTTP 请求在进入 Controller 之前/之后执行的代码片段，构成请求链路：

```
Global Middleware → Route Middleware → Controller Middleware → Controller → Response → Global Middleware
```

## 中间件类型

### 全局中间件
每个请求都经过：

```php
// app/Http/Kernel.php
protected $middleware = [
    TrustProxies::class,
    CheckForMaintenanceMode::class,
    ValidatePostSize::class,
    TrimStrings::class,
];
```

### 路由中间件
指定路由才经过：

```php
Route::middleware(['auth', 'throttle:60,1'])->group(function () {
    Route::get('/orders', [OrderController::class, 'index']);
});
```

### 中间件分组

```php
protected $middlewareGroups = [
    'web' => [/* session, csrf, etc. */],
    'api' => [/* throttle, bindings, etc. */],
];
```

## 自定义中间件

```php
class EnsureTokenIsValid {
    public function handle(Request $request, Closure $next): Response {
        // 前置逻辑
        if ($request->header('X-Token') !== 'valid') {
            return response('Unauthorized', 401);
        }

        $response = $next($request);

        // 后置逻辑
        $response->headers->set('X-Processed', 'true');

        return $response;
    }
}
```

## 中间件执行顺序

```php
// Kernel.php 中定义优先级
protected $middlewarePriority = [
    \Illuminate\Foundation\Http\Middleware\HandlePrecognitiveRequests::class,
    \Illuminate\Cookie\Middleware\EncryptCookies::class,
    // ... auth, throttle, etc.
];
```

## 踩坑记录

- **中间件中开事务导致死锁**：长事务 + 中间件顺序不当 = 数据库连接耗尽
- **Sentry 监控埋点内存泄漏**：中间件内累积数据未释放
- **CORS 顺序问题**：CORS 中间件必须在最前面，否则预检请求被 auth 拦截
- **Request 生命周期内操作数据库**：中间件的 `terminate` 阶段连接可能已释放

## 常见问题

**Q: 中间件的 handle 和 terminate 有什么区别？**
A: `handle` 在请求前执行（前置）；`terminate` 在响应发送后执行（后置），适合做日志、清理。

**Q: 中间件和事件监听器有什么区别？**
A: 中间件拦截请求流，可短路返回；事件监听器是异步解耦，不影响请求流程。
