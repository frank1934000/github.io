# 队列与事件系统

## 定义
Laravel 的队列和事件系统是异步处理和业务解耦的核心基础设施。博客中深度覆盖了 Queue/Horizon、Events/Listeners、Jobs、Scheduler、EventSauce 事件溯源等实践。

## 核心原理

### Queue（队列）
- **驱动**：Redis、SQS、Database、Beanstalkd、sync
- **Job 类**：实现 ShouldQueue 接口即自动异步
- **延迟队列**：delay() 延迟执行
- **批量任务**：Bus::batch() 并行执行一组 Job
- **失败重试**：retryUntil、tries、backoff 配置

### Horizon（队列仪表盘）
- Redis 队列的可视化管理
- 多队列优先级配置
- 指标采集（吞吐量、运行时间、失败率）
- 自动恢复与告警
- 生产环境运维实践

### Events & Listeners（事件监听）
- 事件分发：event(new OrderCreated($order))
- 监听器：同步/异步执行
- 事件订阅者（Subscriber）
- 事件驱动解耦订单/库存/通知

### EventSauce（事件溯源）
- 存储事件序列而非当前状态
- 聚合根重建当前状态
- 快照机制优化重建性能
- 读模型投影（Projection）
- 订单状态机实战

### Scheduler（定时任务）
- 调度频率：everyMinute/hourly/daily/cron
- onOneServer 限制单实例执行
- 多实例部署下的重入保护
- Kubernetes CronJob 取舍

### Notifications（通知）
- 多通道：Mail、SMS、Slack、企业微信、FCM
- 统一通知抽象
- 降级策略（主通道失败切换备用）
- 数据库通知与广播通知

## 常见问题
- **Queue Driver 选哪个？** Redis 性能最佳，Database 适合小规模，SQS 适合 AWS 环境
- **事件和队列的关系？** 事件是同步的（默认），监听器可以实现 ShouldQueue 变成异步
- **onOneServer 为什么在 K8s 中失效？** K8s 多 Pod 共享文件锁不可靠，推荐用 Redis 原子锁或 K8s CronJob
