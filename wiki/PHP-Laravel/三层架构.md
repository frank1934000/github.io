# Controller-Service-Repository 三层架构

## 定义

将业务逻辑分层：Controller 处理 HTTP，Service 处理业务规则，Repository 处理数据访问。

```
HTTP Request
    ↓
Controller（薄）── 接收请求、验证参数、调用 Service、返回响应
    ↓
Service（厚）── 业务逻辑、事务管理、事件触发
    ↓
Repository（薄）── 数据查询、ORM 操作、缓存
    ↓
Database / Cache
```

## 代码示例

```php
// Controller - 只做 HTTP 相关
class OrderController {
    public function __construct(private OrderService $orderService) {}

    public function store(StoreOrderRequest $request): JsonResponse {
        $order = $this->orderService->create($request->validated());
        return response()->json($order, 201);
    }
}

// Service - 业务逻辑
class OrderService {
    public function __construct(
        private OrderRepo $orderRepo,
        private EventDispatcher $events,
    ) {}

    public function create(array $data): Order {
        return DB::transaction(function () use ($data) {
            $order = $this->orderRepo->create($data);
            $this->events->dispatch(new OrderPlaced($order));
            return $order;
        });
    }
}

// Repository - 数据访问
class EloquentOrderRepo implements OrderRepo {
    public function create(array $data): Order {
        return Order::create($data);
    }
    public function find(int $id): ?Order {
        return Order::find($id);
    }
}
```

## 原则

- **Controller 要薄**：不写业务逻辑，只做参数校验和响应格式化
- **Service 要厚**：所有业务规则集中在此，可独立测试
- **Repository 要薄**：只做数据访问封装，不放业务逻辑
- **面向接口**：Repository 定义接口，Service 依赖接口而非实现

## 踩坑记录

- **Service 臃肿**：一个 Service 500+ 行 → 拆分为多个专职 Service
- **Repository 越权**：Repository 里写业务逻辑 → 违反分层原则
- **事务边界不清**：Controller 开事务、Service 又开事务 → 嵌套死锁

## 常见问题

**Q: 三层架构和 DDD 有什么区别？**
A: 三层是按技术分层（HTTP/业务/数据），DDD 是按领域分层（应用/领域/基础设施）。三层适合中小项目，DDD 适合复杂业务。

**Q: 什么时候不需要 Repository 层？**
A: 简单 CRUD 项目，直接在 Service 中用 Eloquent 即可。过度抽象反而增加复杂度。
