# Laravel 事件驱动架构

## 定义

事件驱动 = 发布者不关心谁在监听。Service 只负责 `dispatch` 一个事件，后续逻辑由各自的 Listener 独立处理。

```
┌─────────────┐     dispatch      ┌──────────────────┐
│  Service /  │ ──────────────▶   │   Event Object    │
│  Controller │                   │  (OrderPlaced)    │
└─────────────┘                   └────────┬─────────┘
                                           │
                          ┌────────────────┼────────────────┐
                          ▼                ▼                ▼
                   ┌──────────┐    ┌──────────┐    ┌──────────┐
                   │ Listener │    │ Listener │    │ Listener │
                   │ 减库存    │    │ 发通知    │    │ 记日志    │
                   └──────────┘    └──────────┘    └──────────┘
```

## 创建事件与监听器

```bash
php artisan make:event OrderPlaced
php artisan make:listener SendOrderNotification --event=OrderPlaced
```

```php
// 事件类
class OrderPlaced {
    use Dispatchable, SerializesModels;
    public function __construct(public Order $order) {}
}

// 监听器
class SendOrderNotification {
    public function handle(OrderPlaced $event): void {
        // 发送通知
    }
}

// 注册映射（或用 EventServiceProvider $listen）
protected $listen = [
    OrderPlaced::class => [
        SendOrderNotification::class,
        DeductInventory::class,
        RecordAuditLog::class,
    ],
];

// 触发
event(new OrderPlaced($order));
// 或
OrderPlaced::dispatch($order);
```

## 队列化监听器

```php
class SendOrderNotification implements ShouldQueue {
    public int $tries = 3;
    public int $backoff = 60;

    public function handle(OrderPlaced $event): void {
        // 异步执行
    }
}
```

## 踩坑记录

- **事件监听器执行顺序**：默认按注册顺序，但队列化后顺序不保证
- **事件中传递大对象**：`SerializesModels` 会序列化整个模型，注意内存
- **事件风暴**：一个事件触发 N 个监听器，某个失败不影响其他（默认）
- **事务边界**：事件在事务提交前触发，监听器可能读到未提交数据

## 常见问题

**Q: 事件和观察者有什么区别？**
A: 事件是全局的，任何地方可监听；观察者只针对单个模型的生命周期事件。

**Q: 如何保证事件不丢失？**
A: 队列化监听器 + 失败重试 + 死信队列。关键事件用 Inbox/Outbox 模式。
