# Laravel 服务容器

## 定义

Laravel 服务容器是 IoC（Inversion of Control）的依赖注入容器，是整个框架的核心。它管理对象的创建、生命周期和依赖解析。

## 核心 API

```php
// 绑定（每次 make 都创建新实例）
app()->bind(OrderRepo::class, MysqlOrderRepo::class);

// 单例（全局共享一个实例）
app()->singleton(OrderRepo::class, MysqlOrderRepo::class);

// 实例注册（直接传入已有实例）
app()->instance('config', $config);

// 获取实例
$repo = app(OrderRepo::class);
$repo = app()->make(OrderRepo::class);

// 扩展已绑定的类
app()->extend(OrderRepo::class, function ($repo, $app) {
    return new CachedOrderRepo($repo);
});
```

## 自动解析（Auto-Resolution）

Laravel 利用反射分析构造函数类型提示，自动注入：

```php
class OrderController {
    public function __construct(
        private OrderService $orderService,
        private PaymentService $paymentService,
    ) {}
}
// 容器自动创建 OrderService 和 PaymentService 并注入
```

## 上下文绑定

同一个接口在不同场景下注入不同实现：

```php
$this->app->when(PhotoController::class)
    ->needs(Filesystem::class)
    ->give(LocalFilesystem::class);

$this->app->when(VideoController::class)
    ->needs(Filesystem::class)
    ->give(S3Filesystem::class);
```

## 服务提供者（Service Provider）

```php
class OrderServiceProvider extends ServiceProvider {
    public function register(): void {
        $this->app->singleton(OrderService::class, function ($app) {
            return new OrderService(
                $app->make(OrderRepo::class),
                $app->make(EventDispatcher::class),
            );
        });
    }
}
```

## 踩坑记录

### 坑 1：循环依赖
A 依赖 B，B 依赖 A → 容器无法实例化。解决方案：引入中间服务或用 `LazyProxy`。

### 坑 2：singleton 内存泄漏
singleton 对象持有大量数据不会释放。CLI 长任务中要注意。

### 坑 3：make() 和 new 的区别
`new` 绕过容器，不会触发事件、不会注入依赖。永远用 `app()->make()`。

## 常见问题

**Q: bind 和 singleton 的区别？**
A: `bind` 每次 `make` 都创建新实例；`singleton` 只创建一次，后续复用。

**Q: 为什么用 `app()->make()` 而不是 `new`？**
A: `make()` 会自动解析依赖、触发事件、应用装饰器。`new` 绕过这些机制。
