# 认证方案进阶

> OAuth 2.1 迁移、Magic Link 无密码登录、Device Authorization Flow——Laravel 认证体系的进阶实践。

## 定义

现代应用的认证方案已从简单的用户名/密码演进到多种无密码、设备无关、安全增强的认证方式。本文覆盖 OAuth 2.1 规范演进、无密码登录、设备授权流程等进阶认证方案。

## 核心原理

### OAuth 2.0 → 2.1 演进

| 变更点 | OAuth 2.0 | OAuth 2.1 |
|--------|-----------|-----------|
| PKCE | 可选（公开客户端） | **强制**（所有客户端） |
| 隐式流（Implicit） | 支持 | **废弃** |
| Resource Owner Password | 支持 | **废弃** |
| Refresh Token 轮换 | 可选 | **强制** |
| Redirect URI 精确匹配 | 推荐 | **强制** |

### Magic Link（无密码登录）

```
用户输入邮箱 → 发送一次性链接邮件 → 用户点击链接 → 自动登录
```

**安全考量**：
- Token 一次性使用 + 短过期时间（10-15分钟）
- 防止邮箱轰炸：频率限制、IP 限制
- Token 使用后立即失效

### Device Authorization Flow

适用于无法使用浏览器的设备（智能电视、CLI 工具、IoT 设备）：

```
1. 设备请求 device code → 返回 user_code + verification_url
2. 设备显示 user_code，提示用户在浏览器中打开 URL
3. 用户在浏览器中输入 user_code 并授权
4. 设备轮询 token endpoint → 授权成功后获取 access_token
```

### 双 Token 轮换

```
Access Token (短期: 15min)  ← 用于 API 请求
Refresh Token (长期: 7d)    ← 用于刷新 Access Token
```

**Refresh Token 轮换**：每次使用 Refresh Token 时，同时返回新的 Refresh Token，旧的立即失效。防止 Token 被盗用。

### Magic Link 实现

来自博客：[Magic Link 无密码登录实战：邮件/短信一次性链接](/2026/06/01/magic-link-passwordless-login-laravel/)

```php
class MagicLinkController extends Controller
{
    public function send(Request $request)
    {
        $request->validate(['email' => 'required|email']);

        // 频率限制
        RateLimiter::attempt("magic-link:{$request->email}", 3, function () {
            // 发送邮件
        }, 3600);

        $token = Str::random(64);
        MagicLink::create([
            'email' => $request->email,
            'token' => hash('sha256', $token),
            'expires_at' => now()->addMinutes(10),
        ]);

        Mail::to($request->email)->send(new MagicLinkMail($token));
    }

    public function verify(Request $request)
    {
        $link = MagicLink::where('token', hash('sha256', $request->token))
            ->where('expires_at', '>', now())
            ->where('used', false)
            ->firstOrFail();

        $link->update(['used' => true]);
        Auth::login(User::firstOrCreate(['email' => $link->email]));

        return redirect('/dashboard');
    }
}
```

### Device Authorization Flow

来自博客：[Device Authorization Flow 实战：智能电视/CLI/IoT 设备的 OAuth 无浏览器授权](/2026/06/01/device-authorization-flow-laravel-passport/)

```php
// CLI 工具
$deviceCode = Http::post('/oauth/device/authorization', [
    'client_id' => 'cli-tool',
    'scope' => 'read write',
]);

echo "请在浏览器中打开: {$deviceCode['verification_uri']}";
echo "并输入代码: {$deviceCode['user_code']}";

// 轮询等待授权
while (true) {
    sleep($deviceCode['interval']);
    $token = Http::post('/oauth/token', [
        'grant_type' => 'urn:ietf:params:oauth:grant-type:device_code',
        'device_code' => $deviceCode['device_code'],
        'client_id' => 'cli-tool',
    ]);

    if ($token->successful()) {
        echo "授权成功！";
        break;
    }
}
```

## 常见问题

**Q: OAuth 2.1 是正式标准吗？**
A: OAuth 2.1 是对 OAuth 2.0 的整合和安全增强，将多个 RFC 的最佳实践合并为一个文档。虽然仍在草案阶段，但主要变更（PKCE 强制、隐式流废弃）已被业界广泛采纳。

**Q: Magic Link 安全吗？**
A: 安全性取决于实现。关键措施：Token 一次性使用、短过期时间、频率限制、HTTPS 传输、邮箱所有权验证。风险：邮箱被入侵则账户沦陷。

**Q: 什么时候用 Device Authorization Flow？**
A: 适用于用户无法直接在当前设备上完成浏览器授权的场景：智能电视、CLI 工具、IoT 设备、游戏主机。
