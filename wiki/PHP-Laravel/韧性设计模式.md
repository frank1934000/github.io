# 韧性设计模式

> Circuit Breaker 熔断器、重试退避策略、Outbox 模式——构建容错的 Laravel 微服务应用。

## 定义

韧性设计（Resilience Design）是一系列保护应用在部分组件故障时仍能正常运行或优雅降级的模式。核心目标是**快速失败、优雅降级、自动恢复**。

## 核心原理

### Circuit Breaker（熔断器）

三状态模型：

```
Closed（正常）→ 失败次数超阈值 → Open（熔断）
Open（熔断）→ 超时后 → Half-Open（半开）
Half-Open → 试探成功 → Closed
Half-Open → 试探失败 → Open
```

**核心参数**：
- 失败阈值（failure threshold）：连续失败 N 次触发熔断
- 熔断持续时间（timeout）：Open 状态持续时间
- 半开请求数（half-open max requests）：半开状态允许的试探请求数

### 重试与退避策略

| 策略 | 公式 | 适用场景 |
|------|------|----------|
| 固定间隔 | `delay` | 轻量级重试 |
| 指数退避 | `delay * 2^attempt` | 网络请求、外部 API |
| 指数退避 + 抖动 | `delay * 2^attempt + random(0, jitter)` | 高并发场景（防止惊群效应） |

### Outbox 模式

保证数据库操作与消息发送的最终一致性：

```
1. 在同一事务中：写业务数据 + 写 Outbox 表
2. 后台进程：轮询 Outbox 表，发送消息到 MQ
3. 发送成功后：标记 Outbox 记录为已处理
```

**解决了什么问题**：
- 数据库操作成功但消息发送失败 → 数据不一致
- 消息发送成功但数据库操作失败 → 数据不一致

## 实战案例

### PHP 手写熔断器

来自博客：[Circuit Breaker 深度实战：PHP 手写熔断器 vs Laravel HTTP Client 的 resilience 模式](/2026/06/01/Circuit-Breaker-深度实战-PHP-手写熔断器-vs-Laravel-HTTP-Client-resilience-模式/)

```php
class CircuitBreaker
{
    private string $state = 'closed';
    private int $failureCount = 0;
    private int $threshold = 5;
    private int $timeout = 60;
    private ?int $lastFailureTime = null;

    public function call(callable $action): mixed
    {
        if ($this->state === 'open') {
            if (time() - $this->lastFailureTime > $this->timeout) {
                $this->state = 'half-open';
            } else {
                throw new CircuitOpenException('Circuit is open');
            }
        }

        try {
            $result = $action();
            $this->onSuccess();
            return $result;
        } catch (\Throwable $e) {
            $this->onFailure();
            throw $e;
        }
    }

    private function onSuccess(): void
    {
        $this->failureCount = 0;
        $this->state = 'closed';
    }

    private function onFailure(): void
    {
        $this->failureCount++;
        $this->lastFailureTime = time();
        if ($this->failureCount >= $this->threshold) {
            $this->state = 'open';
        }
    }
}
```

### Outbox 模式实现

来自博客：[Outbox Pattern 实战：保证数据库与消息队列的最终一致性](/2026/06/01/Outbox-Pattern-实战-保证数据库与消息队列的最终一致性-Laravel-Debezium/)

```php
DB::transaction(function () use ($order, $event) {
    // 1. 写业务数据
    Order::create($order);

    // 2. 写 Outbox 表
    OutboxMessage::create([
        'aggregate_type' => 'Order',
        'aggregate_id' => $order['id'],
        'event_type' => 'OrderCreated',
        'payload' => json_encode($event),
    ]);
});

// 3. 后台进程轮询发送
OutboxMessage::where('sent', false)->chunk(100, function ($messages) {
    foreach ($messages as $msg) {
        EventBus::publish($msg->event_type, $msg->payload);
        $msg->update(['sent' => true]);
    }
});
```

## 常见问题

**Q: 熔断器和限流有什么区别？**
A: 熔断器保护调用方（当下游故障时快速失败），限流保护服务方（限制请求速率防止过载）。两者互补。

**Q: Outbox 模式和事务消息有什么区别？**
A: Outbox 是应用层实现（轮询+发送），事务消息是 MQ 原生能力（如 RocketMQ 事务消息）。Outbox 更通用但有延迟。

**Q: 重试次数设多少合适？**
A: 一般 3 次。超过 3 次后成功率提升边际递减，且增加下游压力。配合指数退避（1s → 2s → 4s）和抖动。
