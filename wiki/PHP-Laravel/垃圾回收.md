# PHP 垃圾回收机制（GC）

## 定义

PHP GC = **引用计数为主 + 循环引用收集器为辅**。引用计数归零立刻释放，解决不了的循环引用由后台收集器周期清理。

## 引用计数（refcount）

PHP 每个变量底层是 `zval`，包含 `refcount` 和 `is_ref` 字段：

```php
$a = 'hello';   // refcount = 1
$b = $a;        // refcount = 2（写时复制，COW）
unset($a);      // refcount = 1
unset($b);      // refcount = 0 → 立即释放
```

## 循环引用问题

```php
$a = new Node();
$b = new Node();
$a->next = $b;   // b refcount=2
$b->next = $a;   // a refcount=2

unset($a);       // a refcount=1（b 还指着它）
unset($b);       // b refcount=1（a 还指着它）
// 谁也不为 0，但外部已经访问不到 → 内存泄漏
```

PHP 5.3 引入**循环引用收集器**，基于 IBM 论文算法：

1. **Roots Buffer**：每次 refcount 减少但 ≠ 0 的 zval，加入「可疑列表」
2. 列表满（默认 10000 个）触发：
   - **Mark**：从可疑根出发，遍历能到达的所有 zval，refcount 全部 -1
   - **Scan**：再遍历一次，refcount > 0 的还原（外部引用）
   - **Collect**：refcount = 0 的真正释放

## 手动控制

```php
gc_enabled();          // GC 是否开启
gc_enable();           // 开
gc_disable();          // 关
gc_collect_cycles();   // 手动触发收集，返回回收数量
gc_status();           // PHP 7.3+ 看 GC 统计
```

### 何时手动调？

- **CLI 长任务**（队列消费者、daemon）：每 N 个任务调一次
- **内存敏感的循环**：处理大数组后立刻 `unset()` + `gc_collect_cycles()`
- **Swoole / Workerman**：常驻进程必须关注 GC

## 常见问题

**Q: PHP 7+ 的 zval 有什么变化？**
A: 实现有重大改动（zval 嵌入栈、引用单独 zend_reference），但算法语义没变。

**Q: 如何排查内存泄漏？**
A: 用 `memory_get_usage()` 追踪、`xdebug_debug_zval()` 检查 refcount、或用 Blackfire/Tideways 做内存 Profile。
