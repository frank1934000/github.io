# 向量数据库选型

## 定义

向量数据库存储高维向量（embedding），基于**语义相似度**而非精确匹配进行查询。是 RAG（Retrieval-Augmented Generation）架构的关键基础设施，广泛用于 AI 知识库、语义搜索、推荐系统等场景。

## 核心原理

### 工作流程

```
文本 → Embedding 模型 → [0.12, -0.34, 0.56, ..., 0.78] (1536维向量)
                              ↓
                        存储到向量数据库
                              ↓
查询文本 → Embedding → 查询向量 → 相似度搜索 → Top-K 最相似文档
```

### ANN 算法

精确最近邻搜索在高维空间中计算量巨大（O(n*d)），ANN（Approximate Nearest Neighbor）以极小精度损失换取数量级速度提升。

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| **HNSW** | 分层可导航小世界图 | 查询快，精度高 | 内存占用大，构建慢 |
| **IVF** | 倒排文件索引，向量空间聚类 | 内存效率高，支持增量更新 | 精度略低于 HNSW |
| **ScaNN** | Google 开发，各向异性量化 | 速度极快 | 实现复杂 |
| **DiskANN** | 基于磁盘的 ANN | 支持超大数据集 | 查询延迟高于内存方案 |

### 相似度度量

- **余弦相似度**：最常用，适合文本语义搜索，值域 [-1, 1]
- **欧氏距离（L2）**：值域 [0, ∞)，0 表示完全相同
- **内积（点积）**：适合归一化后的向量

## 选型决策树

```
需要向量数据库？
├── 已有 PostgreSQL？
│   ├── 数据量 < 100万 → pgvector（最简单，SQL 混合搜索）
│   └── 数据量 > 100万 → Qdrant / Weaviate（独立扩展）
├── 需要全托管？
│   ├── 预算充足 → Pinecone（零运维）
│   └── 预算有限 → Qdrant Cloud / Weaviate Cloud
├── 需要大规模分布式？
│   └── Milvus（最强扩展性）
└── 快速原型/开发？
    └── ChromaDB（嵌入式，零配置）
```

## Laravel 集成示例

### pgvector（最简单）

```php
// 使用 PostgreSQL 原生驱动，无需额外依赖
DB::table('documents')
    ->whereRaw('embedding <=> ?::vector < 0.5', [$queryVector])
    ->orderByRaw('embedding <=> ?::vector', [$queryVector])
    ->limit(10)
    ->get();
```

### Qdrant REST API

```php
$response = Http::post('http://localhost:6333/collections/docs/points/search', [
    'vector' => $embedding,
    'limit' => 10,
    'with_payload' => true,
]);
```

## 常见问题

**Q: pgvector 性能够用吗？**
A: 100 万条以下的向量搜索，pgvector 配合 HNSW 索引完全够用。优势是可以用 SQL 做混合搜索（向量 + 元数据过滤），无需引入新组件。

**Q: Embedding 模型怎么选？**
A: 文本搜索推荐 `text-embedding-3-small`（1536 维，性价比高）或多语言的 BGE-M3（1024 维，开源）。图像搜索用 CLIP 系列。

**Q: 向量维度越高越好吗？**
A: 不是。维度越高存储和计算成本越大，且存在"维度灾难"。一般 768-1536 维已足够覆盖大多数语义搜索场景。
