# 数据库连接池

## 定义

连接池是应用和数据库之间的**中间层**，负责管理一组持久化数据库连接，供多个请求复用。它避免了每次请求都建立和销毁 TCP 连接的开销，在高并发场景下显著提升性能和稳定性。

## PHP-FPM 连接模型的问题

PHP-FPM 采用**进程模型**，每个 worker 进程独立拥有自己的数据库连接：

```
Worker 1 ──→ Connection 1 ──→ MySQL
Worker 2 ──→ Connection 2 ──→ MySQL
Worker 3 ──→ Connection 3 ──→ MySQL
   ...          ...
Worker N ──→ Connection N ──→ MySQL
```

### 问题场景

- **连接数爆炸**：50 个 FPM worker × 3 台服务器 = 150 个连接，突发流量时容易触发 `Too many connections`
- **连接建立开销**：每次请求创建连接需要 TCP 三次握手 + MySQL 认证，约 2-5ms
- **资源浪费**：大部分连接在空闲状态，但 MySQL 仍需为其维护内存和线程

## 连接池方案对比

| 特性 | ProxySQL | PgBouncer | Supabase Supavisor |
|------|----------|-----------|-------------------|
| **数据库** | MySQL 专用 | PostgreSQL 专用 | PostgreSQL 专用 |
| **架构** | C++ 高性能 | C 轻量级 | Elixir 云原生 |
| **读写分离** | ✅ 内置 | ❌ 需配合其他工具 | ✅ 内置 |
| **查询路由** | ✅ 规则引擎 | ❌ | ✅ |
| **查询缓存** | ✅ 内置 | ❌ | ❌ |
| **连接模式** | Session/Transaction | Session/Transaction/Statement | Session/Transaction |
| **故障转移** | ✅ 自动 | ❌ 手动 | ✅ 自动 |
| **监控** | 丰富的统计表 | 基础 stats | Dashboard |
| **部署复杂度** | 中等 | 低 | 低（Serverless） |

## Session 模式 vs Transaction 模式

### Session 模式（连接绑定）

客户端获取连接后独占直到断开，等价于直连数据库：

- ✅ 支持所有 MySQL/PostgreSQL 功能（Prepared Statement、SET 命令、临时表）
- ❌ 连接利用率低，连接数 = 并发请求数

### Transaction 模式（事务级复用）

仅在事务执行期间占用后端连接，事务完成后立即归还：

- ✅ 连接复用率极高，10 个后端连接可服务 100+ 前端请求
- ❌ **不支持** Prepared Statement（跨事务后 PreparedStatement 丢失）
- ❌ **不支持** `SET` 命令（如 `SET NAMES`、`SET time_zone`）
- ❌ 不支持会话级临时表
- ⚠️ 需要应用层适配，Laravel 默认使用 Prepared Statement 需要关闭

### 选型建议

| 场景 | 推荐模式 |
|------|----------|
| 简单 CRUD 应用 | Transaction 模式 |
| 需要 Prepared Statement | Session 模式 |
| 连接数紧张 | Transaction 模式 |
| 使用 `SET` 命令或临时表 | Session 模式 |

## ProxySQL 核心功能

### 查询规则路由

```sql
-- 将 SELECT 路由到从库
INSERT INTO mysql_query_rules (match_pattern, destination_hostgroup)
VALUES ('^SELECT.*FOR UPDATE$', 0);  -- 写组
INSERT INTO mysql_query_rules (match_pattern, destination_hostgroup)
VALUES ('^SELECT', 1);               -- 读组
```

### 读写分离

```
应用 ──→ ProxySQL ──┬──→ Master (Hostgroup 0)  ← 写操作
                    ├──→ Slave 1 (Hostgroup 1) ← 读操作
                    └──→ Slave 2 (Hostgroup 1) ← 读操作
```

### 连接复用

ProxySQL 维护到后端 MySQL 的连接池，前端 1000 个连接可以复用后端 50 个连接。

### 查询缓存

```sql
-- 缓存特定查询结果 10 秒
INSERT INTO mysql_query_rules (match_pattern, cache_ttl)
VALUES ('^SELECT.*FROM config', 10000);
```

### 故障转移

自动检测后端 MySQL 健康状态，故障节点自动下线，恢复后自动上线。

## Laravel 配置

### 基础配置（`config/database.php`）

```php
'mysql' => [
    'driver' => 'mysql',
    'sticky' => true,  // 本次请求写入后，后续读操作也走主库

    'read' => [
        'host' => [
            '192.168.1.101',  // 从库 1
            '192.168.1.102',  // 从库 2
        ],
        'port' => '3306',
        'database' => 'mydb',
        'username' => 'reader',
        'password' => 'secret',
    ],
    'write' => [
        'host' => [
            '192.168.1.100',  // 主库
        ],
        'port' => '3306',
        'database' => 'mydb',
        'username' => 'writer',
        'password' => 'secret',
    ],
],
```

### 通过 ProxySQL 连接（简化配置）

```php
'mysql' => [
    'driver' => 'mysql',
    'host' => '127.0.0.1',       // ProxySQL 地址
    'port' => '6033',             // ProxySQL 端口
    'database' => 'mydb',
    'username' => 'app_user',
    'password' => 'secret',
    'options' => [
        PDO::ATTR_EMULATE_PREPARES => true,  // Transaction 模式需要
    ],
],
```

## 最佳实践

### 连接数计算公式

```
最大连接数 = CPU 核心数 × 2 + 磁盘数
```

例如：8 核 CPU + 1 块 SSD = `8 × 2 + 1 = 17` 个连接

### 监控指标

```sql
-- 当前连接数
SHOW STATUS LIKE 'Threads_connected';

-- 历史最大连接数
SHOW STATUS LIKE 'Max_used_connections';

-- 连接使用率
SELECT
    @@max_connections AS max_conn,
    VARIABLE_VALUE AS current_conn,
    ROUND(VARIABLE_VALUE / @@max_connections * 100, 2) AS usage_pct
FROM performance_schema.global_status
WHERE VARIABLE_NAME = 'Threads_connected';
```

### 告警阈值

| 指标 | 告警阈值 |
|------|----------|
| 连接使用率 | > 70% |
| 连接等待数 | > 0 |
| 连接创建速率 | 突增 2x |
