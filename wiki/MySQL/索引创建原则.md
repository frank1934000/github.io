# 索引创建原则

## 定义

索引不是越多越好，需要遵循一定的原则来设计。好的索引设计能让查询性能提升百倍，差的索引设计反而拖慢系统。

## 五大原则

### 1. 最左前缀匹配原则

MySQL 会一直向右匹配，直到遇到范围查询（>、<、BETWEEN、LIKE）就停止。

```sql
-- 索引 (a, b, c, d)
WHERE a = 1 AND b = 2 AND c > 3 AND d = 4
-- d 用不到索引！
-- 建立 (a, b, d, c) 则都可以用到
```

### 2. = 和 IN 可以乱序

```sql
WHERE a = 1 AND b = 2 AND c = 3
-- 索引 (a, b, c) 任意顺序都行
-- MySQL 查询优化器会自动优化
```

### 3. 选择区分度高的列

区分度公式：`COUNT(DISTINCT col) / COUNT(*)`

- 区分度越大，扫描的记录数越少
- 唯一键区分度 = 1
- 状态、性别字段区分度接近 0
- 一般 JOIN 字段要求区分度 > 0.1（即平均 1 条扫描 10 条记录）

### 4. 索引列不能参与计算

```sql
-- ❌ 索引失效
WHERE from_unixtime(create_time) = '2014-05-29'

-- ✅ 改写
WHERE create_time = unix_timestamp('2014-05-29')
```

原因：B+树中存的是字段值，检索时需要把所有元素应用函数才能比较，成本太大。

### 5. 尽量扩展索引，不要新建

```sql
-- 表中已有索引 idx_a (a)
-- 需要 (a, b) 的索引
-- ✅ 直接修改原来的索引
ALTER TABLE t DROP INDEX idx_a, ADD INDEX idx_a_b (a, b);
```

## 何时该建索引？

| 场景 | 建议 |
|------|------|
| WHERE 条件高频字段 | ✅ 建索引 |
| JOIN 关联字段 | ✅ 建索引 |
| ORDER BY 排序字段 | ✅ 建索引 |
| GROUP BY 分组字段 | ✅ 建索引 |
| 数据量小的表 | ❌ 不需要 |
| 频繁更新的字段 | ❌ 谨慎，维护成本高 |
| 区分度低的字段 | ❌ 如性别、状态 |
