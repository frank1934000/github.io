# 主键设计

## 定义

主键（Primary Key）是表中唯一标识每一行的列或列组合。主键设计直接影响聚簇索引结构、插入性能、存储空间和分库分表方案。

## 自增 ID vs UUID

| 维度 | 自增 ID | UUID |
|---|---|---|
| 存储空间 | 4 字节（INT）/ 8 字节（BIGINT） | 16 字节（二进制）/ 36 字节（字符串） |
| 插入性能 | **高**（顺序追加） | **低**（随机插入，触发页分裂） |
| 二级索引影响 | 小（主键小 → 二级索引也小） | 大（每个二级索引都存一份主键） |
| 全局唯一性 | ❌ 单库唯一 | ✅ 全局唯一 |
| 安全性 | 可枚举（/users/1, /users/2） | 不可枚举 |
| 分库分表 | 需要分布式 ID 方案 | 天然全局唯一 |

## 为什么推荐自增 ID

InnoDB 使用聚簇索引，数据按主键顺序物理存储：

```
自增 ID：新行总是追加到 B+ 树末尾
  → 顺序 I/O → 无页分裂 → 插入快

UUID：新行随机插入到 B+ 树中间
  → 随机 I/O → 频繁页分裂 → 插入慢 → 碎片多
```

> 一个数据页大小为 16KB，UUID 随机插入会导致频繁的页分裂和数据挪动。

## 分布式 ID 方案

当需要全局唯一且有序的 ID 时：

| 方案 | 原理 | 优点 | 缺点 |
|---|---|---|---|
| 雪花算法（Snowflake） | 时间戳 + 机器 ID + 序列号 | 有序、高性能 | 时钟回拨问题 |
| 号段模式（Leaf） | 数据库自增 + 缓冲区 | 简单可靠 | 依赖数据库 |
| UUID v7 | 时间有序的 UUID | 全局唯一 + 有序 | 较新，生态支持有限 |

```sql
-- 雪花 ID 用 BIGINT 存储
id BIGINT UNSIGNED PRIMARY KEY
```

## 主键设计原则

1. **尽量用自增整数** —— 性能最优、空间最小
2. **不要用业务字段做主键** —— 业务可能变化，主键不应变
3. **避免复合主键** —— 增加二级索引体积，用单列自增 + UNIQUE 约束替代
4. **BIGINT 而非 INT** —— INT 上限 42 亿，大表可能溢出
5. **无符号 UNSIGNED** —— 范围加倍

## 常见问题

**Q: 用 UUID 做主键有什么问题？**
A: 三大问题：① 随机插入导致页分裂，插入性能差 ② 16 字节存储空间大，二级索引膨胀 ③ 无法按时间排序（UUID v7 可解决排序问题）。

**Q: 已经用了 UUID 做主键，怎么优化？**
A: 可以用 UUID v7 替代 v4（有序），或迁移到雪花 ID。迁移需要双写过渡期，注意外键引用的同步更新。

**Q: 自增 ID 用完了怎么办？**
A: INT 上限约 42 亿，BIGINT 上限约 1844 亿亿。如果 INT 快用完，ALTER TABLE 改为 BIGINT（在线 DDL，但大表需要谨慎）。
