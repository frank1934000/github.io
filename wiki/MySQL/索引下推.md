# 索引下推（Index Condition Pushdown / ICP）

## 定义

MySQL 5.6 引入了**索引下推**优化，默认开启。可以在**减少回表次数**，在 InnoDB 中只针对**二级索引**有效。

```sql
-- 关闭索引下推
SET optimizer_switch = 'index_condition_pushdown=off';
```

## 核心原理

### 没有索引下推（MySQL 5.6 之前）

```
索引匹配 name LIKE '张%' → 拿到 4 条记录的主键 → 全部回表 → 在 Server 层过滤 age=10
```

回表了 4 次，但只有部分满足条件。

### 有索引下推（MySQL 5.6+）

```
索引匹配 name LIKE '张%' → 在索引层直接检查 age=10 → 只有满足的才回表
```

索引下推会**在存储引擎层就过滤掉不满足条件的记录**，减少回表次数。

## 实例对比

```sql
-- 联合索引 (name, age)
SELECT * FROM tuser WHERE name LIKE '张%' AND age = 10 AND ismale = 1;
```

| 阶段 | 无 ICP | 有 ICP |
|------|--------|--------|
| 索引匹配 | name LIKE '张%' 命中 4 条 | name LIKE '张%' 命中 4 条 |
| 索引层过滤 | ❌ 直接回表 4 次 | ✅ 在索引层检查 age=10，过滤掉 2 条 |
| 回表次数 | **4 次** | **2 次** |
| Server 层过滤 | age + ismale | 只过滤 ismale |

## EXPLAIN 中的表现

| Extra 字段 | 含义 |
|-----------|------|
| **Using index condition** | ✅ 使用了索引下推 |
| **Using index** | ✅ 覆盖索引（不需要 ICP） |
| **Using where** | 在 Server 层过滤 |

## ICP 的限制

- 只对**二级索引**有效，聚簇索引无效
- 只适用于 **InnoDB** 和 **MyISAM**
- 不支持**覆盖索引**的场景（覆盖索引已经不需要回表了）
