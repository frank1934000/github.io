# B+树数据结构

## 定义

MySQL InnoDB 存储引擎的索引底层采用 **B+树** 数据结构。理解 B+树是理解 MySQL 索引优化的基础。

## 核心原理

### 为什么用 B+树？

> 提升查询速度，首先要减少磁盘 I/O 次数，也就是要降低树的高度。

| 数据结构 | 问题 |
|---------|------|
| **平衡二叉树 / 红黑树** | 属于二叉树，时间复杂度 O(n)，数据量上千万时树的深度很深，MySQL 读取消耗大量 IO。InnoDB 以**页**为单位读取，二叉树每个节点存一个关键词，空间浪费严重 |
| **B-树** | 非叶子节点存储数据，占用较多空间，导致每个节点的**指针**少很多，无形增加了树的深度 |
| **B+树** ✅ | 数据都存储在叶子节点，非叶子节点只存**键值+指针**，索引树更加扁平，**三层深度可支持千万级表存储**。叶子节点之间通过链表关联，范围查找更快 |

### B+树的优势

1. **扁平结构**：三层深度即可支持千万级数据
2. **磁盘友好**：非叶子节点只存键值+指针，单页能容纳更多索引项
3. **范围查询快**：叶子节点通过双向链表连接
4. **查询稳定**：所有数据查询都要走到叶子节点，查询时间稳定

### B+树 vs B-树

```
B-树:              B+树:
    [10|20]            [10|20]          ← 非叶子节点只存键值+指针
   /  |   \           /   |   \
 [5] [15] [25]     [5]  [15]  [25]
                   ↕    ↕    ↕        ← 叶子节点通过链表连接
```

## 三层 B+树能存多少数据？

假设：
- 主键 ID 为 BIGINT（8 字节）
- 指针大小 6 字节
- InnoDB 页大小 16KB

每页能存的索引项：`16384 / (8+6) ≈ 1170` 个

- 第一层（根节点）：1170 个指针
- 第二层：1170 × 1170 = 1,368,900 个指针
- 第三层（叶子节点存数据）：假设每行 1KB，每页约 16 行

**总数据量 ≈ 1170 × 1170 × 16 ≈ 2190 万行**
