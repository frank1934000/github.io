# 分库分表

## 定义

> **并发量决定是否需要分库，数据量决定是否需要分表。**

MySQL 表的数据量一般控制在**千万级别**，超过就需要考虑分库分表。

## 何时需要？

| 问题 | 解决方案 |
|------|---------|
| 单库并发过高 | **分库** |
| 单表数据量过大 | **分表** |
| 两者都有 | **分库 + 分表** |

## 海量数据优化手段

按复杂度递增：

1. **缓存加速** - Redis 缓存热点数据
2. **读写分离** - 主库写，从库读
3. **垂直拆分** - 按业务拆分表
4. **分库分表** - 水平拆分
5. **冷热数据分离** - 历史数据归档
6. **ES 助力** - 复杂搜索交给 Elasticsearch
7. **NoSQL** - MongoDB 等文档数据库
8. **NewSQL** - TiDB 等分布式数据库

## 分片策略

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| **时间范围分区** | 按时间归档 | 日志、订单历史 |
| **用户 ID 取模** | `user_id % N` | 用户维度查询 |
| **Sharding Key** | 按业务键分片 | 灵活路由 |

## 分表后全局 ID 方案

分库分表后，多张表共用一套全局 ID，原来单表自增不满足要求。

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID** | 简单、全局唯一 | 无序、索引不友好、占空间 |
| **数据库自增 ID** | 有序 | 依赖数据库 |
| **号段模式** | 高性能 | 需要额外维护 |
| **Redis INCR** | 原子自增、有序 | 依赖 Redis |
| **雪花算法** | 高性能、有序、全局唯一 | 时钟回拨问题 |
| **开源框架** | 百度 uid-generator、美团 Leaf、滴滴 Tinyid | 引入外部依赖 |

## 分表后的问题

### 跨分片查询

以电商为例：
- 以 `buyer_id` 路由 → 无法满足卖家查询
- 以 `seller_id` 路由 → 无法满足买家查询

**解决方案**：
1. **买家库 + 卖家库**：买家库做写库（完整数据），异构同步到卖家库（简化字段）
2. **多线程扫描**：分段查找后聚合结果
3. **ES 异构**：存到 ES 支持多维度搜索

### 跨分片 JOIN

- 应用层组装：拆成多次查询后在内存中组装
- 冗余数据：常用字段冗余到各分片
- 全局表：小表广播到所有分片

### 跨分片排序分页

- 各分片查出 N 条后在应用层排序
- 深度分页性能差，建议使用游标分页
