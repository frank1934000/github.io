# 数据库内定时任务（pg_cron + pg_partman）

## 定义

数据库内定时任务是指利用数据库原生扩展（PostgreSQL 的 pg_cron）在数据库进程内部执行定时 SQL 调度，以及利用 pg_partman 实现分区表的自动创建、维护和过期清理。相比传统的应用层 Scheduler（如 Laravel `schedule:run`），数据库内调度消除了 PHP 进程启动、网络连接、Composer 加载等中间环节，提升了可靠性和执行效率。

## 核心原理

### 传统 Scheduler 的痛点

```
操作系统 Crontab
    └── php artisan schedule:run
            └── Kernel.php 中的 Schedule
                    └── Command / Job
                            └── DB::statement() 执行 SQL
```

这条链路的四个根本问题：

| 问题 | 说明 |
|------|------|
| 链路过长、故障点多 | PHP 启动开销 0.8-2s，Composer/OPcache/FPM 任一出错则静默失败 |
| 数据库操作与应用层耦合 | 清理日志/刷新统计等纯数据库操作绕道 PHP 层毫无必要 |
| 分区手动维护噩梦 | 多脚本管理同一张表的分区，互不知晓，偶尔冲突 |
| 多实例重复执行 | K8s 多 Pod 部署需额外分布式锁，缓存故障可能导致重复执行 |

### pg_cron 架构

pg_cron 是 Citus Data（Microsoft 收购）开发的 PostgreSQL 扩展，作为 Background Worker 在数据库进程内运行：

```
┌──────────────────────────────────────────────────────┐
│                   PostgreSQL Server                    │
│                                                       │
│  ┌──────────────┐     ┌────────────────────────┐     │
│  │  pg_cron     │     │  cron.job (元数据表)     │     │
│  │  Background   │────▶│  - schedule (cron 表达式)│     │
│  │  Worker       │     │  - command (SQL 语句)    │     │
│  │  每分钟唤醒    │     │  - active (是否启用)     │     │
│  └──────────────┘     └────────────────────────┘     │
│         │                                             │
│         ▼                                             │
│  ┌──────────────────────────────────────────────────┐│
│  │  cron.job_run_details (执行历史)                    ││
│  │  - status / return_message / start_time           ││
│  └──────────────────────────────────────────────────┘│
└──────────────────────────────────────────────────────┘
```

核心特点：
- **进程内运行**：Background Worker 共享数据库进程空间，无 IPC 开销
- **标准 cron 语法**：`* * * * *` 表达式，运维零学习成本
- **直接执行 SQL**：支持 SELECT/INSERT/UPDATE/DELETE/CALL/PL/pgSQL
- **跨数据库调度**：PostgreSQL 13+ 支持在 A 库调度 B 库的 SQL
- **持久化存储**：任务定义在 `cron.job` 系统表中，数据库重启自动恢复

### pg_partman 自动分区管理

pg_partman 是 PostgreSQL 分区管理扩展，自动处理分区的创建、维护和过期清理：

```sql
-- 创建基于时间范围的分区父表
CREATE TABLE orders (
    id BIGSERIAL,
    user_id BIGINT,
    amount DECIMAL(10,2),
    created_at TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (created_at);

-- 注册 pg_partman 管理
SELECT partman.create_parent(
    p_parent_table := 'public.orders',
    p_control := 'created_at',
    p_type := 'native',
    p_interval := '1 month',
    p_premake := 3  -- 提前创建 3 个未来分区
);

-- 配置自动保留策略：保留 12 个月，过期自动 detach
UPDATE partman.part_config
SET retention = '12 months',
    retention_keep_table = false,  -- 过期分区直接 DROP
    infinite_time_partitions = true
WHERE parent_table = 'public.orders';
```

### pg_cron + pg_partman 联合自动化

```sql
-- pg_cron 每天凌晨 2 点运行 pg_partman 维护
SELECT cron.schedule(
    'partman-maintenance',
    '0 2 * * *',
    $$CALL partman.run_maintenance()$$
);

-- pg_cron 每小时清理过期分区
SELECT cron.schedule(
    'partman-retention',
    '0 * * * *',
    $$SELECT partman.partition_data_proc('public.orders')$$
);
```

## 四大核心应用场景

| 场景 | 说明 | pg_cron 调度示例 |
|------|------|-----------------|
| 分批清理过期数据 | 删除 N 天前的日志/会话数据 | `DELETE FROM audit_log WHERE created_at < NOW() - INTERVAL '90 days' LIMIT 10000` |
| 统计预计算 | 提前计算报表统计结果 | `REFRESH MATERIALIZED VIEW CONCURRENTLY daily_sales_summary` |
| 物化视图刷新 | 定期刷新缓存的查询结果 | 同上 |
| 数据完整性检查 | 检测孤儿记录/数据异常 | `SELECT * FROM orders WHERE user_id NOT IN (SELECT id FROM users)` |

## 常见问题

**Q: pg_cron 任务失败会告警吗？**
A: pg_cron 本身不会主动告警，但执行结果记录在 `cron.job_run_details` 中（`status = 'failed'`）。需配合 `pg_cron.max_jobs_to_log` 和外部监控查询实现告警。

**Q: MySQL 有类似方案吗？**
A: MySQL 原生没有内置调度器。可通过 MySQL Event Scheduler 实现部分场景，但功能远不如 pg_cron 丰富。MySQL 推荐使用 MySQL Shell + AdminAPI 或外部 Orchestrator。

**Q: pg_cron 和 Laravel Scheduler 应该怎么选？**
A: 纯数据库操作（清理、刷新、统计）→ pg_cron。需要调用外部 API/队列/文件系统 → 保持 Laravel Scheduler。两者可共存。

**Q: pg_partman 支持二级分区（sub-partition）吗？**
A: 支持。可以先按 Range 分区，再在每个分区内按 Hash 子分区，适合超大表（数十亿行）的均匀分布。
