# 慢查询监控实战：MySQL Performance Schema + pg_stat_statements

## 定义

慢查询监控是数据库性能优化的第一步——只有知道哪些查询慢、慢在哪里，才能有针对性地优化。MySQL 和 PostgreSQL 各有原生的慢查询监控机制。

## MySQL Performance Schema

### 核心概念

Performance Schema 是 MySQL 5.5+ 内置的性能监控框架，收集服务器事件信息（SQL 执行、锁等待、IO、内存等）。

### 启用与配置

```sql
-- 检查是否启用
SHOW VARIABLES LIKE 'performance_schema';  -- ON

-- 启用语句统计
UPDATE performance_schema.setup_consumers
SET ENABLED = 'YES'
WHERE NAME = 'events_statements_summary_by_digest';

-- 配置慢查询阈值
SET GLOBAL long_query_time = 1;  -- 超过 1 秒的查询记录
SET GLOBAL slow_query_log = ON;
SET GLOBAL log_queries_not_using_indexes = ON;
```

### 核心查询

```sql
-- Top 10 最慢 SQL（按平均耗时排序）
SELECT
    DIGEST_TEXT AS sql_text,
    COUNT_STAR AS exec_count,
    ROUND(AVG_TIMER_WAIT / 1e12, 3) AS avg_time_sec,
    ROUND(MAX_TIMER_WAIT / 1e12, 3) AS max_time_sec,
    SUM_ROWS_EXAMINED AS rows_examined,
    SUM_ROWS_SENT AS rows_sent
FROM performance_schema.events_statements_summary_by_digest
ORDER BY AVG_TIMER_WAIT DESC
LIMIT 10;

-- 未使用索引的查询
SELECT
    DIGEST_TEXT AS sql_text,
    COUNT_STAR AS exec_count,
    SUM_NO_INDEX_USED AS no_index_count
FROM performance_schema.events_statements_summary_by_digest
WHERE SUM_NO_INDEX_USED > 0
ORDER BY SUM_NO_INDEX_USED DESC;

-- 锁等待分析
SELECT
    THREAD_ID,
    EVENT_NAME,
    TIMER_WAIT / 1e12 AS wait_sec,
    OBJECT_SCHEMA,
    OBJECT_NAME
FROM performance_schema.events_waits_current
WHERE EVENT_NAME LIKE 'wait/synch/mutex/innodb%';
```

### 等待事件分析

```sql
-- Top 等待事件
SELECT
    EVENT_NAME,
    COUNT_STAR,
    ROUND(SUM_TIMER_WAIT / 1e12, 3) AS total_wait_sec,
    ROUND(AVG_TIMER_WAIT / 1e12, 3) AS avg_wait_sec
FROM performance_schema.events_waits_summary_global_by_event_name
WHERE COUNT_STAR > 0
ORDER BY SUM_TIMER_WAIT DESC
LIMIT 20;
```

## pt-query-digest

Percona Toolkit 的慢查询分析工具，比 Performance Schema 更易用：

```bash
# 分析慢查询日志
pt-query-digest /var/lib/mysql/slow.log

# 输出示例：
# Rank  Query ID           Response time  Calls  R/Call  V/M
# ==== ================== ============== ====== ======= =====
#    1  0xABC123...        150.0000  35.2%   5000  0.0300  0.01
#    2  0xDEF456...         80.0000  18.8%   2000  0.0400  0.02

# 实时分析（采样 10 秒）
pt-query-digest --processlist h=127.0.0.1 --interval 10

# 过滤特定数据库
pt-query-digest --filter '$event->{db} eq "myapp"' slow.log
```

## PostgreSQL pg_stat_statements

### 启用

```sql
-- postgresql.conf
shared_preload_libraries = 'pg_stat_statements'
pg_stat_statements.max = 10000
pg_stat_statements.track = all

-- 创建扩展
CREATE EXTENSION pg_stat_statements;
```

### 核心查询

```sql
-- Top 10 最耗时 SQL
SELECT
    query,
    calls,
    ROUND(total_exec_time::numeric, 2) AS total_ms,
    ROUND(mean_exec_time::numeric, 2) AS avg_ms,
    ROUND((100 * total_exec_time / SUM(total_exec_time) OVER ())::numeric, 2) AS pct,
    rows
FROM pg_stat_statements
ORDER BY total_exec_time DESC
LIMIT 10;

-- 读取行数最多（可能缺索引）
SELECT
    query,
    calls,
    shared_blks_read,
    shared_blks_hit,
    ROUND((shared_blks_hit::numeric / NULLIF(shared_blks_hit + shared_blks_read, 0)) * 100, 2) AS cache_hit_pct
FROM pg_stat_statements
ORDER BY shared_blks_read DESC
LIMIT 10;

-- 重置统计
SELECT pg_stat_statements_reset();
```

## EXPLAIN 深度分析

### MySQL EXPLAIN

```sql
EXPLAIN SELECT * FROM orders WHERE user_id = 123 AND status = 'paid';
```

| 字段 | 含义 | 关注点 |
|------|------|--------|
| `type` | 访问类型 | ALL(全表扫描) → index → range → ref → eq_ref → const |
| `key` | 实际使用的索引 | NULL 表示未使用索引 |
| `rows` | 预估扫描行数 | 越小越好 |
| `Extra` | 额外信息 | Using filesort / Using temporary 需优化 |

### EXPLAIN ANALYZE（MySQL 8.0.18+）

```sql
EXPLAIN ANALYZE
SELECT * FROM orders WHERE user_id = 123 AND status = 'paid';
-- 输出实际执行时间，不仅是预估
```

## 慢查询治理流程

```
1. 发现慢查询（Performance Schema / pg_stat_statements / 慢查询日志）
   ↓
2. EXPLAIN 分析执行计划
   ↓
3. 定位根因：
   ├── 缺索引 → 创建索引
   ├── 索引失效 → 检查函数/隐式转换/LIKE
   ├── 扫描行数过多 → 优化 WHERE 条件
   ├── filesort/temporary → 优化 ORDER BY / GROUP BY
   └── 锁等待 → 检查事务粒度
   ↓
4. 验证优化效果（EXPLAIN ANALYZE）
   ↓
5. 监控回归（定期审查 Top SQL）
```

## 常见问题

**Q: Performance Schema 影响性能吗？**
A: 有约 5-10% 的性能开销。生产环境建议启用，但只开启必要的 consumers 和 instruments。

**Q: pg_stat_statements 数据会丢失吗？**
A: 重启 PostgreSQL 后统计数据重置。建议定期导出到监控系统（Prometheus/Grafana）。

**Q: 慢查询阈值设多少合适？**
A: OLTP 系统一般 200ms-1s；OLAP 系统可以放宽到 5-10s。根据业务 SLA 调整。
