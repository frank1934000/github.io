# 不可见索引（Invisible Index）

## 定义

不可见索引（Invisible Index）是 MySQL 8.0 引入的索引管理特性。将索引标记为对查询优化器不可见后，优化器在生成执行计划时会完全忽略该索引，但 InnoDB 存储引擎仍会维护其 B+ 树结构（INSERT/UPDATE/DELETE 仍同步更新）。这一机制提供了**零风险的线上索引验证方案**：确认无影响后再安全删除，发现问题则瞬间恢复。

## 核心原理

### 数据字典中的索引可见性

InnoDB 为每个索引维护 `is_visible` 属性（存储在 `mysql.innodb_index_stats` 等内部表中）：

```sql
-- 查看索引可见性
SELECT
    NAME AS index_name,
    TYPE AS index_type,
    IS_VISIBLE
FROM information_schema.INNODB_INDEXES
WHERE TABLE_ID = (
    SELECT TABLE_ID
    FROM information_schema.INNODB_TABLES
    WHERE NAME = 'your_db/your_table'
);
```

### 关键特性

| 特性 | 说明 |
|------|------|
| 优化器不可见 | 查询编译阶段，InnoDB 过滤掉 `is_visible=NO` 的索引 |
| 物理数据保持 | INSERT/UPDATE/DELETE 仍维护 B+ 树，索引数据始终最新 |
| 操作即时且无锁 | 仅修改元数据，毫秒级完成，不阻塞 DML |

### 语法

```sql
-- 设为不可见
ALTER TABLE orders ALTER INDEX idx_user_id INVISIBLE;

-- 恢复可见
ALTER TABLE orders ALTER INDEX idx_user_id VISIBLE;

-- 查看所有索引的可见性
SHOW INDEX FROM orders;  -- Visible 列显示 YES/NO
```

## 实战案例：三步走索引安全删除流程

来自博客文章：[MySQL Invisible Index 实战：线上索引安全验证](/2026/06/06/MySQL-Invisible-Index-实战-线上索引安全验证-EXPLAIN-实际执行计划索引生效分析/)

### 第一步：标记不可见

```sql
-- 怀疑 idx_status_created 冗余，设为不可见
ALTER TABLE orders ALTER INDEX idx_status_created INVISIBLE;
```

### 第二步：观察期（建议 24-72 小时）

```sql
-- EXPLAIN 验证：确认该索引不再出现在执行计划中
EXPLAIN SELECT * FROM orders WHERE status = 'pending' AND created_at > '2026-01-01';

-- EXPLAIN ANALYZE（MySQL 8.0.18+）：真实执行并返回实际统计信息
EXPLAIN ANALYZE SELECT * FROM orders WHERE status = 'pending' AND created_at > '2026-01-01';

-- Performance Schema 慢查询监控
SELECT DIGEST_TEXT, COUNT_STAR, AVG_TIMER_WAIT/1000000 AS avg_ms
FROM performance_schema.events_statements_summary_by_digest
WHERE SCHEMA_NAME = 'your_db'
  AND AVG_TIMER_WAIT > 1000000000  -- 平均超过 1 秒
ORDER BY AVG_TIMER_WAIT DESC;
```

### 第三步：确认安全后删除

```sql
-- 确认无影响后物理删除
ALTER TABLE orders DROP INDEX idx_status_created;

-- 大表建议配合 pt-online-schema-change
pt-online-schema-change --alter "DROP INDEX idx_status_created" \
  D=db,t=orders --execute
```

## EXPLAIN vs EXPLAIN ANALYZE

| 维度 | EXPLAIN | EXPLAIN ANALYZE (8.0.18+) |
|------|---------|--------------------------|
| 是否真实执行 | **否**（仅优化器估算） | **是**（真实执行） |
| 数据来源 | 统计信息 + 优化器模型 | 实际执行统计 |
| rows 估算 | 可能偏差很大 | 返回实际行数 |
| 适用场景 | 快速查看计划 | 验证索引实际生效 |

**关键区别**：在索引可见性验证中，EXPLAIN 可能因统计信息缓存仍显示使用了不可见索引的计划（极端情况），而 EXPLAIN ANALYZE 是真实执行，结果绝对准确。

## 常见问题

**Q: 设为不可见后，索引还占存储空间吗？**
A: 是的。InnoDB 仍维护 B+ 树结构，INSERT/UPDATE/DELETE 仍有维护开销。只有 `DROP INDEX` 才能释放空间和消除维护开销。

**Q: 主键索引可以设为不可见吗？**
A: 不可以。聚簇索引（主键）不支持 INVISIBLE，它是 InnoDB 数据存储的物理基础。

**Q: 为什么建议观察 24-72 小时？**
A: 某些查询可能是月度报表、周末批处理等低频但关键的场景。短时间观察无法覆盖完整业务周期。

**Q: 不可见索引对 `FORCE INDEX` 有效吗？**
A: `FORCE INDEX` 指定不可见索引会报错 `Key 'idx_name' doesn't exist in table`。这是预期行为——不可见索引对所有查询路径均不可见。
