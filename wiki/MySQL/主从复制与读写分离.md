# 主从复制与读写分离

## 定义

**主从复制**是将主库（Master）的数据自动复制到从库（Slave）的机制。**读写分离**是在此基础上，将写操作发到主库、读操作发到从库的架构模式。

## 主从复制原理

### 四步流程

```
Master                                    Slave
  │                                         │
  │  ① 数据更新，写入 binlog                 │
  │ ──────── log dump 线程 ────────────────→ │
  │         ② 通知 Slave 有数据更新           │
  │ ←────── 请求指定 binlog 副本 ─────────── │
  │                                         │
  │         ③ 存到本地 Relay Log（中继日志）    │
  │                                         │
  │         ④ SQL 线程读取 Relay Log          │
  │            执行 redo 操作，保持同步        │
```

### 三个关键线程

| 线程 | 位置 | 作用 |
|------|------|------|
| **Log Dump 线程** | Master | 通知 Slave 有数据更新 |
| **IO 线程** | Slave | 请求并接收 binlog，存入 Relay Log |
| **SQL 线程** | Slave | 读取 Relay Log，在本地执行重放 |

## 主从延迟

### 什么是主从延迟？

写入 SQL 在主库执行完后，将数据完整同步到从库会有一个时间差。

- 主库生成 binlog，记录时间戳 **t1**
- 从库开始执行 binlog，取当前时间 **t2**
- **延迟 = t2 - t1**

### 排查方法

```sql
SHOW SLAVE STATUS;
-- 关注 Seconds_Behind_Master 参数
-- 为零：主从复制良好
-- 正值：主从延迟，数字越大延迟越严重
```

### 延迟解决方案

| 方案 | 说明 |
|------|------|
| **强制走主库** | 不能接受延迟的查询直接走主库 |
| **引入缓存** | 更新主库后同步写入缓存 |
| **提升从库配置** | 提高 binlog 同步效率 |
| **缩短网络距离** | 减少 binlog 网络传输时间 |
| **一主多从 + canal** | 用 canal 增量订阅组件缓解主库压力 |
| **减少大事务** | 大事务分批执行 |
| **多线程复制** | 5.7+ 设置 `slave_parallel_workers > 0` |
| **浮动 IP 切换** | 延迟超阈值时切换到 Master |

## 读写分离架构

```
                    ┌─────────┐
                    │  应用层   │
                    └────┬────┘
                         │
                    ┌────┴────┐
                    │ 中间件   │  ← Laravel 中间件 / ProxySQL / MySQL Router
                    └────┬────┘
                   ┌─────┴─────┐
                   │           │
              ┌────┴───┐  ┌────┴───┐
              │ Master │  │ Slave  │
              │ (写)   │→│ (读)   │
              └────────┘  └────────┘
```

### 实现方式

1. **应用层路由**：代码中指定读写连接
2. **中间件代理**：ProxySQL / MySQL Router / ShardingSphere
3. **Laravel 中间件**：通过中间件自动路由读写请求
