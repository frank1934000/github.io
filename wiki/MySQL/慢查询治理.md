# 慢查询治理

## 定义

慢查询治理是系统化地发现、分析和修复 MySQL 慢 SQL 的方法论。90% 的接口慢问题，根因都指向 MySQL 慢查询。

## 第一步：开启慢查询日志

### my.cnf 配置

```ini
[mysqld]
slow_query_log = 1
slow_query_log_file = /var/log/mysql/slow.log
long_query_time = 1          # 超过 1 秒记录
log_queries_not_using_indexes = 1  # 记录未使用索引的查询
```

### 运行时开启

```sql
SET GLOBAL slow_query_log = 'ON';
SET GLOBAL long_query_time = 1;
SET GLOBAL log_queries_not_using_indexes = 'ON';
```

## 第二步：使用 pt-query-digest 分析

```bash
# 分析慢查询日志，按响应时间排序
pt-query-digest /var/log/mysql/slow.log > slow_report.txt

# 只看最近 1 小时
pt-query-digest --since '1h' /var/log/mysql/slow.log

# 只看某个用户的查询
pt-query-digest --filter '($event->{user} || "") =~ m/^app_user/' slow.log
```

### 报告关键指标

| 指标 | 说明 |
|------|------|
| **Response time** | 总响应时间占比 |
| **Calls** | 执行次数 |
| **R/Call** | 每次平均响应时间 |
| **Rows examine** | 扫描行数 |
| **Rows sent** | 返回行数 |

**重点关注**：Rows examine >> Rows sent 的查询（扫描了大量行但只返回少量）

## 第三步：修复方案

### 方案 1：索引优化

```sql
-- 添加缺失的索引
ALTER TABLE orders ADD INDEX idx_user_status (user_id, status);

-- 使用覆盖索引避免回表
ALTER TABLE orders ADD INDEX idx_cover (user_id, status, amount, created_at);
```

### 方案 2：SQL 重写

```sql
-- ❌ 子查询 → ✅ JOIN
SELECT * FROM orders WHERE user_id IN (SELECT id FROM users WHERE status = 1);
SELECT o.* FROM orders o JOIN users u ON o.user_id = u.id WHERE u.status = 1;

-- ❌ SELECT * → ✅ 只选需要的字段
SELECT * FROM orders WHERE user_id = 100;
SELECT id, amount, status FROM orders WHERE user_id = 100;

-- ❌ 深度分页 → ✅ 游标分页
SELECT * FROM orders ORDER BY id DESC LIMIT 100000, 20;
SELECT * FROM orders WHERE id < 上次最大id ORDER BY id DESC LIMIT 20;
```

### 方案 3：架构优化

- 读写分离：将查询分流到从库
- 缓存加速：热点数据缓存到 Redis
- 分库分表：大数据量表水平拆分
- 冷热分离：历史数据归档
