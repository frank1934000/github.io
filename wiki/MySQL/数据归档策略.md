# 数据归档策略

## 定义

数据归档是将**历史数据**从活跃表迁移到归档表或归档库的过程，目标是保持活跃表轻量，从而提升查询性能、降低存储成本、简化备份恢复。

## 冷热分离策略

### 按时间分区

将数据按时间划分为"热数据"（频繁访问）和"冷数据"（偶尔访问）：

```
活跃表 (orders)          归档表 (orders_archive)
┌─────────────────┐      ┌─────────────────┐
│ 2025-01 ~ 至今   │      │ 2024-01 ~ 2024-12│
│ 热数据，高频查询  │  ←→  │ 冷数据，偶尔查询  │
└─────────────────┘      └─────────────────┘
```

### 保留周期

| 数据类型 | 热数据保留 | 归档策略 |
|----------|-----------|----------|
| 订单数据 | 近 6 个月 | 按月归档 |
| 日志数据 | 近 3 个月 | 按周归档 |
| 用户行为 | 近 1 个月 | 按天归档 |
| 财务数据 | 近 2 年 | 按年归档，永不删除 |

## 归档方式

### 方式一：分区表 (PARTITION BY RANGE)

```sql
-- 按月自动分区
CREATE TABLE orders (
    id BIGINT UNSIGNED AUTO_INCREMENT,
    order_no VARCHAR(32),
    amount DECIMAL(10,2),
    created_at DATETIME,
    PRIMARY KEY (id, created_at)
) PARTITION BY RANGE (TO_DAYS(created_at)) (
    PARTITION p202501 VALUES LESS THAN (TO_DAYS('2025-02-01')),
    PARTITION p202502 VALUES LESS THAN (TO_DAYS('2025-03-01')),
    PARTITION p202503 VALUES LESS THAN (TO_DAYS('2025-04-01')),
    PARTITION p_future  VALUES LESS THAN MAXVALUE
);

-- 删除旧分区（比 DELETE 快得多，不写 binlog）
ALTER TABLE orders DROP PARTITION p202401;

-- 归档到另一张表
ALTER TABLE orders REORGANIZE PARTITION p202401 INTO (
    PARTITION p_archive VALUES LESS THAN (TO_DAYS('2024-02-01'))
);
```

**优点**：分区删除是 DDL 操作，速度快、不写 undo log
**限制**：唯一索引必须包含分区键

### 方式二：pt-archive 工具

Percona Toolkit 提供的在线归档工具：

```bash
# 将 6 个月前的订单归档到 archive 表
pt-archive \
    --source h=master,D=shop,t=orders \
    --dest h=archive,D=shop_archive,t=orders_archive \
    --where "created_at < DATE_SUB(NOW(), INTERVAL 6 MONTH)" \
    --limit 1000 \
    --commit-each \
    --progress 5000 \
    --statistics
```

**优点**：低负载、可控速率、自动分批、可中断恢复
**注意**：需要在业务低峰期执行

### 方式三：Laravel 队列异步归档

```php
// app/Jobs/ArchiveOrdersJob.php
class ArchiveOrdersJob implements ShouldQueue
{
    use Dispatchable, InteractsWithQueue, Queueable, SerializesModels;

    public function handle(): void
    {
        $cutoff = now()->subMonths(6);

        // 分批处理，每批 1000 条
        Order::where('created_at', '<', $cutoff)
            ->orderBy('id')
            ->chunkById(1000, function ($orders) {
                DB::transaction(function () use ($orders) {
                    // 插入归档表
                    OrderArchive::insert($orders->toArray());
                    // 删除原记录
                    Order::whereIn('id', $orders->pluck('id'))->delete();
                });
            });
    }
}

// 每月 1 日凌晨执行
$schedule->job(new ArchiveOrdersJob)->monthlyOn(1, '03:00');
```

## 查询兼容

归档后业务仍需查询历史数据，需要保证透明性：

### 方案一：UNION ALL 视图

```sql
CREATE VIEW orders_all AS
SELECT id, order_no, amount, created_at FROM orders
UNION ALL
SELECT id, order_no, amount, created_at FROM orders_archive;
```

### 方案二：应用层路由

```php
class OrderRepository
{
    public function find(string $orderNo): ?Order
    {
        // 先查活跃表
        $order = Order::where('order_no', $orderNo)->first();
        if ($order) return $order;

        // 回退查归档表
        return OrderArchive::where('order_no', $orderNo)->first();
    }

    public function search(Carbon $from, Carbon $to): Collection
    {
        if ($from->isBefore(now()->subMonths(6))) {
            // 查询范围涉及归档数据
            return DB::query()
                ->from('orders_all')
                ->whereBetween('created_at', [$from, $to])
                ->get();
        }

        return Order::whereBetween('created_at', [$from, $to])->get();
    }
}
```

### 方案三：Laravel Union 查询

```php
$recent = Order::where('created_at', '>=', $cutoff)
    ->select('id', 'order_no', 'amount', 'created_at');

$archived = OrderArchive::where('created_at', '<', $cutoff)
    ->select('id', 'order_no', 'amount', 'created_at');

$results = $recent->unionAll($archived)->orderBy('created_at')->paginate();
```

## 注意事项

### 外键约束

归档前必须处理外键依赖：

```sql
-- 方案 1：临时禁用外键检查
SET FOREIGN_KEY_CHECKS = 0;
-- 执行归档操作
SET FOREIGN_KEY_CHECKS = 1;

-- 方案 2：先归档子表，再归档父表
-- orders → order_items → payments
```

### 触发器

归档操作会触发 `BEFORE DELETE` / `AFTER DELETE` 触发器，可能导致意外行为。建议归档前临时禁用触发器或使用 `pt-archive` 的 `--no-delete` 选项先复制再手动清理。

### 归档期间的锁

大批量 DELETE 会产生行锁，影响正常写入：

- 使用 `LIMIT` 分批删除（每批 1000~5000 条）
- 每批之间 `SLEEP(0.1)` 让出锁
- 在业务低峰期执行

### Binlog 影响

大量归档操作会产生大量 binlog，影响主从复制延迟：

```sql
-- 归档操作使用 session 级别跳过 binlog
SET SESSION sql_log_bin = 0;
-- 执行归档
SET SESSION sql_log_bin = 1;
```

⚠️ 跳过 binlog 意味着从库不同步归档操作，需要在从库也执行相同操作。
