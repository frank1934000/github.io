# MySQL 乐观锁 vs 悲观锁实战

## 定义

并发场景下对共享资源的保护有两种策略：

| 策略 | 核心思想 | 实现方式 | 冲突处理 |
|------|---------|---------|---------|
| **悲观锁** | 假设一定会冲突，先锁再操作 | `SELECT ... FOR UPDATE` | 阻塞等待 |
| **乐观锁** | 假设很少冲突，提交时检测 | 版本号/时间戳 | 重试或报错 |

## 悲观锁（Pessimistic Lock）

### SELECT FOR UPDATE

```sql
BEGIN;
SELECT * FROM orders WHERE id = 1001 FOR UPDATE;  -- 加排他锁
UPDATE orders SET status = 'paid' WHERE id = 1001;
COMMIT;  -- 释放锁
```

- 其他事务对同一行的 `SELECT FOR UPDATE` 会**阻塞等待**
- 适合**写多读少、冲突频繁**的场景
- 行锁基于索引：如果 WHERE 条件没有索引，会退化为**表锁**

### 锁的粒度

```sql
-- 行锁（有索引时）
SELECT * FROM orders WHERE id = 1001 FOR UPDATE;

-- 间隙锁（范围查询）
SELECT * FROM orders WHERE amount > 1000 FOR UPDATE;
-- 锁住 amount > 1000 的所有间隙，防止插入

-- 共享锁（读锁）
SELECT * FROM orders WHERE id = 1001 LOCK IN SHARE MODE;
```

### 悲观锁在 Laravel 中的应用

```php
DB::transaction(function () use ($orderId) {
    $order = Order::where('id', $orderId)->lockForUpdate()->first();
    $order->update(['status' => 'paid', 'amount' => $order->amount - $discount]);
});
```

## 乐观锁（Optimistic Lock）

### 版本号机制

```sql
-- 读取时获取版本号
SELECT id, amount, version FROM orders WHERE id = 1001;
-- version = 5

-- 更新时检查版本号
UPDATE orders SET amount = 900, version = version + 1
WHERE id = 1001 AND version = 5;
-- affected_rows = 1 → 成功
-- affected_rows = 0 → 被其他事务修改，需重试
```

### 时间戳机制

```sql
SELECT id, amount, updated_at FROM orders WHERE id = 1001;
-- updated_at = '2026-06-06 10:00:00'

UPDATE orders SET amount = 900
WHERE id = 1001 AND updated_at = '2026-06-06 10:00:00';
```

### 乐观锁在 Laravel 中的应用

```php
// 方式一：手动版本号
$order = Order::find($orderId);
$affected = Order::where('id', $orderId)
    ->where('version', $order->version)
    ->update([
        'status' => 'paid',
        'version' => $order->version + 1,
    ]);

if ($affected === 0) {
    // 版本冲突，重试或报错
    throw new ConcurrentModificationException('订单被其他操作修改');
}

// 方式二：Laravel 自带 updated_at 检查
$order = Order::find($orderId);
$order->update(['status' => 'paid']);
// Eloquent 自动检查 updated_at 是否匹配
```

## 选型决策

| 维度 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 冲突频率 | 高（>10%）选悲观锁 | 低（<10%）选乐观锁 |
| 响应时间 | 阻塞等待，延迟较高 | 立即返回，延迟低 |
| 实现复杂度 | 简单（数据库原生支持） | 需要重试逻辑 |
| 死锁风险 | 有（需要合理设计） | 无 |
| 适用场景 | 支付扣款、库存扣减 | 商品浏览、信息更新 |
| 数据库压力 | 持锁期间占用连接 | 短事务，连接快速释放 |

## 混合策略

高并发场景下的常见实践：

```
读取阶段（乐观）→ 预检查 → 写入阶段（悲观）→ 确认
```

例如秒杀场景：
1. **Redis 预扣减**（乐观，快速过滤）
2. **数据库 SELECT FOR UPDATE**（悲观，最终一致性）
3. **更新库存 + 创建订单**

## 常见问题

**Q: SELECT FOR UPDATE 没命中索引会怎样？**
A: 退化为表锁，所有行都被锁住，严重影响并发性能。务必确保 WHERE 条件有索引。

**Q: 乐观锁冲突率高时怎么办？**
A: 增加重试次数（指数退避），或切换为悲观锁。冲突率 > 10% 时悲观锁通常更优。

**Q: 如何避免死锁？**
A: 固定访问顺序、减小事务粒度、使用低隔离级别（RC 比 RR 锁范围小）、确保索引覆盖。
