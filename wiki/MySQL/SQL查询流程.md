# SQL 查询流程

## 定义

MySQL 执行一条 SQL 查询时的完整处理链路，从客户端发送到返回结果，涉及 Service 层（连接器、分析器、优化器、执行器）和存储引擎层。

## 执行流程

```
客户端 → 连接器 → 查询缓存(8.0已移除) → 分析器 → 优化器 → 执行器 → 存储引擎 → 返回结果
```

### 1. 连接器

- 验证用户名密码
- 验证权限
- 获取连接（长连接 vs 短连接）

### 2. 查询缓存（MySQL 8.0 已移除）

- MySQL 5.7 及之前：SELECT 语句命中缓存直接返回
- 8.0 起彻底移除，因为表更新会清空整个表的缓存，命中率极低

### 3. 分析器

- **词法分析**：识别关键字、表名、列名
- **语法分析**：检查 SQL 语法是否正确
- **预处理**：检查表和列是否存在，解析别名

### 4. 优化器

- 决定使用哪个索引
- 决定表的连接顺序
- 选择全表扫描还是索引扫描
- 等价改写（如子查询 → JOIN）

### 5. 执行器

- 检查表权限
- 根据优化器的执行计划调用存储引擎接口
- 返回结果集

## 两层架构

```
┌─────────────────────────────────┐
│          Service 层              │
│  连接器 → 分析器 → 优化器 → 执行器 │
├─────────────────────────────────┤
│         存储引擎层               │
│  InnoDB / MyISAM / Memory ...   │
│  （以插件形式可替换）              │
└─────────────────────────────────┘
```

## 常见问题

**Q: 为什么我的 SQL 走了全表扫描而没用索引？**
A: 优化器判断全表扫描比索引扫描更快（如返回行数超过表的 30%）。用 EXPLAIN 查看 type 字段确认。

**Q: 长连接和短连接怎么选？**
A: PHP-FPM 默认短连接（每个请求一个连接）。高并发场景建议用连接池（如 ProxySQL）替代长连接，避免连接风暴。
