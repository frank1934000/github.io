# 分布式工作流引擎

## 定义

分布式工作流引擎是一种在分布式系统中编排、执行和管理长时间运行的业务流程的中间件基础设施。与传统的任务队列不同，工作流引擎提供**持久化状态管理**、**故障自动恢复**、**补偿回滚**和**复杂编排**等能力，是解决分布式长事务问题的核心架构组件。

核心概念包括：

- **Temporal.io**：由 Uber 开源的持久化工作流引擎，基于事件溯源和确定性重放
- **Saga 模式**：分布式事务的补偿模式，分为编排式（Orchestration）和协同式（Choreography）
- **工作流编排（Orchestration）**：由中心编排器协调多个服务的执行顺序
- **事件协同（Choreography）**：各服务通过事件自主协调，无中心控制点
- **Durable Execution**：持久化执行，即使进程崩溃也能从断点恢复

在 Laravel 生态中，虽然队列（Queue）系统能处理简单的异步任务，但对于复杂的多步骤业务流程、需要补偿回滚的分布式事务，以及长时间运行的工作流，专用的工作流引擎提供了更可靠的解决方案。

## 核心原理

### 1. Temporal.io 持久化工作流引擎

Temporal 是由 Uber 开源的分布式工作流引擎（前身是 Cadence），在 Uber、Netflix、Snap 等公司大规模生产验证。

**核心架构：**
```
┌─────────────────────────────────────────────────────────┐
│                   Temporal 架构                          │
│                                                         │
│  ┌──────────────┐    ┌──────────────┐    ┌───────────┐  │
│  │  Workflow     │    │  Temporal    │    │  Worker   │  │
│  │  Client       │───>│  Server      │<──>│  Process  │  │
│  │  (发起工作流) │    │  (编排引擎)  │    │  (执行逻辑)│  │
│  └──────────────┘    └──────┬───────┘    └───────────┘  │
│                             │                           │
│                      ┌──────┴───────┐                   │
│                      │  持久化存储   │                   │
│                      │  (Cassandra/ │                   │
│                      │   MySQL/PG)  │                   │
│                      └──────────────┘                   │
└─────────────────────────────────────────────────────────┘
```

**核心组件：**

1. **Workflow（工作流）**：定义业务流程的逻辑，必须是**确定性**的
2. **Activity（活动）**：执行具体的副作用操作（API 调用、数据库操作等）
3. **Worker（工作者）**：运行工作流和活动代码的进程
4. **Task Queue（任务队列）**：工作流和活动任务的调度队列
5. **Workflow Execution（工作流执行）**：一次工作流实例的运行

**确定性重放（Deterministic Replay）：**

Temporal 的核心创新在于确定性重放机制：
```
第 1 次执行:
  Step 1: 调用 Activity A → 返回结果 X → 记录到事件历史
  Step 2: 调用 Activity B → 返回结果 Y → 记录到事件历史
  Step 3: Worker 崩溃 ❌

恢复执行（重放）:
  Step 1: 从事件历史读取 Activity A 的结果 X（不实际调用）
  Step 2: 从事件历史读取 Activity B 的结果 Y（不实际调用）
  Step 3: 继续执行 Step 3... ✅
```

**为什么工作流必须是确定性的？**

如果工作流中包含随机数、当前时间、UUID 等非确定性操作，在重放时会产生不同的执行路径。Temporal 提供了专用 API 解决这些问题：

```python
# ❌ 错误：非确定性
import random
result = random.randint(1, 100)

# ✅ 正确：使用 Temporal 提供的确定性随机
result = workflow.random().randint(1, 100)

# ❌ 错误：非确定性
now = datetime.now()

# ✅ 正确：使用 Temporal 提供的时间
now = workflow.now()

# ❌ 错误：非确定性
uid = uuid.uuid4()

# ✅ 正确：使用 Temporal 提供的 UUID
uid = workflow.uuid4()
```

### 2. Temporal 工作流定义

**Python SDK 示例：**
```python
from temporalio import workflow, activity
from temporalio.common import RetryPolicy
from datetime import timedelta

# 定义 Activity
@activity.defn
async def charge_payment(order_id: str, amount: float) -> str:
    """调用支付网关扣款"""
    payment_id = await payment_gateway.charge(order_id, amount)
    return payment_id

@activity.defn
async def reserve_inventory(order_id: str, items: list) -> str:
    """预留库存"""
    reservation_id = await inventory_service.reserve(order_id, items)
    return reservation_id

@activity.defn
async def send_notification(user_id: str, message: str) -> None:
    """发送通知"""
    await notification_service.send(user_id, message)

@activity.defn
async def refund_payment(payment_id: str) -> None:
    """退款（补偿操作）"""
    await payment_gateway.refund(payment_id)

@activity.defn
async def release_inventory(reservation_id: str) -> None:
    """释放库存（补偿操作）"""
    await inventory_service.release(reservation_id)

# 定义 Workflow
@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order: Order) -> str:
        payment_id = None
        reservation_id = None

        try:
            # Step 1: 预留库存
            reservation_id = await workflow.execute_activity(
                reserve_inventory,
                args=[order.id, order.items],
                start_to_close_timeout=timedelta(seconds=30),
                retry_policy=RetryPolicy(
                    maximum_attempts=3,
                    initial_interval=timedelta(seconds=1),
                ),
            )

            # Step 2: 扣款
            payment_id = await workflow.execute_activity(
                charge_payment,
                args=[order.id, order.total],
                start_to_close_timeout=timedelta(seconds=30),
                retry_policy=RetryPolicy(maximum_attempts=3),
            )

            # Step 3: 等待外部审批（如果金额 > 阈值）
            if order.total > 10000:
                await workflow.wait_condition(
                    lambda: self.approved or self.rejected
                )
                if self.rejected:
                    raise ApplicationRejected("金额超限，审批被拒")

            # Step 4: 确认订单
            await workflow.execute_activity(
                send_notification,
                args=[order.user_id, f"订单 {order.id} 已确认"],
                start_to_close_timeout=timedelta(seconds=10),
            )

            return f"Order {order.id} completed"

        except Exception as e:
            # 补偿逻辑（Saga 回滚）
            if payment_id:
                await workflow.execute_activity(
                    refund_payment,
                    args=[payment_id],
                    start_to_close_timeout=timedelta(seconds=30),
                )
            if reservation_id:
                await workflow.execute_activity(
                    release_inventory,
                    args=[reservation_id],
                    start_to_close_timeout=timedelta(seconds=30),
                )
            raise
```

### 3. Saga 模式

Saga 模式是管理分布式长事务的经典模式，将大事务拆分为一系列本地事务，每个本地事务有对应的补偿操作。

**Saga 的两种实现方式：**

#### 3.1 编排式 Saga（Orchestration）

由中心编排器（Orchestrator）协调所有参与者的执行顺序：

```
┌──────────┐
│ 编排器    │
│ (Order    │
│ Workflow) │
└────┬─────┘
     │
     ├──> [库存服务] 预留库存 ──成功──> 继续
     │         │
     │         └──失败──> 终止并通知
     │
     ├──> [支付服务] 扣款 ──成功──> 继续
     │         │
     │         └──失败──> [库存服务] 释放库存（补偿）
     │
     ├──> [物流服务] 创建运单 ──成功──> 完成
     │         │
     │         └──失败──> [支付服务] 退款 + [库存服务] 释放库存（补偿）
```

**优点：**
- 集中控制，流程清晰
- 易于理解和调试
- 补偿逻辑集中管理
- Temporal.io 天然支持

**缺点：**
- 编排器是单点（但 Temporal Server 是高可用的）
- 编排器可能成为性能瓶颈

#### 3.2 协同式 Saga（Choreography）

各服务通过事件自主协调，无中心控制点：

```
[订单服务] ──OrderCreated──> [事件总线]
                                │
                    ┌───────────┼───────────┐
                    ▼           ▼           ▼
              [库存服务]   [支付服务]   [通知服务]
              预留库存     扣款        发送确认
              │           │           │
              └──InventoryReserved──> [事件总线]
                                      │
                                  [支付服务] 扣款
                                      │
                                      └──PaymentCompleted──> [事件总线]
                                                              │
                                                          [物流服务] 创建运单
```

**优点：**
- 无单点瓶颈
- 服务间松耦合
- 更容易水平扩展

**缺点：**
- 流程分散，难以追踪
- 补偿逻辑分散在各服务中
- 调试困难
- 循环依赖风险

**编排式 vs 协同式选型：**

| 维度 | 编排式 (Orchestration) | 协同式 (Choreography) |
|------|----------------------|---------------------|
| 控制方式 | 中心编排器 | 分布式事件驱动 |
| 耦合度 | 编排器依赖各服务 | 事件解耦 |
| 可追踪性 | 高（集中日志） | 低（需分布式追踪） |
| 补偿逻辑 | 集中在编排器 | 分散在各服务 |
| 复杂度 | 步骤多时集中管理方便 | 步骤少时更简洁 |
| 适用场景 | 3+ 步骤的复杂流程 | 2-3 步骤的简单事件链 |
| 推荐工具 | Temporal.io | 事件总线 + 各服务自处理 |

### 4. Laravel 队列 vs Temporal.io

Laravel 的队列系统和 Temporal.io 解决的问题层次不同：

| 维度 | Laravel Queues | Temporal.io |
|------|---------------|-------------|
| 任务模型 | 单个独立任务 | 多步骤工作流编排 |
| 状态管理 | 无状态（任务执行后丢弃） | 有状态（持久化事件历史） |
| 故障恢复 | 重试 + 死信队列 | 确定性重放（精确恢复到断点） |
| 补偿机制 | 需手动实现 | 原生支持 Saga 补偿 |
| 长时间运行 | 不适合（超时限制） | 天然支持（小时/天/周） |
| 人工审批 | 需轮询实现 | 原生信号（Signal）和查询（Query） |
| 子工作流 | 不支持 | 原生支持 |
| 版本管理 | 无 | 工作流版本化 |
| 可观测性 | Horizon Dashboard | Temporal Web UI |
| 部署复杂度 | 低（Redis + Worker） | 高（Temporal Server + DB） |
| 适用场景 | 异步邮件/通知/图像处理 | 订单流程/审批链/数据管道 |

**Laravel 队列的局限示例：**
```php
// 用 Laravel 队列实现订单流程 — 不推荐
class CreateOrderJob implements ShouldQueue
{
    public function handle()
    {
        // Step 1: 预留库存
        $reservation = $this->reserveInventory();

        // Step 2: 扣款 — 如果这里失败了怎么办？
        // 库存已经预留，需要手动实现补偿逻辑
        $payment = $this->chargePayment();

        // Step 3: 创建运单 — 如果这里失败了怎么办？
        // 需要退款 + 释放库存
        $this->createShipment();

        // 问题：
        // 1. 任务失败后重试会从头开始，不是从断点恢复
        // 2. 补偿逻辑需要自己写，容易遗漏
        // 3. 无法支持长时间等待（如人工审批）
        // 4. 无法追踪中间状态
    }
}
```

**Temporal 解决方案：**
```python
@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order: Order) -> str:
        # Step 1: 预留库存
        reservation = await workflow.execute_activity(
            reserve_inventory, order.items,
            start_to_close_timeout=timedelta(seconds=30),
        )

        # Step 2: 扣款
        payment = await workflow.execute_activity(
            charge_payment, order.total,
            start_to_close_timeout=timedelta(seconds=30),
        )

        # Step 3: 等待人工审批（支持长时间等待）
        if order.total > 10000:
            approved = await workflow.wait_condition_with_timeout(
                timedelta(days=7),
                lambda: self.approval_received,
            )
            if not approved:
                # 自动补偿
                await self.compensate(payment, reservation)
                return "Order rejected: approval timeout"

        # 完成
        return "Order completed"
```

### 5. 在 Laravel 中集成 Temporal

**架构方案：**
```
[Laravel 应用]
    │
    ├── 发起工作流请求（HTTP/gRPC）
    │
    ▼
[Temporal Server]
    │
    ├── Task Queue: order-processing
    ├── Task Queue: payment-workflow
    │
    ▼
[Worker 进程（Python/Go/PHP SDK）]
    │
    ├── 执行工作流逻辑
    ├── 调用 Activity（API 调用等）
    │
    ▼
[外部服务：支付/库存/物流/通知]
```

**Laravel 作为工作流发起方：**

class TemporalService

    public function __construct()
    {
        $this->client = \Temporal\Client\WorkflowClient::create(
            env('TEMPORAL_ADDRESS', 'localhost:7233')
        );
    }

    public function startOrderWorkflow(Order $order): string
    {
        $workflow = $this->client->newWorkflowStub(
            OrderWorkflowInterface::class,
            \Temporal\Client\WorkflowOptions::new()
                ->withTaskQueue('order-processing')
                ->withWorkflowId('order-' . $order->id)
                ->withWorkflowExecutionTimeout(CarbonInterval::hours(24))
        );

        $run = $this->client->start($workflow, $order->toArray());

        return $run->getExecution()->getID();
    }

    public function signalApproval(string $workflowId, bool $approved): void
    {
        $workflow = $this->client->newWorkflowStub(
            OrderWorkflowInterface::class,
            \Temporal\Client\WorkflowOptions::new()
                ->withWorkflowId($workflowId)
        );

        $workflow->approve($approved);
    }
}

use Workflow\ActivityStub;

class OrderWorkflow extends Workflow
{
    public function execute(Order $order)
    {

        $reservation = yield ActivityStub::make(
            ReserveInventory::class, $order->items
        );

        // Step 2: 扣款
        $payment = yield ActivityStub::make(
            ChargePayment::class, $order->total

        return 'completed';
    }
}

### 7. 工作流设计模式

#### 7.1 补偿模式（Compensating Transaction）
```
Step 1 → Step 2 → Step 3（失败）
                   ↓
        Compensate(Step 2) → Compensate(Step 1)
```

#### 7.2 扇出/扇入模式（Fan-Out/Fan-In）
```
          ┌→ Task A →┐
Start →   ├→ Task B → ├ → Aggregate → End
          └→ Task C →┘
```

#### 7.3 人工审批模式（Human-in-the-Loop）
```
Start → Process → WaitForApproval → Decision
                      ↑ (Signal)       ├── Approved → Continue
                      │                └── Rejected → Compensate
                 [人工审批 UI]
```

#### 7.4 子工作流模式（Child Workflow）
```
Parent Workflow
    ├── Child: PaymentWorkflow
    ├── Child: ShippingWorkflow
    └── Child: NotificationWorkflow
```

#### 7.5 重试与超时模式
```
Activity
    ├── start_to_close_timeout: 单次执行超时
    ├── schedule_to_start_timeout: 排队超时
    ├── schedule_to_close_timeout: 总超时
    ├── heartbeat_timeout: 心跳超时（长任务）
    └── retry_policy: 重试策略（次数、间隔、退避）
```

### 8. 事件溯源与 CQRS 在工作流中的应用

Temporal.io 的内部实现本质上就是事件溯源：
- 每个工作流执行的每一步都记录为事件
- 确定性重放就是从事件历史重建状态
- 事件历史天然支持审计和合规

与 [CQRS模式](CQRS模式.md) 的结合：
- Command 端：发起工作流（写操作）
- Query 端：查询工作流状态和历史（读操作）

### 案例一：Temporal.io 替代 Laravel 长事务编排

**场景描述：** 电商订单流程涉及库存预留、支付扣款、物流下单、通知发送等多个服务的协调，传统 Laravel 队列方案存在补偿逻辑复杂、断点恢复困难等问题。

**方案：** 使用 Temporal.io 编排订单 Saga，实现可靠的分布式事务。

**博客文章参考：** [Temporal.io 实战：持久化工作流引擎-Laravel中的长事务编排与Saga模式的工程化替代方案](/2026/06/01/Temporal-io-实战-持久化工作流引擎-Laravel中的长事务编排与Saga模式的工程化替代方案/)

**架构对比：**
```
[传统方案]
Laravel Job → 调用服务A → 调用服务B → 调用服务C
    │ (失败时)
    └→ 手动补偿逻辑（容易遗漏、难以测试）

[Temporal方案]

    │ (失败时)
    └→ 自动回放 + 确定性补偿（可靠、可测试、可审计）
```

## 常见问题

### Q1: Temporal.io 和 Laravel Queues 应该怎么选？

**使用 Laravel Queues 的场景：**
- 简单的异步任务（发邮件、生成报告、图片处理）
- 不需要跨服务协调
- 不需要补偿回滚逻辑
- 任务执行时间短（秒级）

**使用 Temporal.io 的场景：**
- 多步骤业务流程（订单、审批、注册流程）
- 需要 Saga 补偿回滚
- 需要人工审批等待（小时/天级）
- 需要可靠的状态管理和断点恢复
- 需要复杂编排逻辑（条件分支、并行、子工作流）

### Q2: Temporal 的确定性约束对业务逻辑有什么影响？

**影响：** 工作流代码中不能直接使用随机数、当前时间、UUID、文件 I/O 等非确定性操作。

**解决方案：**
- 时间：使用 `workflow.now()` 代替 `time.now()`
- 随机数：使用 `workflow.random()` 代替 `random.random()`
- UUID：使用 `workflow.uuid4()` 代替 `uuid.uuid4()`
- I/O 操作：放到 Activity 中执行

**原则：** 工作流代码只包含业务决策逻辑，所有副作用放到 Activity 中。

### Q3: Saga 模式的补偿操作失败了怎么办？

1. **重试策略**：Temporal 的 Activity 支持自动重试
2. **幂等性设计**：补偿操作必须是幂等的（可重复执行不产生副作用）
3. **人工干预**：重试耗尽后转人工处理队列
4. **告警监控**：补偿失败是严重问题，需要立即告警
5. **最终一致性**：通过持续重试达到最终一致状态

### Q4: Temporal Server 的高可用如何保障？

1. **数据库后端**：使用 Cassandra（推荐）或 MySQL/PostgreSQL 集群
2. **多节点部署**：Temporal Server 是无状态的，可以水平扩展
3. **跨数据中心**：支持多集群复制（Temporal Nexus）
4. **Worker 高可用**：Worker 进程可以多实例部署
5. **托管服务**：Temporal Cloud 提供托管方案

### Q5: Laravel Workflow 包和 Temporal.io 该选哪个？

- **Laravel Workflow**：轻量级，基于 Laravel 队列，适合简单工作流，无额外基础设施
- **Temporal.io**：重量级，独立基础设施，适合复杂分布式工作流，生产级可靠性
- **建议**：先用 Laravel Workflow 验证业务流程，当复杂度增长后迁移到 Temporal.io

### Q6: 工作流版本化如何处理？

Temporal 支持在同一集群中运行不同版本的工作流代码：

```python
@workflow.defn
class OrderWorkflow:
    @workflow.run
    async def run(self, order: Order) -> str:
        if workflow.info().workflow_id.startswith('order-v2-'):
            # V2 逻辑：新增风控检查
            await workflow.execute_activity(risk_check, order)
        # 通用逻辑
        ...
```

**最佳实践：**
1. 使用工作流 ID 前缀或信号标记版本
2. 新旧版本并行运行，逐步迁移
3. 通过 `workflow.patched()` API 处理版本兼容
4. 保留旧版本 Worker 直到所有旧工作流执行完毕

### Q7: 如何监控和调试工作流执行？

1. **Temporal Web UI**：内置的 Web 界面，查看工作流历史、状态、事件
2. **Temporal CLI**：命令行工具，查询和操作工作流
3. **OpenTelemetry**：集成分布式追踪
4. **日志聚合**：每个 Activity 执行都有独立日志
5. **事件历史导出**：导出完整事件历史用于调试

### Q8: 除了 Temporal 还有哪些工作流引擎值得考虑？

- **AWS Step Functions**：如果你全在 AWS 上，这是最简单的选择
- **Cadence**：Temporal 的前身（Uber 开源版本，仍在维护）
- **Argo Workflows**：如果你的基础设施在 Kubernetes 上
- **Apache Airflow**：如果你的主要需求是数据管道/ETL
- **Durable Functions（Azure）**：如果你在 Azure 上
- **Laravel Workflow**：如果你的需求简单且希望保持 Laravel 生态

### Q9: 工作流引擎和微服务编排有什么区别？

- **微服务编排（API Gateway/BFF）**：同步请求-响应式的多服务调用，通常在单次 HTTP 请求内完成
- **工作流引擎**：异步、持久化、长时间运行的多步骤流程，支持补偿、等待、重试
- **选择标准**：如果需要跨多服务协调且涉及补偿回滚/长时间等待，用工作流引擎；简单的聚合查询用 BFF
