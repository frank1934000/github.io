# CAP 定理

## 定义

CAP 定理指出：分布式系统中，**一致性（Consistency）、可用性（Availability）、分区容错性（Partition Tolerance）三者最多只能满足两个**。又因为网络分区一定会发生（P 不可避免），所以现实中只能在 CP 与 AP 之间二选一。

## 核心原理

### 三个字母

| 字母 | 含义 | 通俗理解 |
|------|------|---------|
| **C**onsistency | 一致性 | 任何时候读到的数据都是最新写入的 |
| **A**vailability | 可用性 | 每次请求都能在合理时间内返回响应 |
| **P**artition tolerance | 分区容错性 | 节点之间网络断开时系统仍能工作 |

### CP vs AP

| 选择 | 特点 | 代表系统 |
|------|------|---------|
| **CP** | 网络分区时，宁可拒绝服务也要保证强一致 | ZooKeeper、etcd、HBase |
| **AP** | 网络分区时，宁可返回旧数据也要保证可用 | Cassandra、Eureka、DNS |

### 具体例子

主从两节点 MySQL，主从间网络断了：
- **CP 派**：从库直接拒绝读请求（避免读到旧数据）→ 一致但不可用
- **AP 派**：从库继续返回旧数据 → 可用但不一致

### BASE 理论：AP 的工程化补丁

| 字母 | 含义 | 实际做法 |
|------|------|---------|
| **B**asically **A**vailable | 基本可用 | 部分功能/性能降级，核心可用 |
| **S**oft state | 软状态 | 允许中间状态存在（订单"处理中"） |
| **E**ventually consistent | 最终一致 | 一段时间后数据收敛到一致 |

### 最终一致的常见手段

- **消息队列**：写主库 → 发 MQ → 其它服务异步消费更新
- **定时对账**：每日 T+1 跑账，发现不一致就补
- **TCC / Saga**：拆成可补偿的子事务
- **读修复 / 反熵**：Cassandra 风格的后台一致性修复

### CAP vs ACID vs BASE

| 视角 | 单机数据库 | 分布式（强一致派） | 分布式（最终一致派） |
|------|-----------|-------------------|---------------------|
| 模型 | ACID | CP | AP + BASE |
| 例子 | MySQL 单实例 | ZooKeeper、etcd | Cassandra、DynamoDB |
| 取舍 | 不用考虑分区 | 牺牲可用性 | 牺牲强一致 |

### 选型速查

| 业务 | 推荐 | 原因 |
|------|------|------|
| 配置中心、服务注册表 | **CP**（etcd、ZK） | 配错比看不到更可怕 |
| 服务发现（高可用优先） | **AP**（Eureka、Nacos AP） | 拿到旧节点列表也比拿不到强 |
| 商品详情页、信息流 | **AP** + 缓存 | 读多写少，最终一致足够 |
| 支付、库存扣减 | **CP** 或强一致中间件 | 钱出错没法解释 |
| 消息系统 | 看场景 | Kafka 偏 AP；RabbitMQ 偏 CP |

## 常见误区

1. **"NoSQL 都是 AP"** — 错。MongoDB 用 majority 写就是 CP
2. **"我有事务就有 C"** — 单机 ACID 的 C 跟 CAP 的 C 不是一回事
3. **"BASE 是 NoSQL 专属"** — BASE 本质是工程取舍思路，MQ + 对账就是典型 BASE
