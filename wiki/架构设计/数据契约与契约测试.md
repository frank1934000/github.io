# 数据契约与契约测试

## 定义

数据契约（Data Contract）定义了服务间数据交换的格式、类型和约束，比 API 契约（API Contract）更深层——API 契约描述"能调用什么端点"，数据契约描述"数据长什么样"。契约测试（Contract Testing）通过消费者驱动的方式验证提供者是否满足消费者的数据期望，防止接口变更导致生产故障。

## 核心原理

### API 契约 vs 数据契约

| 维度 | API 契约 | 数据契 contract |
|------|----------|-----------------|
| 关注点 | 端点、方法、路径、状态码 | 字段结构、类型、约束、枚举值 |
| 标准 | OpenAPI/Swagger | JSON Schema / Avro / Protobuf |
| 覆盖范围 | HTTP API | HTTP + 消息队列 + 共享数据库 |
| 变更检测 | 路径/参数变更 | 字段删除、类型变更、枚举缩减 |

### 三种数据流场景

```
场景 1: 同步 HTTP API
  Consumer → HTTP Request → Provider
  契约: OpenAPI + JSON Schema + Pact

场景 2: 异步事件（消息队列）
  Producer → Event → Message Broker → Consumer
  契约: AsyncAPI + Avro Schema + Pact Async Message

场景 3: 共享数据库访问
  Service A → DB Table → Service B
  契约: Migration Contract + Column Type Agreement
```

### 消费者驱动契约（CDC）with Pact

```
Consumer 测试 → 生成 Pact JSON 文件
                    ↓
              Pact Broker 存储
                    ↓
Provider CI 验证 → 是否满足所有消费者期望？
                    ↓
              can-i-deploy → 安全部署门禁
```

**核心流程**：
1. 消费者编写测试，定义期望的数据格式
2. Pact 库录制交互，生成 Pact JSON 文件
3. 发布到 Pact Broker，提供者拉取验证
4. `can-i-deploy` 命令检查是否可以安全部署

### Breaking Change 检测

自定义 CLI 工具对比 JSON Schema 版本差异：

**破坏性变更类型**：
- 字段删除（Consumer 依赖该字段）
- 类型变更（string → number）
- 枚举值缩减（移除 Consumer 正在使用的值）
- 新增必填字段（Consumer 无法提供）

**安全变更类型**：
- 新增可选字段
- 枚举值扩展
- 字段重命名（带 alias 兼容）

### 事件版本化策略

```
HTTP API    → URL 版本化: /api/v1/orders, /api/v2/orders
消息队列事件 → Payload 内嵌版本: { "event_version": "1.0", ... }
Schema 存储 → contracts/events/OrderCreated/v1.0/schema.json
```

### CI/CD 四阶段流水线

```
Stage 1: Schema Breaking Change 检查（PR Review 阶段）
    ↓
Stage 2: Consumer Pact 生成（单元测试阶段）
    ↓
Stage 3: Provider Pact 验证（集成测试阶段）
    ↓
Stage 4: can-i-deploy 门禁（部署前检查）
```

## 实战案例

来自博客文章：[Data Contract 实战：Pact-style 数据契约——Laravel 微服务间数据格式的版本化、验证与 Breaking Change 检测](/2026/06/05/2026-06-05-Data-Contract-Pact-style-Laravel微服务数据契约版本化验证Breaking-Change检测/)

**关键技术点**：
- `Versionable` trait 实现事件版本化（v1 → v2 平滑迁移）
- `CheckEventContractChanges` Artisan 命令执行 JSON Schema diff
- `pact-foundation/pact-php` 消费者/提供者测试
- GitHub Actions CI/CD 工作流（data-contract-pipeline.yml）
- JSON Schema 存储结构：`contracts/events/OrderCreated/v1.0/schema.json`

**实施效果**：
- 数据格式变更导致的生产故障：从月均 3-5 次降至 0 次
- 回归测试耗时：从 2-3 天降至 30 分钟
- Breaking Change 拦截点：从生产环境前移到 PR Review 阶段

## 常见问题

**Q: Eloquent `$casts` 导致类型不匹配？**
A: Eloquent 的 `$casts` 会改变字段类型（如 `string` → `int`）。Pact 测试中需明确声明转换后的类型，或在 Provider 端使用原始数据库类型验证。

**Q: Consumer 过度约束契约？**
A: Consumer 不应断言所有字段，只需验证自己依赖的字段。过度约束会导致 Provider 无法安全添加新字段。使用 `like()` 匹配器定义类型约束而非精确值。

**Q: Pact Broker Webhook 配置问题？**
A: 确保 Provider 的验证结果能正确回传到 Broker。检查 `pact_broker_base_url`、`publish_verification_results` 配置，以及 CI 环境中的认证 token。

**Q: 版本号排序问题？**
A: 使用语义化版本（SemVer）且在 Schema 存储中使用目录名排序。避免 `v1.0` vs `v1.0.0` 混用，统一采用 `v{major}.{minor}` 格式。
