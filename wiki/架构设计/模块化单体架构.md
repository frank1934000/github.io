# 模块化单体架构

> 模块化单体是在一个部署单元内通过清晰的模块边界实现逻辑分离的架构模式。它是介于传统单体与微服务之间的最佳平衡点——先模块化，后微服务化。

## 定义

**模块化单体（Modular Monolith）** 是一种架构模式：在单一部署单元（单个代码库、单个进程、单个数据库）内，通过清晰的模块边界将业务逻辑划分为独立的模块。每个模块拥有自己的领域模型、服务层、API 端点和测试套件，模块间通过定义良好的契约（接口/事件）通信，而非直接调用内部实现。

## 核心原则

### 1. 先模块化后微服务化
- 从单体开始，在业务边界清晰后才考虑拆分微服务
- 模块化是微服务的"预演"——模块边界 ≈ 未来的服务边界
- 避免过早微服务化带来的分布式复杂性

### 2. 单一代码库 + 模块边界
- 所有模块共享同一个代码库和部署流程
- 模块内部可以自由重构，模块间通过契约通信
- 编译时/运行时强制边界检查（PHP 可用 PHPStan 规则）

### 3. 本地事务强一致性
- 同一数据库内的操作可以使用本地事务
- 避免分布式事务的复杂性（Saga/TCC/2PC）
- 模块间需要最终一致性时才引入领域事件

## 模块通信方式

### 1. 领域事件（Events）— 跨模块首选
```php
// Order 模块发布事件
class OrderPlaced implements ShouldBroadcast
{
    public function __construct(
        public readonly string $orderId,
        public readonly string $userId,
        public readonly Money $totalAmount,
    ) {}
}

// Payment 模块监听事件
class OrderPlacedListener
{
    public function handle(OrderPlaced $event): void
    {
        $this->createPaymentIntent->execute($event->orderId, $event->totalAmount);
    }
}
```

**优点**：松耦合、可扩展、可追溯
**适用**：跨模块异步操作、最终一致性场景

### 2. 接口契约（Interfaces）— 跨模块同步调用
```php
// Shared/Contracts/InventoryServiceInterface.php
interface InventoryServiceInterface
{
    public function checkStock(string $sku, int $quantity): bool;
    public function reserveStock(string $sku, int $quantity, string $orderId): void;
    public function releaseStock(string $sku, int $quantity, string $orderId): void;
}

// Inventory 模块实现
class InventoryService implements InventoryServiceInterface
{
    // 具体实现
}

// Order 模块通过接口调用
class PlaceOrderAction
{
    public function __construct(
        private InventoryServiceInterface $inventory, // 依赖接口，非具体实现
    ) {}
}
```

**优点**：编译时类型安全、可测试（Mock 接口）
**适用**：跨模块同步查询、需要即时结果的操作

### 3. 直接调用（Direct Calls）— 同模块内
```php
// 同一 Order 模块内直接调用
class PlaceOrderAction
{
    public function __construct(
        private OrderRepository $orders,      // 同模块，直接依赖
        private PricingService $pricing,       // 同模块，直接依赖
    ) {}
}
```

**优点**：最简单、性能最好
**适用**：同一模块内的服务调用

## 模块间通信规则总结

| 通信方式 | 耦合度 | 一致性 | 适用场景 |
|----------|--------|--------|----------|
| 领域事件 | 低 | 最终一致 | 跨模块异步操作、状态变更通知 |
| 接口契约 | 中 | 强一致 | 跨模块同步查询、需要即时结果 |
| 直接调用 | 高 | 强一致 | 同模块内的服务协作 |

## 对比：传统单体 vs 模块化单体 vs 微服务

| 维度 | 传统单体 | 模块化单体 | 微服务 |
|------|----------|------------|--------|
| **部署单元** | 1 个 | 1 个 | N 个 |
| **数据库** | 共享，无边界 | 共享，但有逻辑 Schema 边界 | 每服务独立数据库 |
| **模块间通信** | 直接方法调用（任意耦合） | 事件 + 接口（契约约束） | 网络调用（HTTP/gRPC/MQ） |
| **一致性** | 本地事务（强一致） | 本地事务（强一致）+ 事件（最终一致） | 分布式事务（最终一致） |
| **技术复杂度** | 低 | 中 | 高 |
| **团队协作** | 代码冲突频繁 | 模块所有权清晰 | 服务所有权完全独立 |
| **扩展性** | 整体扩展 | 整体扩展 | 按模块独立扩展 |
| **可观测性** | 简单日志 | 模块级日志 + 事件追踪 | 分布式链路追踪 |
| **适用团队** | < 10 人 | 10-50 人 | > 50 人 |

## 何时使用模块化单体

✅ **适合使用的场景**：
- 中小到中大团队（10-50 人）
- 业务复杂度中到高，但变化频率可控
- 需要快速迭代但不想承担分布式复杂性
- 团队对 DDD 有基本认知
- 希望保留本地事务强一致性优势

❌ **不适合的场景**：
- 团队 < 5 人且业务简单（直接传统单体）
- 已有成熟的微服务基础设施和运维能力
- 模块间有严格的资源隔离和独立扩缩需求

## 何时迁移至微服务

当以下信号出现时，可以将特定模块从模块化单体中拆出：

1. **某模块需要独立扩展** — 该模块的流量/计算量远超其他模块
2. **团队规模超过 50 人** — 单一代码库的协作成本过高
3. **技术栈异构需求** — 某模块需要用 Go/Rust 重写以提升性能
4. **部署频率差异大** — 某模块需要每日多次部署，其他模块月级部署
5. **故障隔离需求** — 某模块的故障不应影响整个系统

### 渐进式迁移步骤

```
模块化单体 ──→ 识别热点模块 ──→ 提取模块为独立服务 ──→ 引入 API 网关 ──→ 事件驱动解耦
     │              │                    │                    │               │
     │         性能/团队瓶颈         双写过渡期           统一入口         最终一致性
     │
     └──→ 继续模块化其他部分
```

## Laravel 实现要点

### 1. 模块 ServiceProvider
```php
// app/Modules/Order/Providers/OrderServiceProvider.php
class OrderServiceProvider extends ServiceProvider
{
    public function boot(): void
    {
        $this->loadRoutesFrom(__DIR__.'/../Routes/web.php');
        $this->loadMigrationsFrom(__DIR__.'/../Database/Migrations');
        $this->loadViewsFrom(__DIR__.'/../Resources/Views', 'order');
    }

    public function register(): void
    {
        $this->app->bind(
            InventoryServiceInterface::class,
            InventoryService::class
        );
    }
}
```

### 2. Composer 自动加载
```json
{
    "autoload": {
        "psr-4": {
            "App\\Modules\\Order\\": "app/Modules/Order/",
            "App\\Modules\\Payment\\": "app/Modules/Payment/",
            "App\\Modules\\Shared\\": "app/Modules/Shared/"
        }
    }
}
```

### 3. PHPStan 边界检查
```neon
# phpstan.neon
rules:
    - PHPStan\Rules\Architecture\NoCrossModuleDependencyRule

parameters:
    moduleBoundaries:
        Order:
            allowedDependencies: [Shared]
        Payment:
            allowedDependencies: [Shared]
        Inventory:
            allowedDependencies: [Shared]
```

## 最佳实践

1. **严格模块边界** — 使用 PHPStan/Rector 强制检查跨模块依赖
2. **事件优先** — 跨模块操作优先使用领域事件，而非直接调用
3. **接口隔离** — 模块对外只暴露接口，不暴露内部实现
4. **独立测试** — 每个模块有独立的测试套件，可单独运行
5. **渐进演进** — 从传统单体开始模块化，而非一开始就设计完美架构
6. **文档化边界** — 每个模块的 Module.php 记录依赖关系和公开 API
