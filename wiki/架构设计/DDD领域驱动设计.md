# DDD 领域驱动设计（Domain-Driven Design）

## 定义

DDD 是 Eric Evans 提出的**以业务领域为中心的软件设计方法论**。核心思想是：软件的复杂度来自业务本身，而非技术实现。通过与领域专家的持续协作，建立统一语言（Ubiquitous Language），将业务规则内化到代码模型中。

## 核心原理

### 分层架构

```
┌──────────────────────────────────────────────────────────────┐
│                     DDD 分层架构 (Laravel)                     │
├──────────────────────────────────────────────────────────────┤
│  Presentation Layer（展示层）                                  │
│  ├── Controllers / Requests / Resources                       │
│  └── 只做 DTO 转换，不含业务逻辑                               │
├──────────────────────────────────────────────────────────────┤
│  Application Layer（应用层）                                   │
│  ├── Application Services（用例编排）                          │
│  └── 调用领域层，协调事务，不含领域规则                          │
├──────────────────────────────────────────────────────────────┤
│  Domain Layer（领域层）★ 核心                                  │
│  ├── Entities / Aggregates（实体/聚合根）                      │
│  ├── Value Objects（值对象）                                   │
│  ├── Domain Events（领域事件）                                 │
│  ├── Domain Services（领域服务）                               │
│  └── Repository Interfaces（仓储接口）                        │
├──────────────────────────────────────────────────────────────┤
│  Infrastructure Layer（基础设施层）                             │
│  ├── Eloquent Models / Repositories 实现                      │
│  ├── Event Dispatcher / Queue Workers                        │
│  └── 外部 API 适配器                                          │
└──────────────────────────────────────────────────────────────┘
```

**关键原则**：领域层不依赖任何框架组件，保持纯净。

### 核心构建块

| 概念 | 定义 | 示例 |
|------|------|------|
| **实体（Entity）** | 有唯一标识，生命周期内状态可变 | 订单、用户 |
| **值对象（Value Object）** | 无唯一标识，通过属性值判断相等，不可变 | 价格、地址、金额 |
| **聚合根（Aggregate Root）** | 一组关联对象的入口，保证业务不变量 | 订单（含订单项） |
| **领域事件（Domain Event）** | 表示领域中发生的重要事情 | 订单已创建、库存已扣减 |
| **领域服务（Domain Service）** | 不属于任何实体的业务逻辑 | 跨聚合的库存校验 |
| **仓储（Repository）** | 聚合的持久化抽象 | OrderRepository |

### 值对象 vs 实体

```
值对象（Money）                     实体（Order）
├── amount: 100                    ├── id: 12345
├── currency: "CNY"                ├── status: "pending"
├── 无唯一标识                      ├── items: [...]
├── 属性相同即相等                  ├── 有唯一标识
└── 不可变                          └── 状态可变
```

### 聚合设计原则

1. **聚合边界内保证一致性**：一个事务只修改一个聚合
2. **聚合间通过 ID 引用**：不直接持有对方对象
3. **聚合根是唯一入口**：外部只能通过聚合根操作内部对象
4. **小聚合优于大聚合**：避免巨型聚合导致并发瓶颈

## 常见问题

**Q: DDD 适合小项目吗？**
A: 不建议。DDD 的分层、聚合、值对象会增加代码量。如果业务逻辑简单（CRUD 为主），传统 MVC 更高效。DDD 适合业务复杂度高、需要频繁迭代的项目。

**Q: Laravel + Eloquent 能做 DDD 吗？**
A: 可以，但需要约束。Eloquent Model 放在基础设施层，领域层用纯 PHP 实体。通过 Repository 接口解耦。

**Q: 聚合根怎么确定？**
A: 找「一致性边界」——哪些对象必须在同一个事务中保持一致？它们属于同一个聚合。例如：订单和订单项必须一致，但订单和用户不需要（用户信息变化不影响订单）。
