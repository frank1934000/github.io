# 流批一体计算引擎

> 流批一体计算引擎是指同一套 API 同时处理有界（批）和无界（流）数据的计算框架。Apache Flink 是该领域的事实标准，为 Laravel 应用提供了超越队列+定时任务的实时计算能力。

## 定义

**流批一体（Unified Stream and Batch Processing）** 是一种数据处理范式：使用同一套编程模型和运行时同时处理有界数据集（批处理）和无界数据流（流处理）。批处理是流处理的特例——有界流 = 已知结束边界的流。

核心思想：**流是一等公民，批是流的子集**。

## 为什么 Laravel 需要流批一体引擎

### Laravel 队列+定时任务的局限

| 需求 | Laravel Queue + Cron | 流批一体引擎 |
|------|---------------------|-------------|
| 秒级聚合 | ❌ Cron 最小粒度 1 分钟 | ✅ 亚秒级窗口聚合 |
| 滑动窗口分析 | ❌ 无法维护滑动状态 | ✅ 原生窗口支持 |
| 跨事件流 Join | ❌ 需要应用层手动关联 | ✅ 双流 Join 原语 |
| Exactly-Once 语义 | ❌ 至少一次（需幂等补偿） | ✅ Checkpoint 端到端精确一次 |
| 背压处理 | ❌ 队列积压需人工干预 | ✅ 内置背压机制 |
| 状态管理 | ❌ 无状态（状态在 DB/Redis） | ✅ 内置状态后端（RocksDB） |
| 乱序处理 | ❌ 无（按到达顺序处理） | ✅ Watermark 机制 |

### 典型场景

```
实时风控：用户行为流 → 5分钟滑动窗口 → 异常模式检测 → 触发告警
实时大屏：订单流 → 1分钟滚动窗口 → 销售额聚合 → 推送 Dashboard
实时推荐：用户点击流 + 商品流 → 双流 Join → 个性化推荐 → 写入 Redis
数据同步：MySQL binlog → Flink SQL CDC → 实时 ETL → 写入数据仓库
```

## Apache Flink 分层 API

```
┌─────────────────────────────────────────────────────┐
│                    Flink SQL / Table API              │  ← 声明式（最高层）
│              SELECT, JOIN, GROUP BY, WINDOW           │
├─────────────────────────────────────────────────────┤
│                    DataStream API                      │  ← 核心编程模型
│          map, filter, keyBy, window, process           │
├─────────────────────────────────────────────────────┤
│                    Process Function                    │  ← 底层控制
│          onTimer, sideOutput, checkpointedFunction     │
└─────────────────────────────────────────────────────┘
```

### Flink SQL（声明式）
```sql
-- 实时计算每分钟订单总额
SELECT
    TUMBLE_START(order_time, INTERVAL '1' MINUTE) AS window_start,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM orders
GROUP BY TUMBLE(order_time, INTERVAL '1' MINUTE);
```

### DataStream API（核心）
```java
DataStream<Order> orders = env.addSource(new OrderSource());

orders
    .keyBy(Order::getRegion)
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .aggregate(new OrderAggregateFunction())
    .addSink(new RedisSink<>());
```

### Process Function（底层）
```java
orders
    .keyBy(Order::getUserId)
    .process(new KeyedProcessFunction<String, Order, Alert>() {
        @Override
        public void processElement(Order order, Context ctx, Collector<Alert> out) {
            // 更新状态
            state.update(state.value() + order.getAmount());
            // 注册定时器
            ctx.timerService().registerEventTimeTimer(
                ctx.timerService().currentWatermark() + 300_000
            );
        }
        
        @Override
        public void onTimer(long timestamp, OnTimerContext ctx, Collector<Alert> out) {
            // 定时触发：检查是否超过阈值
            if (state.value() > THRESHOLD) {
                out.collect(new Alert(ctx.getCurrentKey(), state.value()));
            }
        }
    });
```

## 核心概念

### StreamExecutionEnvironment（执行环境）
```java
// 批处理环境
ExecutionEnvironment bEnv = ExecutionEnvironment.getExecutionEnvironment();

// 流处理环境
StreamExecutionEnvironment sEnv = StreamExecutionEnvironment.getExecutionEnvironment();

// 统一环境（Flink 1.12+）
StreamExecutionEnvironment env = StreamExecutionEnvironment.getExecutionEnvironment();
env.setRuntimeMode(RuntimeExecutionMode.STREAMING); // 或 BATCH, AUTOMATIC
```

### DataStream（数据流）
- **Transformation**：map, filter, flatMap, keyBy, window, reduce, aggregate
- **Partitioning**：rebalance（轮询）, broadcast（广播）, keyBy（按键分区）
- **Side Output**：旁路输出，分流处理逻辑

### Operator（算子）
- **Source**：数据源（Kafka, MySQL CDC, 文件）
- **Transformation**：转换逻辑
- **Sink**：数据输出（Redis, MySQL, Kafka, Elasticsearch）

### Window（窗口）
窗口是流处理的核心抽象，用于将无界流切分为有限块进行聚合。

### Watermark（水位线）
Watermark 是处理乱序事件的机制——表示"时间戳 ≤ Watermark 的事件已全部到达"。

```java
DataStream<Order> ordersWithWatermark = orders
    .assignTimestampsAndWatermarks(
        WatermarkStrategy
            .<Order>forBoundedOutOfOrderness(Duration.ofSeconds(5))
            .withTimestampAssigner((order, ts) -> order.getEventTime())
    );
```

### Checkpoint（检查点）
Checkpoint 是 Flink 容错机制的核心——定期将状态快照持久化到外部存储（HDFS/S3），故障时从最近 Checkpoint 恢复。

```java
env.enableCheckpointing(60_000); // 每 60 秒做一次 Checkpoint
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(30_000);
env.getCheckpointConfig().setCheckpointTimeout(120_000);
env.getCheckpointConfig().setExternalizedCheckpointCleanup(
    ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
);
```

## 窗口类型详解

### 1. Tumbling Window（固定窗口/滚动窗口）
```
时间轴: |----W1----|----W2----|----W3----|
事件:   a  b  c    d  e       f  g  h  i

W1: [a, b, c]  W2: [d, e]  W3: [f, g, h, i]
```

```java
.window(TumblingEventTimeWindows.of(Time.minutes(1)))
```

**特点**：固定大小、不重叠、不遗漏
**适用**：每分钟订单统计、每小时 PV 统计

### 2. Sliding Window（滑动窗口）
```
时间轴: |--W1--|
           |--W2--|
              |--W3--|
                 |--W4--|
事件:   a  b  c  d  e  f  g

W1: [a, b, c]  W2: [b, c, d]  W3: [c, d, e]  W4: [d, e, f]
```

```java
.window(SlidingEventTimeWindows.of(Time.minutes(5), Time.minutes(1)))
```

**特点**：固定大小、可重叠、更平滑的聚合
**适用**：5 分钟滑动窗口的实时风控、趋势分析

### 3. Session Window（会话窗口）
```
事件: a  b    c  d         e    f  g
Gap:     <2s>    <10s>   <3s>

Session1: [a, b, c, d]  (gap < timeout)
Session2: [e]           (gap > timeout, 新会话)
Session3: [f, g]
```

```java
.window(EventTimeSessionWindows.withGap(Time.minutes(5)))
```

**特点**：大小不固定、按活跃间隔动态划分
**适用**：用户会话分析、在线时长统计

### 窗口对比

| 窗口类型 | 大小 | 重叠 | 触发条件 | 典型场景 |
|----------|------|------|----------|----------|
| Tumbling | 固定 | 否 | 窗口结束 | 每分钟聚合 |
| Sliding | 固定 | 是 | 滑动步长 | 滑动趋势分析 |
| Session | 动态 | 否 | 会话超时 | 用户行为分析 |
| Global | 无界 | - | 自定义触发 | 累积计算 |

## Exactly-Once 语义

### 端到端精确一次的实现

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  Source  │───▶│  Flink  │───▶│  Sink   │───▶│  目标   │
│ (Kafka)  │    │  引擎   │    │(2PC Sink)│    │ (DB/Redis)│
└─────────┘    └─────────┘    └─────────┘    └─────────┘
     │              │              │
     └──────────────┼──────────────┘
                    │
              Checkpoint 机制
```

1. **Source 端**：Kafka Consumer 记录 offset 到 Checkpoint
2. **Flink 引擎**：状态后端（RocksDB）定期快照
3. **Sink 端**：两阶段提交（2PC），Checkpoint 完成后才提交外部事务

### 两阶段提交（2PC）流程
```
Checkpoint 触发:
  1. Source: 暂停读取，记录当前 offset
  2. Operator: 状态快照到 RocksDB
  3. Sink: 预提交（pre-commit）— 写入临时数据
  4. Checkpoint Coordinator: 收集所有算子快照
  5. Checkpoint 完成: 通知所有算子
  6. Sink: 正式提交（commit）— 使数据可见
  7. Source: 恢复读取
```

## Flink SQL CDC

Flink CDC（Change Data Capture）直接从 MySQL binlog 捕获变更，无需 Debezium + Kafka 中间层。

### Flink SQL CDC 示例
```sql
-- 创建 CDC 表（直接连接 MySQL binlog）
CREATE TABLE orders_cdc (
    order_id BIGINT,
    user_id BIGINT,
    amount DECIMAL(10,2),
    status STRING,
    created_at TIMESTAMP(3),
    PRIMARY KEY (order_id) NOT ENFORCED
) WITH (
    'connector' = 'mysql-cdc',
    'hostname' = 'localhost',
    'port' = '3306',
    'username' = 'root',
    'password' = 'password',
    'database-name' = 'shop',
    'table-name' = 'orders'
);

-- 创建输出表
CREATE TABLE order_stats_sink (
    window_start TIMESTAMP(3),
    window_end TIMESTAMP(3),
    order_count BIGINT,
    total_amount DECIMAL(12,2),
    PRIMARY KEY (window_start) NOT ENFORCED
) WITH (
    'connector' = 'jdbc',
    'url' = 'jdbc:mysql://localhost:3306/analytics',
    'table-name' = 'order_stats',
    'username' = 'root',
    'password' = 'password'
);

-- 实时聚合写入
INSERT INTO order_stats_sink
SELECT
    TUMBLE_START(created_at, INTERVAL '1' MINUTE) AS window_start,
    TUMBLE_END(created_at, INTERVAL '1' MINUTE) AS window_end,
    COUNT(*) AS order_count,
    SUM(amount) AS total_amount
FROM orders_cdc
GROUP BY TUMBLE(created_at, INTERVAL '1' MINUTE);
```

### CDC vs Debezium + Kafka

| 维度 | Flink SQL CDC | Debezium + Kafka |
|------|---------------|------------------|
| 架构复杂度 | 低（Flink 直连 MySQL） | 高（需部署 Kafka Connect） |
| 延迟 | 低（直接消费 binlog） | 中（经过 Kafka 中转） |
| 运维成本 | 低 | 高（需维护 Kafka 集群） |
| 灵活性 | 高（SQL 即可定义） | 中（需配置 Connector） |
| 适用场景 | 简单 CDC + 实时计算 | 复杂事件路由、多消费者 |

### Laravel 端代码

```php
// app/Modules/Order/Services/OrderEventPublisher.php
class OrderEventPublisher
{
    public function __construct(
        private KafkaProducer $kafka,
    ) {}

    public function publishOrderPlaced(Order $order): void
    {
        $this->kafka->produce('order-events', [
            'event_type' => 'order.placed',
            'order_id' => $order->id,
            'user_id' => $order->user_id,
            'amount' => $order->total_amount,
            'region' => $order->shipping_address->region,
            'event_time' => $order->created_at->toIso8601String(),
        ]);
    }
}

// 读取 Flink 聚合结果
class RealtimeDashboardService
{
    public function getMinuteStats(string $date): Collection
    {
        return DB::table('order_stats')
            ->whereDate('window_start', $date)
            ->orderBy('window_start')
            ->get();
    }

    public function getRealtimeCount(): int
    {
        return (int) Redis::get('flink:realtime:order_count') ?: 0;
    }
}
```

## 对比：Laravel Queue vs Redis Stream vs Kafka vs Flink

| 维度 | Laravel Queue | Redis Stream | Kafka | Flink |
|------|--------------|-------------|-------|-------|
| **定位** | 任务队列 | 轻量消息流 | 分布式事件流 | 流计算引擎 |
| **消息持久化** | DB/Redis | 内存+RDB/AOF | 磁盘+副本 | 状态后端 |
| **消费模式** | 竞争消费 | 消费者组 | 消费者组 | 算子链 |
| **窗口聚合** | ❌ | ❌ | ❌ | ✅ |
| **状态管理** | ❌ | ❌ | ❌ | ✅ RocksDB |
| **Exactly-Once** | ❌ | ❌ | 幂等+事务 | ✅ Checkpoint |
| **背压** | ❌ | ❌ | ✅ | ✅ |
| **延迟** | 秒级 | 毫秒级 | 毫秒级 | 毫秒-秒级 |
| **吞吐** | 中 | 高 | 极高 | 极高 |
| **运维复杂度** | 低 | 低 | 高 | 高 |
| **适用场景** | 异步任务 | 简单消息 | 事件总线 | 实时计算 |

### 选型建议

```
简单异步任务（发邮件/生成报表）
  → Laravel Queue

轻量消息流（< 10万/天，简单消费）
  → Redis Stream

中大规模事件总线（多生产者/消费者，需要持久化）
  → Kafka

需要实时计算（窗口聚合/流 Join/模式检测/Exactly-Once）
  → Kafka + Flink

需要简单 CDC + 实时聚合（不想运维 Kafka）
  → Flink SQL CDC（直连 MySQL）
```

## 运维与监控

### 关键指标
- **吞吐量（Throughput）**：每秒处理的记录数
- **延迟（Latency）**：端到端处理延迟
- **Checkpoint 耗时**：Checkpoint 完成时间
- **背压比率**：下游算子处理不过来的比例
- **状态大小**：RocksDB 状态后端的存储大小

### Prometheus 监控
```yaml
# flink-conf.yaml
metrics.reporters: prom
metrics.reporter.prom.factory.class: org.apache.flink.metrics.prometheus.PrometheusReporterFactory
metrics.reporter.prom.port: 9249
```

### 常见调优
- **并行度**：`env.setParallelism(4)` — 根据 CPU 核数设置
- **Checkpoint 间隔**：根据状态大小和延迟要求调整（通常 30s-5min）
- **状态后端**：大状态用 RocksDB，小状态用 HashMapStateBackend
- **Watermark 延迟**：根据数据乱序程度调整（通常 0-30s）

## 最佳实践

1. **从 Flink SQL 开始** — 声明式 SQL 降低学习曲线，复杂场景再用 DataStream API
2. **合理设置 Watermark** — 过大增加延迟，过小丢数据
3. **Checkpoint + 外部存储** — 状态后端选 RocksDB，Checkpoint 存 S3/HDFS
4. **监控背压** — 背压是性能瓶颈的信号，及时调优算子并行度
5. **幂等 Sink** — 即使有 Exactly-Once，Sink 仍建议实现幂等
6. **CDC 表设主键** — Flink SQL CDC 表必须声明 PRIMARY KEY
7. **渐进式引入** — 先用 Flink 解决最痛的实时计算需求，逐步扩展
