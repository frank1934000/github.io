# 限流与高并发（Rate Limiting & High Concurrency）

## 定义

限流是**控制请求速率**的机制，保护系统不被过量请求打爆。高并发场景下，限流是系统稳定性的第一道防线。

## 核心原理

### 三种限流算法

#### 1. 固定窗口（Fixed Window）

```
时间轴：|----1s----|----1s----|
请求数：|  100/100  |  100/100  |

问题：窗口边界处可能瞬间涌入 200 个请求
```

#### 2. 滑动窗口（Sliding Window）

```
时间轴：|----1s----|----1s----|
窗口：  ───[====滑动窗口====]───

优势：平滑限制，避免边界突发
实现：Redis ZSET + 时间戳
```

#### 3. 令牌桶（Token Bucket）

```
  ┌──────────┐
  │ 令牌桶    │ ← 以固定速率 r 添加令牌
  │ 容量 b    │
  └────┬─────┘
       │ 每个请求消耗一个令牌
       ▼
  ┌──────────┐
  │ 处理请求  │
  └──────────┘

特点：允许突发流量（桶满时），长期速率受限于 r
```

### 限流架构分层

```
Client
  │
  ▼
Nginx Limit Req（第一层：连接级限流）
  │
  ▼
Laravel RateLimiter Middleware（第二层：业务级限流）
  │
  ▼
Redis 限流计数（第三层：分布式限流）
  │
  ├── within limit → Controller → Service → DB
  └── exceeded → 429 Too Many Requests + Retry-After Header
```

### 限流粒度

| 粒度 | 场景 | 实现 |
|------|------|------|
| **全局限流** | 保护整体系统 | Nginx limit_req |
| **API 限流** | 保护特定接口 | Laravel RateLimiter |
| **用户限流** | 防止单用户滥用 | Redis 按 user_id 计数 |
| **IP 限流** | 防爬虫/攻击 | Redis 按 IP 计数 |

### 秒杀系统三层防线

```
第一层：前端限流
  ├── 按钮置灰 + 倒计时
  ├── 验证码/答题
  └── 请求排队

第二层：Redis 预扣减（核心）
  ├── 库存预热到 Redis
  ├── Lua 脚本原子扣减
  └── 快速拒绝超量请求

第三层：MQ 异步下单
  ├── Redis 扣减成功 → 发送 MQ 消息
  ├── 消费者异步创建订单
  └── 超时未支付 → 库存回滚
```

### 秒杀 Lua 脚本示例

```lua
-- 原子操作：检查库存 + 扣减
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock and stock > 0 then
    redis.call('DECR', KEYS[1])
    return 1  -- 扣减成功
else
    return 0  -- 库存不足
end
```

### 秒杀的本质矛盾

- **读多写少**：10 万人抢 100 个商品，99.9% 的请求应该被快速拒绝
- **瞬时高并发**：1 秒内涌入 50,000+ QPS，是日常流量的 100 倍
- **强一致性**：库存不能超卖（100 件商品绝不能卖出 101 件）

## 常见问题

**Q: 限流返回什么状态码？**
A: 429 Too Many Requests，附带 `Retry-After` Header 告知客户端何时重试。

**Q: 限流和熔断有什么区别？**
A: 限流是**主动控制**请求速率；熔断是**被动响应**后端故障（当下游错误率过高时自动断开）。

**Q: 秒杀库存超卖怎么办？**
A: Redis Lua 脚本原子扣减 + 数据库乐观锁（WHERE stock > 0）双重保障。
