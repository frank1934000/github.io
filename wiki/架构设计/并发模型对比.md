# 并发模型对比

## 定义

并发模型是指在计算机程序中组织和管理多个同时进行的计算任务的架构范式。不同的编程语言和运行时采用不同的并发模型，每种模型都有其独特的哲学、优势和适用场景。

本文对比以下主流并发模型：

- **Actor 模型（Akka / Elixir / PHP）**：通过消息传递替代共享状态的并发架构
- **Elixir OTP Supervisor 树与 GenServer**：基于 Erlang BEAM 虚拟机的容错并发框架
- **Kotlin 协程与结构化并发**：基于挂起函数的轻量级并发原语
- **Python asyncio 事件循环**：基于事件循环和协程的异步 I/O 框架
- **PHP-FPM 无状态模型**：请求-响应的无状态进程模型
- **PHP Fibers**：PHP 8.1 引入的纤程原语
- **Go Goroutines**：基于 CSP（Communicating Sequential Processes）的轻量级协程

理解这些并发模型的差异，对于在微服务架构中选择合适的技术栈、设计高并发系统至关重要。

## 核心原理

### 1. Actor 模型

Actor 模型由 Carl Hewitt 于 1973 年提出，是一种基于消息传递的并发计算模型。

**核心概念：**
- **Actor（演员）**：并发计算的基本单元，每个 Actor 拥有私有状态
- **消息传递**：Actor 之间通过异步消息通信，不共享内存
- **邮箱（Mailbox）**：每个 Actor 有专属的消息队列
- **行为（Behavior）**：Actor 收到消息后可以做出三种响应：
  1. 创建新的 Actor
  2. 发送消息给其他 Actor
  3. 更新自身状态

**Actor 模型 vs 共享内存模型：**

| 特性 | Actor 模型 | 共享内存模型 |
|------|-----------|-------------|
| 状态共享 | 无共享（消息传递） | 共享可变状态 |
| 并发控制 | 无需锁（单线程处理消息） | 需要锁/Mutex/信号量 |
| 死锁风险 | 不存在 | 可能发生 |
| 数据竞争 | 不存在 | 可能发生 |
| 可伸缩性 | 天然分布式友好 | 受限于共享内存 |
| 调试难度 | 消息追踪 | 竞态条件排查 |

**Akka（JVM）示例：**
```java
public class OrderActor extends AbstractActor {
    private List<OrderItem> items = new ArrayList<>();

    @Override
    public Receive createReceive() {
        return receiveBuilder()
            .match(AddItem.class, this::onAddItem)
            .match(ProcessOrder.class, this::onProcessOrder)
            .build();
    }

    private void onAddItem(AddItem msg) {
        items.add(msg.item);
        getSender().tell(new ItemAdded(msg.item), getSelf());
    }

    private void onProcessOrder(ProcessOrder msg) {
        // 处理订单逻辑
        ActorRef paymentActor = getContext().actorOf(PaymentActor.props());
        paymentActor.tell(new ProcessPayment(items), getSelf());
    }
}
```

**Elixir Actor 模型：**
```elixir
defmodule OrderActor do
  use GenServer

  # 客户端 API
  def start_link(initial_state) do
    GenServer.start_link(__MODULE__, initial_state, name: __MODULE__)
  end

  def add_item(pid, item) do
    GenServer.cast(pid, {:add_item, item})
  end

  def process_order(pid) do
    GenServer.call(pid, :process_order)
  end

  # 服务端回调
  @impl true
  def init(state) do
    {:ok, state}
  end

  @impl true
  def handle_cast({:add_item, item}, state) do
    {:noreply, %{state | items: [item | state.items]}}
  end

  @impl true
  def handle_call(:process_order, _from, state) do
    # 处理订单
    {:reply, :ok, %{state | status: :processed}}
  end
end
```

**PHP Actor 模型（使用 concurrent/futures）：**
```php
// 使用 ReactPHP 或 Amp 实现类 Actor 模式
class OrderActor
{
    private Mailbox $mailbox;
    private array $items = [];

    public function __construct()
    {
        $this->mailbox = new Mailbox();
    }

    public function receive(): void
    {
        Loop::add(function () {
            while (true) {
                $message = yield $this->mailbox->receive();
                match ($message['type']) {
                    'add_item' => $this->addItem($message['item']),
                    'process' => $this->processOrder(),
                };
            }
        });
    }

    public function send(array $message): void
    {
        $this->mailbox->send($message);
    }
}
```

### 2. Elixir OTP Supervisor 树与 GenServer

OTP（Open Telecom Platform）是 Erlang/OTP 提供的构建容错并发系统的框架。

**核心组件：**

**GenServer（Generic Server）：**
- 封装了有状态的服务进程
- 处理同步请求（call）和异步请求（cast）
- 内置超时、代码热更新支持

**Supervisor（监督者）：**
- 监控子进程的生命周期
- 子进程崩溃时自动重启
- 定义重启策略（one_for_one, one_for_all, rest_for_one）

**Supervisor 树：**
```elixir
defmodule MyApp.Supervisor do
  use Supervisor

  def start_link(init_arg) do
    Supervisor.start_link(__MODULE__, init_arg, name: __MODULE__)
  end

  @impl true
  def init(_init_arg) do
    children = [
      # 一级子进程
      {OrderSupervisor, []},      # 订单子树
      {PaymentSupervisor, []},    # 支付子树
      {NotificationActor, []},    # 通知进程
      {CacheActor, strategy: :one_for_one}
    ]

    # 策略：一个子进程崩溃只重启它自己
    Supervisor.init(children, strategy: :one_for_one)
  end
end

defmodule OrderSupervisor do
  use Supervisor

  def init(_arg) do
    children = [
      {OrderProcessor, []},
      {OrderValidator, []},
      {OrderNotifier, []}
    ]

    # 策略：一个子进程崩溃则重启所有子进程
    Supervisor.init(children, strategy: :rest_for_one)
  end
end
```

**Supervisor 重启策略：**

| 策略 | 描述 | 适用场景 |
|------|------|----------|
| `one_for_one` | 只重启崩溃的子进程 | 子进程相互独立 |
| `one_for_all` | 重启所有子进程 | 子进程相互依赖 |
| `rest_for_one` | 重启崩溃子进程及其之后启动的子进程 | 有序依赖关系 |

**Elixir vs PHP-FPM 并发哲学：**

| 维度 | Elixir/OTP | PHP-FPM |
|------|-----------|---------|
| 进程模型 | 长驻轻量进程（数百万） | 短生命周期 Worker |
| 内存模型 | 每进程独立堆，无共享 | 每请求独立，无共享 |
| 容错 | Let it crash + Supervisor | 进程崩溃由 FPM 回收 |
| 状态管理 | 有状态进程（GenServer） | 无状态（状态在外部存储） |
| 消息传递 | 原生支持（进程邮箱） | 需要外部中间件 |
| 热更新 | 支持代码热升级 | 需要重启 Worker |
| 并发单位 | 进程（轻量，~2KB） | 进程（重量，~10MB+） |

### 3. Kotlin 协程与结构化并发

Kotlin 协程是基于挂起函数（Suspending Functions）的轻量级并发框架。

**核心概念：**

**挂起函数（Suspending Functions）：**
```kotlin
suspend fun fetchUser(id: String): User {
    return withContext(Dispatchers.IO) {
        httpClient.get("https://api.example.com/users/$id")
    }
}

suspend fun fetchOrders(userId: String): List<Order> {
    return withContext(Dispatchers.IO) {
        httpClient.get("https://api.example.com/orders?user=$userId")
    }
}
```

**结构化并发（Structured Concurrency）：**
```kotlin
suspend fun loadDashboard(userId: String): Dashboard {
    return coroutineScope {
        // 并发执行，任一失败则取消所有
        val user = async { fetchUser(userId) }
        val orders = async { fetchOrders(userId) }
        val notifications = async { fetchNotifications(userId) }

        Dashboard(
            user = user.await(),
            orders = orders.await(),
            notifications = notifications.await()
        )
    }
}
```

**结构化并发的核心原则：**
1. **生命周期绑定**：子协程的生命周期绑定到父协程的作用域
2. **错误传播**：子协程的异常会传播到父协程
3. **取消传播**：父协程取消时，所有子协程也被取消
4. **无泄漏**：作用域结束时，所有子协程必定完成或取消

**Kotlin Flow（响应式流）：**
```kotlin
fun observeOrders(): Flow<Order> = flow {
    while (true) {
        val newOrders = api.getNewOrders()
        newOrders.forEach { emit(it) }
        delay(1000)
    }
}

// 使用
viewModelScope.launch {
    observeOrders()
        .filter { it.amount > 100 }
        .map { it.toOrderSummary() }
        .collect { summary ->
            updateUI(summary)
        }
}
```

**Kotlin 协程 vs PHP Fibers vs Go Goroutines：**

| 特性 | Kotlin Coroutines | PHP Fibers | Go Goroutines |
|------|------------------|------------|---------------|
| 引入版本 | Kotlin 1.3 | PHP 8.1 | Go 1.0 |
| 并发模型 | 结构化并发 | 非结构化纤程 | CSP（通信顺序进程） |
| 调度器 | 可配置（IO/Default/Main） | 运行时自动 | Go 运行时调度器（M:N） |
| 内存开销 | ~几 KB | ~几 KB | ~几 KB |
| 取消机制 | 协作式取消（CancellationException） | Fiber::suspend/throw | context.Context |
| 错误传播 | 结构化传播 | 需手动处理 | 通过 channel 返回 error |
| Channel | Channel API | 无内置 | chan（原生支持） |
| Select | select 表达式 | 无内置 | select 语句 |
| 生态成熟度 | 高（JetBrains + Google） | 中等（基础原语） | 高（标准库） |

**PHP Fibers 示例：**
```php
class AsyncFetcher
{
    public function fetchMultiple(array $urls): array
    {
        $fibers = [];
        foreach ($urls as $url) {
            $fibers[] = new Fiber(function (string $url) {
                $result = $this->httpGet($url);
                Fiber::suspend($result);
            });
        }

        $results = [];
        foreach ($fibers as $fiber) {
            $fiber->start($url);
            $results[] = $fiber->getReturn();
        }

        return $results;
    }
}
```

### 4. Python asyncio 事件循环

Python asyncio 基于单线程事件循环实现协程调度，是 Python 异步 I/O 的标准框架。

**事件循环架构：**
```
┌─────────────────────────────────────────────────┐
│              Python asyncio 事件循环             │
│                                                 │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │ 协程队列  │  │ 回调队列  │  │ I/O 多路复用│     │
│  │ (coro)   │  │ (callback)│  │ (epoll/kqueue)│   │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘      │
│       │              │              │            │
│       ▼              ▼              ▼            │
│  ┌──────────────────────────────────────┐       │
│  │          事件循环调度器               │       │
│  │  1. 执行到期的回调/协程               │       │
│  │  2. 处理 I/O 事件                     │       │
│  │  3. 执行延迟回调                      │       │
│  │  4. 休眠等待新事件                    │       │
│  └──────────────────────────────────────┘       │
└─────────────────────────────────────────────────┘
```

**基本用法：**
```python
import asyncio
import aiohttp

async def fetch_user(session, user_id):
    async with session.get(f'https://api.example.com/users/{user_id}') as resp:
        return await resp.json()

async def fetch_orders(session, user_id):
    async with session.get(f'https://api.example.com/orders/{user_id}') as resp:
        return await resp.json()

async def load_dashboard(user_id):
    async with aiohttp.ClientSession() as session:
        # 并发执行多个异步任务
        user, orders = await asyncio.gather(
            fetch_user(session, user_id),
            fetch_orders(session, user_id),
        )
        return {'user': user, 'orders': orders}

# 运行
dashboard = asyncio.run(load_dashboard(123))
```

**asyncio TaskGroup（Python 3.11+ 结构化并发）：**
```python
async def load_dashboard_structured(user_id):
    async with asyncio.TaskGroup() as tg:
        async with aiohttp.ClientSession() as session:
            user_task = tg.create_task(fetch_user(session, user_id))
            orders_task = tg.create_task(fetch_orders(session, user_id))
            notifications_task = tg.create_task(fetch_notifications(session, user_id))

    # 所有任务完成或任一异常时退出
    return {
        'user': user_task.result(),
        'orders': orders_task.result(),
        'notifications': notifications_task.result(),
    }
```

**Python asyncio vs PHP-FPM：**

| 特性 | Python asyncio | PHP-FPM |
|------|---------------|---------|
| 并发模型 | 单线程事件循环 + 协程 | 多进程（每请求一进程） |
| I/O 模型 | 非阻塞异步 I/O | 同步阻塞 I/O |
| 并发能力 | 数千并发连接 | 受限于 Worker 进程数 |
| CPU 密集型 | 不适合（GIL 限制） | 适合（每进程独立） |
| 适用场景 | I/O 密集型（Web 爬虫、API 网关） | 传统 Web 应用 |
| 学习曲线 | 中等（需理解异步思维） | 低（同步编程模型） |

### 5. PHP-FPM 无状态模型深度分析

PHP-FPM 的并发模型是基于多进程的请求-响应模型，每个请求由一个独立的 Worker 进程处理。

**请求生命周期：**
```
客户端请求 --> Nginx --> PHP-FPM Master
                          │
                          ├── Worker 1: Boot → Handle → Respond → Destroy
                          ├── Worker 2: Boot → Handle → Respond → Destroy
                          ├── Worker 3: Boot → Handle → Respond → Destroy
                          └── Worker N: Boot → Handle → Respond → Destroy
```

**PHP-FPM 的优势：**
1. **天然隔离**：每个请求独占进程，无共享状态问题
2. **无内存泄漏**：请求结束后进程回收
3. **简单可靠**：编程模型简单，不需要考虑并发问题
4. **成熟稳定**：数十年生产验证

**PHP-FPM 的局限：**
1. **进程开销大**：每个 Worker 约 10-50MB 内存
2. **无法长连接**：不适合 WebSocket 等长连接场景
3. **同步阻塞**：I/O 操作阻塞 Worker 进程
4. **冷启动**：进程启动需要加载框架和代码

**PHP-FPM 向异步的演进：**

```
PHP-FPM (传统)
    │
    ├── Swoole（C 扩展，协程+异步 I/O）
    │       └── Laravel Octane
    │
    ├── ReactPHP（纯 PHP 事件循环）
    │       └── 类似 Node.js 模型
    │
    ├── PHP Fibers（8.1+ 语言级纤程）
    │       └── 基础并发原语
    │
    └── FrankenPHP（Go + PHP，长驻进程）
            └── Laravel Octane 支持
```

### 6. Go Goroutines 与 CSP 模型

Go 的并发模型基于 CSP（Communicating Sequential Processes），核心理念是"通过通信共享内存"。

```go
func loadDashboard(userID string) (*Dashboard, error) {
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()

    // 使用 channel 收集结果
    userCh := make(chan *User, 1)
    ordersCh := make(chan []*Order, 1)
    errCh := make(chan error, 2)

    // 并发获取数据
    go func() {
        user, err := fetchUser(ctx, userID)
        if err != nil {
            errCh <- err
            return
        }
        userCh <- user
    }()

    go func() {
        orders, err := fetchOrders(ctx, userID)
        if err != nil {
            errCh <- err
            return
        }
        ordersCh <- orders
    }()

    // 收集结果
    dashboard := &Dashboard{}
    for i := 0; i < 2; i++ {
        select {
        case user := <-userCh:
            dashboard.User = user
        case orders := <-ordersCh:
            dashboard.Orders = orders
        case err := <-errCh:
            return nil, err
        case <-ctx.Done():
            return nil, ctx.Err()
        }
    }

    return dashboard, nil
}
```

### 7. 并发模型全景对比

| 维度 | Actor (Elixir/OTP) | Kotlin Coroutines | Python asyncio | PHP-FPM | Go Goroutines |
|------|-------------------|-------------------|---------------|---------|---------------|
| 进程/线程模型 | BEAM 虚拟机进程 | JVM 线程池 | 单线程事件循环 | 多进程 | M:N 调度 |
| 最小并发单元 | Erlang 进程 (~2KB) | 协程 (~几KB) | Task (~几KB) | Worker 进程 (~10MB) | Goroutine (~2KB) |
| 并发上限 | 百万级 | 数万级 | 数万级 | 数百级 | 百万级 |
| 共享状态 | 无（消息传递） | 可选 | 无（单线程） | 无（进程隔离） | 通过 channel |
| 容错机制 | Supervisor 树 | 结构化取消 | TaskGroup | FPM 进程回收 | context + error |
| 分布式支持 | 原生（节点间消息） | 需外部库 | 需外部库 | 需外部中间件 | 原生 RPC |
| 热更新 | 支持 | 不支持 | 不支持 | 重启 Worker | 不支持 |
| CPU 密集型 | 支持（脏调度器） | 支持 | 不适合（GIL） | 支持 | 支持 |
| I/O 密集型 | ★★★★★ | ★★★★★ | ★★★★ | ★★ | ★★★★★ |

### 8. 选型决策指南

```
需要高并发系统
├── 需要极高容错和分布式？ → Elixir/OTP（电信级可靠性）
├── Android/服务端 + JVM 生态？ → Kotlin Coroutines
├── Python 生态（AI/ML/数据）？ → asyncio
├── 已有 PHP 生态 + 简单需求？ → PHP-FPM + Laravel
├── 已有 PHP + 需要长连接？ → Laravel Octane (Swoole/FrankenPHP)
├── 需要极高并发 + 简单部署？ → Go Goroutines
└── 需要分布式 Actor？ → Akka (JVM) / Elixir / Microsoft Orleans (.NET)
```

## 常见问题

### Q1: PHP 应用需要学习 Elixir/Go 吗？

**取决于场景。** 如果你的 PHP 应用通过 Laravel Octane、队列、缓存等手段能满足并发需求，无需引入新语言。但如果你需要：
- WebSocket 长连接 + 百万级并发：考虑 Elixir
- 高性能微服务 + 简单部署：考虑 Go
- AI/ML 异步流水线：考虑 Python asyncio

**务实建议：** 先用 PHP 生态内的解决方案（Octane、Swoole、Laravel Queues）优化，再考虑引入新语言。

### Q2: PHP Fibers 和 Swoole 协程有什么区别？

- **PHP Fibers**：语言级原语，是构建协程的基础构件，本身不提供异步 I/O
- **Swoole 协程**：完整的协程运行时，包括异步 I/O、连接池、定时器等
- **关系**：Fibers 类似 Go 的 goroutine 原语，Swoole 类似 Go 的整个运行时
- **使用建议**：在 Swoole/Octane 环境中，底层已经使用 Fibers；纯 PHP 环境下 Fibers 需配合异步库使用

### Q3: Actor 模型的消息丢失如何处理？

- **Elixir/OTP**：Supervisor 监控进程崩溃，自动重启并恢复状态
- **Akka**：支持 At-Most-Once、At-Least-Once、Exactly-Once 语义
- **持久化 Actor**：Akka Persistence、EventStore 支持事件溯源
- **最佳实践**：关键消息使用持久化 + 确认机制，非关键消息使用 At-Most-Once

### Q4: 结构化并发和非结构化并发的区别？

- **结构化并发**（Kotlin Coroutines, Python TaskGroup）：子任务绑定到父作用域，生命周期受控，异常自动传播
- **非结构化并发**（PHP Fibers, 原始线程）：子任务独立于父任务，需要手动管理生命周期和错误处理
- **优势**：结构化并发避免了"泄漏的并发任务"和"孤儿协程"问题

### Q5: 为什么 Elixir/BEAM 能支持百万级并发？

1. **极轻量进程**：每个 Erlang 进程仅占用约 2KB 内存
2. **独立堆**：每个进程有独立的堆，无需 GC 全局停顿
3. **抢占式调度**：BEAM 的 reduction-based 调度确保公平调度
4. **多核利用**：调度器自动分配进程到所有 CPU 核心
5. **零共享内存**：无锁设计，无缓存一致性开销

### Q6: Go 的 Goroutine 泄漏如何避免？

```go
// ❌ 泄漏的 Goroutine
func leaky() {
    go func() {
        for {
            // 永远运行，无法停止
            doSomething()
        }
    }()
}

// ✅ 使用 Context 控制生命周期
func controlled(ctx context.Context) {
    go func() {
        for {
            select {
            case <-ctx.Done():
                return // 收到取消信号，退出
            default:
                doSomething()
            }
        }
    }()
}
```

### Q7: Python asyncio 的 GIL 限制如何解决？

1. **I/O 密集型**：asyncio 不受 GIL 影响（I/O 等待时释放 GIL）
2. **CPU 密集型**：使用 `ProcessPoolExecutor` 将计算分配到多个进程
3. **Python 3.13+**：实验性去除 GIL（`--disable-gil` 编译选项）
4. **替代方案**：Rust/C 扩展处理 CPU 密集计算（如 PyO3）

### Q8: 在 Laravel 中如何渐进式引入并发能力？

**阶段一：** 使用 Laravel Queues 异步化耗时操作
```php
ProcessOrder::dispatch($order)->onQueue('high');
```

**阶段二：** 使用 Laravel Octane 提升吞吐量
```bash
php artisan octane:start --server=swoole --workers=32
```

**阶段三：** 使用 Laravel Reverb + WebSocket 实现实时通信
```bash
php artisan reverb:start
```

**阶段四：** 针对特定场景引入 Elixir/Go 微服务
```
[Laravel 主应用] ←→ [Elixir 实时服务] ←→ [Go 高并发网关]
```
