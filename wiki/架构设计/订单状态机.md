# 订单状态机

## 定义

订单状态机（Order State Machine）是使用有限状态机（Finite State Machine, FSM）模型对订单全生命周期进行建模的架构设计模式。它将订单的各种状态定义为有限集合，将状态之间的转换定义为受规则约束的事件触发过程，从而保证订单流转的合法性、可追溯性和可维护性。

在电商系统中，订单是核心业务实体，其状态流转涉及支付、库存、物流、退款等多个子系统的协作。一个设计良好的订单状态机是整个电商系统的基石。

## 核心原理

### 有限状态机的数学模型

一个有限状态机可以形式化定义为五元组 (S, E, T, s₀, F)：

- **S（States）**：有限状态集合
- **E（Events）**：有限事件集合
- **T（Transitions）**：转换函数 T(S, E) → S，定义在某个状态下收到某个事件后转移到哪个状态
- **s₀（Initial State）**：初始状态
- **F（Final States）**：终态集合

### 订单生命周期状态定义

一个典型的电商订单包含以下状态：

```
PENDING          → 待支付（订单创建成功，等待支付）
PAID             → 已支付（支付成功）
CONFIRMED        → 已确认（商家确认接单）
PICKING          → 拣货中（仓库正在拣货）
SHIPPED          → 已发货（物流已揽收）
DELIVERING       → 配送中（物流运输中）
DELIVERED        → 已签收（买家确认签收）
COMPLETED        → 已完成（售后期结束）
CLOSED           → 已关闭（超时未支付或用户取消）
REFUNDING        → 退款中（申请退款）
REFUNDED         → 已退款（退款成功）
CANCELED         → 已取消（支付前取消）
```

### 状态转换规则

状态机的核心价值在于严格定义哪些状态转换是合法的：

```
PENDING   → PAID          # 支付成功事件
PENDING   → CLOSED        # 超时关闭事件
PENDING   → CANCELED      # 用户主动取消
PAID      → CONFIRMED     # 商家确认事件
PAID      → REFUNDING     # 用户申请退款
CONFIRMED → PICKING       # 开始拣货
PICKING   → SHIPPED       # 揽收发货
SHIPPED   → DELIVERING    # 物流更新
DELIVERING → DELIVERED    # 签收事件
DELIVERED → COMPLETED     # 售后期满
REFUNDING → REFUNDED      # 退款成功
REFUNDING → PAID          # 退款拒绝（回到已支付）
REFUNDED  → COMPLETED     # 退款完成，订单结束
```

**不允许的状态转换示例：**
- COMPLETED → PAID（已完成的订单不能回到已支付）
- SHIPPED → CANCELED（已发货不能直接取消，需要走退款流程）
- PENDING → SHIPPED（必须经过支付和确认步骤）

### 事件驱动的状态转换

状态机中的每次转换都由事件触发：

```php
// 事件定义
class OrderEvents
{
    const PAY_SUCCESS = 'pay_success';         // 支付成功
    const PAY_TIMEOUT = 'pay_timeout';         // 支付超时
    const MERCHANT_CONFIRM = 'merchant_confirm'; // 商家确认
    const START_PICKING = 'start_picking';     // 开始拣货
    const SHIP = 'ship';                       // 发货
    const DELIVER = 'deliver';                 // 配送
    const CONFIRM_RECEIPT = 'confirm_receipt'; // 确认收货
    const APPLY_REFUND = 'apply_refund';       // 申请退款
    const REFUND_SUCCESS = 'refund_success';   // 退款成功
    const REFUND_REJECT = 'refund_reject';     // 退款拒绝
    const USER_CANCEL = 'user_cancel';         // 用户取消
}
```

### 状态机的三种实现方式

#### 1. 状态转换表（Transition Table）

用二维表描述所有合法的状态转换：

| 当前状态 | 事件 | 目标状态 | 守卫条件 |
|---------|------|---------|---------|
| PENDING | PAY_SUCCESS | PAID | 金额校验通过 |
| PENDING | PAY_TIMEOUT | CLOSED | 超过30分钟 |
| PAID | APPLY_REFUND | REFUNDING | 退款金额 ≤ 订单金额 |
| ... | ... | ... | ... |

#### 2. 状态模式（State Pattern）

面向对象的经典实现，每个状态是一个类：

```php
interface OrderState
{
    public function pay(OrderContext $context): void;
    public function cancel(OrderContext $context): void;
    public function refund(OrderContext $context): void;
    public function getName(): string;
}

class PendingState implements OrderState
{
    public function pay(OrderContext $context): void
    {
        // 支付逻辑
        $context->setState(new PaidState());
    }

    public function cancel(OrderContext $context): void
    {
        $context->setState(new CanceledState());
    }

    public function refund(OrderContext $context): void
    {
        throw new InvalidTransitionException('待支付订单不能退款');
    }

    public function getName(): string
    {
        return 'pending';
    }
}
```

#### 3. 状态机引擎（State Machine Engine）

使用专门的状态机库（如 XState）进行声明式定义：

```javascript
const orderMachine = createMachine({
  id: 'order',
  initial: 'pending',
  states: {
    pending: {
      on: {
        PAY_SUCCESS: 'paid',
        PAY_TIMEOUT: 'closed',
        USER_CANCEL: 'canceled'
      }
    },
    paid: {
      on: {
        MERCHANT_CONFIRM: 'confirmed',
        APPLY_REFUND: 'refunding'
      }
    },
    // ... 其他状态
  }
});
```

### 状态机的关键特性

#### 不变量保护（Invariant）
状态机保证系统在任何时刻都处于一个合法状态，不会出现"已支付但未确认却已发货"这种非法状态。

#### 守卫条件（Guard）
状态转换可以附加业务条件，只有满足条件才能转换：
```php
// 申请退款的守卫条件
'paid' => [
    'apply_refund' => [
        'target' => 'refunding',
        'guard' => fn($order) => $order->canRefund() && $order->refundAmount <= $order->totalAmount
    ]
]
```

#### 副作用/动作（Action/Effect）
状态转换时可以触发副作用：
```php
'order.paid' => function ($order) {
    // 触发库存扣减
    // 发送支付成功通知
    // 记录状态变更日志
    event(new OrderPaidEvent($order));
}
```

### 事件溯源集成

将状态机与[事件溯源](事件驱动架构.md)结合，每次状态变更记录为一个事件：

```
OrderCreated → PaymentReceived → OrderConfirmed → OrderShipped → OrderDelivered → OrderCompleted
```

这提供了完整的审计追踪能力，任何时刻都可以重放事件序列重建订单状态。

### Laravel 状态机实现

在 Laravel 中实现订单状态机，通常结合以下组件：

**数据库迁移：**
```php
Schema::table('orders', function (Blueprint $table) {
    $table->string('status', 32)->default('pending')->index();
    $table->timestamp('status_changed_at')->nullable();
});
```

**状态变更日志表：**
```php
Schema::create('order_status_logs', function (Blueprint $table) {
    $table->id();
    $table->foreignId('order_id');
    $table->string('from_status');
    $table->string('to_status');
    $table->string('event');
    $table->json('metadata')->nullable();
    $table->timestamps();
});
```

**状态机服务：**
```php
class OrderStateMachine
{
    private array $transitions = [
        'pending' => ['paid', 'closed', 'canceled'],
        'paid' => ['confirmed', 'refunding'],
        'confirmed' => ['picking'],
        'picking' => ['shipped'],
        'shipped' => ['delivering'],
        'delivering' => ['delivered'],
        'delivered' => ['completed'],
        'refunding' => ['refunded', 'paid'],
        'refunded' => ['completed'],
    ];

    public function transition(Order $order, string $event): Order
    {
        $targetState = $this->resolveTargetState($order->status, $event);

        if (!$this->can($order, $event)) {
            throw new InvalidTransitionException(
                "订单 {$order->no} 不允许从 {$order->status} 转换到 {$targetState}"
            );
        }

        DB::transaction(function () use ($order, $targetState, $event) {
            $fromStatus = $order->status;
            $order->update([
                'status' => $targetState,
                'status_changed_at' => now(),
            ]);

            // 记录状态变更日志
            OrderStatusLog::create([
                'order_id' => $order->id,
                'from_status' => $fromStatus,
                'to_status' => $targetState,
                'event' => $event,
            ]);

            // 触发副作用
            $this->dispatchSideEffects($order, $fromStatus, $targetState, $event);
        });

        return $order;
    }

    public function can(Order $order, string $event): bool
    {
        return in_array(
            $this->resolveTargetState($order->status, $event),
            $this->transitions[$order->status] ?? []
        );
    }
}
```

## 常见问题

### Q: 状态机和简单的 if-else 有什么区别？
状态机提供了形式化的状态定义和转换规则，保证只有合法的状态转换才能发生。if-else 容易遗漏边界条件，随着状态和事件增加，代码复杂度呈指数增长。状态机的可维护性、可测试性和可审计性远优于 if-else。

### Q: 如何处理并发状态转换？
使用乐观锁（版本号）或悲观锁保证同一订单同时只有一个状态转换在进行：
```php
$order = Order::lockForUpdate()->find($id);
// 或
$order = Order::where('id', $id)->where('version', $version)->update([...]);
```

### Q: 状态机如何处理并行状态？
有些场景需要并行状态（如订单的支付状态和物流状态独立流转），可以使用 XState 的 parallel states 或维护多个独立的状态字段。

### Q: 如何设计可扩展的状态机？
- 将状态和转换规则配置化（存数据库或配置文件）
- 使用策略模式处理不同状态下的业务逻辑
- 通过事件监听器解耦状态变更的副作用
- 支持动态注册新的状态和转换规则

### Q: 状态机与工作流引擎有什么关系？
状态机专注于单个实体的状态管理，工作流引擎管理跨多个实体的复杂流程编排。简单场景用状态机即可，复杂业务流程（如审批流、多步骤订单处理）可能需要工作流引擎。

### Q: 如何测试状态机？
- 测试所有合法转换路径
- 测试所有非法转换被正确拒绝
- 测试守卫条件的边界情况
- 测试副作用是否正确触发
- 基于状态图自动生成测试用例
