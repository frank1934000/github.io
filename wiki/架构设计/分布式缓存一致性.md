# 分布式缓存一致性

## 定义

分布式缓存一致性（Distributed Cache Consistency）是指在分布式系统中，保证缓存层（如 Redis、Memcached）与持久化存储层（如 MySQL、PostgreSQL）之间数据保持一致状态的工程实践与理论体系。在高并发电商、金融等场景下，缓存与数据库的双写一致性是系统设计中最具挑战性的问题之一。

核心挑战在于：缓存和数据库是两个独立的存储系统，无法在一个原子事务中完成双写，任何顺序的写入都可能在中间状态下被并发读取打断，导致数据不一致。

## 核心原理

### 一致性模型分类

缓存一致性从严格程度上可分为：

- **强一致性**：任何时刻读取的数据都是最新的，代价是性能下降
- **最终一致性**：在一段时间窗口内数据可能不一致，但最终会收敛到一致状态
- **会话一致性**：同一会话内保证读取到自己写入的数据

实际工程中大多选择最终一致性，因为它在性能和一致性之间取得了较好平衡。

### Cache-Aside 模式（旁路缓存）

Cache-Aside 是最常用的缓存访问模式，应用代码同时管理缓存和数据库：

**读取流程：**
1. 先查缓存，命中则直接返回
2. 缓存未命中，查询数据库
3. 将数据库结果写入缓存
4. 返回结果

**写入流程：**
1. 先更新数据库
2. 再删除缓存（而非更新缓存）

**为什么是删除缓存而非更新缓存？**
- 更新缓存在并发场景下可能出现时序问题：线程 A 更新缓存值为 1，线程 B 更新缓存值为 2，但数据库写入顺序可能是 B 先 A 后，导致缓存值（1）与数据库值（旧值）不一致
- 删除缓存将重建工作交给下次读取，避免了并发更新的竞态条件

**为什么先更新数据库再删除缓存？**
- 先删缓存再更新数据库存在更大不一致窗口：删除缓存后、数据库更新前，另一个读请求会从数据库读到旧值并回填缓存
- 先更新数据库再删除缓存的不一致窗口极小，需要在数据库更新完成到缓存删除之间的极短时间窗口内，另一个读请求恰好缓存未命中并读到旧值

### Write-Through 模式（写穿透）

Write-Through 要求缓存层作为写入的唯一入口，所有写操作先写缓存，由缓存层同步写入数据库：

**写入流程：**
1. 应用写入缓存
2. 缓存层同步写入数据库
3. 两者都成功后才返回成功

**优点：**
- 数据一致性好，缓存与数据库始终同步
- 应用代码简单，只需操作缓存

**缺点：**
- 写入延迟较高，因为需要等待数据库写入完成
- 缓存层需要承担同步写数据库的责任，增加了缓存组件的复杂度

### Write-Behind 模式（写回/异步写穿透）

Write-Behind 是 Write-Through 的异步变体：

**写入流程：**
1. 应用写入缓存
2. 立即返回成功
3. 缓存层异步批量写入数据库

**优点：**
- 写入延迟极低，接近纯缓存操作的性能
- 支持批量写入，减少数据库压力

**缺点：**
- 数据可能丢失：缓存宕机时未同步到数据库的数据会丢失
- 一致性保证最弱，适合可容忍短暂不一致的场景（如计数器、浏览量）

### 双写问题（Dual-Write Problem）

双写问题是分布式缓存一致性中最棘手的场景。当需要同时更新缓存和数据库时：

**问题本质：** 两个独立系统的写入无法原子化，存在中间状态。

**常见失败场景：**
- 更新数据库成功，删除缓存失败 → 缓存脏数据
- 删除缓存成功，更新数据库失败 → 数据库旧数据，缓存无数据（下次回填旧值）
- 两者都成功，但顺序被打乱 → 并发下的时序不一致

**解决方案：**

#### 方案一：延迟双删
```
1. 先更新数据库
2. 删除缓存
3. 延迟一段时间（如 500ms）
4. 再次删除缓存
```
第二次删除是为了清理在步骤 1-2 之间被其他请求回填的旧缓存。延迟时间需要大于一次读请求的耗时。

#### 方案二：基于消息队列的可靠删除
```
1. 更新数据库
2. 将缓存删除操作发送到消息队列
3. 消费者处理消息，删除缓存
4. 删除失败则重试
```
借助消息队列的可靠性保证最终删除成功。

#### 方案三：基于 Binlog 的异步同步（CDC）
```
1. 应用只更新数据库
2. 监听数据库 Binlog（如使用 Canal、Debezium）
3. Binlog 解析服务消费变更事件
4. 异步删除/更新缓存
```
这是最解耦的方案，应用无需关心缓存操作，适合微服务架构。

### 缓存失效策略

#### TTL（Time-To-Live）过期
- 为缓存设置过期时间，到期自动失效
- 简单可靠，但存在数据不一致的时间窗口
- 适合对一致性要求不高的场景

#### 主动失效（Active Invalidation）
- 数据变更时主动删除或更新缓存
- 需要可靠的失效机制（如消息队列 + 重试）
- 适合对一致性要求较高的场景

#### 版本号/时间戳校验
- 缓存数据携带版本号或时间戳
- 读取时校验版本，过期则重新加载
- 可与 TTL 结合使用

### 缓存穿透、击穿、雪崩

这三个问题是缓存系统必须面对的经典挑战：

- **缓存穿透**：查询不存在的数据，缓存永远未命中，请求直达数据库。解决方案：布隆过滤器、缓存空值
- **缓存击穿**：热点 key 过期瞬间，大量并发请求同时击穿到数据库。解决方案：互斥锁（singleflight）、永不过期 + 异步更新
- **缓存雪崩**：大量 key 同时过期或缓存服务宕机，请求全部打到数据库。解决方案：TTL 加随机偏移、多级缓存、熔断降级

## 实战案例

来自博客文章：[分布式缓存一致性实战：Cache-Aside、Write-Through、Write-Behind 在 Laravel 中的工程化落地](/2026/06/01/分布式缓存一致性实战-Cache-Aside-Write-Through-Write-Behind在Laravel中的工程化落地/)

### Laravel 中的 Cache-Aside 实现

在 Laravel 中实现 Cache-Aside 模式，通常结合 Repository 模式：

```php
class ProductRepository
{
    public function findById(int $id): ?Product
    {
        return Cache::remember("product:{$id}", 3600, function () use ($id) {
            return Product::find($id);
        });
    }

    public function update(int $id, array $data): bool
    {
        $result = DB::transaction(function () use ($id, $data) {
            return Product::where('id', $id)->update($data);
        });

        if ($result) {
            Cache::forget("product:{$id}");
        }

        return $result;
    }
}
```

### Laravel 中的 Write-Behind 实现

利用 Laravel Queue 实现异步写回：

```php
// 写入时立即更新缓存，异步同步数据库
class WriteBehindCache
{
    public function put(string $key, mixed $value, int $ttl = 3600): void
    {
        Cache::put($key, $value, $ttl);
        WriteBehindJob::dispatch($key, $value);
    }
}

// 队列任务异步写入数据库
class WriteBehindJob implements ShouldQueue
{
    public function handle(): void
    {
        // 批量写入数据库
        DB::table('cache_writes')->insert($this->batch);
    }
}
```

### 基于 Canal + Laravel 的 Binlog 同步

在微服务架构中，使用 Canal 监听 MySQL Binlog，通过消息队列通知 Laravel 应用失效缓存：

```
MySQL → Canal → Kafka/RabbitMQ → Laravel Consumer → Redis DEL
```

## 常见问题

### Q: Cache-Aside 为什么推荐删除缓存而不是更新缓存？
在并发场景下，更新缓存可能导致时序不一致。线程 A 和线程 B 同时更新同一数据，可能因为执行顺序不同导致缓存值与数据库值不一致。删除缓存将重建工作交给下次读取，避免了这个问题。

### Q: 先更新数据库还是先删除缓存？
推荐先更新数据库再删除缓存。先删缓存再更新数据库的不一致窗口更大：删缓存后、更新数据库前的读请求会从数据库读到旧值并回填缓存，导致缓存长期持有脏数据。

### Q: 删除缓存失败怎么办？
可以使用延迟双删、消息队列可靠重试、或基于 Binlog 的 CDC 方案。其中 Binlog 方案解耦最彻底，应用无需关心缓存操作的成败。

### Q: 如何选择 Cache-Aside、Write-Through、Write-Behind？
- **Cache-Aside**：最通用，适合读多写少的场景
- **Write-Through**：适合对一致性要求高、写入延迟可接受的场景
- **Write-Behind**：适合写入密集、可容忍短暂不一致的场景（如计数器、日志）

### Q: 多级缓存（L1 + L2）如何保证一致性？
通常 L1 为本地缓存（如 PHP APCu），L2 为分布式缓存（如 Redis）。失效时需要同时通知所有节点的 L1 缓存失效，可通过消息广播（如 Redis Pub/Sub）实现。

### Q: 缓存一致性如何与 DDD 结合？
在 DDD 中，Repository 是聚合根的持久化抽象。可以在 Repository 实现中封装缓存逻辑，保持领域层对缓存无感知。领域事件可以触发缓存失效，实现关注点分离。
