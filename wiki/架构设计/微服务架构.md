# 微服务架构（Microservices Architecture）

## 定义

微服务架构将单体应用拆分为**一组小型、自治的服务**，每个服务围绕一个业务能力构建，独立开发、部署和扩缩容。服务间通过轻量级协议（HTTP/gRPC/消息队列）通信。

## 核心原理

### 何时该拆？决策矩阵

| 信号 | 指标 | 阈值 |
|------|------|------|
| **部署耦合** | 单次部署平均影响模块数 | > 5 个模块 |
| **团队扩展瓶颈** | Git merge conflict 每周 | > 15 次 |
| **性能热点隔离** | 单模块 CPU/内存占用 | > 总量 60% |

### 不该拆的反信号

| 反信号 | 真实案例 | 后果 |
|--------|---------|------|
| 「微服务是潮流」 | 项目初期就拆了 8 个服务 | 2 人团队维护不过来 |
| 「单体太慢了」 | 实际瓶颈在 N+1 查询 | 拆分后问题依然存在 |
| 「想要独立部署」 | 其实只需要更好的 CI/CD | K8s 运维成本远超预期 |

**经验法则：团队 < 5 人，且没有弹性扩缩需求，先把单体写好。**

### 服务边界识别

用 **Event Storming** 识别 Bounded Context，而非凭直觉划分：

```
订单上下文          支付上下文          库存上下文
├── Order           ├── Payment         ├── Inventory
├── OrderItem       ├── Transaction     ├── Stock
├── OrderStatus     ├── Refund          ├── Warehouse
└── 事件:           └── 事件:           └── 事件:
    OrderCreated        PaymentCompleted    StockReserved
    OrderCancelled      RefundIssued        StockAdjusted
```

### 通信模式对比

| 模式 | 协议 | 适用场景 | 优缺点 |
|------|------|---------|--------|
| **同步 HTTP** | REST/JSON | 简单查询、实时响应 | 简单但有级联超时风险 |
| **gRPC** | Protobuf | 高性能内部调用 | 高效但调试困难 |
| **消息队列** | RabbitMQ/Kafka | 异步解耦、事件驱动 | 可靠但增加复杂度 |
| **事件总线** | Kafka/NATS | 领域事件广播 | 松耦合但顺序难保证 |

### 渐进式演进策略

```
阶段一：模块化单体
  单体 Laravel + 清晰的模块边界（app/Modules/）

阶段二：提取 BFF 层
  单体 + Laravel BFF 聚合层

阶段三：核心服务拆分
  BFF + 订单服务 + 支付服务 + 库存服务

阶段四：全面微服务
  API Gateway + 服务注册发现 + 链路追踪
```

## 常见问题

**Q: 微服务用 Monorepo 还是 Polyrepo？**
A: 小团队（< 10 人）推荐 Monorepo（代码复用、原子提交）；大团队推荐 Polyrepo（独立版本、独立 CI/CD）。

**Q: 服务间用 HTTP 还是 gRPC？**
A: 对外用 HTTP REST（兼容性好），对内用 gRPC（性能高、类型安全）。

**Q: 怎么处理分布式事务？**
A: 优先避免分布式事务（合并聚合）；必须时用 Saga 模式保证最终一致性。详见 [分布式事务](分布式事务.md)。
