# 实时通信方案

## 定义

实时通信方案是指在客户端与服务器之间建立低延迟、双向或单向数据传输通道的技术体系。在现代 Web 架构中，实时通信是构建即时通知、在线协作、实时数据仪表盘、聊天系统、AI 流式输出等场景的核心基础设施。

常见的实时通信技术包括：

- **SSE（Server-Sent Events）**：基于 HTTP 的单向服务器推送协议
- **WebSocket**：基于 TCP 的全双工双向通信协议
- **HTTP Streaming**：利用 HTTP 分块传输编码实现流式数据推送
- **Long Polling**：客户端轮询的降级方案
- **gRPC Streaming**：基于 HTTP/2 的双向流式 RPC

在 Laravel BFF（Backend for Frontend）架构中，选择合适的实时通信方案直接影响系统复杂度、运维成本和用户体验。

## 核心原理

### 1. SSE（Server-Sent Events）

SSE 是 W3C 标准化的服务器推送技术，基于 HTTP/1.1 或 HTTP/2 的长连接。

**协议特点：**
- 基于标准 HTTP 协议，无需特殊代理配置
- 单向通信：仅服务器向客户端推送
- 内置自动重连机制（`retry` 字段）
- 文本协议，数据格式为 `data: ...\n\n`
- 支持事件类型（`event:` 字段）和事件 ID（`id:` 字段）
- 最大并发连接数受浏览器限制（通常 6 个/域名）

**工作流程：**
```
客户端 --GET /events--> 服务器
客户端 <--Content-Type: text/event-stream-- 服务器
客户端 <--data: {"message":"hello"}\n\n-- 服务器
客户端 <--data: {"message":"world"}\n\n-- 服务器
...（持续推送直到连接关闭）
```

**在 Laravel 中的实现：**
```php
// Laravel SSE 响应
Route::get('/events', function () {
    return response()->stream(function () {
        while (true) {
            $data = Redis::blPop('notifications', 0);
            echo "data: " . json_encode($data) . "\n\n";
            ob_flush();
            flush();
        }
    }, 200, [
        'Content-Type' => 'text/event-stream',
        'Cache-Control' => 'no-cache',
        'Connection' => 'keep-alive',
        'X-Accel-Buffering' => 'no', // Nginx 禁用缓冲
    ]);
});
```

**适用场景：**
- AI/LLM 流式输出（ChatGPT 风格的逐字输出）
- 实时通知推送
- 股票行情、体育比分更新
- 构建进度展示
- 日志流式查看

### 2. WebSocket

WebSocket 是在单个 TCP 连接上提供全双工通信的协议，通过 HTTP Upgrade 握手建立连接。

**协议特点：**
- 全双工通信：客户端和服务器可随时互发消息
- 二进制和文本数据均支持
- 帧协议开销小（2-14 字节头部）
- 需要专门的服务器支持（不能使用传统 HTTP 服务器模型）
- 需要处理心跳保活（Ping/Pong）
- 需要处理连接状态管理

**握手过程：**
```
客户端 --> GET /chat HTTP/1.1
           Upgrade: websocket
           Connection: Upgrade
           Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
服务器 <-- HTTP/1.1 101 Switching Protocols
           Upgrade: websocket
           Connection: Upgrade
           Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=
```

**在 Laravel 中的实现：**
```php
// 使用 Laravel WebSocket（beyondcode/laravel-websockets）
// 或使用 Pusher 协议

// 广播事件
class OrderStatusUpdated implements ShouldBroadcast
{
    public function broadcastOn(): array
    {
        return [new PrivateChannel('orders.' . $this->orderId)];
    }

    public function broadcastAs(): string
    {
        return 'order.status.updated';
    }
}

// 客户端监听
// Echo.private(`orders.${orderId}`)
//     .listen('.order.status.updated', (e) => { ... });
```

**适用场景：**
- 在线聊天系统
- 多人协作编辑（如 Google Docs）
- 在线游戏
- 实时白板/画布协作
- 需要客户端频繁向服务器发送数据的场景

### 3. HTTP Streaming

HTTP Streaming 利用 HTTP 分块传输编码（Chunked Transfer Encoding）实现流式数据推送。

**协议特点：**
- 基于标准 HTTP 协议
- 使用 `Transfer-Encoding: chunked`
- 支持 HTTP/1.1 和 HTTP/2
- 与 SSE 类似但更底层，没有内置事件格式
- 可传输二进制数据

**与 SSE 的区别：**

| 特性 | SSE | HTTP Streaming |
|------|-----|---------------|
| 协议标准 | W3C 标准 | HTTP 传输特性 |
| 数据格式 | 纯文本，`data:` 前缀 | 任意格式 |
| 自动重连 | 内置 `retry` | 需手动实现 |
| 事件类型 | 支持 `event:` 字段 | 不支持 |
| 浏览器 API | `EventSource` API | `fetch` + `ReadableStream` |
| 二进制支持 | 不支持 | 支持 |

### 4. 选型决策矩阵

| 评估维度 | SSE | WebSocket | HTTP Streaming |
|----------|-----|-----------|---------------|
| 通信方向 | 服务器→客户端 | 双向 | 服务器→客户端 |
| 协议基础 | HTTP | WebSocket（独立协议） | HTTP |
| 防火墙/代理友好度 | ★★★★★ | ★★★ | ★★★★★ |
| CDN/负载均衡兼容性 | ★★★★★ | ★★★ | ★★★★ |
| 浏览器原生支持 | EventSource API | WebSocket API | fetch + ReadableStream |
| 自动重连 | 内置 | 需手动 | 需手动 |
| 二进制传输 | ❌ | ✅ | ✅ |
| 连接开销 | 低 | 低 | 低 |
| 移动端后台保活 | 较差 | 较差 | 较差 |
| Laravel 生态支持 | ★★★★ | ★★★ | ★★★ |
| 实现复杂度 | 低 | 高 | 中 |
| 扩展性（水平） | 中等 | 较难 | 中等 |

### 5. 工程选型指南

**选择 SSE 的场景：**
- 单向数据推送（服务器→客户端）
- 需要与现有 HTTP 基础设施无缝集成
- 需要通过 CDN 缓存和代理
- 团队对 WebSocket 运维经验不足
- AI 流式输出场景（当前最主流选择）

**选择 WebSocket 的场景：**
- 需要双向实时通信
- 客户端需要频繁发送小消息
- 低延迟要求极高（< 50ms）
- 已有成熟的 WebSocket 基础设施

**选择 HTTP Streaming 的场景：**
- 需要传输二进制流数据
- 需要更精细的流控制
- 与 SSE 类似但需要自定义数据格式

### 6. Laravel BFF 架构中的实践

在 BFF 架构中，实时通信方案通常作为 BFF 层的一个端点暴露给前端：

```
[前端 SPA/Mobile] <--SSE/WS--> [Laravel BFF] <--> [微服务集群]
                                    |
                                    ├── 通知服务（SSE）
                                    ├── 聊天服务（WebSocket）
                                    └── 日志流（HTTP Streaming）
```

**Laravel Reverb（官方 WebSocket 服务器）：**
- Laravel 11+ 推出的官方 WebSocket 解决方案
- 替代 Pusher 的自托管方案
- 支持 Pusher 协议兼容
- 与 Laravel Echo 无缝集成

**SSE 在 Laravel 中的工程化封装：**
```php
class SseStream
{
    private Generator $dataSource;
    private int $retryInterval;

    public function __construct(Generator $dataSource, int $retryInterval = 3000)
    {
        $this->dataSource = $dataSource;
        $this->retryInterval = $retryInterval;
    }

    public function response(): StreamedResponse
    {
        return response()->stream(function () {
            echo "retry: {$this->retryInterval}\n\n";

            foreach ($this->dataSource as $event) {
                if (connection_aborted()) break;

                if (isset($event['id'])) {
                    echo "id: {$event['id']}\n";
                }
                if (isset($event['event'])) {
                    echo "event: {$event['event']}\n";
                }

                $data = is_string($event['data']) 
                    ? $event['data'] 
                    : json_encode($event['data']);

                foreach (explode("\n", $data) as $line) {
                    echo "data: {$line}\n";
                }
                echo "\n";

                ob_flush();
                flush();
            }
        }, 200, [
            'Content-Type' => 'text/event-stream',
            'Cache-Control' => 'no-cache',
            'X-Accel-Buffering' => 'no',
        ]);
    }
}
```

### 7. 水平扩展挑战

**SSE 扩展方案：**
- 使用 Redis Pub/Sub 作为消息总线
- 每个 SSE 进程订阅 Redis 频道
- 负载均衡器使用 IP Hash 或 Sticky Session
- 或使用专门的推送服务（如 Pusher、Ably）

**WebSocket 扩展方案：**
- 使用 Redis 作为跨节点消息广播
- Sticky Session 保证连接一致性
- 使用专门的 WebSocket 网关（如 Soketi、Laravel Reverb）

## 实战案例

### 案例一：AI 流式输出（SSE）

**场景描述：** 集成 OpenAI/Claude API，实现 ChatGPT 风格的逐字输出效果。

**方案选择：** SSE，因为是单向推送（服务器→客户端），且需要与 HTTP 基础设施兼容。

**实现要点：**
```php
Route::post('/chat/stream', function (Request $request) {
    $messages = $request->input('messages');
    
    return response()->stream(function () use ($messages) {
        $stream = OpenAI::chat()->createStreamed([
            'model' => 'gpt-4',
            'messages' => $messages,
        ]);
        
        foreach ($stream as $response) {
            $content = $response->choices[0]->delta->content ?? '';
            echo "data: " . json_encode(['content' => $content]) . "\n\n";
            ob_flush();
            flush();
        }
        
        echo "data: [DONE]\n\n";
        ob_flush();
        flush();
    }, 200, [
        'Content-Type' => 'text/event-stream',
        'Cache-Control' => 'no-cache',
        'X-Accel-Buffering' => 'no',
    ]);
});
```

### 案例二：实时通知系统（SSE + Redis Pub/Sub）

**场景描述：** 为多个微服务提供统一的实时通知推送能力。

**架构设计：**
```
[业务服务] --> Redis Pub/Sub --> [Laravel SSE Worker] --> [客户端浏览器]
                                         |
                                    [Laravel Horizon 监控]
```

### 案例三：在线客服系统（WebSocket）

**场景描述：** 客户与客服之间的双向实时聊天。

**方案选择：** WebSocket，因为需要双向通信，客户端和服务器都需要主动发送消息。

## 常见问题

### Q1: SSE 和 WebSocket 哪个更适合 AI 流式输出？

**SSE 更适合。** 原因：
1. AI 流式输出是典型的单向推送场景（服务器→客户端）
2. SSE 基于标准 HTTP，无需额外基础设施
3. 浏览器原生 `EventSource` API 内置自动重连
4. 与 CDN、反向代理（Nginx、Cloudflare）完全兼容
5. ChatGPT、Claude 等主流 AI 产品均使用 SSE

### Q2: SSE 在 Nginx 反向代理下为什么需要特殊配置？

Nginx 默认会缓冲响应体，导致 SSE 消息无法实时推送。需要：
```nginx
location /events {
    proxy_buffering off;
    proxy_cache off;
    proxy_http_version 1.1;
    proxy_set_header Connection '';
    chunked_transfer_encoding off;
}
```

### Q3: WebSocket 连接断开后如何自动恢复？

需要在客户端实现重连逻辑：
```javascript
function connectWithRetry(url, maxRetries = 5) {
    let retries = 0;
    const connect = () => {
        const ws = new WebSocket(url);
        ws.onclose = () => {
            if (retries < maxRetries) {
                retries++;
                setTimeout(connect, Math.pow(2, retries) * 1000); // 指数退避
            }
        };
        ws.onopen = () => { retries = 0; };
    };
    connect();
}
```

### Q4: Laravel Octane 环境下 SSE 有什么注意事项？

Laravel Octane 使用长驻进程，在 Octane 下实现 SSE 时需要注意：
1. 不能使用 `sleep()` 阻塞整个进程
2. 需要使用协程或异步 IO（Swoole 环境下使用 `co::sleep`）
3. 建议将 SSE 端点与 Octane 主进程分离
4. 考虑使用独立的 WebSocket 服务器（如 Laravel Reverb）

### Q5: 如何在负载均衡环境下保证 SSE/WebSocket 连接的一致性？

1. **Sticky Session**：负载均衡器将同一客户端的请求路由到同一后端
2. **Redis 广播**：所有后端实例共享消息状态，任意实例均可服务连接
3. **独立推送网关**：将实时通信层从应用服务器中分离，使用专门的推送服务
4. **IP Hash 负载均衡**：基于客户端 IP 的一致性路由

### Q6: SSE 连接数有浏览器限制，如何解决？

浏览器对同一域名的 SSE 连接数限制为 6 个（HTTP/1.1）。解决方案：
1. 使用 HTTP/2（连接限制为每个连接 100 个流）
2. 使用多域名分片（events1.example.com, events2.example.com）
3. 合并多个事件流到单个连接
4. 使用 WebSocket 替代（无此限制）

### Q7: 移动端应用如何处理实时通信的后台保活？

移动操作系统会在应用进入后台时限制网络连接：
1. **iOS**：后台连接可能在 30 秒内被关闭
2. **Android**：Doze 模式会限制网络访问
3. **推荐方案**：使用 APNs/FCM 推送通知保底，前台时使用 SSE/WebSocket
4. **混合方案**：前台实时通道 + 后台静默推送

### Q8: 在微服务架构中，如何设计跨服务的实时事件传播？

```
[服务A] --事件--> [事件总线/RabbitMQ/Kafka] --> [推送网关] --> [客户端]
[服务B] --事件--> [事件总线/RabbitMQ/Kafka] --> [推送网关] --> [客户端]
```

关键设计：
1. 使用统一的事件格式（CloudEvents 规范）
2. 推送网关订阅所有需要实时推送的事件主题
3. 推送网关维护客户端订阅关系（哪些客户端关心哪些事件）
4. 使用 Redis 或内存缓存维护在线用户列表
