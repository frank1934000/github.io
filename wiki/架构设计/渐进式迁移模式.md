# 渐进式迁移（Strangler Fig Pattern）

## 定义

绞杀者无花果模式（Strangler Fig Pattern）是一种**将遗留系统逐步替换为新系统**的架构迁移策略。得名于热带雨林中绞杀榕缠绕宿主树木、逐渐取代其结构的生长方式。在软件领域，新系统像藤蔓一样逐步包裹旧系统，通过代理层将流量从旧系统路由到新系统，直到旧系统被完全替换。

## 核心原理

### 迁移三阶段

```
阶段一：共存（Coexist）
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  Client  │────>│  Proxy/Facade│────>│ Legacy System│
└──────────┘     └──────────────┘     └──────────────┘
                      │
                      │ (新功能)
                      ▼
                 ┌──────────────┐
                 │  New Service  │
                 └──────────────┘

阶段二：蚕食（Migrate）
┌──────────┐     ┌──────────────┐     ┌──────────────┐
│  Client  │────>│  Proxy/Facade│─X─>│ Legacy System│ (流量减少)
└──────────┘     └──────────────┘     └──────────────┘
                      │
                      ▼ (更多模块迁移)
                 ┌──────────────┐
                 │  New Services │
                 └──────────────┘

阶段三：退役（Retire）
┌──────────┐     ┌──────────────┐
│  Client  │────>│  New Services │
└──────────┘     └──────────────┘
                 (Legacy 已完全退役)
```

### Anti-Corruption Layer (ACL)

新系统不应被旧系统的数据模型"污染"。ACL 是新旧系统之间的翻译层：

```php
// Anti-Corruption Layer：旧系统 DTO → 新领域模型
class OrderACLayer
{
    public function translateFromLegacy(array $legacyOrder): OrderAggregate
    {
        return new OrderAggregate(
            orderId: OrderId::fromLegacy($legacyOrder['order_id']),
            customer: CustomerRef::fromLegacyCrm($legacyOrder['cust_no']),
            items: collect($legacyOrder['items'])->map(
                fn($item) => OrderItem::fromLegacy($item)
            )->all(),
            // 旧系统用状态码 1/2/3，新系统用枚举
            status: OrderStatus::fromLegacyCode($legacyOrder['status_code']),
        );
    }
}
```

### 事件驱动双轨策略

在迁移期间，新旧系统可能需要同时处理同一业务事件：

```
Legacy System ──→ 旧事件总线 ──→ Legacy Handler
      │
      └──→ ACL 适配器 ──→ 新事件总线 ──→ New Handler
                                        (新系统并行处理)
```

```php
// 双写期间的事件适配
class OrderCreatedAdapter
{
    public function handle(LegacyOrderCreated $event): void
    {
        // 转换为新领域事件
        $newEvent = new OrderCreatedV2(
            orderId: OrderId::fromLegacy($event->legacyOrderId),
            total: Money::fromLegacy($event->amount, $event->currency),
            items: $this->acl->translateItems($event->items),
        );

        // 发布到新事件总线
        EventBus::dispatch($newEvent);
    }
}
```

## 实战策略

### 1. 按功能模块迁移

不要尝试一次性重写整个系统。按业务模块逐个迁移：

```
✅ 推荐：先迁移"用户通知"模块（独立、低风险）
         再迁移"订单管理"模块（核心但边界清晰）
         最后迁移"支付结算"模块（最复杂、最敏感）

❌ 避免：同时迁移所有模块
```

### 2. 路由层设计

```nginx
# Nginx 路由规则示例
location /api/v2/orders {
    proxy_pass http://new-order-service;  # 新系统
}
location /api/v1/orders {
    proxy_pass http://legacy-system;       # 旧系统
}
location /api/v1/inventory {
    # 渐进式切换：10% 流量到新系统
    split_clients "${request_uri}" $backend {
        10%  http://new-inventory-service;
        *    http://legacy-system;
    }
    proxy_pass $backend;
}
```

### 3. 数据同步策略

迁移期间新旧系统可能共享数据库或需要数据同步：

| 策略 | 适用场景 | 复杂度 |
|------|---------|--------|
| 共享数据库 | 短期共存 | 低 |
| CDC 双向同步 | 长期共存 | 高 |
| 写新读旧 | 新系统先接写入 | 中 |
| 双写+对账 | 关键业务 | 高 |

## 常见问题

**Q: 旧系统一直有新需求怎么办？**
A: 迁移期间旧系统仍然需要维护。建议设置"冻结期"——迁移中的模块不再添加新功能，新功能只在新系统开发。

**Q: 如何判断迁移完成？**
A: 当旧系统的某个模块流量降为零，且数据已完全迁移到新系统，该模块就可以退役。通过监控确认无流量后再下线。

**Q: 回滚策略是什么？**
A: 代理层可以随时将流量切回旧系统。这就是为什么代理层如此重要——它是迁移的安全网。
