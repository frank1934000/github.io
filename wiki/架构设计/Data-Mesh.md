# 数据网格（Data Mesh）

## 定义

数据网格（Data Mesh）是一种**去中心化的数据架构范式**，由 Zhamak Dehghani 提出。核心思想是将数据所有权从中心化数据团队转移给各业务领域团队，每个领域团队负责自己的数据产品（Data Product），通过自助数据平台和联邦治理实现跨域数据协作。

## 核心原理

### 1. 四大原则

| 原则 | 说明 |
|------|------|
| **领域数据所有权** | 每个业务领域（如订单、用户、商品）拥有并管理自己的数据 |
| **数据即产品** | 领域团队将数据视为产品，提供标准化接口、SLA、文档 |
| **自助数据平台** | 提供基础设施即代码，让领域团队自主构建数据管道 |
| **联邦计算治理** | 全局标准（命名、安全、互操作性）由跨领域团队共同制定 |

### 2. 与传统数据架构对比

```
传统模式（中心化）：
  业务系统 → ETL → 数据仓库 → BI 报表
                ↑
          中心化数据团队管理

Data Mesh（去中心化）：
  订单域 ──→ 订单数据产品 ──┐
  用户域 ──→ 用户数据产品 ──┼──→ 消费者（分析/AI/其他域）
  商品域 ──→ 商品数据产品 ──┘
        ↑                    ↑
  领域团队自治          自助平台 + 联邦治理
```

### 3. 数据产品的核心特征

一个合格的数据产品应具备：

- **可发现**：在数据目录中注册，有元数据描述
- **可寻址**：有稳定的端点地址（API/Topic/表）
- **可信任**：有 SLA、数据质量检查、血缘追踪
- **可互操作**：遵循全局命名和格式标准
- **自描述**：包含 Schema、文档、示例
- **安全**：访问控制、脱敏、审计

### 4. 实施路径

```
Phase 1: 识别领域边界
  └── DDD Bounded Context → 数据域划分

Phase 2: 构建自助平台
  └── 数据管道模板 / Schema 注册 / 目录服务 / 质量检查

Phase 3: 试点数据产品
  └── 选择 1-2 个域，端到端构建数据产品

Phase 4: 联邦治理
  └── 全局命名规范 / 安全策略 / 互操作标准

Phase 5: 规模推广
  └── 培训、工具、文化变革
```

### Laravel 微服务落地

1. **领域数据产品 API**：每个 Laravel 微服务暴露标准化的数据产品 API（REST/GraphQL）
2. **Schema 注册**：使用 Confluent Schema Registry 或自建 Schema 服务管理数据契约
3. **事件驱动同步**：通过 Kafka/NATS 发布领域事件，下游消费者订阅
4. **数据目录**：集成 DataHub/OpenMetadata 实现数据产品可发现
5. **质量检查**：在数据管道中嵌入 Great Expectations 或自定义规则

## 常见问题

### Q: Data Mesh 适合什么规模的团队？
A: 通常适合 50+ 人的数据组织。小团队用中心化数据仓库更高效。

### Q: Data Mesh 会取代数据仓库吗？
A: 不会完全取代。Data Mesh 更适合跨域协作场景，数据仓库在单一域内仍有价值。两者可共存。

### Q: 如何保证数据质量？
A: 通过数据产品 SLA + 自动化质量检查 + 数据目录中的评分体系。质量不达标的产品会被标记为"不可信"。

### Q: 与数据湖（Data Lake）的关系？
A: Data Lake 是存储层，Data Mesh 是组织+架构层。Data Mesh 可以构建在 Data Lake 之上，也可以用其他存储。
