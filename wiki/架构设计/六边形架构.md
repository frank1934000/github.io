# 六边形架构（Hexagonal Architecture）

## 定义

六边形架构（又称 **Ports and Adapters**）是 Alistair Cockburn 提出的设计模式。核心思想是**依赖反转**：业务逻辑（核心域）不依赖任何外部设施，而是通过「端口」定义契约，由「适配器」提供具体实现。

## 核心原理

### 架构图

```
                    ┌─────────────────────────────────┐
                    │         Driving Adapters         │
                    │  (HTTP Controller / CLI / gRPC)  │
                    └──────────┬──────────────────────┘
                               │ 调用
                    ┌──────────▼──────────────────────┐
                    │          Input Ports             │
                    │   (Use Case / Service Interface) │
                    └──────────┬──────────────────────┘
                               │
                    ┌──────────▼──────────────────────┐
                    │        Domain Layer (核心域)      │
                    │  ├── Entities (实体)              │
                    │  ├── Value Objects (值对象)       │
                    │  ├── Domain Services (领域服务)   │
                    │  └── Domain Events (领域事件)     │
                    └──────────┬──────────────────────┘
                               │ 依赖反转
                    ┌──────────▼──────────────────────┐
                    │         Output Ports             │
                    │    (Repository Interface)         │
                    └──────────┬──────────────────────┘
                               │ 实现
                    ┌──────────▼──────────────────────┐
                    │       Driven Adapters            │
                    │  (Eloquent / Redis / S3 / Stripe)│
                    └─────────────────────────────────┘
```

### 核心思想

- **端口（Port）**：核心域定义的接口契约（如 `OrderRepositoryInterface`）
- **适配器（Adapter）**：端口的具体实现（如 `EloquentOrderRepository`）
- **依赖方向**：外部依赖核心域，而非核心域依赖外部

### Driving vs Driven

| 类型 | 方向 | 示例 |
|------|------|------|
| **Driving Adapter** | 调用核心域 | HTTP Controller、CLI、gRPC Server |
| **Driven Adapter** | 被核心域调用 | 数据库、Redis、外部 API、消息队列 |

### 传统 MVC vs 六边形架构

```
传统 Laravel MVC：
  Controller → Service → Eloquent Model → MySQL
  问题：Service 直接依赖 Eloquent，无法独立测试

六边形架构：
  Controller → UseCase (Port) → Domain → Repository (Port)
                                                ↑
                                        Eloquent Adapter
  优势：Domain 不依赖 Eloquent，可用 Mock 测试
```

### 与 DDD 的关系

六边形架构是 DDD 基础设施层的**具体实现方案**：

- DDD 的 Repository Interface = 六边形架构的 Output Port
- DDD 的 Application Service = 六边形架构的 Input Port
- DDD 的 Infrastructure Layer = 六边形架构的 Driven Adapter

### 依赖反转的三大好处

| 好处 | 说明 |
|------|------|
| **可测试** | 核心域不依赖基础设施，Mock 即可测试 |
| **可替换** | 换数据库/支付网关只需换 Adapter，核心域不变 |
| **可独立** | 核心域独立于框架，可移植到不同技术栈 |

## 常见问题

**Q: 六边形架构会增加代码量吗？**
A: 会。需要定义接口（Port）+ 实现（Adapter），但换来的是可测试性和可替换性。对于长期维护的项目，收益大于成本。

**Q: 所有代码都要用六边形架构吗？**
A: 不需要。CRUD 为主的小项目用传统 MVC 即可。业务复杂、需要频繁切换基础设施的项目才值得。

**Q: Laravel 中怎么落地？**
A: 核心域放 `app/Domains/`，Port 放 `app/Domains/*/Contracts/`，Adapter 放 `app/Infrastructure/`。
