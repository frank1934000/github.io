# 事件通知与状态传输（Event Notification vs Event-Carried State Transfer）

## 定义

事件驱动架构中有两种核心的事件模式，它们在**信息量**和**耦合程度**之间做出不同的权衡：

- **Event Notification（事件通知）**：事件仅携带"发生了什么"的最小信息，消费者需要回调源服务获取详细数据
- **Event-Carried State Transfer（事件携带状态传输）**：事件携带消费者所需的全部数据，消费者无需回调源服务

## 核心原理

### 对比图

```
Event Notification（事件通知）
┌──────────┐  order.created   ┌──────────┐
│  Order   │────────────────>│  Notify  │
│  Service │  {order_id: 42}  │  Service │
└──────────┘                  └────┬─────┘
      ▲                            │
      │  GET /orders/42            │ 需要详情
      └────────────────────────────┘

Event-Carried State Transfer（状态传输）
┌──────────┐  order.created          ┌──────────┐
│  Order   │─────────────────────>│  Notify  │
│  Service │  {order_id: 42,       │  Service │
└──────────┘   customer: "Alice",  └──────────┘
               items: [...],        (无需回调，数据自足)
               total: 299.00}
```

### 详细对比

| 维度 | Event Notification | Event-Carried State Transfer |
|------|-------------------|------------------------------|
| 事件大小 | 小（仅 ID + 事件类型） | 大（包含完整业务数据） |
| 服务耦合 | 高（消费者依赖源服务 API） | 低（消费者自足） |
| 源服务可用性 | 消费者依赖源服务在线 | 消费者可独立运行 |
| 数据新鲜度 | 实时（回调获取最新数据） | 事件时刻的快照 |
| 事件 Schema 稳定性 | 稳定（Schema 变化少） | 易变（业务字段变化频繁） |
| 网络开销 | 高（每次消费都回调） | 低（一次性传输） |
| 适用场景 | 消费者只需要通知 | 消费者需要完整数据 |

### 选型决策树

```
消费者需要什么？
├─ 仅需知道"事件发生了" → Event Notification
│  例：发送通知、触发审计日志
│
├─ 需要完整的业务数据 → Event-Carried State Transfer
│  例：搜索索引更新、报表聚合、缓存预热
│
└─ 需要实时最新数据 → Event Notification + 回调
   例：价格变更（需要最新价格）
```

## 实战案例

### Event Notification：订单创建通知

```php
// 事件：最小信息
class OrderCreated
{
    public function __construct(
        public readonly string $orderId,
        public readonly string $orderNo,
        public readonly string $eventType = 'order.created',
    ) {}
}

// 消费者：需要详情时回调
class NotificationHandler
{
    public function handle(OrderCreated $event): void
    {
        // 回调获取完整订单信息
        $order = $this->orderClient->getOrder($event->orderId);
        $customer = $this->customerClient->getCustomer($order->customerId);

        $this->notify($customer, "订单 {$order->orderNo} 已创建");
    }
}
```

### Event-Carried State Transfer：搜索索引更新

```php
// 事件：完整数据
class OrderCreatedWithState
{
    public function __construct(
        public readonly string $orderId,
        public readonly string $orderNo,
        public readonly string $customerName,
        public readonly string $customerEmail,
        public readonly array $items,
        public readonly float $totalAmount,
        public readonly string $currency,
        public readonly string $createdAt,
    ) {}
}

// 消费者：自足，无需回调
class SearchIndexHandler
{
    public function handle(OrderCreatedWithState $event): void
    {
        // 直接用事件中的数据更新搜索索引
        $this->searchIndex->index('orders', [
            'id' => $event->orderId,
            'order_no' => $event->orderNo,
            'customer' => $event->customerName,
            'items' => $event->items,
            'total' => $event->totalAmount,
            'created_at' => $event->createdAt,
        ]);
    }
}
```

### 混合模式

实际项目中，同一个事件总线上可以同时存在两种模式：

```php
// 订单聚合事件（Event-Carried State Transfer）
// → 发送给：搜索服务、报表服务、推荐服务
class OrderPlaced { /* 完整数据 */ }

// 订单通知事件（Event Notification）
// → 发送给：通知服务、审计服务
class OrderPlacedNotification { /* 仅 orderId + eventType */ }
```

## 常见问题

**Q: Event-Carried State Transfer 会导致事件 Schema 频繁变化吗？**
A: 是的，这是主要缺点。需要通过 Schema Registry 和契约测试来管理变更。建议事件 Schema 做向后兼容的演进。

**Q: 如果消费者只需要部分字段怎么办？**
A: 可以发送"精简版"状态传输事件，只包含消费者需要的字段。或者使用 GraphQL 风格的事件订阅，让消费者声明需要的字段。

**Q: 事件体积过大会影响性能吗？**
A: 会。建议控制事件大小（通常 < 1MB），大对象用引用（ID + URI）代替内联。Redis Stream 和 Kafka 都有消息大小限制。
