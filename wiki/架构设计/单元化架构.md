# 单元化架构（Cell-Based Architecture）

## 定义

单元化架构（Cell-Based Architecture）是一种将系统拆分为多个**自治单元（Cell）**的架构模式。每个 Cell 是一个独立的、自包含的服务集合，拥有自己的数据库、缓存、消息队列和完整业务能力。Cell 之间通过明确的路由层隔离，单个 Cell 故障不会扩散到其他 Cell。

## 核心原理

### 1. Cell 的定义

一个 Cell 不是微服务，而是**一组微服务的组合**，代表一个完整的业务能力单元：

```
┌─────────────────────────────────────┐
│              Router Layer            │
│  (用户路由 / 流量分配 / Cell 发现)    │
└──────┬──────────┬──────────┬────────┘
       │          │          │
  ┌────▼────┐ ┌───▼────┐ ┌──▼─────┐
  │ Cell A  │ │ Cell B │ │ Cell C │
  │─────────│ │────────│ │────────│
  │ Service │ │Service │ │Service │
  │ DB      │ │DB      │ │DB      │
  │ Cache   │ │Cache   │ │Cache   │
  │ MQ      │ │MQ      │ │MQ      │
  └─────────┘ └────────┘ └────────┘
```

### 2. 核心设计原则

| 原则 | 说明 |
|------|------|
| **故障隔离** | 单 Cell 故障不影响其他 Cell，爆炸半径可控 |
| **独立扩缩** | 每个 Cell 可独立水平扩展，按业务负载分配资源 |
| **数据隔离** | 每个 Cell 拥有独立数据存储，避免跨 Cell 数据竞争 |
| **路由驱动** | 通过 Cell Router 按用户 ID、地域、租户等维度分配流量 |
| **无跨 Cell 调用** | Cell 之间尽量同步调用，避免分布式事务 |

### 3. Cell 路由策略

```
用户请求 → Cell Router → 根据路由键分配到目标 Cell

路由键选择：
- 用户 ID % Cell 数量（最常见）
- 地域（亚太 → Cell-A，欧美 → Cell-B）
- 租户 ID（SaaS 多租户隔离）
- 业务类型（订单 Cell、内容 Cell、社交 Cell）
```

### 4. Cell 与微服务的区别

| 维度 | 微服务 | 单元化 |
|------|--------|--------|
| 粒度 | 单个服务 | 一组服务的组合 |
| 隔离 | 共享数据库/消息队列 | 每个 Cell 独立基础设施 |
| 故障影响 | 可能级联 | 限制在单个 Cell 内 |
| 扩缩 | 按服务扩缩 | 按 Cell 整体扩缩 |
| 数据 | 通常共享 | Cell 内闭环 |
| 复杂度 | 服务间依赖 | Cell 间依赖更少 |

## 实战案例

来自博客文章：[Cell-Based Architecture 实战：单元化架构在 Laravel 微服务中的落地——故障隔离、独立扩缩与跨单元路由](/2026/06/01/Cell-Based-Architecture-实战-单元化架构在Laravel微服务中的落地-故障隔离独立扩缩与跨单元路由/)

### Laravel 落地要点

1. **Cell Router 中间件**：在 Laravel 中实现路由中间件，按用户 ID 哈希分配到目标 Cell
2. **数据库隔离**：每个 Cell 独立的 MySQL 实例，通过 Laravel `config/database.php` 动态切换
3. **Redis 隔离**：每个 Cell 独立的 Redis 集群，避免 Key 冲突
4. **消息队列隔离**：每个 Cell 独立的队列 Worker，避免消息路由混乱
5. **跨 Cell 查询**：通过 API Gateway 聚合，避免直接数据库跨 Cell 查询

### 适用场景

- **多租户 SaaS**：每个租户独立 Cell，数据物理隔离
- **电商大促**：按用户维度分流，单 Cell 故障不影响其他用户
- **全球化部署**：按地域部署 Cell，就近接入降低延迟
- **金融系统**：合规要求数据隔离，Cell 天然满足

## 常见问题

### Q: Cell 数量如何确定？
A: 初期 2-4 个 Cell 即可，根据业务增长逐步扩展。Cell 数量通常是 2 的幂次方便于哈希分配。

### Q: 跨 Cell 数据查询怎么办？
A: 三种方案：(1) 聚合层 API 查询；(2) 异步数据同步到全局只读副本；(3) 搜索引擎（ES）全局索引。

### Q: Cell 与 Kubernetes Namespace 的关系？
A: K8s Namespace 是部署层面的隔离，Cell 是业务层面的隔离。一个 Cell 可能对应多个 K8s Namespace，但通常 1:1 映射。

### Q: 如何处理 Cell 间的通信？
A: 尽量避免同步调用。必要时通过事件总线（Kafka/NATS）异步通信，或通过 API Gateway 中转。
