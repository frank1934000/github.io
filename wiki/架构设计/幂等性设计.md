# 幂等性设计（Idempotency）

## 定义

幂等性是指**同一个操作执行一次和执行多次的效果完全相同**。在分布式系统中，由于网络超时、消息重试、用户重复点击等原因，同一个请求可能被发送多次，幂等性设计确保这些重复请求不会导致数据不一致。

## 核心原理

### 为什么需要幂等

- **网络不可靠**：请求超时后客户端重试，服务端可能已处理但客户端不知道
- **消息队列重复投递**：At-Least-Once 语义下，消息可能被消费多次
- **用户重复点击**：前端防抖不够，同一笔订单可能被提交两次
- **分布式事务补偿**：Saga 模式的补偿操作可能被重复执行

### 三层防护体系

```
┌─────────────────────────────────────────────────┐
│  第一层：请求去重（Idempotency Key）               │
│  客户端生成唯一 Key → Redis SETNX → 已处理则返回缓存结果  │
├─────────────────────────────────────────────────┤
│  第二层：业务唯一约束（数据库层兜底）                 │
│  UNIQUE INDEX / 唯一业务号 → 重复插入直接报错          │
├─────────────────────────────────────────────────┤
│  第三层：分布式锁（互斥执行）                        │
│  Redis SET NX EX → 保证同一时刻只有一个请求在处理       │
└─────────────────────────────────────────────────┘
```

### Idempotency Key 机制

```
客户端                        服务端
  │                              │
  │  POST /orders                │
  │  Idempotency-Key: abc-123   │
  │ ────────────────────────────>│
  │                              │── Redis GET "idem:abc-123"
  │                              │   ├─ 存在 → 返回缓存结果
  │                              │   └─ 不存在 →
  │                              │      ├── Redis SETNX "idem:abc-123" "processing"
  │                              │      ├── 执行业务逻辑
  │                              │      ├── Redis SET "idem:abc-123" result (TTL)
  │                              │      └── 返回 result
  │ <────────────────────────────│
  │  201 Created (order_id=42)   │
  │                              │
  │  POST /orders（重试）         │
  │  Idempotency-Key: abc-123   │
  │ ────────────────────────────>│
  │                              │── Redis GET "idem:abc-123"
  │                              │   └─ 存在 → 返回缓存结果
  │ <────────────────────────────│
  │  201 Created (order_id=42)   │
```

## 实战案例

### 支付回调幂等

```php
// 支付网关回调可能多次到达
public function handlePaymentCallback(Request $request)
{
    $paymentNo = $request->input('payment_no');

    // 第一层：Redis 去重
    $lock = Cache::lock("payment_callback:{$paymentNo}", 30);
    if (!$lock->get()) {
        return response()->json(['message' => 'Processing']);
    }

    try {
        // 第二层：数据库状态检查
        $payment = Payment::where('payment_no', $paymentNo)->first();
        if ($payment->status === 'completed') {
            return response()->json(['message' => 'Already processed']);
        }

        // 执行业务逻辑
        $payment->update(['status' => 'completed']);
        event(new PaymentCompleted($payment));

        return response()->json(['message' => 'Success']);
    } finally {
        $lock->release();
    }
}
```

### Laravel 队列消息幂等

```php
// 实现 ShouldBeUnique 接口
class ProcessOrder implements ShouldQueue, ShouldBeUnique
{
    public $orderId;

    public function uniqueId(): string
    {
        return $this->orderId; // 同一订单只处理一次
    }

    public function uniqueFor(): int
    {
        return 3600; // 1小时内去重
    }
}
```

## 何时需要幂等

| 场景 | 是否需要幂等 | 原因 |
|------|-------------|------|
| 创建订单 | ✅ 必须 | 重复创建导致重复扣款 |
| 支付回调 | ✅ 必须 | 网关可能多次通知 |
| 发送通知 | ⚠️ 推荐 | 重复发送影响用户体验 |
| 查询操作 | ❌ 不需要 | 天然幂等 |
| 更新状态 | ⚠️ 看情况 | 如果是幂等赋值（SET）则天然幂等 |

## 常见问题

**Q: Idempotency Key 应该由谁生成？**
A: 客户端生成（UUID v4），因为重试时客户端才知道"这是同一个请求"。

**Q: Idempotency Key 的 TTL 应该设多长？**
A: 根据业务最长处理时间设置。支付回调通常 24-48 小时，普通 API 请求 1-24 小时。

**Q: 幂等和防重有什么区别？**
A: 幂等是"重复执行效果相同"，防重是"阻止重复执行"。幂等更优雅——允许重复，但保证结果一致。
