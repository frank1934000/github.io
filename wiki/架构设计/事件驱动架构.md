# 事件驱动架构（Event-Driven Architecture）

## 定义

事件驱动架构以**事件**作为系统间通信的核心机制。当领域中发生重要事情时，发布事件；其他组件订阅感兴趣的事件并做出响应。这种模式实现了发布者和订阅者的彻底解耦。

## 核心原理

### 三种事件类型

| 类型 | 定义 | 示例 |
|------|------|------|
| **领域事件（Domain Event）** | 领域中发生的有意义的事情 | 订单已创建、库存已扣减 |
| **集成事件（Integration Event）** | 跨服务边界传播的事件 | 用户注册成功 → 通知服务 |
| **CommandEvent** | 触发某个操作的命令 | 发送确认邮件 |

### Domain Events 解耦模式

```
❌ 胖 Service：所有副作用耦合在一个方法中
OrderService::placeOrder()
  → inventoryService->deduct()
  → mailer->sendOrderConfirmation()
  → memberService->addPoints()
  → slack->notify()
  → auditLog->record()

✅ 事件驱动：Order 聚合发布事件，各 Handler 独立响应
OrderService::placeOrder()
  → $order = Order::create(...)
  → event(new OrderPlaced($order))
      ├── InventoryHandler::handle()    → 扣减库存
      ├── MailHandler::handle()         → 发送邮件
      ├── MemberPointsHandler::handle() → 更新积分
      ├── SlackHandler::handle()        → 推送通知
      └── AuditHandler::handle()        → 写审计日志
```

### Event Sourcing（事件溯源）

**核心思想**：不存储当前状态，而是存储所有导致当前状态的事件序列。当前状态通过「重放事件」得到。

```
传统 CRUD:    购物车表 → { items: [...], total: 500 }  (只有最终状态)

Event Sourcing:
  → CartCreated { cart_id, user_id }
  → ItemAdded { product_id: 1, qty: 2, price: 100 }
  → ItemAdded { product_id: 2, qty: 1, price: 300 }
  → ItemQuantityChanged { product_id: 1, qty: 3 }
  (可以回到任意时间点的状态)
```

**适用场景**：
- 订单状态回溯：「这个订单经历了哪些状态变更？」
- 库存变更审计：「商品 A 的库存在昨天为什么突然少了 50 件？」
- 用户行为重放：「如果当时购物车里有这些商品，推荐引擎会返回什么？」

### Event Storming（事件风暴）

一种**协作式领域建模方法**，通过便利贴在大白墙上梳理业务事件流：

```
[用户注册]──→(用户聚合)──→[用户已验证]
     │                         │
     ▼                         ▼
[商品浏览]──→(浏览聚合)──→[推荐已更新]
     │
     ▼
[加入购物车]──→(购物车聚合)──→[库存已预留]
     │
     ▼
[下单]──→(订单聚合)──→[订单已创建]──→[支付已发起]
```

## 实战案例

来自博客文章：[Domain Events 解耦实战：用事件驱动替代 Service Layer 直接调用](/2026/05/05/domain-events-guide-service-layer/)

**踩坑经验**：
- `OrderService::placeOrder()` 500 行的「上帝方法」→ 事件驱动后每个 Handler 不超过 50 行
- 同步事件失败会阻塞主流程 → 关键路径用同步，非关键路径用队列异步
- 事件命名用过去式：`OrderPlaced` 而非 `PlaceOrder`

来自博客文章：[Laravel Event-Sourcing 入门实战：事件溯源在 B2C 电商中的应用](/2026/05/05/laravel-event-sourcing-getting-startedguide-b2c-use-cases/)

**适用与不适用**：
- ✅ 订单生命周期、库存变更追踪、审计日志
- ❌ 简单 CRUD、读多写少的配置表、实时性要求极高的场景

## 常见问题

**Q: 领域事件用同步还是异步？**
A: 关键路径（库存扣减）用同步，非关键路径（发邮件、推通知）用队列异步。

**Q: Event Sourcing 性能如何？**
A: 事件表会越来越大。通过定期创建快照（Snapshot）来优化重放性能。

**Q: 事件顺序怎么保证？**
A: 单聚合内的事件用版本号排序；跨聚合的事件用时间戳 + 事件总线的顺序保证。
