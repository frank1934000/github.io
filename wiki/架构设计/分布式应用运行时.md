# 分布式应用运行时

## 定义

分布式应用运行时（Distributed Application Runtime）是一种以 **Sidecar 模式** 运行在应用进程旁边的基础设施层，它将分布式系统中常见的横切关注点（服务调用、发布订阅、状态管理、可观测性、密钥管理等）从业务代码中剥离出来，以标准化 API 的形式提供给任意语言、任意框架的应用使用。

最具代表性的实现是 **Dapr（Distributed Application Runtime）**，由微软于 2019 年开源，现为 CNCF 孵化项目。Dapr 通过 HTTP/gRPC API 暴露构建块（Building Blocks），开发者无需在业务代码中引入特定 SDK 即可完成服务间通信、消息发布订阅、状态读写、Actor 模型等操作。

与传统的"SDK 集成"方式（如在 Laravel 中直接引入 RabbitMQ SDK、Redis Client）相比，分布式应用运行时的核心差异在于：

| 维度 | SDK 集成 | 分布式应用运行时（Sidecar） |
|------|---------|--------------------------|
| 语言耦合 | 高，每种语言需各自集成 | 低，Sidecar 与语言无关 |
| 基础设施耦合 | 高，切换中间件需改代码 | 低，通过 Component 声明式切换 |
| 升级影响 | 需重新构建部署应用 | Sidecar 独立升级 |
| 运维复杂度 | 低 | 略高，多一个 Sidecar 进程 |
| 性能开销 | 低 | 多一跳网络延迟（通常 < 1ms） |

## 核心原理

### 1. Sidecar 模式

Sidecar 模式的核心思想是：每个应用实例旁边部署一个独立的进程（Sidecar），两者通过本地网络（localhost）通信。Sidecar 负责处理所有与分布式基础设施的交互，应用只关注业务逻辑。

```
┌─────────────────────────┐
│  Application Container  │
│  ┌───────────────────┐  │
│  │  Laravel App      │  │
│  │  (port 8000)      │  │
│  └─────┬─────────────┘  │
│        │ localhost       │
│  ┌─────▼─────────────┐  │
│  │  Dapr Sidecar     │  │
│  │  (port 3500)      │  │
│  └───────────────────┘  │
└─────────────────────────┘
```

### 2. 构建块（Building Blocks）

Dapr 提供以下核心构建块：

- **Service Invocation（服务调用）**：服务间 HTTP/gRPC 调用，内置重试、mTLS 加密、可观测性
- **Pub/Sub（发布订阅）**：异步消息通信，支持 RabbitMQ、Kafka、Redis Streams 等
- **State Management（状态管理）**：键值对状态存储，支持 Redis、PostgreSQL、DynamoDB 等
- **Bindings（绑定）**：与外部系统（数据库、消息队列、SaaS）的输入/输出绑定
- **Actors**：虚拟 Actor 模型，适合有状态的并发场景
- **Observability（可观测性）**：分布式追踪、指标、日志自动注入
- **Configuration（配置）**：集中式配置管理
- **Distributed Lock（分布式锁）**：跨实例互斥访问

### 3. Component 模型

基础设施的具体实现通过 **Component** 声明式配置，无需修改业务代码即可切换：

```yaml
# components/pubsub.yaml
apiVersion: dapr.io/v1alpha1
kind: Component
metadata:
  name: orders-pubsub
spec:
  type: pubsub.rabbitmq
  version: v1
  metadata:
    - name: host
      value: "amqp://rabbitmq:5672"
    - name: durable
      value: "true"
```

将 `type` 从 `pubsub.rabbitmq` 改为 `pubsub.kafka` 即可切换消息中间件，应用代码完全不变。

### 4. Laravel 集成示例

在 Laravel 微服务中，通过 HTTP 调用 Dapr Sidecar 的 API 完成服务调用和发布订阅：

```php
// 服务调用：调用 inventory 服务
$response = Http::withHeaders([
    'Content-Type' => 'application/json',
])->post('http://localhost:3500/v1.0/invoke/inventory-service/method/api/stock/check', [
    'product_id' => $productId,
]);

// 发布事件：发布订单创建事件
Http::post('http://localhost:3500/v1.0/publish/orders-pubsub/order.created', [
    'order_id' => $order->id,
    'user_id'  => $order->user_id,
    'total'    => $order->total,
]);

// 状态管理：读写订单状态
Http::put('http://localhost:3500/v1.0/state/statestore/' . $orderId, [
    'status' => 'paid',
]);
$state = Http::get("http://localhost:3500/v1.0/state/statestore/{$orderId}");
```

### 5. 与直接库集成的对比分析

**Dapr/Sidecar 的优势**：
- 多语言团队无需各自维护中间件 SDK 集成
- 基础设施切换零代码变更
- 统一的可观测性，无需每种语言单独配置
- 适合 [微服务架构](微服务架构.md) 中多语言异构系统

**直接库集成的优势**：
- 更低的延迟，无额外网络跳
- 更简单的部署拓扑，无需管理 Sidecar
- 更深的框架集成（如 Laravel 的 Event/Queue 原生支持）
- 适合同构 PHP/Laravel 团队

### 6. 部署模式

- **Kubernetes**：通过 Dapr 注解自动注入 Sidecar 到 Pod
- **Docker Compose**：手动添加 Sidecar 容器
- **自托管（Self-hosted）**：`dapr run` 命令启动应用和 Sidecar
- **Service Mesh 集成**：与 Istio、Linkerd 共存（注意功能重叠）

### 7. Dapr vs Service Mesh

| 维度 | Dapr | Service Mesh |
|------|------|-------------|
| 关注点 | 应用级分布式能力 | 网络级流量管理 |
| API 提供 | 有，应用通过 API 调用 | 无，对应用透明 |
| 状态管理 | 有 | 无 |
| Pub/Sub | 有 | 无 |
| 流量控制 | 基础 | 强大（金丝雀、熔断） |
| 协同使用 | 推荐组合使用 |

## 常见问题

### Q: Dapr Sidecar 带来多少性能开销？

本地 localhost 通信通常增加 < 1ms 延迟。对于内部服务间调用（本身就有数十毫秒网络延迟），这个开销可以忽略。但对于高频低延迟场景（如交易系统），需要评估。

### Q: 已经用了 Laravel 的 Queue 和 Event，还需要 Dapr 吗？

如果团队是纯 PHP/Laravel 同构架构，Laravel 原生的 Queue（队列）和 Event（事件）机制已经足够。Dapr 的价值在于：
1. 多语言团队需要统一的分布式基础设施接口
2. 需要频繁切换底层中间件（如从 RabbitMQ 切换到 Kafka）
3. 需要统一的可观测性，而非每种框架各自配置

### Q: Dapr 和 Laravel Octane 的关系？

两者互补：Laravel Octane 提升单应用的并发处理能力（通过 Swoole/RoadRunner），Dapr 提供跨服务的分布式能力。可以同时使用。

### Q: 如何在本地开发环境运行 Dapr？

```bash
# 安装 Dapr CLI
brew install dapr/tap/dapr-cli

# 初始化本地开发环境
dapr init

# 运行 Laravel 应用 + Dapr Sidecar
dapr run --app-id order-service --app-port 8000 -- php artisan serve --port=8000
```

### Q: 生产环境如何管理 Dapr Component 配置？

在 Kubernetes 中，Component 以 CRD 形式存在，通过 kubectl 或 Helm 管理。敏感信息（如密码）使用 Kubernetes Secret 引用。在非 K8s 环境中，Component YAML 文件通过挂载卷或环境变量注入。

### Q: Sidecar 模式和 Ambassador 模式有什么区别？

Sidecar 与应用一对一部署，共享网络命名空间；Ambassador 是一种特殊的 Sidecar，专注于代理出站连接。在 Dapr 语境中，Sidecar 同时处理入站和出站流量。
