# 事件最终一致性

## 定义

事件最终一致性（Eventual Consistency through Events）是一种分布式系统设计模式，通过异步事件传递机制保证系统中多个服务或组件的数据最终达到一致状态。与分布式事务的强一致性不同，事件最终一致性接受短暂的数据不一致窗口，换取更高的可用性和性能。

在电商系统中，订单创建需要同时扣减库存、创建支付记录、发送通知等操作。这些操作分布在不同微服务中，无法使用传统的 ACID 事务。事件最终一致性通过"先完成本地事务，再异步通知其他服务"的方式，在保证系统可用性的同时实现数据的最终一致。

## 核心原理

### CAP 定理与最终一致性

根据 CAP 定理，分布式系统在网络分区（P）发生时，必须在一致性（C）和可用性（A）之间做出权衡：

- **CP 系统**：选择一致性，牺牲可用性（如 ZooKeeper、etcd）
- **AP 系统**：选择可用性，接受最终一致性（如 Cassandra、DynamoDB）

电商系统通常选择 AP，因为用户体验要求高可用，而短暂的数据不一致在业务上可以接受。

### BASE 理论

BASE 是对 CAP 中 AP 方向的实践总结：

- **Basically Available（基本可用）**：系统保证基本功能可用，允许部分性能降级
- **Soft State（软状态）**：系统允许存在中间状态，不要求实时一致
- **Eventually Consistent（最终一致性）**：经过一段时间后，数据最终达到一致

### 核心模式一：Outbox 模式（发件箱模式）

Outbox 模式是实现事件最终一致性的基石。它解决了"数据库事务 + 消息发送"的原子性问题。

**问题背景：**
```
// 这两步不在同一个事务中，可能只成功一个
DB::commit();                    // ✅ 数据库事务提交成功
MessageQueue::publish($event);   // ❌ 消息发送失败 → 数据已变，但下游服务不知道
```

**Outbox 解决方案：**

```
┌─────────────────────────────────────┐
│           同一个数据库事务            │
│                                     │
│  1. 业务表写入（如 orders）          │
│  2. outbox 表写入（事件消息）        │
│                                     │
│  两个操作要么同时成功，要么同时失败   │
└─────────────────────────────────────┘
                    │
                    ▼
        ┌───────────────────┐
        │   Outbox Relay    │  （独立进程）
        │  轮询 outbox 表    │
        │  发送到消息队列    │
        └───────────────────┘
                    │
                    ▼
            ┌──────────────┐
            │  消息队列     │
            └──────────────┘
                    │
                    ▼
            ┌──────────────┐
            │  下游消费者    │
            └──────────────┘
```

**Outbox 表设计：**
```sql
CREATE TABLE outbox_events (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    aggregate_type VARCHAR(255) NOT NULL,   -- 聚合类型（如 Order）
    aggregate_id VARCHAR(255) NOT NULL,     -- 聚合ID（如订单ID）
    event_type VARCHAR(255) NOT NULL,       -- 事件类型（如 OrderCreated）
    payload JSON NOT NULL,                  -- 事件负载
    created_at TIMESTAMP NOT NULL,
    published_at TIMESTAMP NULL,            -- 发布时间，NULL 表示未发布
    retry_count INT DEFAULT 0,
    INDEX idx_unpublished (published_at, created_at)
);
```

### 核心模式二：Inbox 模式（收件箱模式）

Inbox 模式解决消费者端的幂等性问题。相同事件可能被重复投递（at-least-once），Inbox 表记录已处理的事件，避免重复消费。

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   消息队列    │ ──→ │   Inbox 表   │ ──→ │   业务处理    │
│              │     │  去重检查     │     │  本地事务     │
└──────────────┘     └──────────────┘     └──────────────┘
```

**Inbox 表设计：**
```sql
CREATE TABLE inbox_events (
    id VARCHAR(255) PRIMARY KEY,        -- 事件唯一ID（来自消息）
    event_type VARCHAR(255) NOT NULL,
    payload JSON NOT NULL,
    received_at TIMESTAMP NOT NULL,
    processed_at TIMESTAMP NULL,
    INDEX idx_unprocessed (processed_at, received_at)
);
```

**处理流程：**
1. 接收消息
2. 尝试将事件 ID 插入 inbox 表（唯一约束保证幂等）
3. 插入成功 → 处理业务逻辑 → 更新 processed_at
4. 插入失败（重复） → 跳过，ACK 消息

### 核心模式三：Saga 模式

Saga 将长事务拆分为一系列本地事务，每个本地事务有对应的补偿操作：

**编排式 Saga（Orchestration）：**
由一个中心协调器（Saga Orchestrator）指挥各服务按顺序执行：

```
OrderSaga:
  1. 创建订单（Order Service）
  2. 扣减库存（Inventory Service）
  3. 创建支付（Payment Service）
  4. 发送通知（Notification Service）

  若步骤3失败：
    补偿2: 恢复库存（Inventory Service）
    补偿1: 取消订单（Order Service）
```

**协同式 Saga（Choreography）：**
没有中心协调器，各服务通过事件相互通信：

```
OrderCreated → InventoryService（扣减库存）
                │
                ▼
          InventoryReserved → PaymentService（创建支付）
                                │
                                ▼
                          PaymentCompleted → NotificationService（发送通知）

若 PaymentFailed → InventoryService（释放库存）→ OrderService（取消订单）
```

### 幂等性（Idempotency）

幂等性是事件最终一致性的关键保障。无论同一操作执行多少次，结果都相同。

**实现方式：**

1. **唯一请求 ID**：每个请求携带唯一 ID，服务端检查是否已处理
2. **数据库唯一约束**：利用数据库唯一索引防止重复插入
3. **状态检查**：操作前检查当前状态是否允许执行
4. **乐观锁/版本号**：通过版本号防止重复更新

```php
class PaymentService
{
    public function process(PaymentCommand $command): Payment
    {
        // 幂等性检查：使用外部订单号作为唯一键
        $existing = Payment::where('external_no', $command->orderNo)->first();
        if ($existing) {
            return $existing; // 已处理，直接返回
        }

        return DB::transaction(function () use ($command) {
            return Payment::create([
                'external_no' => $command->orderNo,
                'amount' => $command->amount,
                'status' => 'pending',
            ]);
        });
    }
}
```

### 补偿事务（Compensating Transaction）

补偿事务用于撤销已成功执行的操作。它不是简单的"回滚"，而是执行一个语义上相反的新操作：

| 原始操作 | 补偿操作 |
|---------|---------|
| 扣减库存 | 恢复库存 |
| 创建支付 | 发起退款 |
| 发送通知 | 发送取消通知 |
| 生成积分 | 扣除积分 |

**补偿事务的特点：**
- 补偿操作本身也可能失败，需要重试机制
- 补偿操作必须是幂等的
- 补偿操作有时间窗口限制（如退款有期限）
- 某些操作不可补偿（如已发送的短信），需要通过设计避免

### 事件顺序保证

在分布式环境中保证事件的顺序性是一个挑战：

1. **分区键有序**：同一聚合根的事件路由到同一分区（如按订单 ID 哈希），保证同一订单的事件有序
2. **序列号/版本号**：每个事件携带递增序列号，消费者按序处理
3. **因果一致性**：通过因果 ID 追踪事件的因果关系
4. **最终顺序**：不保证严格顺序，通过状态检查保证最终正确

### 死信队列与重试

处理失败的事件不能简单丢弃：

```
正常消费 → 失败 → 重试（指数退避）
                   → 最大重试次数后进入死信队列
                   → 人工介入或自动修复
```

重试策略：
- 立即重试：适用于瞬时故障（网络抖动）
- 指数退避：适用于需要等待的场景（下游服务恢复）
- 定时重试：适用于业务限制（如支付渠道限额恢复）

## 实战案例

来自博客文章：[Eventual Consistency in E-commerce Engineering](/2026/06/01/eventual-consistency-in-ecommerce-engineering/)

### Laravel Outbox 模式实现

```php
// 业务服务：在同一个事务中写入业务数据和 Outbox 事件
class CreateOrderService
{
    public function execute(CreateOrderCommand $command): Order
    {
        return DB::transaction(function () use ($command) {
            // 1. 创建订单
            $order = Order::create([...]);

            // 2. 写入 Outbox 事件（同一个事务）
            OutboxEvent::create([
                'aggregate_type' => 'Order',
                'aggregate_id' => $order->id,
                'event_type' => 'OrderCreated',
                'payload' => [
                    'order_id' => $order->id,
                    'user_id' => $order->user_id,
                    'total_amount' => $order->total_amount,
                    'items' => $order->items->toArray(),
                ],
            ]);

            return $order;
        });
    }
}

// Outbox Relay：独立进程，轮询发布事件
class OutboxRelayJob implements ShouldQueue
{
    public function handle(): void
    {
        $events = OutboxEvent::whereNull('published_at')
            ->orderBy('created_at')
            ->limit(100)
            ->get();

        foreach ($events as $event) {
            try {
                EventDispatcher::dispatch($event->event_type, $event->payload);
                $event->update(['published_at' => now()]);
            } catch (\Throwable $e) {
                $event->increment('retry_count');
                Log::error('Outbox relay failed', ['event_id' => $event->id]);
            }
        }
    }
}
```

### Laravel Inbox 模式实现

```php
class OrderEventHandler
{
    public function handle(OrderCreatedEvent $event): void
    {
        DB::transaction(function () use ($event) {
            // 1. 幂等性检查：插入 Inbox 记录
            $inbox = InboxEvent::create([
                'id' => $event->eventId,
                'event_type' => 'OrderCreated',
                'payload' => $event->toArray(),
                'received_at' => now(),
            ]);

            // 2. 执行业务逻辑（扣减库存）
            $this->inventoryService->reserve($event->items);

            // 3. 标记已处理
            $inbox->update(['processed_at' => now()]);
        });
    }
}
```

### 补偿事务实现

```php
class OrderSagaOrchestrator
{
    public function compensate(Order $order): void
    {
        // 按逆序执行补偿操作
        $this->releaseInventory($order);    // 恢复库存
        $this->refundPayment($order);       // 退款
        $this->cancelOrder($order);         // 取消订单
        $this->sendCancelNotification($order); // 发送取消通知
    }

    private function releaseInventory(Order $order): void
    {
        InventoryReleasedEvent::dispatch($order->id, $order->items);
    }

    private function refundPayment(Order $order): void
    {
        RefundCommand::dispatch($order->payment->id);
    }
}
```

## 常见问题

### Q: Outbox 模式和直接发消息有什么区别？
直接发消息面临"双写问题"：数据库事务提交后消息发送失败，导致数据已变更但下游不知道。Outbox 模式将事件写入与业务数据写入放在同一个数据库事务中，保证原子性，然后通过 Relay 异步发布到消息队列。

### Q: 如何保证事件不丢失？
1. Outbox 模式保证事件写入的原子性
2. Relay 进程持续轮询未发布的事件
3. 消息队列的持久化保证事件不丢失
4. 消费端确认机制保证至少处理一次
5. Inbox 模式保证消费端的幂等性

### Q: 如何处理长时间未一致的数据？
1. 监控 Outbox 表中长时间未发布的事件
2. 设置事件过期时间，过期后报警
3. 提供手动修复工具
4. 定期对账，发现不一致自动修复

### Q: Saga 和 2PC（两阶段提交）的区别？
- **2PC**：同步阻塞，强一致性，性能差，存在协调者单点问题
- **Saga**：异步非阻塞，最终一致性，性能好，需要补偿机制
电商系统通常选择 Saga，因为可以接受短暂不一致，但不能接受长时间阻塞。

### Q: 如何设计幂等键？
- 使用业务唯一标识（如订单号、支付流水号）
- 使用请求 ID（UUID）作为幂等键
- 组合键：服务名 + 操作类型 + 业务ID
- 幂等键需要设置合理的过期时间

### Q: 事件风暴（Event Storming）和事件最终一致性有什么关系？
Event Storming 是一种协作式建模方法，用于发现业务领域中的事件。通过 Event Storming 识别出的领域事件，可以直接作为事件最终一致性架构中的事件定义。它是从业务到技术的桥梁。

### Q: 如何监控事件最终一致性系统的健康状态？
1. 监控 Outbox 中未发布事件的数量和滞留时间
2. 监控消息队列的积压情况
3. 监控消费者处理延迟
4. 设置数据对账任务，定期检查各服务数据一致性
5. 监控死信队列中的消息数量
