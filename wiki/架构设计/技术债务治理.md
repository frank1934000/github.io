# 技术债务治理（Technical Debt Quadrant）

## 定义

技术债务是**为了短期利益而做出的次优技术决策**，这些决策会在未来产生额外的维护成本。Martin Fowler 的技术债务象限将技术债务分为四个类型，帮助团队区分"合理的权衡"和"需要偿还的债务"。

## 技术债务四象限

```
                    谨慎 (Prudent)
                        │
            ┌───────────┼───────────┐
            │  谨慎+有意  │  谨慎+无意  │
            │           │           │
            │ "我们知道   │ "我们写完   │
            │  有债务，   │  才发现     │
            │  但现在     │  设计有     │
            │  必须发布"  │  问题"     │
   不负责任 ─┼───────────┼───────────┼─ 有意 (Deliberate)
   (Reckless)│  鲁莽+有意  │  鲁莽+无意  │
            │           │           │
            │ "没时间     │ "什么是     │
            │  写测试"    │  单元测试?" │
            │           │           │
            └───────────┼───────────┘
                        │
                    不负责任 (Reckless)
```

### 四种类型详解

| 类型 | 特征 | 示例 | 处理策略 |
|------|------|------|---------|
| **谨慎+有意** | 知道风险，主动权衡 | "先用 JSON 存，后面再改表结构" | 设定偿还时间点 |
| **谨慎+无意** | 事后才发现问题 | "上线后才发现 Eloquent N+1" | 加入技术待办列表 |
| **鲁莽+有意** | 明知有害但不改正 | "没时间写测试，先上线" | 需要团队讨论 |
| **鲁莽+无意** | 不知道什么是好实践 | "不知道有 SOLID 原则" | 需要培训和知识分享 |

## 债务识别方法

### 代码度量

```bash
# PHPStan 静态分析（类型安全）
vendor/bin/phpstan analyse --level=8

# Rector 代码坏味道检测
vendor/bin/rector process --dry-run

# Laravel Pint 代码风格检查
vendor/bin/pint --test

# 复杂度分析
phploc app/
```

### 债务标记

```php
// 在代码中标记技术债务
/**
 * TODO: 这里用了 N+1 查询，需要 eager loading 优化
 * DEBT: 临时方案，下个 Sprint 重构
 * HACK: 支付回调处理，需要迁移到队列
 */
```

## 债务偿还策略

### 1. 男孩童军规则（Boy Scout Rule）

> "离开时比来时更干净。"

每次修改文件时，顺手改进周围的代码。不需要专门的"重构 Sprint"。

### 2. 20% 规则

每个 Sprint 留出 20% 的时间处理技术债务。

```
Sprint 容量：40 story points
├─ 功能开发：32 points (80%)
└─ 技术债务：8 points (20%)
    ├─ 补充单元测试：3 points
    ├─ 重构 N+1 查询：2 points
    └─ 升级依赖版本：3 points
```

### 3. 专项重构

大型债务需要专门的重构项目：

- **绞杀者模式**逐步替换（见 [渐进式迁移模式](渐进式迁移模式.md)）
- **并行运行**新旧系统对比结果
- **Feature Flag**控制重构切换

## 债务追踪

### 技术待办列表

```markdown
## 技术债务待办

### 🔴 高优先级（影响开发效率）
- [ ] OrderService 3000 行 → 拆分为领域服务
- [ ] 数据库 N+1 查询 → 添加 eager loading
- [ ] 测试覆盖率 40% → 提升到 70%

### 🟡 中优先级（影响系统稳定性）
- [ ] Redis 连接池配置 → 添加舱壁隔离
- [ ] 日志格式不统一 → 接入 OpenTelemetry

### 🟢 低优先级（技术升级）
- [ ] PHP 8.1 → 8.3 升级
- [ ] Laravel 10 → 11 升级
```

## 常见问题

**Q: 如何说服管理层投入时间处理技术债务？**
A: 用数据说话——"因为 N+1 查询，API 响应时间增加了 200ms，影响了 15% 的转化率"比"代码质量不好"更有说服力。

**Q: 技术债务和 Bug 有什么区别？**
A: Bug 是"代码行为与预期不符"，技术债务是"代码能工作但不是最优方案"。Bug 需要立即修复，技术债务可以计划性偿还。
