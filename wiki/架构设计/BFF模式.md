# BFF 模式（Backend-for-Frontend）

## 定义

BFF 是一个**面向特定前端的中间层聚合服务**，位于前端和后端微服务之间。每个前端（Web、App、小程序）有专属的 BFF，负责数据聚合、格式转换、版本管理。

## 核心原理

### 为什么需要 BFF

| 问题 | 描述 | 影响 |
|------|------|------|
| **网络抖动** | 3-5 个串行调用 → 延迟叠加 | 页面加载慢 |
| **版本管理复杂** | iOS v1/v2，Android v2/v3 | 后端需多套 API |
| **带宽浪费** | 前端接收大量无用字段 | 流量成本高 |
| **数据格式差异** | Android 要 `id+name`，iOS 要 `uuid+displayName` | 前端适配困难 |

### BFF vs GraphQL vs Direct API

| 维度 | Laravel BFF | GraphQL | Direct API |
|------|-------------|---------|------------|
| **数据聚合** | 中间层处理 | 客户端合并 | 前端多次请求 |
| **版本管理** | 简单（URI + Query） | 麻烦（schema 迁移慢） | 中等 |
| **缓存友好度** | 高 | 低（Query 组合爆炸） | 中 |
| **学习成本** | 低 | 高 | 低 |
| **适用场景** | 多客户端、数据聚合复杂 | 客户端需求多变 | 简单场景 |

### BFF 的核心职责

1. **数据聚合**：合并多个微服务的响应
2. **格式转换**：为不同客户端提供不同数据格式
3. **版本管理**：通过 URI 或 Header 管理 API 版本
4. **缓存**：在 BFF 层做响应缓存，减少后端压力
5. **鉴权**：统一认证逻辑，后端服务无需重复实现

## 实战案例

来自博客文章：[BFF Laravel 中间层聚合实战](/2026/05/04/bff-laravel/)

**踩坑经验**：
- KKday 项目中 BFF 聚合 Search/Recommend/Member 三个 Java 服务
- 并行请求（Promise/Guzzle Pool）减少串行延迟
- BFF 层做字段裁剪，减少不必要的数据传输

来自博客文章：[BFF vs GraphQL：何时用 BFF 而非直接调用 API？](/2026/05/02/bff-vs-graphql/)

**选型建议**：
- 客户端类型多、数据聚合复杂 → BFF
- 客户端需求多变、查询灵活 → GraphQL
- 简单 CRUD、单一客户端 → Direct API

## 常见问题

**Q: BFF 和 API Gateway 有什么区别？**
A: API Gateway 负责路由、鉴权、限流等横切关注点；BFF 负责业务级别的数据聚合和格式转换。两者互补。

**Q: 每个客户端都要一个 BFF 吗？**
A: 不一定。如果客户端需求差异不大，可以共用一个 BFF。差异大时再拆分。

**Q: BFF 用什么语言？**
A: 跟团队技术栈一致。Laravel BFF 适合 PHP 团队，Node.js BFF 适合前端团队。
