# 分布式锁选型（Redis vs Zookeeper vs etcd）

## 定义

分布式锁是**跨多个进程/服务实例的互斥机制**，确保同一时刻只有一个执行者能操作共享资源。在分布式系统中，本地锁（如 PHP 的 `flock`）无法跨进程生效，需要借助外部存储实现。

## 三种方案对比

| 维度 | Redis Redlock | Zookeeper | etcd |
|------|--------------|-----------|------|
| 一致性模型 | AP（异步复制） | CP（ZAB 协议） | CP（Raft 协议） |
| 锁安全性 | 中（时钟漂移风险） | 高（顺序节点 + Watch） | 高（Lease + Revision） |
| 性能 | 最高（内存操作 + 单节点） | 中等（ZK 集群选举开销） | 中等（Raft 共识开销） |
| 运维复杂度 | 低（已有 Redis 则零成本） | 高（需维护 ZK 集群） | 中（K8s 自带 etcd） |
| 适用场景 | 缓存层锁、短时互斥 | 强一致要求、Hadoop 生态 | K8s 生态、服务选举 |
| PHP 生态 | Predis/phpredis 原生支持 | php-zk 扩展（不常用） | etcd PHP 客户端（较少） |
| 锁续期 | 需自行实现（看门狗） | Session 自动续期 | Lease 自动续期 |
| 可重入 | 需自行实现 | 原生支持（临时顺序节点） | 需自行实现 |
| 公平性 | 不公平（随机竞争） | 公平（顺序排队） | 不公平 |

## Redis 分布式锁详解

### 基础实现

```php
// Redis 分布式锁核心命令
SET resource_name $unique_value NX EX 30

// 解锁必须用 Lua 保证原子性
if redis.call("GET", KEYS[1]) == ARGV[1] then
    return redis.call("DEL", KEYS[1])
else
    return 0
end
```

### Laravel 实现

```php
use Illuminate\Support\Facades\Cache;

// 基础锁
$lock = Cache::lock('order:42:process', 30);
if ($lock->get()) {
    try {
        // 处理订单
    } finally {
        $lock->release();
    }
}

// 阻塞锁（等待最多 10 秒）
$lock = Cache::lock('order:42:process', 30);
$lock->block(10); // 阻塞等待
try {
    // 处理订单
} finally {
    $lock->release();
}
```

### Redlock 算法

当 Redis 部署为主从架构时，单节点锁可能丢失（主从切换期间）。Redlock 通过向 N 个独立 Redis 节点加锁来提高安全性：

```
1. 获取当前时间 T1
2. 依次向 N 个 Redis 节点请求锁（相同的 key + unique_value + TTL）
3. 获取当前时间 T2
4. 如果在 N/2+1 个节点上成功获取锁，且 T2-T1 < TTL，则加锁成功
5. 锁的有效时间 = TTL - (T2-T1)
6. 如果加锁失败，向所有节点发送解锁请求
```

### 常见失效场景

| 场景 | 原因 | 解决方案 |
|------|------|---------|
| 锁过期但业务未完成 | TTL 太短 | 看门狗续期（Laravel `block()` 自带） |
| 解错了别人的锁 | unique_value 不唯一 | 使用 UUID 作为锁的值 |
| 主从切换丢锁 | Redis 异步复制 | Redlock 或使用 Redis Cluster |
| GC 停顿导致锁过期 | PHP GC 暂停 | 增大 TTL + 业务层校验 |

## Zookeeper 分布式锁

### 原理

```
1. 在 /locks/resource 下创建临时顺序节点
   /locks/resource/lock-0000000001
   /locks/resource/lock-0000000002
   /locks/resource/lock-0000000003

2. 获取 /locks/resource 下所有子节点
3. 如果自己是最小节点 → 获得锁
4. 否则 Watch 前一个节点（避免惊群效应）
5. 当前一个节点被删除时，收到通知，检查自己是否成为最小节点
```

### 优势

- **公平排队**：按顺序获取锁，避免饥饿
- **自动释放**：临时节点在会话断开时自动删除
- **Watch 机制**：精确通知下一个等待者，无惊群效应

## etcd 分布式锁

### 原理

```
1. 创建 Lease（带 TTL）
2. 用 Lease 的 Revision 作为锁的"序号"
3. 检查是否有比自己更小的 Revision 的 key
4. 如果没有 → 获得锁
5. 如果有 → Watch 前一个 Revision 的 key
6. Lease 自动续期，锁不会意外过期
```

### 与 Kubernetes 集成

etcd 是 K8s 的数据存储，如果应用已经运行在 K8s 上，可以直接利用 etcd 实现分布式锁，无需额外基础设施。

## 选型建议

```
你的项目已有 Redis？
├─ 是 → 优先 Redis 分布式锁（成本最低）
│       └─ 需要强一致？ → 考虑 Redlock 或 etcd
│
└─ 否 → 运行在 K8s 上？
         ├─ 是 → 考虑 etcd（已有基础设施）
         └─ 否 → 需要公平锁？
                  ├─ 是 → Zookeeper
                  └─ 否 → Redis（部署最简单）
```

## 常见问题

**Q: Redlock 到底安全吗？**
A: Martin Kleppmann 和 Antirez 的著名辩论指出 Redlock 依赖时钟同步假设。如果需要强一致，建议用 etcd/Zookeeper；如果可以接受极小概率的并发冲突，Redis 单节点锁 + 业务层校验即可。

**Q: PHP 项目推荐哪种？**
A: 99% 的 PHP/Laravel 项目推荐 Redis 分布式锁。PHP 生态中 Redis 客户端最成熟，Laravel 原生支持，运维成本最低。
