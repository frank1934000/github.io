# 开发者门户与平台工程

## 定义

开发者门户（Developer Portal）是平台工程（Platform Engineering）的核心产物，为开发团队提供统一的自助服务界面来管理服务目录、文档、模板脚手架和基础设施资源。Spotify 开源的 Backstage 是这一领域的事实标准，通过 Software Catalog（服务目录）、TechDocs（文档即代码）和 Scaffolder（模板脚手架）三大模块，解决"30+ 仓库无人知道整个系统长什么样"的问题。

## 核心原理

### Software Catalog（软件目录）
将散落在各处的服务、库、API、基础设施资源统一注册到一个可搜索的目录中：

```yaml
# catalog-info.yaml — 放在每个仓库根目录
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: b2c-api
  description: B2C 电商平台后端 API
  annotations:
    github.com/project-slug: org/b2c-api
    backstage.io/techdocs-ref: dir:.
  tags: [laravel, api, production]
spec:
  type: service
  lifecycle: production
  owner: team-backend
  system: b2c-platform
  dependsOn:
    - resource:b2c-database
    - resource:b2c-redis
```

Catalog 支持的实体类型：
- **Component**：服务、库、网站
- **API**：接口定义（OpenAPI/AsyncAPI）
- **Resource**：基础设施资源（数据库、队列、S3 桶）
- **System**：相关实体的逻辑分组
- **Domain**：业务领域

### TechDocs（文档即代码）
将文档与代码放在同一个仓库中，使用 Markdown 编写，Backstage 自动构建和渲染：

```
b2c-api/
├── docs/
│   ├── index.md         # 文档首页
│   ├── architecture.md  # 架构说明
│   └── api-guide.md     # API 使用指南
├── catalog-info.yaml
└── mkdocs.yml           # MkDocs 配置
```

### Scaffolder（模板脚手架）
通过 YAML 模板定义新项目的创建流程，实现标准化的项目初始化：

```yaml
# template.yaml
apiVersion: scaffolder.backstage.io/v1beta3
kind: Template
metadata:
  name: create-laravel-api
  title: Create Laravel API Service
  description: 创建标准化的 Laravel API 服务
spec:
  parameters:
    - title: 项目信息
      properties:
        name:
          title: 服务名称
          type: string
        owner:
          title: 负责团队
          type: string
          ui:field: OwnerPicker
  steps:
    - id: fetch-template
      action: fetch:template
      input:
        url: ./skeleton
        targetPath: ${{ parameters.name }}
    - id: publish
      action: github:repo:create
      input:
        repoUrl: github.com?owner=org&repo=${{ parameters.name }}
    - id: register
      action: catalog:register
      input:
        repoContentsUrl: ${{ steps.publish.output.repoContentsUrl }}
```

### Plugin 生态
Backstage 的核心能力通过插件扩展：
- **Kubernetes** — 查看 Pod 状态和部署信息
- **GitHub Actions** — 查看 CI/CD 状态
- **SonarQube** — 代码质量指标
- **PagerDuty** — 值班信息
- **Sentry** — 错误追踪集成

## 平台工程落地价值

### 新人 Onboarding 时间缩短
| 阶段 | 没有 Backstage | 有 Backstage |
|---|---|---|
| 了解系统全貌 | 2-3 天（问人+翻代码） | 30 分钟（看目录） |
| 找到相关文档 | 半天（不知道在哪） | 5 分钟（搜索） |
| 创建新服务 | 1-2 天（复制+改配置） | 10 分钟（模板脚手架） |
| 了解依赖关系 | 问架构师 | 目录中直接看 |

### 服务所有权清晰化
每个 Catalog 实体都有明确的 `owner` 字段，谁负责一目了然。配合 `lifecycle` 字段（experimental/staging/production）可以快速识别服务成熟度。

## 部署方案

### Kubernetes 部署
```yaml
# Helm values.yaml
backstage:
  image:
    registry: ghcr.io
    repository: backstage/backstage
    tag: latest
  appConfig:
    app:
      baseUrl: https://backstage.example.com
    backend:
      baseUrl: https://backstage.example.com
      database:
        client: pg
        connection:
          host: backstage-db
```

### 与 Laravel CI/CD 集成
在每个 Laravel 仓库中添加 `catalog-info.yaml` 和 `mkdocs.yml`，GitHub Actions 中添加自动注册步骤：

```yaml
- name: Register in Backstage
  run: |
    curl -X POST https://backstage.example.com/api/catalog/locations \
      -H "Content-Type: application/json" \
      -d '{"type": "url", "target": "https://github.com/org/b2c-api/blob/main/catalog-info.yaml"}'
```

## 常见问题

### Catalog 同步延迟
- 配置 GitHub Discovery 自动扫描仓库中的 `catalog-info.yaml`
- 设置合理的刷新间隔（默认 30 分钟）
- 使用 Webhook 触发即时刷新

### 性能问题
- 使用 PostgreSQL 而非 SQLite 作为后端数据库
- 启用缓存层减少 Catalog API 调用
- 对大型目录使用分页查询
