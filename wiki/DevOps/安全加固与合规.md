# 安全加固与合规

## 定义

安全加固与合规是 DevOps 实践中不可或缺的维度。**安全加固** 通过操作系统级（AppArmor/SELinux/seccomp）、应用级（OWASP Top 10 防护）、供应链级（SBOM/依赖审计）的多层防御，降低系统被攻击的风险。**合规** 则确保系统满足行业法规（GDPR/PCI DSS/个人信息保护法）的要求，避免法律风险和罚款。现代 DevSecOps 将安全左移（Shift Left），在 CI/CD 流水线中自动执行安全检查，而非事后补救。

## 核心原理

### 安全加固三层模型

```
┌─────────────────────────────────────────────────┐
│                应用层安全                         │
│  OWASP Top 10 · 输入校验 · CSRF/XSS 防护        │
│  认证授权 · API 安全 · 速率限制                   │
├─────────────────────────────────────────────────┤
│                容器层安全                         │
│  最小权限 · 只读文件系统 · 非 root 运行          │
│  镜像扫描 · 运行时防护 · 网络策略                 │
├─────────────────────────────────────────────────┤
│                操作系统层安全                     │
│  AppArmor · SELinux · seccomp                    │
│  内核参数加固 · 文件系统权限 · 审计日志            │
└─────────────────────────────────────────────────┘
```

### Linux 安全模块

| 模块 | 机制 | 典型用途 |
|---|---|---|
| **AppArmor** | 基于路径的访问控制（Profile 模式） | 限制容器可访问的文件路径、网络端口 |
| **SELinux** | 基于标签的强制访问控制（MAC） | 企业级多租户隔离、K8s 安全上下文 |
| **seccomp** | 系统调用白名单/黑名单 | 禁止容器执行 `mount`、`reboot` 等危险 syscall |

**Docker 安全最佳实践**：
- `--read-only`：只读文件系统，防止恶意写入
- `--cap-drop ALL --cap-add`：丢弃所有 Linux Capabilities，只添加必要的
- `--security-opt no-new-privileges`：禁止提权
- `--user 1000:1000`：非 root 用户运行
- 镜像扫描（Trivy/Snyk）：在 CI 阶段扫描已知 CVE

### 供应链安全（SBOM）

Software Bill of Materials（SBOM）是软件的「成分表」，列出所有直接和传递依赖。当某个依赖被发现存在漏洞时（如 Log4Shell），可以通过 SBOM 快速定位受影响的系统。

**SBOM 生成工具**：
- **Syft**：从容器镜像、文件系统生成 SBOM（SPDX/CycloneDX 格式）
- **Trivy**：同时做漏洞扫描 + SBOM 生成
- **Grype**：基于 SBOM 的漏洞匹配工具

**CI 集成流水线**：
```
代码提交 → 构建镜像 → 生成 SBOM（Syft）→ 漏洞扫描（Grype）
    → 高危漏洞阻断 CI → SBOM 归档 → 合规审计
```

### 合规框架

| 框架 | 适用场景 | 核心要求 |
|---|---|---|
| **GDPR** | 涉及欧盟用户数据 | 数据最小化、用户同意、数据可携带、被遗忘权 |
| **个人信息保护法** | 涉及中国用户数据 | 知情同意、最小必要、数据本地化、跨境传输审批 |
| **PCI DSS** | 处理支付卡数据 | Token 化、加密传输、访问控制、审计日志、网络分段 |

**GDPR 技术落地要点**：
- **数据主体权利 API**：提供数据导出、数据删除、同意撤回接口
- **数据分类标签**：PII（个人可识别信息）字段标记，查询审计
- **跨境传输**：数据本地化存储，跨境传输需 SCC（标准合同条款）或用户明确同意
- **同意管理**：记录用户同意的版本、时间、范围，支持撤回

**PCI DSS 技术落地要点**：
- **Token 化**：卡号不落库，用 Token 替代（Stripe/Braintree Token）
- **网络分段**：支付服务独立网段，CDE（Cardholder Data Environment）隔离
- **加密**：传输层 TLS 1.2+，存储层 AES-256
- **审计日志**：所有对卡数据的访问必须记录，保留 1 年

### 密钥管理（Secrets Management）

| 工具 | 特点 | 适用场景 |
|---|---|---|
| **HashiCorp Vault** | 动态秘钥、自动轮换、审计日志 | 企业级密钥管理 |
| **SOPS** | 文件级加密，Git 友好 | 配置文件加密 |
| **age** | 简单的文件加密工具 | 小团队密钥管理 |

**密钥轮换策略**：
- 生产密钥每 90 天轮换一次
- 使用 Vault 的 Dynamic Secrets 生成临时数据库凭据（TTL 24h）
- CI/CD 密钥注入通过 OIDC 与云厂商 IAM 集成，避免长期凭据

## 常见问题

**Q: AppArmor 和 SELinux 该选哪个？**
A: Ubuntu/Debian 默认用 AppArmor（Profile 模式更简单），RHEL/CentOS 默认用 SELinux（MAC 模式更严格）。Docker/K8s 两者都支持。如果团队没有 SELinux 经验，先从 AppArmor 开始。

**Q: SBOM 生成后放哪里？**
A: 三个去处：(1) 归档到制品仓库（如 S3/OSS），用于合规审计；(2) 导入漏洞管理平台（如 Dependency-Track），持续监控新 CVE；(3) 在 CI 中作为构建产物保留。

**Q: 小团队也需要合规吗？**
A: 如果你的应用处理用户个人信息（即使是邮箱），至少需要遵守个人信息保护法/GDPR 的基本要求：知情同意、数据最小化、提供删除接口。PCI DSS 只在处理卡号时需要（如果用 Stripe 等第三方支付，卡号不经过你的服务器，合规负担大幅降低）。
