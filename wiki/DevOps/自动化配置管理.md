# 自动化配置管理

## 定义

自动化配置管理是通过声明式代码管理服务器配置和应用部署的实践。与 Terraform 管理"基础设施资源"不同，配置管理工具（如 Ansible）管理的是"服务器上发生了什么"——安装哪些软件包、部署哪些代码、运行哪些服务。两者互补：Terraform 管"资源有无"，Ansible 管"配置好坏"。

## 核心原理

### Ansible 架构
Ansible 是 Red Hat 开源的无代理（Agentless）自动化工具，通过 SSH 连接目标主机执行任务：

- **控制节点**：运行 Ansible 的机器（开发机或 CI Runner）
- **被管节点**：通过 SSH 连接的目标服务器，无需安装任何 Agent
- **Inventory**：定义被管节点的清单文件
- **Playbook**：用 YAML 描述的自动化任务集
- **Role**：可复用的任务组织单元

### Inventory 设计
```ini
# inventory/production
[web]
web-01 ansible_host=10.0.1.11
web-02 ansible_host=10.0.1.12

[db]
db-master ansible_host=10.0.2.10

[all:vars]
ansible_user=deploy
ansible_ssh_private_key_file=~/.ssh/deploy_key
```

### Playbook 编写
```yaml
# deploy.yml
---
- name: Deploy Laravel Application
  hosts: web
  become: true
  vars:
    app_dir: /var/www/b2c-api
    app_env: production

  tasks:
    - name: Pull latest code
      git:
        repo: "git@github.com:org/b2c-api.git"
        dest: "{{ app_dir }}"
        version: main

    - name: Install Composer dependencies
      composer:
        command: install
        working_dir: "{{ app_dir }}"
        no_dev: true
        optimize_autoloader: true

    - name: Run migrations
      command: php artisan migrate --force
      args:
        chdir: "{{ app_dir }}"

    - name: Restart PHP-FPM
      service:
        name: php8.3-fpm
        state: restarted
```

### Role 组织
```
roles/
├── common/          # 基础配置（时区、NTP、SSH 加固）
│   └── tasks/
│       └── main.yml
├── php/             # PHP 环境安装与配置
│   ├── tasks/
│   ├── templates/
│   └── handlers/
├── nginx/           # Nginx 配置
│   ├── tasks/
│   └── templates/
└── deploy/          # 应用部署
    └── tasks/
        └── main.yml
```

### Vault 密钥管理
Ansible Vault 用于加密敏感数据（数据库密码、API Key）：

```bash
# 创建加密文件
ansible-vault create secrets.yml

# 编辑加密文件
ansible-vault edit secrets.yml

# 运行时解密
ansible-playbook deploy.yml --ask-vault-pass
```

## 部署策略

### 滚动部署（Rolling Deployment）
逐台服务器更新，保证始终有实例提供服务：

```yaml
- hosts: web
  serial: 1          # 每次更新 1 台
  max_fail_percentage: 0  # 任何失败都停止
  tasks:
    - name: Remove from load balancer
      # ...
    - name: Deploy new version
      # ...
    - name: Health check
      # ...
    - name: Add back to load balancer
      # ...
```

### 零停机发布
结合 Laravel 的维护模式与健康检查：

1. 从负载均衡器摘除实例
2. 进入维护模式（`php artisan down`）
3. 部署新代码
4. 运行迁移
5. 退出维护模式（`php artisan up`）
6. 健康检查通过后重新加入负载均衡器

## 幂等性保障
幂等性是 Ansible 的核心设计原则：同一个 Playbook 运行多次，结果应该相同。关键实践：
- 使用 `creates`/`removes` 参数避免重复执行
- 使用 `when` 条件判断，仅在需要时执行
- 使用模块（`apt`/`yum`/`service`）而非裸 `command`

## 常见问题

### 幂等性违背
- `command`/`shell` 模块默认不幂等，使用 `creates`/`removes` 或 `changed_when`
- 使用 `check_mode: yes`（dry run）预览变更

### Handler 时序问题
- Handler 在 Play 结束时统一执行，不是在 Task 触发时立即执行
- 使用 `meta: flush_handlers` 强制立即执行

### SSH 连接超时
- 在 `ansible.cfg` 中设置 `timeout = 30`
- 使用 ControlPersist 复用 SSH 连接
- 对大规模场景考虑使用 Mitogen 插件加速
