# 蓝绿部署与零停机发布

## 定义

蓝绿部署（Blue-Green Deployment）是一种零停机发布策略，通过维护两套完全相同的生产环境（蓝色 = 当前版本，绿色 = 新版本），在新版本验证通过后，通过负载均衡器一键切换流量，实现秒级发布和回滚。与滚动更新不同，蓝绿部署在整个切换过程中始终有一套完整的旧环境可用，回滚只需将流量切回旧环境，无需重新部署。

## 核心原理

### 蓝绿部署流程

```
                    ┌──── 负载均衡器 ────┐
                    │                    │
            ┌───────┴───────┐    ┌───────┴───────┐
            │  Blue 环境    │    │  Green 环境   │
            │  (当前版本)    │    │  (新版本)     │
            │  v1.2.3       │    │  v1.2.4       │
            │  ← 流量 100%  │    │  流量 0% →    │
            └───────────────┘    └───────────────┘

步骤 1: 部署新版本到 Green 环境
步骤 2: 在 Green 环境运行冒烟测试
步骤 3: 切换流量: Blue 0% ← → Green 100%
步骤 4: Blue 保留作为回滚目标
步骤 5: 验证稳定后，Blue 更新为下一个版本
```

### 关键挑战：数据库迁移

蓝绿部署最大的挑战不是流量切换，而是 **数据库 Schema 迁移**。因为蓝绿两个版本可能同时连接同一个数据库，Schema 变更必须兼容两个版本：

**向前兼容迁移策略**：
1. **只增不删**：新版本先添加列/表，不删除旧列
2. **分步迁移**：
   - Step 1: 添加新列，代码同时写入新旧列
   - Step 2: 迁移历史数据到新列
   - Step 3: 切换代码读取新列
   - Step 4: 下次发布时删除旧列
3. **Feature Flag**：通过配置开关控制新旧逻辑，而非依赖 Schema 变更

### 多区域部署

全球化应用需要在多个地理区域部署，以降低延迟并满足数据合规要求：

```
┌──────────────────────────────────────────────────┐
│                 全球流量调度（DNS/CDN）             │
│                                                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐       │
│  │ US-East  │  │ EU-West  │  │ AP-East  │       │
│  │ Laravel  │  │ Laravel  │  │ Laravel  │       │
│  │ MySQL    │  │ MySQL    │  │ MySQL    │       │
│  │ Redis    │  │ Redis    │  │ Redis    │       │
│  └────┬─────┘  └────┬─────┘  └────┬─────┘       │
│       │              │              │             │
│       └──────────────┴──────────────┘             │
│              跨区域数据同步                        │
│         (MySQL 主从 / CRDT / CDC)                │
└──────────────────────────────────────────────────┘
```

**多区域一致性策略**：
- **强一致**：所有区域写入同一主库（延迟高，但数据一致）
- **最终一致**：各区域独立写入，通过 CDC/Replication 异步同步（延迟低，但有冲突风险）
- **混合策略**：核心交易走强一致，非核心数据走最终一致

### 发布策略对比

| 策略 | 停机时间 | 回滚速度 | 资源开销 | 数据库兼容性要求 |
|---|---|---|---|---|
| **蓝绿部署** | 零 | 秒级 | 2x | 高（两版本共用 DB） |
| **滚动更新** | 零 | 中等（逐 Pod 回滚） | 1x | 中 |
| **金丝雀发布** | 零 | 秒级 | 1.1x | 高 |
| **A/B 测试** | 零 | 秒级 | 1.1x | 高 |

## 常见问题

**Q: 蓝绿部署需要双倍服务器资源吗？**
A: 是的，这是蓝绿部署的主要成本。在 K8s 环境中可以通过 HPA（Horizontal Pod Autoscaler）在切换后缩容旧环境来降低开销。如果资源有限，可以考虑金丝雀发布（只需额外 10% 资源）。

**Q: 数据库迁移怎么做才能不影响旧版本？**
A: 核心原则是「只增不删」和「分步迁移」。新版本先添加新列/表，旧列保留不动。代码层兼容读写新旧列。确认所有实例都更新到新版本后，再删除旧列。工具推荐 Laravel 的 `php artisan migrate` 配合 `doctrine/dbal` 做安全的列变更。

**Q: 如何验证新版本没问题再切流量？**
A: 三道关卡：(1) CI 流水线的自动化测试；(2) 部署到 Green 环境后的冒烟测试（Smoke Test）；(3) 切换 10% 流量到 Green 环境观察 5-10 分钟的错误率和延迟指标。
