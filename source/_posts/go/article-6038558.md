# Go 模块化架构：从单体到微服务的工程化实践

## 概述

PHP 开发者第一次接触 Go 项目时，最常见的困惑是：没有 Composer、没有 autoload，代码组织该怎么做？Go 的模块系统看似简单（一个 `go.mod` 就完事了），但在大型项目中，如何划分包、如何控制依赖方向、如何让模块既能独立编译又能灵活组合——这些才是工程化的真正挑战。

本文从一个实际场景出发：你有一个运行了 3 年的 Go 单体服务，代码量超过 10 万行，启动一次编译要 5 分钟，团队从 3 人扩展到 15 人，代码冲突频繁，每次发版都要协调 5 个团队。如何通过模块化重构，将它拆分为可独立部署、可独立开发的微服务架构？

## 核心概念

### Go Module vs PHP Composer

在深入架构设计之前，先理清 Go Module 和 PHP Composer 的对应关系：

| 概念 | PHP Composer | Go Module |
|------|-------------|-----------|
| 包定义 | `composer.json` | `go.mod` |
| 版本约束 | `^1.2` / `~1.2.3` | `v1.2.3` / `v1.x.x` |
| 私有仓库 | `repositories` + `auth.json` | `GOPRIVATE` + `.netrc` |
| 命名空间 | PSR-4 autoload | 包路径 = import 路径 |
| 版本锁定 | `composer.lock` | `go.sum` |
| 依赖下载 | `vendor/` | `$GOPATH/pkg/mod/` |

关键区别：PHP 的 autoload 是运行时机制，Go 的包引用是编译时绑定。这意味着 Go 的包边界更严格——你不能在运行时"偷偷"调用一个没导入的包。

### Go 的包设计原则

Go 社区有一条黄金法则：**包的职责应该小到可以用一句话描述**。

```go
// ❌ 错误：一个包做太多事
package myapp

func CreateUser(...) error { ... }
func SendEmail(...) error { ... }
func ProcessPayment(...) error { ... }
func GenerateReport(...) error { ... }

// ✅ 正确：每个包一个职责
package user
func Create(...) error { ... }

package mail
func Send(...) error { ... }

package payment
func Process(...) error { ... }

package report
func Generate(...) error { ... }
```

这和 PHP 的 SRP（单一职责原则）异曲同工，但 Go 的包系统强制你把职责分离写到目录结构里——你没法在一个文件里"顺便"写一个不相关的函数。

### 内部包（internal）：Go 独有的可见性控制

Go 1.4 引入的 `internal` 目录是控制包可见性的杀手锏。任何放在 `internal/` 下的包，只有其父目录及其子目录能导入：

```
myproject/
├── cmd/
│   └── server/
│       └── main.go
├── internal/
│   ├── auth/          # 只有 cmd/server 能用
│   │   └── auth.go
│   └── billing/       # 只有 cmd/server 能用
│       └── billing.go
├── pkg/               # 所有人都能用的公共包
│   └── httputil/
│       └── response.go
└── go.mod
```

这相当于 PHP 中的 `private` 访问级别，但作用于包级别。在 PHP 中，你只能通过命名约定（如 `_` 前缀）来暗示"这是内部的"，但 Go 编译器会强制执行这个约束。

## 实战：单体到模块化的重构

### 第一步：识别模块边界

先梳理当前单体的代码，找出高内聚的代码块：

```bash
# 分析 import 关系，找出耦合度高的包
go list -f '{{.ImportPath}}: {{join .Imports ", "}}' ./... | \
  awk -F: '{print $1}' | sort | uniq -c | sort -rn

# 找出循环依赖的包
go build ./... 2>&1 | grep "import cycle"
```

以一个典型的电商系统为例，我们可以识别出这些模块：

```
myproject/
├── cmd/
│   ├── api-gateway/       # API 网关
│   ├── user-service/      # 用户服务
│   ├── order-service/     # 订单服务
│   └── payment-service/   # 支付服务
├── internal/
│   ├── domain/            # 领域模型（DDD）
│   │   ├── user/
│   │   ├── order/
│   │   └── payment/
│   ├── application/       # 应用服务层
│   │   ├── user/
│   │   ├── order/
│   │   └── payment/
│   ├── infrastructure/    # 基础设施层
│   │   ├── database/
│   │   ├── cache/
│   │   └── messaging/
│   └── shared/            # 跨模块共享
│       ├── errors/
│       ├── logger/
│       └── config/
├── pkg/                   # 可被外部项目引用的公共库
│   └── httputil/
└── go.mod
```

### 第二步：定义模块接口

模块之间通过接口通信，而不是直接依赖具体实现。这是 Go 的"依赖倒置"实践：

```go
// internal/domain/user/repository.go
package user

import "context"

// Repository 定义用户存储接口
type Repository interface {
    FindByID(ctx context.Context, id int64) (*User, error)
    FindByEmail(ctx context.Context, email string) (*User, error)
    Save(ctx context.Context, user *User) error
    Delete(ctx context.Context, id int64) error
}

// Service 定义用户业务逻辑接口
type Service interface {
    Register(ctx context.Context, input RegisterInput) (*User, error)
    GetProfile(ctx context.Context, id int64) (*User, error)
    UpdateProfile(ctx context.Context, id int64, input UpdateInput) error
}

type RegisterInput struct {
    Email    string `json:"email" validate:"required,email"`
    Password string `json:"password" validate:"required,min=8"`
    Name     string `json:"name" validate:"required"`
}
```

```go
// internal/infrastructure/database/user_repo.go
package database

import (
    "context"
    "database/sql"
    
    "myproject/internal/domain/user"
)

// PostgresRepository 实现 user.Repository
type PostgresRepository struct {
    db *sql.DB
}

func NewPostgresRepository(db *sql.DB) *PostgresRepository {
    return &PostgresRepository{db: db}
}

func (r *PostgresRepository) FindByID(ctx context.Context, id int64) (*user.User, error) {
    var u user.User
    err := r.db.QueryRowContext(ctx,
        "SELECT id, email, name, created_at FROM users WHERE id = $1", id,
    ).Scan(&u.ID, &u.Email, &u.Name, &u.CreatedAt)
    if err == sql.ErrNoRows {
        return nil, user.ErrNotFound
    }
    if err != nil {
        return nil, fmt.Errorf("query user by id: %w", err)
    }
    return &u, nil
}

// FindByEmail 和 Save 方法类似，这里省略
```

### 第三步：依赖注入——Go 的"容器"

Go 没有 Laravel 的 IoC 容器，但我们可以用更简单的方式实现依赖注入：

```go
// internal/application/user/service.go
package user

import (
    "context"
    "fmt"
    
    "myproject/internal/domain/user"
)

// ApplicationService 用户应用服务
type ApplicationService struct {
    repo       user.Repository       // 依赖接口，不依赖具体实现
    mailer     Mailer                 // 邮件服务
    cache      Cache                  // 缓存服务
}

// Mailer 和 Cache 也是接口，定义在各自模块
type Mailer interface {
    Send(ctx context.Context, to, subject, body string) error
}

type Cache interface {
    Get(ctx context.Context, key string) (string, error)
    Set(ctx context.Context, key, value string, ttl int) error
}

// NewApplicationService 构造函数注入依赖
func NewApplicationService(
    repo user.Repository,
    mailer Mailer,
    cache Cache,
) *ApplicationService {
    return &ApplicationService{
        repo:   repo,
        mailer: mailer,
        cache:  cache,
    }
}

// Register 用户注册
func (s *ApplicationService) Register(ctx context.Context, input user.RegisterInput) (*user.User, error) {
    // 检查邮箱是否已存在
    existing, err := s.repo.FindByEmail(ctx, input.Email)
    if err != nil && err != user.ErrNotFound {
        return nil, fmt.Errorf("check existing user: %w", err)
    }
    if existing != nil {
        return nil, user.ErrEmailExists
    }

    // 创建用户
    u := &user.User{
        Email:    input.Email,
        Name:     input.Name,
        Password: hashPassword(input.Password), // 实际应用中用 bcrypt
    }
    
    if err := s.repo.Save(ctx, u); err != nil {
        return nil, fmt.Errorf("save user: %w", err)
    }

    // 发送欢迎邮件（异步）
    go func() {
        _ = s.mailer.Send(ctx, u.Email, "Welcome!", "Hello "+u.Name)
    }()

    return u, nil
}
```

### 第四步：Wire——自动依赖注入

手动写构造函数在依赖多时会很痛苦。Google 的 `wire` 工具可以自动生成依赖注入代码：

```go
// cmd/api-gateway/wire.go
//go:build wireinject
// +build wireinject

package main

import (
    "database/sql"
    
    "myproject/internal/application/user"
    "myproject/internal/infrastructure/database"
    "myproject/internal/infrastructure/mail"
    "myproject/internal/infrastructure/cache"
)

func InitializeUserService(db *sql.DB) *user.ApplicationService {
    wire.Build(
        database.NewPostgresRepository,
        mail.NewSMTPMailer,
        cache.NewRedisCache,
        user.NewApplicationService,
    )
    return nil
}
```

运行 `wire ./cmd/api-gateway` 会自动生成：

```go
// cmd/api-gateway/wire_gen.go
// Code generated by Wire. DO NOT EDIT.

package main

import (
    "database/sql"
    
    "myproject/internal/application/user"
    "myproject/internal/infrastructure/cache"
    "myproject/internal/infrastructure/database"
    "myproject/internal/infrastructure/mail"
)

func InitializeUserService(db *sql.DB) *user.ApplicationService {
    postgresRepository := database.NewPostgresRepository(db)
    smtpMailer := mail.NewSMTPMailer()
    redisCache := cache.NewRedisCache()
    applicationService := user.NewApplicationService(postgresRepository, smtpMailer, redisCache)
    return applicationService
}
```

这就像 PHP 中的 `config/services.php`，但类型安全，编译时检查。

## 踩坑记录

### 坑 1：循环依赖——Go 编译器会直接拒绝

PHP 中循环依赖是运行时问题（autoload 触发），Go 中是编译时错误：

```
import cycle not allowed
package myproject/internal/domain/user
  imports myproject/internal/application/user
  imports myproject/internal/domain/user
```

**解决方案**：引入第三包（通常是 interface 包或 event 包）打破循环：

```
domain/user/     → 定义 User 模型和 Repository 接口
domain/event/    → 定义事件接口
application/user → 实现业务逻辑，依赖 domain/user
```

### 坑 2：internal 包的陷阱

`internal` 包的可见性规则是基于目录路径的，不是基于包名的。如果你把 `internal` 放错位置：

```
# ❌ 这样 internal 不会限制访问
myproject/
├── pkg/
│   └── internal/    # 这个 internal 不生效！
│       └── secret/
└── go.mod

# ✅ internal 必须在模块根目录下
myproject/
├── internal/        # 这样才生效
│   └── secret/
└── go.mod
```

### 坑 3：Go 没有"真正的"私有方法

Go 的首字母大小写控制可见性（大写=导出，小写=未导出），但这只在包级别生效。同一包内的任何函数都能访问未导出的字段：

```go
// package user
type User struct {
    password string  // 小写，包外不可见
}

func (u *User) GetPassword() string {
    return u.password  // 同包内可以访问
}

// package other
func Hack(u *user.User) {
    // u.password  // 编译错误：未导出字段
    // 但如果 other 和 user 是同一个包... 就能访问
}
```

**解决方案**：用 `internal` 目录 + 小写字段名实现双重保护。

### 坑 4：go.mod 的 replace 指令——本地开发利器

在微服务拆分过程中，你可能需要同时修改多个模块。`replace` 指令可以让你在本地同时开发多个模块：

```go
// go.mod
module myproject

go 1.22

require (
    myproject/shared v1.2.3
    myproject/user-service v2.0.0
)

// 本地开发时替换为本地路径
replace myproject/shared => ../shared
replace myproject/user-service => ../user-service
```

这就像 PHP 的 `path repository`，但更简单直接。

## 模块边界检查

为了防止团队成员无意间打破模块边界，可以在 CI 中加入检查：

```bash
#!/bin/bash
# scripts/check-boundaries.sh

echo "=== 检查模块边界 ==="

# 检查 internal 包是否被外部引用
ERRORS=0
for pkg in $(go list ./... | grep /internal/); do
    importers=$(go list -f '{{join .Imports "\n"}}' ./... | \
        grep -v "^$pkg" | \
        grep "$(echo $pkg | sed 's|/internal/.*||')" || true)
    
    # 更精确的检查：看是否有非 internal 子包导入了 internal 包
done

# 检查 import cycle
echo "=== 检查循环依赖 ==="
go build ./... 2>&1 | grep "import cycle" && {
    echo "发现循环依赖！"
    exit 1
}

echo "=== 边界检查通过 ==="
```

## PHP 到 Go 的思维转换

| PHP 思维 | Go 思维 |
|---------|---------|
| 一个文件多个类 | 一个文件一个类型（或紧密相关的一组） |
| autoload 按需加载 | 编译时静态链接，不需要 autoload |
| 接口是可选的 | 接口是隐式实现的，鼓励大量使用 |
| 依赖注入容器 | 构造函数注入，Wire 自动生成 |
| 路由定义在一处 | `cmd/` 下每个入口点独立 |
| 中间件链 | `net/http` Handler 包装 |
| 全局配置 | `config` 包 + 环境变量 |

## 总结

Go 的模块化不是"把代码放到不同目录"这么简单。它的核心思想是：

1. **包要小**：一个包只做一件事，用一句话能描述清楚
2. **接口要窄**：只暴露调用者需要的方法，不要做"胖接口"
3. **依赖要反**：高层依赖接口，低层实现接口
4. **边界要硬**：用 `internal` 强制执行，不要靠约定

从 PHP 转 Go 的过程中，最大的收获是：Go 的编译器会替你守护架构边界。PHP 中需要靠团队自律遵守的规则，在 Go 中变成了编译错误。这种"编译器即架构师"的理念，是 Go 在大型项目中保持代码质量的关键。

如果你正在考虑将一个 PHP 单体服务用 Go 重写，建议分阶段进行：先用 Go 重写最核心的模块（通常是订单和支付），保留 PHP 处理业务逻辑较复杂的部分（如报表和营销），通过 gRPC 或 REST API 通信。不要试图一次性重写所有东西——那是在给自己挖坑。
