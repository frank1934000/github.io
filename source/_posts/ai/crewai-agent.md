# CrewAI 实战：多角色 Agent 协作与任务分解策略

在单 Agent 应用还停留在“一个模型接一个提示词”的阶段时，多 Agent 协作已经进入了更接近真实团队运作的工程范式。一个 Agent 负责调研，一个 Agent 负责规划，一个 Agent 负责执行，一个 Agent 负责审校——这种结构不是为了“看起来高级”，而是为了让复杂任务在**职责分离、上下文隔离、结果可审查、流程可扩展**的前提下稳定落地。

CrewAI 正是在这样的背景下受到开发者欢迎的。它没有把重点放在“让一个万能 Agent 做所有事情”，而是强调：**把复杂目标拆成多个角色驱动的任务，让 Agent 像团队一样协作**。这种设计对内容生产、代码审查、研究分析、运营自动化、企业内部知识流程都非常有吸引力。

本文会从实战角度系统拆解 CrewAI 的核心思想与工程方法，重点覆盖以下内容：

- CrewAI 的架构与设计哲学
- Agent / Task / Crew 三个核心抽象
- 角色定义、目标设计与 Backstory 编写技巧
- Sequential 与 Hierarchical 两种流程模式的差异
- 任务委派、任务依赖与结果传递策略
- Memory 与 Context Sharing 的使用边界
- Python 代码示例与落地建议
- 三类典型业务实践：内容生成、代码评审、研究团队

如果你已经会写基础 Prompt，但在构建多角色系统时经常遇到这些问题——

- Agent 角色定义很花哨，但执行结果高度重复
- 上游任务输出无法真正被下游利用
- 多个 Agent 之间互相覆盖上下文，越跑越混乱
- 分层调度时成本失控，流程难以调试
- 任务拆得太细导致系统“看起来协作，实际上低效”

那么 CrewAI 是一个很适合用来建立工程直觉的框架。

## 一、为什么多角色 Agent 协作值得认真设计

很多团队第一次接触多 Agent，会把它理解成“多个 LLM 并排调用”。但从系统设计角度看，多 Agent 不只是并行调用模型，而是把一个复杂目标映射为一组**有边界、有责任、有依赖关系**的执行单元。

这背后有三个现实原因。

### 1. 单 Agent 容易在复杂任务里失真

单 Agent 在以下场景中通常表现不稳定：

1. 任务目标很长，包含多个阶段
2. 既要检索事实，又要规划步骤，还要输出最终成果
3. 需要多轮审查和修正
4. 不同阶段的成功标准不同

例如“写一篇技术白皮书”并不是一个单一动作，而是：

- 确定主题与目标受众
- 收集资料并验证来源
- 提炼结构与关键论点
- 生成初稿
- 进行事实核验与风格编辑
- 输出最终版本

把这些步骤强行塞给同一个 Agent，通常会出现两种情况：

- 要么 prompt 过长，模型无法稳定遵守所有约束
- 要么输出看似完整，但事实、结构、表达和目标受众之间不一致

### 2. 不同任务天然需要不同“人格”和约束

负责研究的 Agent 需要保守、谨慎、重证据；
负责创作的 Agent 需要结构感、表达力和读者意识；
负责审查的 Agent 需要挑错能力和标准化检查表；
负责经理角色的 Agent 需要从全局角度安排资源和任务顺序。

这些角色即便底层调用的是同一个模型，也应该通过不同的系统提示、工具权限、输入上下文和输出格式来体现差异。CrewAI 的价值就在于，它把这种差异明确建模为框架能力，而不是让开发者靠一堆零散 prompt 手搓。

### 3. 多角色系统更适合工程化治理

从生产系统角度，工程团队真正关心的是：

- 哪个阶段失败了
- 哪个任务的输入有问题
- 哪个角色输出质量最差
- 哪些步骤可以复用
- 哪些阶段必须加人工审核
- 哪个环节最耗 token、最耗时

只有当任务被拆成多个 Agent 与 Task，问题才变得可观察、可调优、可替换。CrewAI 在这方面提供了很好的抽象基础。

## 二、CrewAI 的架构与设计哲学

CrewAI 的核心理念可以概括成一句话：**把 AI 工作流从“提示词编排”提升为“团队协作编排”**。

它不是单纯追求链式调用，而是强调三个层面的结构化：

1. **角色结构化**：谁来做
2. **任务结构化**：做什么
3. **团队结构化**：如何协作完成

这三个层面正对应 CrewAI 的三个核心对象：

- `Agent`
- `Task`
- `Crew`

### 1. Agent：执行者与责任边界

Agent 不是“一个模型实例”这么简单，它代表的是一个有角色定位的执行单元。一个 CrewAI Agent 通常包含：

- role：角色名称
- goal：目标
- backstory：背景故事
- tools：可用工具
- memory / verbose / delegation 等行为配置
- llm：所使用的语言模型

在工程语义上，Agent 是“**谁对这类任务负责**”的描述。

### 2. Task：任务定义与输出契约

Task 是 Agent 要完成的具体工作描述，它通常包含：

- description：任务说明
- expected_output：期望输出形式
- agent：执行该任务的 Agent
- context：依赖的上游任务输出
- async_execution / output_file 等执行配置

Task 是“**要交付什么成果**”的描述，而不是泛泛告诉模型“帮我做一下”。

### 3. Crew：协作团队与流程控制器

Crew 把多个 Agent 和 Task 编排成一个可执行团队。它负责：

- 定义任务执行顺序
- 管理上下文传递
- 决定流程模式（如 sequential 或 hierarchical）
- 控制是否允许 delegation
- 管理共享记忆与整体执行过程

可以把它理解为一个运行时团队容器。

### 4. 设计哲学：先分工，再协作，最后汇总

CrewAI 的思想与很多“万能 AI 助手”路线不同。它默认认为：

- 复杂任务不应该一次性解决
- 每个角色应只承担有限责任
- 上下游输出要形成明确接口
- 协作关系要比 prompt 技巧更重要

这意味着，在 CrewAI 中，好的结果往往不是来自“写出最牛的总 prompt”，而是来自：

- 角色划分是否合理
- 任务边界是否清晰
- 输出契约是否可消费
- 依赖关系是否正确
- 过程模式是否与问题复杂度匹配

这也是为什么很多新手写的 CrewAI Demo 看起来能跑，但一旦换成真实业务数据就崩：问题通常不在模型，而在任务分解。

## 三、Agent / Task / Crew 三个核心概念的工程理解

接下来我们不只讲定义，而是从“怎么设计才稳定”的角度展开。

## 3.1 Agent：不是角色名称，而是职责封装

很多人定义 Agent 时只写一句：

- 研究员
- 编辑
- 程序员
- 审核员

这远远不够。一个真正可用的 Agent 至少要回答四个问题：

1. 它负责什么
2. 它不负责什么
3. 它依据什么标准判断输出质量
4. 它有哪些工具和上下文权限

例如一个“技术研究员”Agent 的有效定义不应该只是“擅长研究”，而应该像这样：

- 负责检索、归纳、比对技术资料
- 不负责产出最终营销文案
- 必须标记结论依据、区分事实与推测
- 可以访问搜索工具与内部知识库，但不直接调用部署工具

这样的定义才能避免角色漂移。

## 3.2 Task：最小可交付单元，而不是动作列表

任务设计最常见的错误是把 Task 写成流水账，例如：

- 查资料
- 分析一下
- 输出报告

这种写法没有定义交付标准，下游也无法消费结果。更好的 Task 设计应该具备：

- 明确输入范围
- 明确处理动作
- 明确输出格式
- 明确质量标准

比如：

> 请针对给定主题收集 8-10 条高可信资料，按“事实、来源、关键结论、争议点”四栏整理，并输出 Markdown 表格，供后续内容策划任务使用。

这就不是“做个研究”，而是一个能被下游可靠复用的任务。

## 3.3 Crew：不是容器，而是协作机制

很多示例中，Crew 只是把几个 Agent 和 Task 塞进去执行。但在实际系统里，Crew 承担的是“工作组织者”的职责。它决定：

- 是按固定顺序依次推进，还是由管理者动态调度
- 上游结果是否自动注入下游
- 是否允许某个 Agent 再把任务委托出去
- 是否启用记忆机制
- 是否需要保留执行轨迹用于调试

也就是说，Crew 决定的是“团队制度”，不是简单的“运行入口”。

## 四、角色定义与 Backstory 设计：为什么它比你想象中更重要

在 CrewAI 中，很多人最容易轻视的字段就是 `backstory`。从表面看，它像是一个“人格设定”；但从模型行为引导看，它实际上承担着非常关键的约束作用。

### 4.1 Backstory 的本质：给模型一个稳定的决策偏好

LLM 在不同 prompt 下会表现出不同风格，但如果只有 role 和 goal，模型往往仍然缺少足够稳定的行为偏向。Backstory 的作用是告诉模型：

- 你为什么这样思考
- 你习惯如何判断好坏
- 你在哪些方面特别谨慎或特别擅长
- 你应该优先关注什么，忽略什么

例如下面两个 Agent 看起来都叫“编辑”：

**版本 A**：
> 你是一名技术编辑，负责润色文章。

**版本 B**：
> 你曾长期担任开发者媒体的资深技术编辑，擅长将复杂工程概念改写为结构清晰、逻辑严谨、术语一致的中文技术文章。你非常重视事实准确性、章节衔接和示例可复现性，讨厌空洞宣传和未经验证的结论。

显然，版本 B 更可能让模型在输出时保持“技术编辑”而不是“市场文案编辑”的风格。

### 4.2 好的 Backstory 应包含哪些元素

建议从以下五个维度编写：

1. **经验来源**：这个角色曾做过什么
2. **核心偏好**：它优先关注什么
3. **方法论**：它通常如何处理任务
4. **质量标准**：它怎样定义高质量结果
5. **禁忌事项**：它会避免什么错误

例如：

```python
researcher = Agent(
    role="AI 技术研究员",
    goal="收集并验证与主题相关的高可信技术资料，为后续写作提供可靠事实基础",
    backstory=(
        "你曾为技术咨询公司和开发者媒体提供前沿技术调研，"
        "擅长从官方文档、RFC、技术博客和社区讨论中识别高价值信息。"
        "你重视来源可信度、时间新鲜度和术语准确性，"
        "会主动区分事实、经验结论与未经验证的推测。"
        "如果信息存在争议，你会明确指出，而不是强行给出确定答案。"
    ),
    verbose=True
)
```

### 4.3 Backstory 不是越长越好

需要注意的是，Backstory 并不是文学创作。它太短会失去约束，太长则会：

- 增加 token 成本
- 稀释关键信号
- 让不同 Agent 的人设过度重叠

建议把 Backstory 控制在“足以形成稳定偏好，但不挤压任务上下文”的长度。通常 80 到 200 字的中文描述已经可以达到很好的效果。

### 4.4 角色设计中的一个关键原则：互补，而不是重复

多角色系统失败的一个高频原因，是多个 Agent 的设定实际上高度重复。比如：

- 战略分析师：负责分析
- 研究顾问：负责研究与分析
- 内容专家：负责内容分析与输出

这种定义很容易让三者都在做相似的事情，结果是输出同质化严重，协作没有真正发生。

正确做法是让角色之间形成**能力互补**：

- 研究员：找事实
- 策划者：定结构
- 写作者：写成文
- 审校者：找问题
- 经理：调度和收敛

只有边界清晰，协作才有价值。

## 五、Sequential 与 Hierarchical：两种流程模式怎么选

CrewAI 中最关键的运行设计之一，就是 `Process.sequential` 与 `Process.hierarchical` 的选择。这不是一个“高级模式和普通模式”的区别，而是两种完全不同的组织逻辑。

## 5.1 Sequential：固定顺序、强约束、易调试

顺序执行模式最容易理解：任务按预设次序依次执行，上一个任务的结果作为下一个任务的上下文或输入。

它适合以下场景：

- 业务流程稳定，步骤确定
- 依赖链清晰
- 需要高可控性和易调试性
- 不希望模型自行决定流程

典型例子：

1. 调研主题
2. 输出文章大纲
3. 生成初稿
4. 校验事实与表达
5. 输出终稿

这类流程强烈建议先用 sequential，因为：

- 每一步都能单独验证
- 更容易定位是哪一步污染了结果
- 更容易做缓存与重试
- 更适合生产环境的稳定任务

### Sequential 示例

```python
from crewai import Agent, Task, Crew, Process

planner = Agent(
    role="内容策划师",
    goal="根据研究资料制定结构完整、逻辑清晰的文章大纲",
    backstory="你是一名擅长技术专题策划的编辑，重视读者路径、章节层次和信息密度。",
    verbose=True
)

writer = Agent(
    role="技术作者",
    goal="根据大纲和研究资料写出清晰、可读、可复现的技术文章",
    backstory="你专注于面向开发者的中文技术写作，擅长把抽象概念拆解成可理解的例子。",
    verbose=True
)

editor = Agent(
    role="技术审校",
    goal="审查文章中的事实准确性、逻辑漏洞与表达问题",
    backstory="你习惯用审稿清单检查术语一致性、因果关系和示例完整性。",
    verbose=True
)

outline_task = Task(
    description="基于给定研究资料，设计一篇关于 CrewAI 多角色协作的详细文章大纲。",
    expected_output="Markdown 格式的大纲，包含一级标题、二级标题和每节核心要点。",
    agent=planner
)

writing_task = Task(
    description="根据大纲写出完整文章初稿，语言要求专业、清晰，包含示例和实践建议。",
    expected_output="一篇结构完整的 Markdown 技术文章初稿。",
    agent=writer,
    context=[outline_task]
)

review_task = Task(
    description="审查文章初稿，指出事实、结构、表达与重复问题，并给出修改后的终稿。",
    expected_output="经过修订的最终 Markdown 文章。",
    agent=editor,
    context=[writing_task]
)

crew = Crew(
    agents=[planner, writer, editor],
    tasks=[outline_task, writing_task, review_task],
    process=Process.sequential,
    verbose=True
)

result = crew.kickoff()
print(result)
```

## 5.2 Hierarchical：由管理者动态分派，更接近真实团队

层级模式则更像一个“经理 + 执行团队”的组织结构。管理者 Agent 会根据目标、任务状态和各角色能力动态决定：

- 该把任务分给谁
- 是否需要进一步拆解
- 哪个任务先做
- 是否追加检查或补充信息

这种模式适合：

- 任务路径不固定
- 复杂目标需要动态规划
- 有较多不确定性和中间决策点
- 希望系统具备一定自治能力

例如“为一个新产品进入海外市场制定竞争分析方案”，管理者可能会根据当前信息决定先做市场研究，再做竞品拆解，或者先做风险清单，再回头补行业趋势。

### Hierarchical 的优点

- 适应不确定问题
- 能根据中间结果重组流程
- 更接近人类团队的协作形态

### Hierarchical 的代价

- token 消耗更高
- 执行轨迹更复杂
- 更难保证结果完全稳定
- 调试成本明显增加

对于生产系统，我的建议是：**先用 Sequential 验证任务拆解是否合理，再考虑是否值得升级到 Hierarchical**。不要为了“更智能”而过早选择更复杂的流程。

### Hierarchical 示例

```python
from crewai import Agent, Task, Crew, Process

manager = Agent(
    role="项目经理",
    goal="根据总体目标动态分配任务，协调研究、写作和审校角色完成交付",
    backstory=(
        "你曾负责跨职能内容团队的项目管理，善于判断任务依赖、发现瓶颈并安排合适的人完成工作。"
    ),
    allow_delegation=True,
    verbose=True
)

researcher = Agent(
    role="研究分析师",
    goal="收集事实并形成结构化资料",
    backstory="你重视信息来源的可信度与可追溯性。",
    verbose=True
)

writer = Agent(
    role="技术写作者",
    goal="将资料和结构转化为高质量技术内容",
    backstory="你擅长兼顾准确性与可读性。",
    verbose=True
)

reviewer = Agent(
    role="质量审查员",
    goal="发现文稿中的逻辑问题、术语错误与风险表述",
    backstory="你会严格对照交付标准审查结果。",
    verbose=True
)

main_task = Task(
    description="围绕 CrewAI 多角色协作与任务分解策略，完成一篇面向开发者的深度技术文章。",
    expected_output="最终可发布的 Markdown 技术文章。",
    agent=manager
)

crew = Crew(
    agents=[manager, researcher, writer, reviewer],
    tasks=[main_task],
    process=Process.hierarchical,
    manager_agent=manager,
    verbose=True
)

result = crew.kickoff()
print(result)
```

## 六、任务委派与任务依赖：多 Agent 成败的真正分水岭

多角色系统真正难的地方，不是“定义几个 Agent”，而是让任务之间形成有效依赖，而不是信息重复。

## 6.1 什么是任务委派

任务委派（delegation）指的是某个 Agent 不直接完成全部工作，而是把部分子任务交给更合适的 Agent。它强调的是：

- 当前角色知道自己不能或不该独立完成全部事情
- 它能识别需要哪些补充角色
- 它能将目标转化为其他角色可执行的子任务

例如项目经理 Agent 接到“产出一份竞品分析报告”，它可能会委派：

- 给研究员：搜集竞品资料
- 给分析师：构建对比维度
- 给写作者：生成对外报告初稿
- 给审核者：检查事实与措辞风险

委派能力让系统更灵活，但也更容易失控。

## 6.2 什么时候应该允许 delegation

以下情况适合开启：

- 任务路径不确定
- 需要管理型 Agent 动态拆解任务
- 子任务选择依赖中间结果

以下情况不建议开启：

- 工作流非常固定
- 成本敏感
- 需要高度可预测结果
- 团队还没有验证过角色边界

很多项目一开始就打开 delegation，结果出现两类问题：

1. 任务被过度拆解，执行轮次暴增
2. 多个 Agent 来回转发信息，最终没有实质推进

所以 delegation 最好只给“经理型”或“协调型”角色，而不是给所有 Agent。

## 6.3 任务依赖：下游必须消费上游，而不是忽略上游

在 CrewAI 中，Task 可以通过 `context` 引用其他任务输出。这看起来简单，但设计上非常关键。

错误做法通常是：

- 上游任务输出一大段自由文本
- 下游任务只“参考一下”
- 最终下游重新生成内容，完全没有利用结构化结果

正确做法是让上游任务输出具备明确接口。例如：

- 研究任务输出表格或要点清单
- 规划任务输出文章结构树
- 审查任务输出问题列表 + 修改建议

这样下游才能真正“吃掉”上游结果。

### 任务依赖示例

```python
research_task = Task(
    description=(
        "收集 CrewAI 在多角色协作中的关键概念、流程模式、记忆机制与常见应用案例，"
        "按主题整理为结构化研究笔记。"
    ),
    expected_output=(
        "Markdown 列表，分为：架构概念、流程模式、任务拆解、内存机制、实战案例五部分，"
        "每部分至少包含 5 条要点。"
    ),
    agent=researcher
)

planning_task = Task(
    description="根据研究笔记设计文章大纲，确保结构层层递进，覆盖概念、方法、代码与案例。",
    expected_output="一份详细 Markdown 大纲，每节标明写作目标与关键信息来源。",
    agent=planner,
    context=[research_task]
)

writing_task = Task(
    description="根据研究笔记和文章大纲撰写完整初稿。",
    expected_output="一篇完整文章初稿。",
    agent=writer,
    context=[research_task, planning_task]
)
```

这里的关键不是“有 context”，而是：

- 研究任务提供事实基础
- 规划任务提供结构骨架
- 写作任务同时消费两类不同价值的信息

这才是真正的任务依赖设计。

## 七、Memory 与 Context Sharing：何时共享，何时隔离

多 Agent 系统中，另一个高频误区是“共享越多越智能”。实际上，**上下文共享过度，往往比共享不足更危险**。

### 7.1 Context Sharing 的价值

上下文共享的意义在于避免重复劳动，让下游 Agent 站在上游结果之上继续推进。例如：

- 研究员已经整理了事实，写作者无需重新检索
- 规划者已经确定结构，编辑无需重新定义文章主线
- 审核者可以直接查看初稿与任务标准的差距

如果没有上下文共享，多角色系统会退化成多个互不相干的单 Agent。

### 7.2 共享过度的风险

但如果所有历史对话、所有任务输出、所有草稿版本都无差别注入给下游，会出现三个问题：

1. **噪音过大**：模型难以抓住关键约束
2. **目标漂移**：不同版本内容相互干扰
3. **成本失控**：token 快速膨胀

因此最重要的原则是：**共享结果，不共享杂音；共享结构，不共享全部过程。**

### 7.3 Memory 的作用边界

CrewAI 中的 Memory 通常用于让 Agent 或团队保留一部分历史状态，用来改善连续任务表现。但要清楚，它不是万能记忆库，而是一个需要精心治理的状态层。

常见的 Memory 用法包括：

- 记录任务执行历史
- 保存中间结论
- 维护偏好信息
- 在多轮交互中避免重复决策

但 Memory 也会带来风险：

- 旧结论污染新任务
- 错误信息被长期保留
- 提示膨胀导致质量下降

所以在生产实践里，我更建议采用：

- **短期上下文**：本轮任务明确需要的上游结果
- **长期记忆**：经过筛选、结构化、可验证的稳定信息

不要把所有中间推理都塞进长期记忆。

### 7.4 一个实用策略：分层上下文模型

一个比较稳的设计是把上下文分成三层：

1. **任务输入层**：当前任务必须看到的信息
2. **团队共享层**：多个角色都需要的共识信息，如目标、受众、交付标准
3. **长期记忆层**：历史项目中沉淀下来的稳定知识

例如写文章团队里：

- 当前任务输入层：本次主题研究笔记、大纲、初稿
- 团队共享层：目标读者、文章长度、风格规范、禁用表达
- 长期记忆层：术语表、品牌语气指南、历史最佳实践

这样能显著降低上下文污染。

## 八、Python 实战：构建一个内容生产 Crew

下面我们做一个更完整的实战示例：一个用于技术博客写作的多 Agent 团队。它包含四个角色：

- 研究员：搜集事实与案例
- 策划师：设计文章结构
- 作者：撰写正文
- 审校者：检查质量并输出终稿

### 8.1 安装与准备

```bash
pip install crewai
```

如果你通过虚拟环境管理项目，建议配合 `.env` 保存模型 API Key，例如 OpenAI、Anthropic 或兼容 OpenAI API 的服务。

```python
import os
from crewai import Agent, Task, Crew, Process
from langchain_openai import ChatOpenAI
```

### 8.2 定义角色

```python
researcher = Agent(
    role="技术研究员",
    goal="收集与主题相关的事实、案例、框架机制和实践建议",
    backstory=(
        "你长期负责开发者技术调研，善于从官方文档、社区经验和工程案例中整理出"
        "高可信、可复用的信息，并明确区分事实和经验性判断。"
    ),
    llm=llm,
    verbose=True
)

planner = Agent(
    role="内容策划师",
    goal="根据研究资料制定对开发者友好的文章结构",
    backstory=(
        "你是一名资深技术编辑，擅长根据读者认知路径安排章节层次，"
        "确保文章从概念到实战逐步深入。"
    ),
    llm=llm,
    verbose=True
)

writer = Agent(
    role="技术作者",
    goal="输出兼顾准确性、可读性与可操作性的中文技术文章",

)

    role="技术审校",
    goal="从事实、结构、术语与表达一致性角度优化文章质量",
    backstory=(
        "你有丰富的技术审稿经验，会严格检查逻辑跳跃、术语误用、重复段落与不完整示例。"
    ),
    llm=llm,
    verbose=True

```

### 8.3 定义任务

```python
research_task = Task(
    description=(
        "围绕主题‘CrewAI 实战：多角色 Agent 协作与任务分解策略’，"
        "整理以下内容：架构理念、Agent/Task/Crew 核心概念、角色设计、"
        "Sequential 与 Hierarchical 差异、任务委派、任务依赖、Memory 与 Context Sharing、"
        "以及内容生成、代码审查、研究分析三个典型场景。"
    ),
    expected_output=(
        "结构化研究笔记，使用 Markdown 二级标题分组，每组包含关键概念、实践建议和注意事项。"
    ),
    agent=researcher
)

planning_task = Task(
    description="基于研究笔记输出一份层次清晰、适合深度博客的文章大纲。",
    expected_output="详细 Markdown 大纲，至少包含 10 个一级或二级主题，并给出每节写作目标。",
    agent=planner,
    context=[research_task]
)

writing_task = Task(
    description=(
        "根据研究笔记和文章大纲撰写完整博客文章。要求语言专业、中文自然、"
        "兼顾概念解释、工程实践与代码示例。"
    ),
    expected_output="完整 Markdown 初稿。",
    agent=writer,
    context=[research_task, planning_task]
)

review_task = Task(
    description=(
        "审查初稿中的事实准确性、段落衔接、重复表达、术语一致性与代码解释清晰度，"
        "并输出最终可发布版本。"
    ),
    expected_output="最终 Markdown 成稿。",
    agent=reviewer,
    context=[writing_task]
)
```

### 8.4 组装 Crew 并执行

```python
crew = Crew(
    agents=[researcher, planner, writer, reviewer],
    tasks=[research_task, planning_task, writing_task, review_task],
    process=Process.sequential,
    verbose=True
)

result = crew.kickoff()
print(result)
```

### 8.5 为什么这个结构比“一个写作 Agent”更稳

因为它把四类能力拆开了：

- **研究真实性** 与 **表达能力** 不再混在一起
- **结构规划** 成为独立中间产物
- **审校** 不依赖写作者自我修正
- 每一步都能单独评估与替换

如果你要进一步工程化，还可以对每个 Task 增加：

- 输出格式校验
- 重试策略
- 人工审批节点
- 结果落盘与缓存
- tracing / logging / evaluation

## 九、实战场景一：内容生成团队

这是 CrewAI 最适合落地的典型场景之一，因为内容生产天然就是多角色协作。

### 9.1 典型角色划分

一个稳定的内容生成团队可以这样设计：

- **Research Agent**：收集资料、提炼事实、标记来源
- **Outline Agent**：设计结构、控制信息层次
- **Writer Agent**：生成初稿、解释概念、串联案例
- **Editor Agent**：润色语言、减少重复、保证术语一致
- **SEO Agent**（可选）：优化标题、摘要、关键词布局

### 9.2 任务拆分建议

很多团队容易犯的错误，是让写作者同时负责研究、结构、正文和 SEO。短期看调用次数少，长期看质量波动很大。

更合理的拆分是：

1. 研究任务只负责“信息正确”
2. 大纲任务只负责“结构合理”
3. 写作任务只负责“表达完整”
4. 编辑任务只负责“质量优化”
5. SEO 任务只负责“搜索可发现性”

这样每个环节都能单独打分和迭代。

### 9.3 适合 Sequential 还是 Hierarchical

- 固定栏目、批量生产：优先 Sequential
- 专题策划、主题探索：可以尝试 Hierarchical

对于博客、知识库、白皮书、课程讲义这类内容，Sequential 往往已经足够强大。

## 十、实战场景二：代码评审团队

代码审查是另一个非常适合多 Agent 的方向，因为它天然包含不同维度的检查。

### 10.1 为什么单个代码审查 Agent 不够

一次高质量 Code Review 通常要同时看：

- 业务逻辑是否正确
- 安全风险是否存在
- 性能问题是否明显
- 可维护性与可读性是否达标
- 测试覆盖是否足够

让一个 Agent 同时承担全部维度，常常会导致“泛泛指出几个问题”，但缺少系统性。更好的做法是把审查维度拆开。

### 10.2 一个实用的代码评审 Crew

可以设计以下角色：

- **架构审查 Agent**：关注模块边界、依赖关系、抽象层次
- **安全审查 Agent**：关注注入、鉴权、数据泄露、敏感操作
- **性能审查 Agent**：关注复杂度、IO、缓存与数据库访问模式
- **测试审查 Agent**：关注单元测试、边界情况、回归风险
- **汇总 Agent**：整合各方意见，输出最终 Review 报告

### 10.3 任务输出格式建议

代码评审最怕各个 Agent 输出“想法很多，但没法执行”。因此任务输出最好规范成：

- 问题级别：Critical / High / Medium / Low
- 影响范围：模块、函数、接口
- 触发条件：在什么情况下发生
- 修复建议：可操作方案
- 是否需要补测试：是 / 否

如果上游 Agent 都按同一种结构输出，汇总 Agent 的工作会非常顺畅。

### 10.4 一个示例思路

```python
security_review_task = Task(
    description="审查代码中的安全风险，包括输入校验、权限控制、敏感数据处理和外部调用风险。",
    expected_output="结构化安全审查报告，按风险级别列出问题、原因、影响和修复建议。",
    agent=security_reviewer
)

performance_review_task = Task(
    description="审查代码中的性能隐患，包括重复查询、无效循环、序列化开销与阻塞式操作。",
    expected_output="结构化性能审查报告。",
    agent=performance_reviewer
)

summary_task = Task(
    description="汇总安全与性能审查结果，形成统一的代码评审结论与优先级修复建议。",
    expected_output="最终 Code Review 报告。",
    agent=lead_reviewer,
    context=[security_review_task, performance_review_task]
)
```

这种设计比“一个 Agent 直接 review 全部代码”更适合企业团队，因为它能对齐真实审查职责。

## 十一、实战场景三：研究分析团队

研究类任务是 Hierarchical 模式最容易发挥价值的场景之一。

例如你要做一份“AI Agent 框架选型报告”，其中可能包含：

- 产品定位比较
- 架构设计比较
- 社区生态比较
- 成本与性能比较
- 安全与治理能力比较
- 场景适配建议

这类任务路径并不固定，管理者可能需要根据某一方向发现的新信息，临时追加分析任务。

### 11.1 一个典型研究 Crew 的角色

- **Research Lead**：管理研究路径，决定拆解方向
- **Market Analyst**：分析市场定位和用户群
- **Technical Analyst**：分析架构与工程特征
- **Benchmark Analyst**：比较成本、性能和运维复杂度
- **Report Writer**：汇总观点并形成可读报告

### 11.2 为什么研究任务特别需要任务依赖设计

研究任务最大的风险不是“写不出来”，而是：

- 不同子研究结论标准不一致
- 数据来源不统一
- 观点和事实混在一起
- 最终报告只是信息堆砌

因此你需要强制每个分析子任务输出统一模板，例如：

- 研究对象
- 数据来源
- 发现结论
- 不确定因素
- 对最终决策的影响

这样 Report Writer 才能把多个子任务拼成真正可决策的报告。

### 11.3 Hierarchical 在研究任务中的优势

研究型管理者 Agent 可以根据中间发现动态决定：

- 某个框架是否值得深挖
- 是否需要补安全治理维度
- 是否需要增加时间成本测算
- 是否要让报告写作延后，先补事实缺口

这类动态调整是 Sequential 很难优雅表达的。

### 11.4 CrewAI vs AutoGen vs LangGraph 选型对比

当研究主题本身就是“多 Agent 框架怎么选”时，建议在报告中加入一张统一比较表，避免团队只凭印象做决策。

| 维度 | CrewAI | AutoGen | LangGraph |
| --- | --- | --- | --- |
| 核心范式 | 角色 + 任务驱动的团队协作 | 对话驱动的多 Agent 协作 | 有状态图驱动的流程编排 |
| 最适合场景 | 内容生产、研究分析、职责明确的分工任务 | 多角色讨论、代码执行、自动修复闭环 | 生产级工作流、复杂分支、强状态控制 |
| 结构表达 | Agent / Task / Crew 清晰分层 | Agent 间消息交换与群聊机制 | Node / Edge / State 显式建模 |
| 优势 | 上手快、分工直观、任务委派自然 | 协作感强、对话灵活、代码沙箱能力突出 | 可恢复、可追踪、可治理，适合长流程系统 |
| 挑战 | 动态状态流转表达不如图式框架灵活 | 容易出现上下文膨胀与终止条件失控 | 学习成本更高，前期建模工作更多 |
| 选型建议 | 先验证团队式协作流程时优先考虑 | 需要“讨论—执行—修复”闭环时优先考虑 | 对稳定性、审计性、流程控制要求高时优先考虑 |

## 十二、任务分解策略：如何避免“伪协作”

很多多 Agent 系统看起来很复杂，实际上只是把同一个 prompt 拆成了三份而已。要避免这种“伪协作”，需要掌握几个任务分解原则。

### 12.1 按职责拆，不按句子拆

错误拆法：

- Agent A：先介绍背景
- Agent B：再介绍架构
- Agent C：最后总结

这只是按写作段落拆，不是按能力拆。

正确拆法：

- Agent A：负责搜集和验证背景事实
- Agent B：负责基于事实设计内容结构
- Agent C：负责把结构和事实转化为清晰文章
- Agent D：负责质量把关

### 12.2 按决策点拆，不按形式拆

复杂任务里最重要的是决策点，而不是文本长度。例如：

- 是否需要补充资料
- 哪种结构最适合目标读者
- 哪些问题风险最高
- 哪些任务必须人工介入

如果某一步包含关键判断，就值得成为独立 Task。

### 12.3 保持每个 Task 的“单一可验证目标”

一个好的 Task 应该能被独立验收。比如：

- “产出一份包含 8 个章节的大纲”可以验收
- “写得更好一点”不能验收

### 12.4 降低跨任务语义损耗

跨任务交接时，要尽量减少自由文本的歧义。方法包括：

- 使用结构化输出
- 强制字段模板
- 给出示例格式
- 用评分标准约束输出质量

这会显著提升多 Agent 的稳定性。

---

## 十三、生产落地建议：从 Demo 到系统的几个关键点

CrewAI 很适合快速搭建原型，但真正落地时，需要补上很多工程能力。

### 13.1 先做可观测性，再做规模化

你至少需要记录：

- 每个 Task 的输入与输出摘要
- 每个 Agent 的执行耗时
- token 消耗
- 错误与重试次数
- 任务间依赖链

否则一旦结果不稳定，你根本不知道问题出在哪一层。

### 13.2 不要过度依赖“自治”

层级型协作看起来很聪明，但生产环境最重要的是确定性和成本可控。对于高频任务，通常应优先：

- 固定流程
- 固定输出模板
- 限制 delegation
- 限制最大迭代轮数

### 13.3 给关键任务增加人工审核门槛

对于以下场景，强烈建议加入人工审批：

- 对外发布内容
- 涉及法律、合规、安全风险的结论
- 自动提交代码或执行部署操作
- 涉及采购、财务、客户承诺的信息

多 Agent 不等于可以完全无人监管。

### 13.4 把高价值中间产物沉淀为资产

例如：

- 研究笔记模板
- 大纲模板
- 审稿清单
- 代码评审规则库
- 行业术语词典

这些资产一旦结构化积累，CrewAI 的效果会比单纯堆 prompt 好很多。

## 十四、常见误区与避坑建议

### 14.1 误区一：Agent 越多越强

错误。Agent 数量增加意味着：

- 更多 token 开销
- 更长执行路径
- 更多上下文传递损耗
- 更复杂的调试难度

如果两个 Agent 的职责区分不明显，通常应该合并。

### 14.2 误区二：Backstory 写得越戏剧化越好

错误。Backstory 的目标是形成稳定行为偏好，不是塑造小说人物。重点应该是专业经验、判断标准和约束边界。

### 14.3 误区三：所有任务都适合 Hierarchical

错误。层级模式适合高不确定性问题，不适合高频稳定流水线。多数内容生产和标准审查任务，Sequential 更稳。

### 14.4 误区四：共享全部上下文最保险

错误。共享过多会稀释关键信号，还会增加成本。应共享“必要结果”，而不是“所有历史”。

### 14.5 误区五：任务描述足够详细就不需要 expected_output

错误。`expected_output` 是输出契约，没有它，下游很难可靠消费结果。特别是在多任务依赖链中，这个字段价值极高。

## 十五、一个可复用的设计模板

如果你要从零开始搭建 CrewAI 系统，可以按下面的步骤来：

### 步骤 1：先画出业务流程

明确：

- 最终交付物是什么
- 中间需要哪些阶段成果
- 哪些阶段存在明显不同的质量标准

### 步骤 2：识别角色，而不是先写 Agent

问自己：

- 谁负责找事实
- 谁负责规划
- 谁负责执行
- 谁负责审核
- 谁负责协调

### 步骤 3：为每个角色定义边界

包括：

- 负责什么
- 不负责什么
- 输入是什么
- 输出是什么
- 成功标准是什么

### 步骤 4：把阶段成果写成 Task 契约

每个 Task 都要写清：

- description
- expected_output
- 上游依赖
- 是否允许委派

### 步骤 5：优先用 Sequential 验证

只有当你发现：

- 固定流程覆盖不了变化
- 管理型角色确实需要动态拆分
- 中间决策对路径影响很大

再考虑 Hierarchical。

### 步骤 6：最后再考虑 Memory

不要一开始就把 Memory 当核心能力。先证明任务拆解与上下文接口本身是合理的，再决定哪些信息值得长期保留。

## 十六、总结：CrewAI 真正有价值的不是“多 Agent”，而是“可治理的协作”

CrewAI 之所以值得学习，并不只是因为它能让多个 Agent 一起工作，而是因为它把复杂任务的工程化组织方式清晰地抽象出来了。

它让我们开始用团队协作的视角思考 AI 系统：

- 谁来负责
- 做什么任务
- 任务如何拆解
- 上下游如何交接
- 什么时候需要管理者
- 哪些信息应该共享，哪些应该隔离
- 如何在质量、成本、自治性之间取得平衡

从实践经验来看，CrewAI 最适合解决的不是“让 AI 一次性做完所有事”，而是“让复杂工作被拆成多个可验证、可协同、可优化的步骤”。

如果你刚开始尝试多 Agent，我建议按下面的顺序推进：

1. 先从 2-4 个角色的 Sequential 流程开始
2. 用清晰的 Task 输出契约替代模糊描述
3. 把 Backstory 当作行为约束，而不是装饰文字
4. 严格控制上下文共享范围
5. 在确认任务边界稳定后，再尝试 delegation 和 hierarchical

当你真正把这些原则落到代码和流程里时，就会发现：多 Agent 系统的关键竞争力，从来不是“角色多”，而是**分工清晰、任务可验收、协作可治理**。

这才是 CrewAI 在实战中最值得掌握的部分。
