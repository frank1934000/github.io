# 三大框架多平台能力对比：传输层实现、格式适配、群聊行为策略

## 前言

2026 年，AI Agent 已经从「实验室里的对话机器人」进化为「嵌入开发者日常工作流的智能助手」。而衡量一个 AI Agent 框架是否成熟的关键指标之一，就是它能覆盖多少消息平台、在不同平台间的行为是否一致、以及面对平台差异时的适配能力。

Hermes Agent、OpenClaw 和 OpenHuman 是当前开源 AI Agent 领域的三大主流框架。它们各自代表了不同的设计哲学：Hermes 以注册表驱动和技能系统见长，OpenClaw 以文件原生心智架构取胜，OpenHuman 则以本地优先和记忆树独树一帜。但在多平台能力这个维度上，三者的差异更加微妙且实用。

本文将从**传输层实现、消息格式适配、群聊行为策略**三个核心维度，深入对比三大框架的多平台能力，帮助开发者在实际选型时做出更明智的决策。

---

## 一、支持平台全景概览

### 1.1 Hermes Agent 的平台矩阵

Hermes Agent 采用**插件化传输层架构**，通过 `PluginContext` 注册不同的消息通道。当前官方支持的平台包括：

| 平台 | 支持状态 | 传输方式 | 备注 |
|------|---------|---------|------|
| Telegram | ✅ 完整支持 | Bot API (HTTP) | 主力平台，功能最全 |
| Discord | ✅ 完整支持 | Gateway WebSocket | Bot + Slash Commands |
| 微信 | ⚠️ 社区支持 | iLink 协议 / 企业微信 API | 非官方方案，需自行适配 |
| WhatsApp | ⚠️ 实验性 | WhatsApp Business API | 需要 Business 账号 |
| Slack | ✅ 完整支持 | Socket Mode / Events API | 企业级集成 |
| 飞书 | ✅ 完整支持 | 飞书开放平台 API | 国内团队常用 |
| 终端 (CLI) | ✅ 内置 | stdio | 开发调试主力 |
| Web UI | ✅ 内置 | HTTP/SSE | 自带 Chat 界面 |

Hermes 的平台支持策略是**核心平台深度集成 + 外围平台通过 Plugin 扩展**。Telegram 和 Discord 作为一等公民，获得了最完整的功能支持，包括文件上传/下载、内联键盘、消息编辑等。

### 1.2 OpenClaw 的平台矩阵

OpenClaw 采用**文件原生的配置驱动模式**，平台集成通过 `SOUL.md` 和各平台的 connector 脚本实现：

| 平台 | 支持状态 | 传输方式 | 备注 |
|------|---------|---------|------|
| Telegram | ✅ 完整支持 | python-telegram-bot | 核心平台 |
| Discord | ✅ 完整支持 | discord.py | 多频道管理 |
| 微信 | ✅ 深度支持 | iLink 协议 | 官方维护的微信集成 |
| WhatsApp | ✅ 官方支持 | whatsapp-web.js / Baileys | 社区活跃 |
| Slack | ✅ 完整支持 | Bolt SDK | 企业场景 |
| 飞书 | ✅ 完整支持 | 飞书 SDK | daily-report.py 五通道之一 |
| QQ | ✅ 社区支持 | go-cqhttp / NapCat | 国内特色 |
| Signal | ⚠️ 实验性 | signal-cli | 隐私场景 |

OpenClaw 的平台覆盖范围在三者中**最为广泛**，尤其是对国内平台（微信、QQ、飞书）的支持更为深入。其 `daily-report.py` 实现了五通道统一分发，是多平台能力的典型体现。

### 1.3 OpenHuman 的平台矩阵

OpenHuman 采用**源适配器（Source Adapter）架构**，通过统一的数据摄入管道处理多平台消息：

| 平台 | 支持状态 | 传输方式 | 备注 |
|------|---------|---------|------|
| Telegram | ✅ 完整支持 | Telegram Bot API | 核心通道 |
| Discord | ✅ 完整支持 | Discord.js | 社区管理 |
| Slack | ✅ 完整支持 | Slack SDK | 工作流集成 |
| Gmail | ✅ 深度支持 | Google OAuth + IMAP | 邮件处理 |
| GitHub | ✅ 深度支持 | GitHub API | Issue/PR 集成 |
| Notion | ✅ 完整支持 | Notion API | 知识库同步 |
| Google Meet | ✅ 独特支持 | Webview 嵌入 | 实时会议参与 |
| 微信 | ⚠️ 社区支持 | 第三方桥接 | 非官方方案 |

OpenHuman 的平台策略与其他两者有显著差异：它不追求消息平台的数量，而是追求**集成的深度**。118+ 集成中包含了大量非消息类平台（Gmail、Notion、GitHub、Google Calendar 等），体现了其「AI 超级智能助手」的定位。

## 二、传输层实现深度对比

### 2.1 Hermes Agent：注册表驱动的插件化传输

Hermes 的传输层架构基于 `ProviderProfile` 和 `PluginContext` 双注册表机制：

```
┌─────────────────────────────────────────┐
│              Hermes Core                │
│  ┌───────────┐  ┌────────────────────┐  │
│  │ Provider  │  │  PluginContext     │  │
│  │ Profile   │  │  ┌──────────────┐  │  │
│  │ Registry  │  │  │ TelegramPlugin│  │  │
│  │           │  │  ├──────────────┤  │  │
│  │ - model   │  │  │ DiscordPlugin │  │  │
│  │ - provider│  │  ├──────────────┤  │  │
│  │ - fallback│  │  │ SlackPlugin   │  │  │
│  └───────────┘  │  ├──────────────┤  │  │
│                 │  │ WebPlugin     │  │  │
│                 │  └──────────────┘  │  │
│                 └────────────────────┘  │
└─────────────────────────────────────────┘
```

**Telegram 传输层实现特点：**

- 使用 HTTP 长轮询（getUpdates）或 Webhook 两种模式
- 支持消息编辑（editMessage）、内联键盘（InlineKeyboardMarkup）
- 文件处理通过 Bot API 的 getFile/downloadFile 链路
- 支持消息分块发送（Telegram 单条消息 4096 字符限制）

**Discord 传输层实现特点：**

- 基于 Gateway WebSocket 的全双工通信
- 支持 Slash Commands 和交互组件（按钮、下拉菜单）
- Embed 消息格式支持丰富的卡片展示
- 消息长度限制 2000 字符，支持自动分块

**核心设计决策：**

Hermes 在传输层的关键设计是**将传输协议与消息处理逻辑解耦**。每个 Plugin 只负责「收消息」和「发消息」，所有 AI 推理、记忆管理、工具调用都在 Core 层完成。这意味着新增一个平台只需要实现 Plugin 接口的 `receive()` 和 `send()` 方法。

```python
# Hermes Plugin 伪代码示例
class TelegramPlugin(Plugin):
    def register(self, context: PluginContext):
        context.register_tool("telegram_send", self.send_message)
        context.register_hook("on_message", self.handle_incoming)
    
    async def handle_incoming(self, message: IncomingMessage):
        # 统一转换为 InternalMessage
        return InternalMessage(
            content=message.text,
            platform="telegram",
            user_id=message.from_user.id,
            metadata={"chat_id": message.chat_id}
        )
    
    async def send_message(self, target: str, content: str):
        # 处理分块、格式转换、错误重试
        chunks = split_message(content, max_length=4096)
        for chunk in chunks:
            await self.bot.send_message(target, chunk, parse_mode="Markdown")
```

### 2.2 OpenClaw：Python 脚本驱动的灵活传输

OpenClaw 的传输层更加「脚本化」，每个平台都有独立的 Python 脚本负责消息收发：

```
openclaw/
├── connectors/
│   ├── telegram_bot.py      # python-telegram-bot
│   ├── discord_bot.py       # discord.py
│   ├── wechat_link.py       # iLink 协议
│   ├── whatsapp_baileys.js  # Node.js 桥接
│   ├── slack_bolt.py        # Slack Bolt SDK
│   └── feishu_sdk.py        # 飞书 SDK
├── daily-report.py          # 五通道统一分发
└── heartbeat-notify.py      # 多通道告警
```

**OpenClaw 的传输层特色——`daily-report.py` 五通道分发：**

这是 OpenClaw 多平台能力的典型代表。一个脚本同时向 Telegram、Slack、飞书、QQ、微信五个平台发送日报：

```python
# daily-report.py 核心逻辑（简化）
async def broadcast_daily_report(report: str):
    channels = [
        TelegramChannel(bot_token=os.getenv("TELEGRAM_TOKEN")),
        SlackChannel(webhook=os.getenv("SLACK_WEBHOOK")),
        FeishuChannel(app_id=os.getenv("FEISHU_APP_ID")),
        QQChannel(cqhttp_url=os.getenv("CQHTTP_URL")),
        WeChatChannel(ilink_url=os.getenv("ILINK_URL")),
    ]
    
    for channel in channels:
        try:
            formatted = channel.adapt_format(report)
            await channel.send(formatted)
        except Exception as e:
            log.warning(f"Channel {channel.name} failed: {e}")
            # 不阻塞其他通道
```

**微信集成——iLink 协议深度剖析：**

OpenClaw 对微信的支持是三者中最深入的。其 `wechat_link.py` 使用 iLink 协议，通过 `X-WECHAT-UIN` 头部机制实现消息的收发：

```python
# iLink 协议核心
class WeChatILinkConnector:
    def __init__(self, ilink_server: str, bot_token: str):
        self.server = ilink_server
        self.token = bot_token
    
    async def send_message(self, uin: str, content: str):
        # iLink 使用 HTTP API 转发消息
        headers = {"X-WECHAT-UIN": uin, "Authorization": f"Bearer {self.token}"}
        await self.http.post(f"{self.server}/api/send", 
                           json={"to": uin, "content": content},
                           headers=headers)
    
    async def listen(self):
        # WebSocket 长连接接收消息
        async with websockets.connect(f"{self.server}/ws") as ws:
            async for message in ws:
                yield self.parse_message(message)
```

### 2.3 OpenHuman：源适配器架构的统一管道

OpenHuman 的传输层采用了最系统化的架构——**源适配器管道（Source Adapter Pipeline）**：

```
┌──────────────────────────────────────────────────────┐
│                  OpenHuman Adapter Pipeline           │
│                                                      │
│  ┌─────────┐   ┌──────────┐   ┌──────────┐         │
│  │ Source   │──▶│Normalize │──▶│ Chunking │──▶ ...  │
│  │ Adapter  │   │ Layer    │   │ Engine   │         │
│  └─────────┘   └──────────┘   └──────────┘         │
│       │                                              │
│  ┌────┴──────────────────────────────────────┐      │
│  │  Telegram │ Discord │ Gmail │ GitHub │ ... │      │
│  └───────────────────────────────────────────┘      │
│                                                      │
│  统一消息模型：CanonicalMessage                       │
│  - content: str                                      │
│  - platform: str                                     │
│  - timestamp: datetime                               │
│  - attachments: List[Attachment]                     │
│  - thread_id: Optional[str]                          │
│  - metadata: Dict[str, Any]                          │
└──────────────────────────────────────────────────────┘
```

**源适配器的核心设计原则：**

1. **摄入（Ingestion）**：每个平台有独立的 Adapter，负责认证和原始数据获取
2. **规范化（Normalization）**：将平台特有的消息格式转换为 `CanonicalMessage`
3. **分块（Chunking）**：长消息按语义边界切分，适合后续的 Memory Tree 处理
4. **索引（Indexing）**：提取实体、关键词，建立可搜索的索引

```python
# OpenHuman Source Adapter 伪代码
class GmailAdapter(SourceAdapter):
    platform = "gmail"
    
    async def ingest(self, since: datetime) -> List[RawMessage]:
        """从 Gmail 拉取新邮件"""
        service = build('gmail', 'v1', credentials=self.creds)
        results = service.users().messages().list(
            userId='me', q=f'after:{since.strftime("%Y/%m/%d")}'
        ).execute()
        return [self._fetch_message(m['id']) for m in results.get('messages', [])]
    
    def normalize(self, raw: RawMessage) -> CanonicalMessage:
        """将 Gmail 消息规范化"""
        return CanonicalMessage(
            content=extract_body(raw),
            platform="gmail",
            timestamp=parse_date(raw['Date']),
            attachments=extract_attachments(raw),
            metadata={"subject": raw['Subject'], "from": raw['From']}
        )
```

**AutoFetch 调度器——每 20 分钟的智能上下文拉取：**

OpenHuman 独特的 AutoFetch 机制会每 20 分钟遍历所有已连接的源适配器，自动拉取新数据：

```python
class AutoFetchScheduler:
    INTERVAL = 20 * 60  # 20 分钟
    
    async def run_cycle(self):
        for connection in self.connection_manager.get_active():
            adapter = self.adapter_registry.get(connection.platform)
            new_messages = await adapter.ingest(since=connection.last_sync)
            
            for msg in new_messages:
                canonical = adapter.normalize(msg)
                await self.memory_tree.ingest(canonical)
            
            connection.last_sync = datetime.now()
            connection.sync_state.save()
```

| 维度 | Hermes Agent | OpenClaw | OpenHuman |
|------|-------------|---------|----------|
| 架构模式 | 插件注册表 | Python 脚本 | 源适配器管道 |
| 新增平台成本 | 实现 Plugin 接口 | 新增 connector 脚本 | 新增 SourceAdapter |

## 三、消息格式适配策略

### 3.1 跨平台格式差异的挑战

不同消息平台对富文本的支持程度差异巨大：

| 平台 | Markdown | HTML | 富文本 | 卡片消息 | 代码块 | 文件 | 图片 |
|------|---------|------|--------|---------|--------|------|------|
| Telegram | ✅ 子集 | ❌ | ❌ | InlineKeyboard | ✅ | ✅ | ✅ |
| Discord | ✅ 子集 | ❌ | ❌ | Embed | ✅ | ✅ | ✅ |
| 微信 | ❌ | ❌ | ❌ | 小程序卡片 | ❌ 部分 | ✅ | ✅ |
| Slack | ✅ mrkdwn | ❌ | ✅ Block Kit | ✅ | ✅ | ✅ | ✅ |
| 飞书 | ✅ 子集 | ❌ | ✅ 消息卡片 | ✅ Interactive | ✅ | ✅ | ✅ |
| WhatsApp | ✅ 极小子集 | ❌ | ❌ | ❌ | ✅ | ✅ | ✅ |

### 3.2 Hermes Agent 的格式适配策略

Hermes 采用**Markdown-first + 平台降级**策略：

```python
class MarkdownAdapter:
    """统一的 Markdown 格式适配器"""
    
    def adapt(self, markdown: str, platform: str) -> str:
        # 先解析 Markdown AST
        ast = parse_markdown(markdown)
        
        # 根据平台能力裁剪
        if platform == "telegram":
            return self._to_telegram_markdown(ast)
        elif platform == "discord":
            return self._to_discord_markdown(ast)
        elif platform == "slack":
            return self._to_slack_mrkdwn(ast)
        elif platform == "wechat":
            return self._to_plain_text(ast)  # 降级为纯文本
        # ...
    
    def _to_telegram_markdown(self, ast) -> str:
        """Telegram MarkdownV2 特殊字符转义"""
        # Telegram 的 Markdown 实现有特殊要求
        # 需要转义: _ * [ ] ( ) ~ ` > # + - = | { } . !
        text = render_markdown(ast)
        return escape_telegram_special_chars(text)
    
    def _to_slack_mrkdwn(self, ast) -> str:
        """Slack mrkdwn 格式（注意不是标准 Markdown）"""
        # Slack 使用 *bold* 而非 **bold**
        # 使用 _italic_ 而非 *italic*
        # 代码块用 ``` 而非缩进
        return convert_to_slack_mrkdwn(ast)
```

**Hermes 的消息分块策略：**

```python
def split_message(content: str, max_length: int) -> List[str]:
    """
    智能分块：优先在段落边界切分，其次在句子边界，最后在换行符处
    """
    if len(content) <= max_length:
        return [content]
    
    chunks = []
    current = ""
    
    for paragraph in content.split("\n\n"):
        if len(current) + len(paragraph) + 2 <= max_length:
            current += ("\n\n" if current else "") + paragraph
        else:
            if current:
                chunks.append(current)
            # 段落本身超过限制，需要进一步切分
            if len(paragraph) > max_length:
                chunks.extend(split_by_sentence(paragraph, max_length))
            else:
                current = paragraph
    
    if current:
        chunks.append(current)
    
    return chunks
```

### 3.3 OpenClaw 的格式适配策略

OpenClaw 采用**平台原生格式 + 适配层**策略，每个 connector 自行处理格式转换：

```python
# OpenClaw 的格式适配分散在各 connector 中
class TelegramConnector:
    def format_response(self, content: str) -> str:
        # 根据 HEARTBEAT_OK 规则决定是否发送
        if self.should_be_silent(content):
            return None
        
        # 检测内容类型，选择最佳格式
        if self.is_code_block(content):
            return f"```\n{content}\n```"
        elif self.has_tables(content):
            return self.convert_table_to_text(content)
        else:
            return self.adapt_markdown(content)

class WeChatConnector:
    def format_response(self, content: str) -> str:
        # 微信不支持 Markdown，需要特殊处理
        # 代码块转换为截图或纯文本
        # 链接保留，但去掉 Markdown 语法
        return self.markdown_to_plain(content)
```

**OpenClaw 的群聊格式适配——反应礼仪：**

OpenClaw 在群聊中有一套独特的「反应礼仪」规则：

```python
# heartbeat-notify.py 中的 hash 去重
class ReactionPolicy:
    def should_react(self, message: Message, context: ChatContext) -> bool:
        # 1. 检查是否被 @ 提及
        if message.mentions_me:
            return True
        
        # 2. 检查 HEARTBEAT_OK 状态
        if not self.heartbeat_ok:
            return False
        
        # 3. hash 去重：避免对同一条消息重复反应
        msg_hash = hashlib.md5(message.content.encode()).hexdigest()
        if msg_hash in self.reaction_cache:
            return False
        self.reaction_cache.add(msg_hash)
        
        return True
```

### 3.4 OpenHuman 的格式适配策略

OpenHuman 采用**CanonicalMessage + 渲染器分离**的架构：

```python
# OpenHuman 的格式处理是管道化的一等公民
class FormatPipeline:
    def __init__(self):
        self.renderers = {
            "telegram": TelegramRenderer(),
            "discord": DiscordRenderer(),
            "slack": SlackRenderer(),
            "gmail": EmailRenderer(),
            "web": HTMLRenderer(),
        }
    
    def render(self, canonical: CanonicalMessage, platform: str) -> str:
        renderer = self.renderers[platform]
        
        # 1. 如果有结构化数据，优先使用平台原生卡片
        if canonical.structured_data and renderer.supports_cards:
            return renderer.render_card(canonical.structured_data)
        
        # 2. 否则渲染 Markdown
        if renderer.supports_markdown:
            return renderer.render_markdown(canonical.content)
        
        # 3. 最终降级为纯文本
        return renderer.render_plain(canonical.content)
```

**TokenJuice 压缩与格式优化：**

OpenHuman 独特的 TokenJuice 引擎不仅压缩 Token，还会优化输出格式：

```python
class TokenJuiceFormatter:
    def compress_for_platform(self, content: str, platform: str) -> str:
        # 1. HTML → Markdown 转换
        if "<html>" in content.lower():
            content = html_to_markdown(content)
        
        # 2. URL 缩短（保留域名信息）
        content = self.shorten_urls(content)
        
        # 3. 输出去重（移除重复段落）
        content = self.deduplicate_paragraphs(content)
        
        # 4. 正则噪声过滤
        content = self.filter_noise(content)
        
        # 5. 平台特定优化
        if platform == "telegram":
            content = self.optimize_for_telegram(content)
        
        return content
```

## 四、群聊行为策略深度对比

### 4.1 问题定义：AI Agent 在群聊中的「度」

群聊是 AI Agent 面临的最复杂场景之一。Agent 需要决定：

1. **何时说话？** —— 被 @ 提及时必须回复，但其他时候呢？
2. **何时沉默？** —— 群聊闲聊时是否应该参与？
3. **如何参与？** —— 是主动提供价值还是被动等待指令？
4. **上下文如何处理？** —— 群聊中多个用户的对话如何区分？

### 4.3 OpenClaw 的群聊策略：HEARTBEAT_OK 静默机制

OpenClaw 的群聊策略围绕 **HEARTBEAT_OK** 状态标志展开：

```python
# OpenClaw 群聊行为状态机
class GroupBehaviorStateMachine:
    """
    HEARTBEAT_OK 控制 Agent 在群聊中的活跃度
    
    状态转换：
    - IDLE → ACTIVE: 收到 @ 提及或关键词触发
    - ACTIVE → IDLE: 超时无交互（默认 5 分钟）
    - ANY → QUIET: 进入安静时段
    - QUIET → IDLE: 安静时段结束
    """
    
    def should_respond(self, message: GroupMessage) -> Decision:
        # 1. 直接 @ 提及 → 必须回复
        if message.mentions_me:
            return Decision.RESPOND
        
        # 2. 回复 Agent 的消息 → 回复
        if message.is_reply_to_me:
            return Decision.RESPOND
        
        # 3. HEARTBEAT_OK 为 False → 静默
        if not self.heartbeat_ok:
            return Decision.SILENT
        
        # 4. 安静时段 → 静默
        if self.in_quiet_hours():
            return Decision.SILENT
        
        # 5. 消息 hash 去重
        msg_hash = self.compute_hash(message)
        if msg_hash in self.seen_hashes:
            return Decision.SILENT
        self.seen_hashes.add(msg_hash)
        
        # 6. 内容相关性判断（由 LLM 决策）
        return Decision.ASK_LLM
```

### 4.4 OpenHuman 的群聊策略：上下文感知 + 记忆分区

OpenHuman 的群聊策略更加智能化，基于其 **Memory Tree** 的上下文理解能力：

```python
class OpenHumanGroupStrategy:
    """
    OpenHuman 群聊策略：
    - 上下文感知：理解群聊中的对话流，而非简单关键词匹配
    - 记忆分区：群聊记忆与私聊记忆完全隔离
    - 隐私感知：不在群聊中泄露私聊内容
    """
    
    def __init__(self, memory_tree: MemoryTree):
        self.memory = memory_tree
        self.group_context = GroupContextWindow(max_messages=50)
    
    async def decide_response(self, message: GroupMessage) -> Optional[str]:
        # 1. 更新群聊上下文窗口
        self.group_context.add(message)
        
        # 2. 检查是否需要响应
        if not self.should_respond(message):
            return None
        
        # 3. 从 Memory Tree 检索相关记忆
        #    注意：只检索群聊分区的记忆
        relevant_memories = self.memory.search(
            query=message.content,
            scope="group",  # 关键：不搜索私聊记忆
            top_k=5
        )
        
        # 4. 构建响应上下文
        context = self.build_context(message, relevant_memories)
        
        # 5. 生成响应（考虑群聊的简洁性要求）
        response = await self.llm.generate(
            system_prompt=self.group_system_prompt,
            user_message=context,
            max_tokens=500  # 群聊中保持简洁
        )
        
        return response
    
    def should_respond(self, message: GroupMessage) -> bool:
        """多因素决策"""
        # 直接提及
        if message.mentions_me:
            return True
        
        # 问题检测（即使没有 @，有价值的问题也值得回答）
        if self.is_valuable_question(message):
            return True
        
        # 默认静默
        return False
    
    def is_valuable_question(self, message: GroupMessage) -> bool:
        """
        使用轻量级模型判断消息是否是有价值的问题
        避免对每条群消息都调用大模型
        """
        # 简单规则过滤
        if len(message.content) < 10:
            return False
        if not any(kw in message.content for kw in ["?", "？", "怎么", "为什么", "如何"]):
            return False
        
        # 调用快速模型判断
        return self.fast_model.classify(message.content) == "valuable_question"
```

## 五、实际集成踩坑与最佳实践

### 5.1 Telegram 集成常见坑

**坑 1：MarkdownV2 特殊字符转义**

```
问题：Telegram 的 MarkdownV2 模式要求转义大量特殊字符
      直接发送 Markdown 会报错 "Bad Request: can't parse entities"

解决：
- 使用 parse_mode="MarkdownV2" 并严格转义
- 或使用 parse_mode="HTML" 作为更安全的替代
- 或使用 reply_markup 发送纯文本 + 格式化按钮

Hermes 方案：内置 TelegramMarkdownEscaper，在 Plugin 层自动处理
OpenClaw 方案：telegram_bot.py 中有 format_for_telegram() 函数
OpenHuman 方案：TelegramRenderer 自动选择最佳 parse_mode
```

**坑 2：消息长度限制（4096 字符）**

```
问题：AI 生成的长回复超过 4096 字符会被截断

解决：智能分块发送，优先在段落边界切分

三大框架都有各自的分块策略，但处理方式不同：
- Hermes：在传输层自动分块，对上层透明
- OpenClaw：在 connector 层手动处理
- OpenHuman：在渲染器层处理，支持「继续阅读」按钮
```

**坑 3：文件大小限制（50MB）**

```
问题：Telegram Bot API 文件上传限制 50MB

解决：
- 小文件直接上传
- 大文件压缩后上传或提供下载链接
- 图片自动压缩到平台推荐尺寸
```

### 5.2 Discord 集成常见坑

**坑 1：Gateway 连接断开重连**

```
问题：Discord Gateway WebSocket 在长时间无活动后会断开

解决：
- 实现心跳机制（Discord 要求每 41.25 秒发送一次心跳）
- 监听 RESUME 事件实现无缝重连
- 持久化 session_id 用于断线恢复

Hermes 方案：Plugin 层封装，自动重连
OpenClaw 方案：discord.py 库内置重连逻辑
OpenHuman 方案：ConnectionManager 统一管理
```

**坑 2：Slash Commands 注册延迟**

```
问题：Discord Slash Commands 注册后需要最多 1 小时才能全局生效

解决：
- 开发阶段使用 Guild Commands（即时生效）
- 生产环境使用 Global Commands
- 启动时检查并同步命令列表
```

### 5.3 微信集成常见坑

**坑 1：非官方 API 的封号风险**

```
问题：微信没有官方 Bot API，第三方方案存在封号风险

缓解措施：
- 使用企业微信 API（官方支持）
- iLink 协议通过 Web 协议模拟，风险相对较低
- 控制消息频率，模拟人类行为模式
- 使用专用微信号，不使用个人号
```

**坑 2：消息格式极度受限**

```
问题：微信不支持 Markdown、不支持富文本、代码块显示异常

解决：
- 长文本转为图片发送（微信对图片大小限制较宽松）
- 代码块截图后发送
- 关键信息用纯文本 + emoji 辅助排版
```

## 六、选型建议

### 6.1 按平台需求选型

| 你的需求 | 推荐框架 | 理由 |
|---------|---------|------|
| 主要使用 Telegram/Discord | Hermes Agent | 核心平台深度集成，Profile 隔离适合多场景 |

|---------|---------|------|

## 七、总结

三大框架在多平台能力上各有千秋：

- **Hermes Agent** 的优势在于**架构化和隔离性**。插件化传输层、Profile 隔离、子代理工具限制等设计，让它在企业级多平台场景中表现最佳。但其平台覆盖范围不如 OpenClaw 广泛。

- **OpenClaw** 的优势在于**平台覆盖广度和配置简洁性**。五通道统一分发、iLink 微信深度集成、群聊 HEARTBEAT_OK 机制等，让它成为需要覆盖国内平台的开发者的首选。但其「脚本化」的架构在大规模场景下可能面临维护挑战。

- **OpenHuman** 的优势在于**集成深度和数据管道化**。源适配器架构、AutoFetch 调度、Memory Tree 上下文理解等，让它在处理异构数据源时最为强大。但它对传统消息平台的支持不如前两者深入。

最终的选择取决于你的具体需求：**要广度选 OpenClaw，要深度选 OpenHuman，要架构化选 Hermes**。而随着 AI Agent 生态的成熟，三大框架也在不断借鉴彼此的优点，未来的差距可能会进一步缩小。
