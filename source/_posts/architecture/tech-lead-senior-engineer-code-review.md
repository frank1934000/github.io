## 前言

你是一个出色的 Senior Engineer。你写的代码优雅高效，你解决的技术难题让团队刮目相看，你的 Code Review 总能发现别人遗漏的问题。然后有一天，你的 Manager 找你谈话："我们觉得你适合做 Tech Lead。"

这一刻，很多人以为这只是 title 的变化——从 Senior 变成 Lead，继续写代码就好。但实际上，这是一个**根本性的角色转变**。你不再只是对自己写的代码负责，而是对整个团队的技术产出负责。你不再只是解决问题，而是要**定义问题**、**分配问题**、**确保问题被正确解决**。

本文基于我在多个 Laravel B2C 项目中担任 Tech Lead 的实战经验，完整拆解这个角色跃迁过程中的关键挑战与应对策略。

## 一、角色认知的根本转变

### 1.1 Senior Engineer vs Tech Lead

很多新晋 Tech Lead 最大的误区是：**继续用 Individual Contributor（IC）的方式工作，只是多了一些会议**。

```text
┌─────────────────────────────────────────────────────────────┐
│              Senior Engineer 的工作模式                       │
├─────────────────────────────────────────────────────────────┤
│ 80% 写代码 + 10% Review + 5% 设计 + 5% 沟通                  │
│                                                             │
│ 核心产出：自己写的代码质量                                      │
│ 成功标准：我能解决多难的技术问题                                │
│ 影响范围：自己负责的模块                                       │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│              Tech Lead 的工作模式                             │
├─────────────────────────────────────────────────────────────┤
│ 30% 写代码 + 25% Review + 25% 设计 + 20% 沟通                │
│                                                             │
│ 核心产出：团队的技术决策质量和交付效率                           │
│ 成功标准：团队能解决多复杂的问题                                │
│ 影响范围：整个技术栈和团队的技术文化                             │
└─────────────────────────────────────────────────────────────┘
```

### 1.2 三个必须放下的执念

**执念 1："我是团队最强的程序员"**

作为 Tech Lead，你可能仍然是团队中编码能力最强的人。但你的价值不再是写出最好的代码，而是**让团队中每个人都能写出好代码**。

```text
错误心态：这段代码我来写更快 → 自己写
正确心态：这段代码如果让小王写，他能学到什么？ → 指导小王写 + Review

短期效率下降，但长期团队能力提升
```

**执念 2："我的技术方案一定是最好的"**

Tech Lead 容易陷入"架构师陷阱"——所有技术决策都要自己做。但真正好的 Tech Lead 会**创造一个让好方案自然浮现的环境**。

```text
错误方式：我设计好方案，你们来执行
正确方式：我定义问题边界和约束，我们一起讨论方案，最终我做决策

Tech Lead 的角色不是"最好的设计师"，而是"最好的决策者"
```

**执念 3："我要了解所有技术细节"**

面对日益复杂的技术栈，试图了解每个细节会让你精疲力竭。Tech Lead 需要的是**足够的深度来做出正确判断**，而非事无巨细的了解。

```text
需要深度了解的：
- 系统架构的整体设计和关键约束
- 核心业务逻辑的技术实现
- 技术债务的分布和优先级
- 团队成员的能力和成长方向

可以委托了解的：
- 具体的实现细节（通过 Review 来把关）
- 第三方库的选择（让负责的同学做技术调研）
- 性能优化的微观层面（让专精的同学来处理）
```

### 1.3 Tech Lead 的五重身份

```text
┌──────────────────────────────────────────┐
│              Tech Lead                    │
│                                          │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 架构师   │  │ 导师     │  │ 守门人   │  │
│  │ 设计系统 │  │ 培养团队 │  │ 保证质量 │  │
│  └─────────┘  └─────────┘  └─────────┘  │
│                                          │
│  ┌─────────┐  ┌─────────┐               │
│  │ 翻译者   │  │ 消除者   │               │
│  │ 沟通桥梁 │  │ 消除障碍 │               │
│  └─────────┘  └─────────┘               │
└──────────────────────────────────────────┘
```

## 二、时间分配模型：编码 vs 设计 vs Review vs 沟通

### 2.1 理想的时间分配

一个成熟的 Tech Lead 的一周时间分配（40 小时工作制）：

```text
┌─────────────────────────────────────────────────────┐
│ 周一                                               │
│ 09:00-09:30  周会 + Sprint Planning                  │
│ 09:30-10:30  邮件/Slack 处理 + 优先级梳理             │
│ 10:30-12:00  Code Review（集中批量处理）              │
│ 13:30-15:30  技术设计或编码（深度工作时间）             │
│ 15:30-16:00  1-on-1 会议（团队成员 A）                │
│ 16:00-17:30  跨团队沟通/技术方案讨论                   │
├─────────────────────────────────────────────────────┤
│ 周二-周四                                          │
│ 类似分布，但 1-on-1 分散到不同时间段                  │
│ 周三下午预留 2 小时"技术债务治理"时间                  │
├─────────────────────────────────────────────────────┤
│ 周五                                               │
│ 上午：编码/设计（不被打扰的深度工作）                   │
│ 下午：周回顾 + 下周规划 + 技术分享/学习                │
└─────────────────────────────────────────────────────┘

每周总计：
- Code Review: 8-10h (20-25%)
- 编码: 10-12h (25-30%)
- 技术设计: 8-10h (20-25%)
- 沟通协调: 6-8h (15-20%)
- 1-on-1 辅导: 3-4h (8-10%)
```

### 2.2 保护深度工作时间

Tech Lead 最大的挑战是：**你的时间会被无数的打断消耗殆尽**。

```text
常见的时间杀手：
- Slack 消息通知（平均每天 50-100 条需要回复）
- 临时技术问题咨询
- 会议邀请（"需要你来参加一下"）
- 生产环境告警
- 跨团队对齐会议

应对策略：
1. 设置 Slack "Focus Time" 状态，非紧急问题异步处理
2. 每天固定 2 个 Code Review 时段，而非随到随审
3. 对会议邀请说"不"——先看 Agenda，没有明确议题的不参加
4. 建立问题升级机制，团队成员先互相解决，解决不了再找你
5. 生产告警由 On-Call 轮值处理，TL 只处理 P0
```

### 2.3 随团队规模调整

```text
团队规模    编码时间    Review    设计    沟通    1-on-1
3-5 人     40-50%     15-20%   15%    10-15%  5-10%
6-10 人    20-30%     20-25%   20%    15-20%  10-15%
11-15 人   10-15%     20-25%   20%    25-30%  15-20%

超过 10 人的团队，TL 几乎不写业务代码
应该把编码时间花在工具链、自动化脚本、POC 验证上
```

## 三、架构决策框架

### 3.1 ADR（Architecture Decision Record）

ADR 是记录架构决策的标准格式，核心价值在于**记录决策的上下文和权衡过程**，而非仅仅记录结论。

**ADR 模板：**

```markdown
# ADR-0012: 订单系统从 MySQL 迁移到 TiDB

## 状态
Proposed | Accepted | Deprecated | Superseded

## 背景
- 当前订单表单表已超过 5000 万行，单库写入成为瓶颈
- 每年数据量增长约 30%，预计 2 年后达到 1.5 亿行
- 读写分离已部署但写入瓶颈无法水平扩展

## 决策
采用 TiDB 作为订单系统的存储引擎，保持 MySQL 协议兼容。

## 选项分析

### 选项 A：MySQL 分库分表
- 优点：技术成熟，团队经验丰富
- 缺点：应用层改造大，跨片查询复杂，运维成本高
- 工作量估算：3 人月

### 选项 B：TiDB 分布式数据库
- 优点：MySQL 兼容、自动分片、水平扩展
- 缺点：团队需要学习、TiDB 运维经验不足
- 工作量估算：2 人月（含学习成本）
- 风险：TiDB 在高并发写入场景下的性能需要验证

### 选项 C：PostgreSQL + Citus
- 优点：PostgreSQL 生态强大
- 缺点：与现有 Laravel 项目不兼容，迁移成本最高
- 排除原因：成本过高

## 后果
- 需要投入 2 周进行 TiDB 性能测试
- 需要更新 CI/CD 流水线支持 TiDB
- 团队需要 TiDB 运维培训
- 如果 TiDB 性能不达标，需要回退到选项 A

## 参与决策者
- 张三（TL）、李四（DBA）、王五（Backend）
```

### 3.2 决策矩阵

当面对多个技术选项时，使用加权决策矩阵：

```text
场景：选择 Laravel 的队列驱动

┌────────────┬──────┬────────┬────────┬─────────┬──────────┐
│ 评估维度    │ 权重 │ Redis  │ SQS    │ RabbitMQ│ Database │
├────────────┼──────┼────────┼────────┼─────────┼──────────┤
│ 性能        │ 25%  │ 9 (2.25)│ 7 (1.75)│ 9 (2.25)│ 4 (1.0)  │
│ 可靠性      │ 25%  │ 7 (1.75)│ 9 (2.25)│ 9 (2.25)│ 8 (2.0)  │
│ 运维成本    │ 20%  │ 8 (1.6) │ 9 (1.8) │ 6 (1.2) │ 9 (1.8)  │
│ 团队熟悉度  │ 15%  │ 9 (1.35)│ 5 (0.75)│ 4 (0.6) │ 9 (1.35) │
│ 扩展性      │ 15%  │ 8 (1.2) │ 9 (1.35)│ 8 (1.2) │ 3 (0.45) │
├────────────┼──────┼────────┼────────┼─────────┼──────────┤
│ 总分        │ 100% │ 8.15   │ 7.90   │ 7.50    │ 6.60     │
└────────────┴──────┴────────┴────────┴─────────┴──────────┘

决策：选择 Redis Queue 作为主要队列驱动
理由：综合得分最高，团队最熟悉，已有 Redis 基础设施
注意：需要配置 Redis 持久化以保证消息可靠性
```

### 3.3 架构决策的沟通

```text
Tech Lead 做架构决策时的沟通策略：

1. 提前沟通（决策前）
   - 在方案讨论阶段邀请相关方参与
   - 分享你的思考方向，收集反馈
   - 不要让人觉得"被通知"

2. 决策时
   - 使用 ADR 记录完整的决策过程
   - 明确说明"为什么选 A 而不选 B"
   - 诚实地列出风险和后续计划

3. 决策后
   - 在团队会议中同步决策结论
   - 创建执行计划和里程碑
   - 定期回顾决策效果（Decision Review）

关键原则：
- 决策要快，但不要草率
- 可以逆转的决策快速做，不可逆转的谨慎做
- 每个决策都要有"如果失败了怎么办"的 Plan B
```

## 四、Code Review 文化建设

### 4.1 Code Review 的价值分层

```text
Code Review 的三个价值层次：

Level 1：找 Bug（基础价值）
  - 发现语法错误、逻辑漏洞、边界条件
  - 这是最低层次的价值，应该由 CI/CD 和自动化工具承担大部分

Level 2：知识传递（核心价值）
  - Reviewer 学习新的代码模式
  - Author 了解团队的最佳实践
  - 新成员通过 Review 快速了解系统

Level 3：设计讨论（最高价值）
  - "这段代码应该放在 Service 层还是 Repository 层？"
  - "这个 API 的设计是否符合 RESTful 规范？"
  - "这个功能是否应该做成异步的？"
```

### 4.2 Review Checklist

```markdown
## Laravel Code Review Checklist

### 功能正确性
- [ ] 代码实现了需求描述的功能
- [ ] 边界条件已处理（空值、极大值、并发）
- [ ] 异常情况有合理的错误处理

### 代码质量
- [ ] 遵循 PSR-12 编码规范
- [ ] 方法长度不超过 30 行（超过则考虑拆分）
- [ ] 类职责单一（不超过 3 个 public 方法的 Controller）
- [ ] 命名清晰，避免缩写（$o → $order）

### Laravel 最佳实践
- [ ] 使用 Form Request 验证输入（不在 Controller 中写验证逻辑）
- [ ] 业务逻辑在 Service 层，Controller 只做协调
- [ ] 使用 Eloquent 的 lazy loading 避免 N+1 查询
- [ ] 大量数据操作使用 chunk 而非一次性加载
- [ ] 敏感操作使用 DB::transaction 包裹

### 安全性
- [ ] SQL 查询使用参数绑定（无字符串拼接）
- [ ] 文件上传有类型和大小限制
- [ ] 敏感数据（密码、token）未出现在日志中
- [ ] API 权限检查使用 Policy/Gate

### 性能
- [ ] 数据库查询已索引（检查 WHERE/JOIN 条件）
- [ ] 没有 N+1 查询问题
- [ ] 缓存策略合理（不要缓存频繁变化的数据）
- [ ] 大量数据使用分页

### 可维护性
- [ ] 添加了必要的注释（特别是业务逻辑复杂的部分）
- [ ] 编写了对应的测试用例
- [ ] 配置项使用 env() 而非硬编码
```

### 4.3 如何给出建设性的 Review 评论

```text
❌ 不好的 Review 评论：
"这段代码写得不好" — 太模糊，没有建设性
"你应该用 X 方法" — 只给了结论，没有解释
"这里有 bug" — 没有说明是什么 bug，怎么修

✅ 好的 Review 评论：

技术建议型：
"这里使用了 DB::select() 原生查询，建议改用 Eloquent 的 where() 方法。
原因：1) Eloquent 自动处理参数绑定，更安全
     2) 方便后续添加 Global Scope
     3) 保持项目风格统一

示例：
// Before
$orders = DB::select('SELECT * FROM orders WHERE user_id = ?', [$userId]);

// After
$orders = Order::where('user_id', $userId)->get();"

问题引导型：
"我注意到这里没有处理并发场景——如果两个请求同时到达，
库存可能会被扣成负数。你觉得是否需要加锁？
如果是的话，我建议使用 Redis 分布式锁而非数据库悲观锁，
因为这里不是强一致性的场景。"

认可型：
"这个 Service 的设计很好！将支付逻辑从 Controller 中抽离出来，
职责清晰，后续添加新的支付方式会很方便。"
```

### 4.4 异步 Review 流程

```text
推荐的 Code Review 流程：

1. 提交 PR
   - PR 描述模板：背景、改动说明、测试方式、截图（如有 UI 变更）
   - 自动触发 CI（单元测试 + 代码风格检查 + 静态分析）

2. 分配 Reviewer
   - 至少 1 个 Reviewer（Tech Lead 对架构相关 PR 必须 Review）
   - 使用 GitHub 的 CODEOWNERS 自动分配

3. Review 过程
   - Reviewer 在 4 小时内完成（工作时间内）
   - 使用 Conventional Comments 前缀：
     * [suggestion] 建议性改动
     * [issue] 必须修复的问题
     * [question] 需要解释的地方
     * [praise] 值得表扬的做法
     * [nit] 小问题（可以不改）

4. 修复与合并
   - Author 修复问题后 @ Reviewer 再次确认
   - 所有 [issue] 标记的问题必须解决
   - [suggestion] 和 [nit] 由 Author 决定是否采纳
   - 至少获得 1 个 Approve 后才能合并
```

## 五、技术债务管理

### 5.1 技术债务的分类

```text
技术债务四象限：

                    故意的
                 ┌─────────────────────────────────┐
                 │                                 │
    有意识的      │  战略性债务          鲁莽性债务   │
                 │  "我们知道这不理想，  "我们没有    │
                 │   但为了赶上线先这    时间做设计"  │
                 │   么做，下个迭代再                 │
                 │   重构"                          │
                 │                                 │
                 ├─────────────────────────────────┤
                 │                                 │
    无意识的      │  谨慎性债务          不可避免的   │
                 │  "我们现在知道这     无知性债务   │
                 │   样更好了，当时     "我们当时    │
                 │   的做法也不算错"    根本不知    │
                 │                    道这是个     │
                 │                    问题"        │
                 │                                 │
                 └─────────────────────────────────┘

处理优先级：
鲁莽性债务 > 战略性债务（已逾期的）> 谨慎性债务 > 无知性债务
```

### 5.2 技术债务登记簿

```php
<?php

namespace App\Services\TechDebt;

/**
 * 技术债务登记簿（Tech Debt Registry）
 * 将技术债务像 Bug 一样管理起来
 */
class TechDebtRegistry
{
    /**
     * 债务条目格式
     */
    public static function template(): array
    {
        return [
            'id' => 'TD-001',
            'title' => '订单查询接口缺少复合索引',
            'description' => 'orders 表在 user_id + status + created_at 条件查询时未使用索引，全表扫描',
            'category' => 'performance',  // performance | maintainability | security | reliability
            'severity' => 'high',          // low | medium | high | critical
            'impact' => [
                'technical' => '查询延迟 P95 达到 800ms',
                'business' => '影响订单列表页加载速度，可能影响转化率',
                'team' => '每次改动订单相关功能都需要额外注意性能',
            ],
            'effort_estimate' => '2h',     // 修复工作量估算
            'created_at' => '2026-01-15',
            'assigned_to' => null,
            'target_sprint' => null,
            'status' => 'open',            // open | in_progress | resolved | wontfix
            'related_pr' => null,
        ];
    }

    /**
     * 债务优先级评分公式
     * 优先级 = 严重程度 × 影响范围 × 利息速率
     */
    public static function calculatePriority(array $debt): int
    {
        $severityScore = match ($debt['severity']) {
            'critical' => 4,
            'high'     => 3,
            'medium'   => 2,
            'low'      => 1,
        };

        $impactScore = match (true) {
            str_contains($debt['impact']['technical'] ?? '', 'P0') => 4,
            str_contains($debt['impact']['technical'] ?? '', '数据丢失') => 4,
            str_contains($debt['impact']['business'] ?? '', '营收') => 3,
            str_contains($debt['impact']['business'] ?? '', '用户体验') => 2,
            default => 1,
        };

        // 利息速率：债务存在越久，修复成本越高
        $ageMonths = \Carbon\Carbon::parse($debt['created_at'])->diffInMonths(now());
        $interestRate = min(4, 1 + (int) ($ageMonths / 3));

        return $severityScore * $impactScore * $interestRate;
    }
}
```

### 5.3 Tech Debt Sprint 策略

```text
技术债务偿还策略（推荐：20% 法则）：

每个 Sprint（2 周）分配 20% 的容量给技术债务
- 10 个开发者 × 10 天 = 100 人天
- 20% = 20 人天用于技术债务

分配方式：
- 周三下午：团队"技术债务时间"（每人半天）
- 每个 Sprint 固定安排 1-2 个"技术债务 Ticket"
- 大型重构单独安排 Sprint（每季度 1 次，1-2 天）

选择哪些债务来偿还：
1. 影响当前开发效率的债务（每天都踩的坑）
2. 影响系统可靠性的债务（生产环境频繁报警的根因）
3. 影响新功能开发的债务（阻碍架构扩展的技术限制）
4. 安全相关的债务（不修复就是定时炸弹）
```

## 六、团队赋能

### 6.1 1-on-1 技术辅导

```text
1-on-1 会议模板（每两周 30-45 分钟）：

1. 个人状态（5 分钟）
   - 最近工作感觉如何？
   - 有什么困难或压力？

2. 技术成长（15 分钟）
   - 上次制定的学习计划进展如何？
   - 最近有什么技术发现或灵感？
   - 需要我提供什么技术指导？

3. 项目进展（10 分钟）
   - 当前任务的状态和阻塞点
   - 对项目方向的看法和建议

4. 反馈（5-10 分钟）
   - 我可以怎样更好地支持你？
   - 你对团队的流程有什么改进建议？

关键原则：
- 固定时间，不轻易取消
- 以倾听为主，少说多问
- 记录 Action Items，下次跟进
- 关注成长而非监控
```

### 6.2 Pair Programming 策略

```text
有效的 Pair Programming 模式：

1. Driver-Navigator 模式
   - Driver：写代码的人
   - Navigator：看代码、思考方向、发现错误
   - 每 25 分钟切换角色（Pomodoro 节奏）

2. 适用场景
   - 新成员 Onboarding（老成员 Driver → 新成员 Driver）
   - 复杂功能开发（TL Navigator + 开发者 Driver）
   - 知识传递（让团队成员了解不熟悉的模块）
   - 代码重构（一起讨论重构方案）

3. TL 参与 Pair Programming 的频率
   - 每周 2-3 次，每次 1-2 小时
   - 优先与初级开发者和新成员配对
   - 目标：3 个月内让新人能独立负责一个模块

4. 不适合 Pair Programming 的场景
   - 简单的 CRUD 功能
   - 探索性调研（各自看资料效率更高）
   - 写文档、配置环境等机械性工作
```

### 6.3 Tech Talk 机制

```text
团队 Tech Talk 制度：

频率：每周一次，周五下午 4:00-4:45
形式：团队成员轮流分享，每人每 6-8 周分享一次

分享主题参考：
- 最近解决的一个技术难题（深度 Case Study）
- 新技术/工具的调研报告
- 一个 Code Review 中发现的有趣模式
- 生产事故的 Postmortem 分享
- 开源项目源码解读
- 软技能分享（时间管理、沟通技巧）

TL 的角色：
- 第一个月由 TL 先做几次示范分享
- 之后引导团队成员主动认领 Topic
- 在分享后引导讨论，鼓励提问
- 将好的分享沉淀为团队 Wiki

效果衡量：
- 团队成员是否能独立解决更复杂的问题
- Code Review 中是否出现了新的代码模式
- 新成员的 Onboarding 速度是否加快
```

## 七、向上管理与跨团队协作

### 7.1 向非技术 Stakeholders 解释技术决策

```text
场景：需要向 Product Manager 解释为什么要花 2 周做数据库重构

❌ 技术语言：
"我们需要对 orders 表做分库分表，使用 ShardingSphere-Proxy
 按 user_id 哈希分 16 个片，每个片预估 300 万行，
 配合 Snowflake ID 生成器解决分布式 ID 问题。"

PM 的内心："我完全听不懂，但听起来很贵。"

✅ 业务语言：
"目前订单系统每月只能处理 50 万笔订单。按照业务增长速度，
 3 个月后系统会开始变慢，用户下单可能需要等待 10 秒以上。

 我们花 2 周升级后，系统能支持每月 500 万笔订单，
 足够支撑未来 2 年的业务增长。

 如果不升级，3 个月后我们可能面临：
 - 用户下单失败，直接影响营收
 - 需要紧急加班修复，影响团队士气
 - 临时修复可能引入新 Bug"

PM 的内心："我明白了，这是投资未来，值得做。"
```

### 7.2 与 Product Manager 的协作框架

```text
Tech Lead × Product Manager 协作模型：

Product Manager 负责：
- 定义做什么（What）和为什么做（Why）
- 确定需求优先级
- 代表用户和业务方的声音

Tech Lead 负责：
- 决定怎么做（How）和做多久（How long）
- 评估技术可行性和风险
- 代表技术质量和团队产能的声音

最佳实践：
1. 需求评审前 TL 先看技术可行性（30 分钟预研）
2. 需求拆分时 TL 参与，确保技术上可独立交付
3. 每周与 PM 对齐技术债务和架构改进的需求
4. TL 可以建议"先做 MVP 验证，再做完整方案"
```

### 7.3 跨团队技术对齐

```text
跨团队协作中的 Tech Lead 角色：

场景：你的团队负责订单服务，另一个团队负责支付服务，
     需要对接支付回调接口。

你的职责：
1. 定义接口契约（API Schema）
   - 使用 OpenAPI Spec 描述接口
   - 明确请求/响应格式、错误码、幂等性要求
   - 双方 Review 并确认

2. 约定 SLA
   - 响应时间承诺（P95 < 200ms）
   - 可用性承诺（99.9%）
   - 超时和重试策略

3. 建立沟通机制
   - 接口变更的通知流程
   - 联调测试的环境和时间安排
   - 线上问题的排查和升级路径

4. 事后维护
   - 定期 Review 接口使用情况
   - 收集对方团队的反馈
   - 计划接口演进（版本控制）
```

## 八、危机时刻的领导力

### 8.1 线上故障时 Tech Lead 的角色

```text
P0 故障发生时，Tech Lead 的职责：

不是：冲上去自己修代码
而是：指挥团队高效协作修复故障

具体行为：

T+0min（故障发生）
  ✅ 确认是否有人在处理（On-Call 是否已响应）
  ✅ 如果 On-Call 5 分钟内无响应，立即接手
  ✅ 创建 Incident Channel（Slack: #inc-20260602-001）

T+5min（初步评估）
  ✅ 确定故障严重级别（P0/P1/P2）
  ✅ 通知相关 stakeholders（PM、Manager）
  ✅ 指定 Incident Commander（通常是你自己）
  ✅ 指定 Technical Lead（负责排查问题的人）

T+15min（深入排查）
  ✅ 确保排查方向正确（问对问题比给答案更重要）
  ✅ 控制信息流：一个地方同步进展，避免多头沟通
  ✅ 评估是否需要回滚、扩容或降级

T+30min（修复执行）
  ✅ 快速决策：选择最优的恢复方案
  ✅ 监控修复效果
  ✅ 准备沟通材料（给 PM 和 Stakeholders）

T+60min+（恢复后）
  ✅ 确认系统恢复正常
  ✅ 组织 Postmortem（24-48 小时内）
  ✅ 创建 Action Items 并跟踪执行
```

### 8.2 决策框架：修复 vs 回滚 vs 降级

```text
故障处理的决策树：

系统异常
  │
  ├── 最近有代码变更？
  │     ├── 是 → 快速回滚（< 5 分钟可行）
  │     │        ├── 回滚后恢复正常 → 问题定位不紧急
  │     │        └── 回滚后仍异常 → 继续排查
  │     │
  │     └── 否 → 检查基础设施
  │              ├── 服务器/网络问题 → 联系云服务商
  │              ├── 数据库问题 → 检查慢查询/锁/连接池
  │              └── 流量异常 → 检查是否有爬虫/DDoS
  │
  ├── 能否在 30 分钟内修复？
  │     ├── 是 → 直接修复
  │     └── 否 → 先降级保核心功能，再慢慢修
  │
  └── 降级策略
        ├── 关闭非核心功能（推荐、评论、个性化）
        ├── 开启全站缓存（降级为缓存数据）
        ├── 限流（保护后端不被压垮）
        └── 静态页面兜底（返回预设的静态页面）
```

## 九、真实案例：一次架构重构的完整决策过程

### 9.1 背景

某 Laravel B2C 项目的优惠券系统需要重构。原有系统存在以下问题：

```text
问题清单：
1. 优惠券规则硬编码在 Controller 中，每次新增券类型都要改代码
2. 优惠券计算逻辑与订单耦合，无法独立测试
3. 没有幂等性保障，并发领取可能重复发放
4. 查询用户可用优惠券的 SQL 全表扫描，P95 延迟 2s
```

### 9.2 决策过程

```text
Step 1：问题定义（30 分钟团队讨论）
  - 明确约束：不能停服迁移、2 周内完成、后端独立重构不改前端
  - 确定目标：支持 10+ 券类型、查询延迟 < 200ms、100% 幂等

Step 2：方案设计（TL 主导 + 2 名开发者参与）

  方案 A：策略模式重构
  - 定义 CouponStrategy 接口
  - 每种券类型实现独立策略类
  - 使用 Laravel Service Container 注入策略

  方案 B：规则引擎（Drools/Rules Engine）
  - 使用第三方规则引擎配置券规则
  - 运营可自行配置新券类型

  决策：选择方案 A
  理由：
  - 方案 B 学习成本高，团队不熟悉规则引擎
  - 策略模式改动小，可渐进式重构
  - Laravel 原生的 Container 机制天然支持策略注入
  - 10 种券类型的复杂度可控，不需要通用规则引擎

Step 3：任务拆分（TL 拆分 + 团队认领）

  Sprint 1（第 1 周）：
  - 定义 CouponStrategy 接口 + 3 个核心策略实现（张三）
  - 幂等性保障：Redis 分布式锁 + 领取记录表（李四）
  - 数据库索引优化 + 查询重写（TL + DBA）

  Sprint 2（第 2 周）：
  - 剩余 7 个策略实现（张三 + 新成员）
  - 集成测试 + 压测（李四）
  - 灰度发布 + 监控告警配置（TL）

Step 4：执行与调整
  - 每天 Standup 同步进展
  - Day 3 发现 Redis 锁方案有极端并发问题，TL 决定改用 DB 唯一约束
  - Day 5 压测发现策略模式引入了额外的反射开销，TL 决定缓存策略实例
```

### 9.3 结果

```text
重构效果：
- 代码行数：减少 35%（去掉了重复的 if-else）
- 新增券类型：从需要 3 天开发 → 0.5 天（只需实现新策略类）
- 查询延迟：P95 从 2000ms → 150ms
- 幂等性：100% 保障，压测 10000 并发无重复发放
- 测试覆盖率：从 20% → 75%

TL 的关键贡献：
- 做出了"策略模式而非规则引擎"的正确决策
- 在 Redis 锁方案失败时快速调整为 DB 约束方案
- 拆分任务让新成员也能参与，促进了团队成长
- 确保了灰度发布的安全性
```

## 十、常见陷阱与成长建议

### 10.1 新晋 Tech Lead 的 10 个常见陷阱

```text
1. 🔴 继续做 IC 工作，忽视团队管理
   → 每周花 5 分钟审视时间分配

2. 🔴 所有技术决策都自己做
   → 学会委派，定义问题边界而非解决方案

3. 🔴 忽视 Code Review
   → 把 Review 当作最重要的工作之一

4. 🔴 不敢说"我不知道"
   → 诚实面对不懂的领域，带领团队一起研究

5. 🔴 忽视技术债务
   → 建立债务登记簿，每个 Sprint 还一点

6. 🔴 不做 1-on-1
   → 这是你了解团队状态的最重要渠道

7. 🔴 会议太多，没有深度工作时间
   → 设置 Focus Time，学会说"不"

8. 🔴 过度设计（Over-engineering）
   → YAGNI 原则，当前需要什么就做什么

9. 🔴 忽视向上管理
   → 定期与 Manager 同步技术方向和团队状态

10. 🔴 不做 Postmortem
    → 每次故障都是学习机会，无责复盘
```

### 10.2 成长路径

```text
Tech Lead 的成长阶梯：

Senior Engineer
  ↓ 角色转变：从个人贡献到团队赋能
Tech Lead（初级）
  ├── 能独立做架构决策
  ├── 能指导 1-2 个初级开发者
  └── 能管理单个系统的质量
  ↓
Tech Lead（高级）
  ├── 能定义团队的技术方向
  ├── 能管理跨系统的技术架构
  ├── 能影响多个团队的技术标准
  └── 能平衡技术理想与业务现实
  ↓
Staff Engineer / Engineering Manager
  ├── Staff：专注技术影响力，跨团队架构治理
  └── Manager：专注人员管理，团队建设与组织发展
```

## 总结

从 Senior Engineer 到 Tech Lead 的跃迁，本质上是从**"做正确的事"**到**"让团队做正确的事"**的转变。这需要你在以下方面持续精进：

1. **架构决策**：用 ADR 和决策矩阵记录和优化你的决策过程
2. **Code Review**：建立团队的 Review 文化，让知识在 Review 中流动
3. **技术债务**：像管理 Bug 一样管理技术债务，量化、排序、偿还
4. **团队赋能**：通过 1-on-1、Pair Programming、Tech Talk 提升团队能力
5. **沟通协调**：向上管理、跨团队协作，让技术决策被理解和接受
6. **危机领导**：线上故障时指挥团队高效协作，而非自己冲上去修代码

最重要的是：**保持学习，保持谦逊**。Tech Lead 不是终点，而是一段新的旅程的开始。
