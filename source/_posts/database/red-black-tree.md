MySQL 索引是数据库性能优化的核心，而理解索引底层的数据结构是做好索引优化的前提。本文将从最基础的二叉树出发，逐步讲解红黑树、B-Tree、B+Tree、Hash 等数据结构的特点与演进，深入分析 MySQL InnoDB 选择 B+Tree 的原因，并结合聚簇索引、联合索引、回表等实战概念，帮助你全面掌握 MySQL 索引的底层原理。

## 一、索引数据结构演进

### 1.1 二叉搜索树（BST）

二叉搜索树是最基础的索引结构，规则很简单：

- 比当前节点大的值放在右子树
- 比当前节点小的值放在左子树

```
        10
       /  \
      5    15
     / \     \
    3   7     20
```

**问题**：当数据按顺序插入时（如 1, 2, 3, 4, 5），二叉搜索树会退化为链表，查找复杂度从 O(log n) 变为 O(n)，相当于全表扫描。

```
1
 \
  2
   \
    3
     \
      4
       \
        5    ← 退化为链表，查找效率 O(n)
```

### 1.2 红黑树（Red-Black Tree）

红黑树是一种自平衡的二叉搜索树，通过旋转和变色操作保证树的平衡，查找时间复杂度稳定在 O(log n)。

**红黑树特性**：
- 每个节点非红即黑
- 根节点是黑色
- 红色节点的两个子节点必须是黑色（不能连续出现两个红色节点）
- 从任一节点到其所有叶子节点的路径上，黑色节点数量相同

```
         10(B)
        /    \
      5(R)   15(R)
     / \        \
   3(B) 7(B)   20(B)
```

**问题**：红黑树是**二叉**树，每个节点最多只有两个子节点。当数据量很大时（如千万级），树的高度会很高（h ≈ log₂N，N=1000万时 h≈24），每查找一个节点就可能需要一次**磁盘 I/O**，磁盘 I/O 次数 = 树的高度。数据量大时，树的高度不可控，I/O 次数过多，性能急剧下降。

### 1.3 B-Tree（多路平衡搜索树）

B-Tree 是一种**多路平衡搜索树**，每个节点可以拥有多个子节点（即多叉），解决了红黑树高度过高的问题。

**B-Tree 特性**：
- 每个节点最多有 M 个子节点（M 为树的阶）
- **非叶子节点也会存储数据**（键值 + 对应的数据记录）
- 所有叶子节点位于同一层
- 节点内的键值有序排列

```
           [10 | 20]
          /    |    \
    [3|5|7] [15|18] [25|30|35]
```

**树的高度计算**：假设每个非叶子节点可以存 N 个键值（由非叶子节点的页叉数决定），树高度为 h，则可存储的数据量为 N^h。例如 N=16, h=3，则可存储 16³ = 4096 条记录。

**问题**：B-Tree 非叶子节点也存储数据，导致每个节点能容纳的键值数量减少，树的高度相对较高。同时 B-Tree 没有维护叶子节点之间的双向链表，范围查询效率较低。

### 1.4 B+Tree（B-Tree 的变种）

B+Tree 是 MySQL InnoDB 引擎实际使用的索引结构，是 B-Tree 的优化变种。

**B+Tree 与 B-Tree 的关键区别**：

| 特性 | B-Tree | B+Tree |
|------|--------|--------|
| 非叶子节点存储数据 | ✅ 存储数据 | ❌ 只存储键值（冗余） |
| 叶子节点链表 | 无 | ✅ 双向链表连接 |
| 范围查询 | 需要中序遍历 | 叶子节点链表顺序扫描 |
| 数据存储位置 | 所有节点 | 仅叶子节点 |

**B+Tree 结构示意**：

![B+Tree索引结构](/images/content/databases-01-content-1.jpg)

```
非叶子节点（只存键值，不存数据）
         [10 | 20]
        /    |    \
    [3|5|7] [10|15|18] [20|25|30|35]
     ←→       ←→          ←→          ← 叶子节点双向链表
```

**核心优势**：
- **非叶子节点不存储数据**，每个节点可以容纳更多的键值，树的高度更低
- **叶子节点双向链表**，天然支持范围查询，范围查询只需遍历链表
- **数据全部在叶子节点**，查找路径上每经过一个节点就加载一个页（16KB），I/O 次数 = 树的高度

**MySQL InnoDB 的页大小为 16KB（16384 字节）**。假设主键为 BIGINT（8 字节），指针大小为 6 字节，则每个非叶子节点可存储：

```
16384 / (8 + 6) ≈ 1170 个键值
```

三层 B+Tree 可存储的记录数：

```
1170 × 1170 × 16 ≈ 2190 万条记录
```

其中叶子节点假设每条记录占 1KB，则每个叶子节点可存 16 条记录。也就是说，**约 2000 万条数据只需要 3 次磁盘 I/O 就能定位到目标记录**，这就是 B+Tree 的威力。

### 1.5 B+Tree 的插入与删除过程

#### 插入过程

B+Tree 插入时，新数据总是插入到叶子节点中：

1. **找到目标叶子节点**：从根节点开始，根据键值比较找到应该插入的叶子节点
2. **叶子节点未满**：直接在叶子节点中按序插入，如果该键值也存在于非叶子节点中则无需更新非叶子节点
3. **叶子节点已满，发生分裂**：
   - 将该叶子节点从中间分裂为两个节点
   - 将右半部分移到新节点中
   - 将新节点中的最小键值**上提到父节点**作为索引
   - 如果父节点也满了，则递归向上分裂
4. **极端情况下**，分裂可能一直传播到根节点，此时根节点分裂，树的高度加 1

```
插入前（假设每个节点最多3个键）：
        [10 | 20]
       /    |    \
  [3|5|7] [15|18] [25|30]

插入键值 8 → 第一个叶子节点满，分裂：
        [7 | 10 | 20]
       /    |    |    \
  [3|5] [7|8] [15|18] [25|30]

如果根节点也满了，继续向上分裂，树高+1
```

#### 删除过程

B+Tree 删除时需要保证非叶子节点中的键值是叶子节点中某条记录的"副本"：

1. **找到目标叶子节点**并删除对应记录
2. **叶子节点记录数低于下限**：
   - 尝试从**相邻兄弟节点借一条记录**（旋转）
   - 如果兄弟节点也无多余记录，则与兄弟节点**合并**
3. **合并后**，父节点中对应的索引键值需要删除
4. 如果父节点的键值数也低于下限，递归向上调整
5. 删除过程中，非叶子节点中的键值可能需要用叶子节点中的最大/最小值来替换

### 1.6 Hash 索引

Hash 索引基于哈希表实现，通过对键值进行 Hash 运算直接定位存储位置。

**特点**：
- **等值查询 O(1)**：只需一次 Hash 运算 + 一次 I/O
- **不支持范围查询**：Hash 运算后数据无序，无法使用 `BETWEEN`、`>`、`<` 等范围条件
- **不支持排序**：无法利用索引做 `ORDER BY`
- **Hash 冲突**：不同键值可能映射到同一位置，需要链表处理冲突
- **不支持联合索引的最左前缀匹配**

**适用场景**：Memory 存储引擎支持 Hash 索引，适合纯等值查询的场景。InnoDB 引擎的自适应哈希索引（AHI）会自动为热点数据构建 Hash 索引，但用户无法手动控制。

## 二、为什么 InnoDB 选择 B+Tree？

MySQL InnoDB 选择 B+Tree 而非其他数据结构，是基于**磁盘 I/O 特性**和**查询模式**的综合考量：

### 对比分析

| 数据结构 | 树高（2000万数据） | 磁盘 I/O 次数 | 范围查询 | 排序 | 适用场景 |
|---------|-------------------|--------------|---------|------|---------|
| 二叉搜索树 | ~24 | ~24 | ❌ | ❌ | 仅教学 |
| 红黑树 | ~24 | ~24 | ❌ | ❌ | 内存数据结构 |
| B-Tree | ~3 | ~3 | ⚠️ 较慢 | ⚠️ | 部分数据库 |
| **B+Tree** | **~3** | **~3** | **✅ 高效** | **✅** | **MySQL InnoDB** |
| Hash | 1 | 1 | ❌ | ❌ | 纯等值查询 |

### 核心原因

1. **磁盘 I/O 是瓶颈**：数据库数据存储在磁盘上，每次读取一个节点就是一次磁盘 I/O（读取一个 16KB 的页）。B+Tree 非叶子节点不存数据，单个节点能容纳更多键值，树的高度更低，I/O 次数更少。

2. **范围查询高效**：业务中大量查询是范围查询（如 `WHERE age BETWEEN 20 AND 30`）。B+Tree 叶子节点通过双向链表连接，范围查询只需定位到起始叶子节点，然后沿链表顺序扫描即可。B-Tree 需要中序遍历整棵树，效率远低于 B+Tree。

3. **排序天然支持**：叶子节点链表本身是有序的，`ORDER BY` 操作可以利用索引的有序性避免额外排序。

4. **查询性能稳定**：B+Tree 所有数据都在叶子节点，任何查询的路径长度都相同（根→叶子），性能稳定可预测。

5. **磁盘预读友好**：操作系统和数据库的预读机制会提前加载相邻的页到内存，B+Tree 的叶子节点链表排列方式天然适合顺序预读。

## 三、聚簇索引与非聚簇索引

### 3.1 聚簇索引（Clustered Index）

聚簇索引是 InnoDB 特有的概念。**聚簇索引的叶子节点存储的是完整的数据行**，数据本身就是按主键顺序存储的。

> **注意**：InnoDB 表必须有聚簇索引。如果定义了主键，主键即为聚簇索引；如果没有主键，MySQL 会选择第一个非空唯一索引；如果都没有，InnoDB 会创建一个隐藏的 6 字节 ROW_ID 作为聚簇索引。

**聚簇索引的特点**：
- 一张表只能有一个聚簇索引
- 叶子节点存储完整行数据
- 按主键顺序物理存储

```
聚簇索引（主键索引）结构：

非叶子节点：
         [id:10 | id:20]
        /       |       \
叶子节点（存完整行数据）：
  [id:1, name:'Alice', age:20]  ←→  [id:5, name:'Bob', age:25]  ←→  [id:10, ...]  ←→  [id:15, ...]  ←→  [id:20, ...]
```

**主键选择建议**：
- 使用**整型**：数值比较效率高于字符串比较
- 使用**自增主键**：自增插入总是追加到叶子节点末尾，避免页分裂
- **存储空间要小**：主键越小，非叶子节点能容纳的键值越多，树高越低

### 3.2 非聚簇索引（Secondary Index）

非聚簇索引（二级索引）的叶子节点存储的**不是完整数据行，而是主键值**。

```
二级索引（如以 name 为索引）结构：

非叶子节点：
         [name:'Mike' | name:'Tom']
        /         |           \
叶子节点（存主键值）：
  [name:'Alice' → id:1]  ←→  [name:'Bob' → id:5]  ←→  [name:'Mike' → id:10]  ←→  [name:'Tom' → id:20]
```

### 3.3 回表查询

当通过二级索引查询非索引字段时，需要**回表**：

```sql
-- 假设 name 字段有索引 idx_name
SELECT * FROM user WHERE name = 'Alice';
```

**查询过程**：
1. 在 `idx_name` 二级索引中找到 `name = 'Alice'`，获取主键 `id = 1`
2. **回表**：用主键 `id = 1` 到聚簇索引中查找完整行数据

> 这里经历了**两次 B+Tree 查找**。如果只需要查主键字段，可以避免回表（覆盖索引）。

### 3.4 ASCII 比较说明

字符串索引比较是基于 ASCII 码逐字符比较的，效率低于整型数值比较。例如 `'abc' < 'abd'` 需要逐字符比较，而 `100 < 200` 只需一次整数比较。这也是为什么推荐使用整型自增主键的原因之一。

## 四、联合索引与最左前缀原则

### 4.1 联合索引结构

联合索引（复合索引）是对多个字段建立的索引，索引内部按键值顺序**依次排序**。

例如，对 `(age, name, city)` 建立联合索引，叶子节点的排列方式为：

```
先按 age 排序，age 相同按 name 排序，name 再相同按 city 排序：

(age=18, name='Alice', city='Beijing')
(age=18, name='Bob',   city='Shanghai')
(age=20, name='Alice', city='Beijing')
(age=20, name='Bob',   city='Guangzhou')
(age=22, name='Charlie', city='Shenzhen')
```

### 4.2 最左前缀原则

联合索引遵循**最左前缀匹配原则**：查询条件必须从索引的最左列开始，且不能跳过中间的列，否则索引会失效。

**创建示例表**：

```sql
CREATE TABLE `user` (
  `id` bigint NOT NULL AUTO_INCREMENT,
  `age` int NOT NULL,
  `name` varchar(50) NOT NULL,
  `city` varchar(50) NOT NULL,
  `email` varchar(100) DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `idx_age_name_city` (`age`, `name`, `city`)
) ENGINE=InnoDB;
```

**EXPLAIN 验证最左前缀原则**：

```sql
-- ✅ 使用了索引（匹配最左列 age）
EXPLAIN SELECT * FROM user WHERE age = 20;
/*
+----+------+------------------+---------+------+-------+
| id | type | key              | key_len | rows | Extra |
+----+------+------------------+---------+------+-------+
|  1 | ref  | idx_age_name_city| 4       |   10 | NULL  |
+----+------+------------------+---------+------+-------+
*/

-- ✅ 使用了索引（匹配 age + name）
EXPLAIN SELECT * FROM user WHERE age = 20 AND name = 'Alice';

-- ✅ 使用了索引（匹配 age + name + city，完整匹配）
EXPLAIN SELECT * FROM user WHERE age = 20 AND name = 'Alice' AND city = 'Beijing';

-- ❌ 未使用索引（跳过了最左列 age）
EXPLAIN SELECT * FROM user WHERE name = 'Alice' AND city = 'Beijing';
/*
+----+------+---------+------+-------+
| id | type | key     | rows | Extra |
+----+------+---------+------+-------+
|  1 | ALL  | NULL    | 5000 | Using where |
+----+------+---------+------+-------+
type=ALL 表示全表扫描，索引失效！
*/

-- ⚠️ 部分使用索引（匹配 age，name 使用了索引的排序，但 city 不能用索引定位）
EXPLAIN SELECT * FROM user WHERE age = 20 AND city = 'Beijing';
-- 只用到了 age 列的索引，city 条件需要在回表后过滤
```

> 联合索引跳过最左字段，后面的数据实际上是**无序**的，无法利用 B+Tree 的有序性进行查找。

## 五、建索引踩坑案例

### 案例 1：对低选择性字段建索引

```sql
-- 性别字段只有 'M' 和 'F' 两个值，选择性很低
CREATE INDEX idx_gender ON user(gender);
```

**问题**：索引的选择性（区分度）太低，MySQL 优化器可能判断全表扫描比走索引更快（走索引还需要回表），导致索引形同虚设。

**建议**：对低选择性字段，考虑与其他字段建立联合索引，或使用复合条件。

### 案例 2：在函数/表达式上使用字段导致索引失效

```sql
-- ❌ 索引失效：对索引字段使用了函数
EXPLAIN SELECT * FROM user WHERE YEAR(create_time) = 2024;

-- ✅ 改写为范围查询，索引生效
EXPLAIN SELECT * FROM user WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';

-- ❌ 索引失效：隐式类型转换
-- phone 是 varchar 类型，传入了 int
EXPLAIN SELECT * FROM user WHERE phone = 13800138000;

-- ✅ 使用正确的类型
EXPLAIN SELECT * FROM user WHERE phone = '13800138000';
```

### 案例 3：联合索引列顺序不当

```sql
-- 业务查询 90% 是 WHERE status = 1 AND user_id = ?
-- 错误：把低选择性的 status 放在最左
CREATE INDEX idx_status_user ON orders(status, user_id);

-- 正确：把高选择性的 user_id 放在最左
CREATE INDEX idx_user_status ON orders(user_id, status);
```

**原则**：联合索引中，把**选择性高**（区分度大）的列放在前面，可以更快缩小扫描范围。

### 案例 4：过多索引拖慢写入

```sql
-- 对一张频繁写入的表建了 10 个索引
-- 每次 INSERT/UPDATE/DELETE 都需要维护所有索引的 B+Tree
```

**问题**：索引越多，写入时维护成本越高。建议：
- 单表索引数量控制在 5 个以内
- 定期用 `pt-duplicate-key-checker` 检查冗余索引
- 使用 `EXPLAIN` 验证每个索引确实被使用

### 案例 5：LIKE 以通配符开头

```sql
-- ❌ 索引失效
EXPLAIN SELECT * FROM user WHERE name LIKE '%Alice';

-- ✅ 索引生效
EXPLAIN SELECT * FROM user WHERE name LIKE 'Alice%';
```

> 以 `%` 开头的 LIKE 查询无法利用 B+Tree 的有序性，会导致全表扫描。

## 六、MVCC 与 Buffer Pool 简述

### MVCC（多版本并发控制）

MVCC 是 InnoDB 实现高并发的核心机制，通过维护数据的多个版本，在不加锁的情况下实现读写并发。MVCC 与索引配合，使得一致性非锁定读成为可能。

### Buffer Pool 缓存机制

InnoDB 使用 Buffer Pool 缓存磁盘上的数据页和索引页，减少磁盘 I/O。Buffer Pool 使用 **LRU（最近最少使用）算法**管理缓存页的淘汰。

- **脏页（Dirty Page）**：Buffer Pool 中被修改但尚未刷到磁盘的数据页
- **改进的 LRU 算法**：将 LRU 列表分为 young 区和 old 区，新读入的页先放入 old 区，避免全表扫描等操作污染热数据缓存
